- [Splash screen](features/splash-screen.md) — system-managed cold-launch splash via `androidx.core:core-splashscreen:1.0.1` (#80); launcher `<activity android:theme="@style/Theme.PyrycodeMobile.SplashScreen">` resolves a `Theme.SplashScreen`-parented style (compat library, no `android:` namespace) with `windowSplashScreenBackground = @color/ic_launcher_background` (#FF32628D, reused from the launcher icon — no parallel color name) + `windowSplashScreenAnimatedIcon = @drawable/ic_splash_logo` (new 288×288dp viewport with all paths inside the inner 192dp safe area; single white-filled r=64 circle centred at 144,144 — placeholder, not brand identity) + `postSplashScreenTheme = @style/Theme.PyrycodeMobile`; `MainActivity.onCreate` calls `installSplashScreen()` as its **first** statement before `super.onCreate(...)` (AndroidX swaps the theme to `postSplashScreenTheme` before the window is realised — calling after `super` leaks the splash theme into the first app frame on some OEMs) with the returned `SplashScreen` handle intentionally discarded — no `setKeepOnScreenCondition { ... }`, no exit-animation listener (the [Navigation](features/navigation.md) #13 conditional start destination + `produceState`+empty-`Surface` placeholder already covers the only flash window). Splash drawables resolve **outside** the app theme so fills must be literal hex (`#FFFFFFFF`), not `?attr/...` references. No automated tests — splash is a system-window concern that `androidx.compose.ui.test` attaches after and `ActivityScenario` bypasses; verification surface is the build + one manual Android 13+ cold launch (Android 11/12 compat fallback is solid-background-no-animation-envelope, expected not a regression). ## Decisions

- [ADR 0001 — kotlinx-datetime for data layer](decisions/0001-kotlinx-datetime-for-data-layer.md) — every `data/` timestamp uses `kotlinx.datetime.Instant`, never `java.time.Instant`, to keep a Compose Multiplatform walk-back cheap.
- [ADR 0002 — markdown renderer library](decisions/0002-markdown-renderer-library.md) — use `org.jetbrains:markdown` (pure-Kotlin CommonMark parser) plus a hand-rolled Compose renderer (`MarkdownText`); reject `dev.jeziellago:compose-markdown` because it's a Markwon-on-`AndroidView` wrapper (would introduce an Android-View interop seam the codebase doesn't have today) and its heading sizes are relative multipliers on a single base style (couldn't satisfy AC5's "each heading maps to a distinct M3 typography slot — `headlineSmall` / `titleLarge` / `titleMedium`" without writing a custom Markwon plugin roughly the same size as a hand-rolled visitor walk). Owning the renderer means "no `.sp` literals; every text style resolves from `MaterialTheme.typography`" holds by construction, and CMP walk-back stays cheap (parser is KMP-native). CommonMark flavour by default; swap to `GFMFlavourDescriptor()` if Claude API responses lean on tables / strikethrough / task lists.
- [ADR 0003 — syntax highlighter library](decisions/0003-syntax-highlighter-library.md) — use `dev.snipme:highlights` 1.1.0 (pure-Kotlin KMP tokeniser) for fenced code blocks inside [`MarkdownText`](features/markdown-text.md); reject `io.github.kbiakov:codeview-android` and Markwon's syntax-highlight plugin (both reintroduce the `AndroidView` seam [ADR 0002](decisions/0002-markdown-renderer-library.md) avoided). Library returns `CodeStructure` with `PhraseLocation(start, end)` lists per category and the renderer owns the colour mapping — bound to `MaterialTheme.colorScheme.tertiary` (keywords/annotations + Medium weight), `.secondary` (strings), `.primary` (literals), `.onSurfaceVariant` italic (comments), inherited `LocalContentColor.current` (punctuation/operators) — so AC6 "no hardcoded hex" holds by construction. Same three-line wire-up shape as [ADR 0002](decisions/0002-markdown-renderer-library.md) (one `[versions]` line, one `[libraries]` line, one `implementation(...)`). Language coverage bounded by the library's `SyntaxLanguage` enum: Kotlin / Shell direct, JSON via `JAVASCRIPT` (subset substitute — no dedicated JSON lexer), Markdown + anything else fall through to plain monospace; `tokeniseCode` wraps the library call in `runCatching { … }.getOrNull()` so library failures degrade to monospace, not a crash.
