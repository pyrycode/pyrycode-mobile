# #22 — Channel list FAB: new discussion

Adds a Material 3 `FloatingActionButton` to `ChannelListScreen`. Tap calls `ConversationRepository.createDiscussion()` (workspace = `null`) and navigates to `conversation_thread/{newId}`. Introduces the first VM-side reducer on `ChannelListViewModel` and the first one-shot navigation channel in the project — establishes the `Channel<…Navigation>` + `receiveAsFlow()` pattern for suspend-shaped side effects whose nav target depends on the suspend's return value.

## Summary

- New sealed-interface variant `ChannelListEvent.CreateDiscussionTapped` (data object) joins `RowTapped(conversationId)` and `SettingsTapped` (#21). Producer is the FAB's `onClick`; consumer is `MainActivity`'s destination-level `when (event)`, which forwards into `vm.onEvent(event)` rather than routing to `navController` directly — because the target route depends on the id returned by the suspend.
- New sealed interface `ChannelListNavigation` (top-level, same file as `ChannelListViewModel`) with one variant: `data class ToThread(val conversationId: String)`. Co-located with its emitter; promotes to its own file when a second screen needs the same one-shot-nav shape.
- `ChannelListViewModel`'s constructor parameter `repository` is now `private val` (promoted from a non-property parameter). Existing tests don't pattern-match on the constructor shape, so this is non-breaking.
- New on the VM: `private val navigationChannel = Channel<ChannelListNavigation>(capacity = Channel.BUFFERED)` exposed as `val navigationEvents: Flow<ChannelListNavigation> = navigationChannel.receiveAsFlow()`. `fun onEvent(event: ChannelListEvent)` launches the create in `viewModelScope` and sends `ToThread(newConversation.id)`. `RowTapped` / `SettingsTapped` arms collapse to `Unit` for compiler exhaustiveness — `MainActivity` does not dispatch those into `onEvent`.
- `ChannelListScreen`'s `Scaffold` grew a `floatingActionButton` slot. Conditional inside the slot's composable lambda (not on the slot assignment) gates rendering to `state is Loaded || state is Empty` — Loading and Error states have no FAB. Default `FabPosition.End`. Icon `Icons.Default.Add` with `contentDescription = stringResource(R.string.cd_new_discussion)`.
- `MainActivity`'s `composable(Routes.ChannelList)` block grew a `LaunchedEffect(vm) { vm.navigationEvents.collect { … } }` that translates `ChannelListNavigation.ToThread` into `navController.navigate("conversation_thread/${event.conversationId}")`. The destination's existing `when (event)` got a third branch: `ChannelListEvent.CreateDiscussionTapped -> vm.onEvent(event)`.
- New string `R.string.cd_new_discussion = "New discussion"` (a-11y content description; `cd_` prefix from #21).
- Two new unit tests on `ChannelListViewModelTest`. Both use `FakeConversationRepository()` (the production fake — reachable from `main`) rather than extending the file's existing `stubRepo` helper, which still returns `TODO("not used")` for `createDiscussion`.
- `./gradlew assembleDebug` and `./gradlew test` both pass.

## Patterns established

- **`Channel<ScreenNavigation>(BUFFERED)` + `receiveAsFlow()` for one-shot nav events whose target depends on a suspend's return value.** Not `MutableSharedFlow` — replay would re-fire on rotation, no-replay would drop in-flight taps. Not a `StateFlow<Navigation?>` with manual reset — racy against the consumer. The `Channel.BUFFERED` shape gives at-most-once delivery, survives the recomposition window between tap and consume, and cancels atomically with the VM scope. Mirror this shape on the next screen that needs a suspend-then-navigate flow (promotion dialog → thread, archive → list, workspace change → re-mount).
- **`fun onEvent(event: ChannelListEvent)` on the VM is opt-in per variant; `MainActivity` decides what to forward.** The VM's `when` has a `RowTapped, SettingsTapped -> Unit` arm purely for compiler exhaustiveness — those events never enter `onEvent` today because `MainActivity` resolves them directly to `navController.navigate(...)`. The decision rule: events with no VM-side side effect stay routed at the destination; events that need a suspend or VM state mutation get forwarded into `onEvent`. Don't preemptively funnel every event through the VM "for consistency" — keeps the dispatch surface narrow and the `Unit` arm legible.
- **Sealed `<Screen>Navigation` co-located with its `<Screen>ViewModel`, not in a `navigation/` subpackage.** Same call as `<Screen>UiState` and `<Screen>Event` — one-event-source-per-screen makes co-location strictly better than a shared hierarchy until a second screen needs the same pattern. The next screen that needs one-shot nav extracts a `ScreenNavigation` interface only when there's a *cross-screen* consumer (there isn't one in this codebase's foreseeable future).
- **Conditional FAB rendering goes inside the `floatingActionButton` slot's composable lambda, not on the slot's assignment.** `Scaffold`'s `floatingActionButton` parameter is a non-nullable `@Composable () -> Unit`; an `if … else` at the assignment site does not type-check. Pattern: `floatingActionButton = { if (state is Loaded || state is Empty) { FloatingActionButton(…) { … } } }`. The slot's recomposition is driven by `state` reads inside the lambda — FAB appears/disappears on state transitions automatically.
- **Constructor-parameter → `private val` promotion is non-breaking when no test pattern-matches the constructor.** Was a bare `repository: ConversationRepository` parameter used only in the `state` initializer; now `private val repository: ConversationRepository` because `onEvent` needs to reach it. Existing tests' `ChannelListViewModel(stubRepo(source))` call sites compile unchanged. Apply the same promotion freely on later tickets that need to capture a constructor dep into a method body.
- **Unit tests for suspend-shaped event handlers use `FakeConversationRepository()` directly, not the file's `stubRepo` helper.** The `stubRepo` helper exists for *state-derivation* tests (it lets `MutableSharedFlow` drive emission timing). The fake is the production implementation and gives a faithful integration of `observeConversations` + `createDiscussion` — exactly what an `onEvent` test needs. The two existed-as-one-file pattern: state tests use `stubRepo`, side-effect tests use the fake.
- **`async { vm.navigationEvents.first() }` is the test-side shape for capturing a one-shot nav emission.** Launched *before* the triggering `vm.onEvent(...)` so the collector is attached when the channel sends; `advanceUntilIdle()` between trigger and `.await()` to drain the in-flight launch. `runTest`-friendly. Mirrors `kotlinx-coroutines-test`'s recommended capture pattern for one-shot channels.

## Lessons learned

- **`LaunchedEffect(vm)` is the correct key for a one-shot-nav collector at a destination.** Not `LaunchedEffect(Unit)` (would survive VM identity changes across configuration in unusual NavBackStackEntry handoffs); not `LaunchedEffect(navController)` (stable across the whole NavHost — would not restart per destination scope). `vm` is stable per `NavBackStackEntry` scope (Compose Navigation 2.9 auto-wires `LocalViewModelStoreOwner` to the back-stack entry), so the collector restarts exactly when the VM identity changes — which is when a fresh subscription is wanted.
- **The composable function is destroyed/recreated on navigation away/back; `LaunchedEffect` cancels on disposal — events emitted while away would be lost, but the only emitter is `onEvent(CreateDiscussionTapped)` which can only fire when the screen is composed.** No race window in practice. If a future flow can emit while the screen is absent (e.g. a long-running background create from a worker), the channel-send-then-leave window matters and the consumer needs lifecycle-aware retention — `Channel.BUFFERED` then becomes load-bearing rather than precautionary.
- **`Icons.Default.Add` is in `material-icons-core` — no `-extended` dependency churn.** Continues the #21 lesson; the "obvious UI verbs" list (Settings, ArrowBack, Menu, Search, Close, Add, Check, Clear, MoreVert, Done, Edit, Refresh, Share, Info, Warning, Email, Star, Favorite, Home, Person, Lock) is reliable. This ticket needed only `Icons.Default.Add`, no catalog edit, no `build.gradle.kts` edit. Future single-glyph needs from that list: same shape — additive import only.
- **Do not wrap `repository.createDiscussion()` in `try/catch`.** The fake's implementation can't throw on this path; speculative catches were called out as forbidden by the spec and CLAUDE.md's "evidence-based fix selection" principle. Phase 4's `RemoteConversationRepository` adds the catch *and* the error UI together; this ticket adds neither.
- **No debounce / single-flight on `CreateDiscussionTapped` either.** Two rapid taps create two discussions. AC reads "single tap creates exactly one new discussion" — that's per-tap, not "duplicate-prevent". No observed failure mode justifies a guard today; defer to a real bug.
- **Fake repo's `createDiscussion` is fast enough that `advanceUntilIdle()` is sufficient — no `runCurrent()` finesse needed.** The whole create + send happens within one `viewModelScope.launch`; `advanceUntilIdle()` drains it. Phase 4's network-backed create will likely need `runCurrent()` between dispatcher hops to assert intermediate state — flag for the ticket that adds it.
- **`Channel.send` is suspending; the whole create-then-send body runs inside `viewModelScope.launch { … }`.** No need to wrap the `send` separately. `trySend` is the non-suspending alternative for non-coroutine callers (event-bus dispatch from a UI callback), but inside a launched coroutine `send` is correct and gives the right back-pressure semantics — irrelevant here because `BUFFERED` doesn't apply pressure, but the shape transfers if a future channel switches to `Channel.RENDEZVOUS`.
- **`floatingActionButton` slot gates on the same `state` reads as the body — no extra `derivedStateOf` needed.** `state` is a snapshot read; Compose tracks reads inside the slot's lambda and recomposes the slot in isolation when `state` transitions. The "FAB appears on Loading→Loaded" transition is observably correct in the preview.
