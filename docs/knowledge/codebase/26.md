# #26 — 'Recent discussions (N) →' pill on Channel list

Surfaces the unpromoted-tier count on the main Channel list via a stateless `RecentDiscussionsPill` composable that sits below the `LazyColumn` (Loaded) or above the centred empty copy (Empty), navigating to `Routes.DiscussionList` on tap. Hides itself when `count <= 0`. `ChannelListViewModel.state` now `combine`s two repository flows (Channels + Discussions); `ChannelListUiState.Loaded` and `.Empty` widened to carry `recentDiscussionsCount: Int` (required, no defaults). Wires the discussions route into the live graph — until this ticket the `discussions` destination from #24 was unreachable.

## Patterns established

- **`combine` two repository flows in a single `StateFlow<UiState>` rather than holding two parallel `StateFlow`s.** When a screen needs two pieces of data from the data layer and the UI contract is "show them together", fold both into one `UiState` via `combine` upstream of `.catch` / `.stateIn`. This keeps error semantics atomic (a throw on either source collapses to `Error`) and lets the screen take a single `state by vm.state.collectAsStateWithLifecycle()` read. The cost is the `combine`-pre-emit Loading window — neither flow has emitted ⇒ `state.value == Loading` — which is exactly what we want for "initial repository round-trip not yet complete". Don't introduce a sibling `StateFlow<DiscussionsCount>` even if it feels more decomposed; future readers have to remember to subscribe to both, and the `is Loaded` / `is Empty` smart casts no longer carry the count.
- **Widen `UiState` variants with required positional params instead of nullable / defaulted fields.** When new state lands on existing sealed variants, prefer `data class Loaded(channels, recentDiscussionsCount: Int)` over `Loaded(channels, recentDiscussionsCount: Int = 0)`. The compiler then surfaces every existing call site (VM constructors, previews, tests) as a non-compiling call — a five-minute cascade fix beats a silently-zero default that masks a missed wire-up. Five call sites needed touching for this ticket; all five surfaced at the first compile.
- **Stateless composable owns its own zero-state guard (`if (count <= 0) return`).** When a composable has a "render nothing in state X" rule, put the early return inside the composable, not at the call site. The screen's `when (state)` branch always calls `RecentDiscussionsPill(count = state.recentDiscussionsCount, onClick = …)` — it doesn't have to wrap the call in an `if`. Two screens, two branches, one rule — one place to maintain.
- **Trailing affordance glyphs as plain text in the label, not as a separate `Icon`.** The pill's `→` is part of the `strings.xml` string, not `Icons.AutoMirrored.Filled.ArrowForward`. AC explicitly specified the visible glyph as part of the label. Plain text auto-mirrors with locale (RTL gets `←` in the translated string), reads as one atom to TalkBack with the merged-descendants semantics, and avoids the cross-axis alignment dance that an icon + text `Row` introduces. Reach for an `Icon` only when the affordance is icon-only (e.g. trailing chevron with no label) or when the icon itself is the action (e.g. close button on a chip).

## Lessons learned

- **`AssistChip` chrome competes with `ConversationRow` on a `LazyColumn`.** The spec's open question deferred the chrome choice to the implementer. `AssistChip(elevatedAssistChipElevation())` reads as "another tappable list item" because it carries the same kind of outlined / elevated visual weight as a row — the eye doesn't distinguish "pill below the list" from "row in the list". `Surface(tonalElevation = 2.dp, shape = MaterialTheme.shapes.medium)` reads as a small below-the-fold affordance because its rounded shape and tonal fill differ from the row's flat `ListItem`. If a future ticket wants the pill to look more chip-like (e.g. on a screen with no list above it), revisit; on the channel list, `Surface` is right.
- **`minimumInteractiveComponentSize()` is required even on a `Surface`-with-`clickable`.** The `Surface` itself does **not** enforce the 48×48dp minimum touch target — that's a `Button` / `Chip` / `IconButton` thing. With the pill's inner padding the surface measured ~36dp tall in the layout inspector before the fix; below the 48dp Material a-11y minimum. Adding `Modifier.minimumInteractiveComponentSize()` between `padding(...)` and `clickable(...)` reserves the layout slot to at least 48dp without changing the surface's drawn size (the surface still draws at ~36dp; the click region extends out symmetrically). The accessibility review flagged this in code-review and the follow-up commit fixed it — pattern for future `Surface`-as-button affordances is the same modifier order.
- **`Modifier.semantics(mergeDescendants = true) { contentDescription = … }` is necessary for `Surface` containing `Text`.** Without it, TalkBack reads the inner `Text`'s label and a separate "double-tap to activate" cue from the `clickable` — two announcements, awkward. With `mergeDescendants = true`, the semantics tree collapses into a single node that announces the content description (the more useful "Open recent discussions, N available" string) and the `Role.Button` cue. Apply the same shape to any future composite affordance built from a `Surface` / `Box` plus a child `Text`. `Modifier.clearAndSetSemantics { … }` is an even stronger reset that drops the children's semantics entirely; prefer `mergeDescendants` because it keeps the inner text's label as a fallback for inspector/testing tools.
- **`Role.Button` declared at the `clickable` call site is what TalkBack reads, not the underlying composable type.** `Surface` has no semantic role; `clickable` defaults to no role unless told. Without `role = Role.Button`, TalkBack reads "Open recent discussions, 5 available, double-tap to activate" — functional but the affordance type is ambiguous. With `role = Role.Button`, it reads "Open recent discussions, 5 available, button, double-tap to activate" — affordance is announced. Apply to any non-`Button` composable that's been made clickable.
- **`combine` will not emit until *both* upstream flows have emitted at least once.** This is what makes the `Loading` initial state observable in tests but also what gates the pill — it cannot appear on screen before the data layer has answered both queries. Don't try to "seed" one side with `.onStart { emit(emptyList()) }` to get a faster first paint: the pill would then render with `count = 0` (i.e. silently absent) before the real discussions count lands, then re-render after the second emission — a visible pop-in. `combine`'s wait-for-both semantics give the right pre-paint timing for free. The `loadingPersists_untilBothFlowsEmit` test exists to pin this behavior against a future refactor (e.g. someone swapping `combine` for `combineTransform` with a seeded initial value).
- **Test helper churn is fine when the VM's data-flow shape changes.** `stubRepo(source)` took one flow and returned it for every filter. The new VM subscribes twice with different filters; the helper had to grow a second parameter (`channels`, `discussions`). Three existing test methods became `stubRepo(channels, discussions = emptyFlow())` style calls. Resist the temptation to wrap the old shape with a `stubRepo(source) = stubRepo(source, source)` overload — the call sites should make the two-subscription structure visible, and a defaulted overload obscures it. The `erroringRepo` helper stayed single-throwing because any subscription on either filter is enough to trigger `.catch`.
- **`Empty` as `data object` → `data class` is a no-op for `is` smart casts but breaks `assertEquals(ChannelListUiState.Empty, …)`.** Equality on a `data object` was singleton-identity; on a `data class(0)` it's component-wise. Existing tests like `assertEquals(ChannelListUiState.Empty, vm.state.value)` had to become `assertEquals(ChannelListUiState.Empty(0), vm.state.value)`. Two such call sites in `ChannelListViewModelTest`. The FAB-guard at `ChannelListScreen.kt:67` uses `state is ChannelListUiState.Empty` (class check, not equality) — unchanged.
