# #130 — feat(ui/thread): code block rendering with syntax highlighting

Replaces the flat `CodeBlock` fallback shipped by [#129](./129.md) with a syntax-highlighted, horizontally-scrolling code surface. Wires `dev.snipme:highlights` 1.1.0 (KMP-native, pure-Kotlin tokeniser), threads the fenced-code info string through to a top-right language label, and binds token colours to `MaterialTheme.colorScheme` slots so AC6 ("no hardcoded hex values") holds by construction. The composable stays file-private inside `MarkdownText.kt`; no public API changes, no new source file.

## Summary

- Library: **`dev.snipme:highlights` 1.1.0** — pure-Kotlin KMP tokeniser, no AndroidX, no reflection, no proguard rules. Wire-up is the same three-line shape as [#129](./129.md): one `[versions]` line, one `[libraries]` line, one `implementation(libs.snipme.highlights)`. Library returns a `CodeStructure` of `PhraseLocation(start, end)` lists per token category; the renderer owns the colour mapping — we never ask the library for a colour, so themability is automatic.
- `CodeBlock` signature widened from `(content: String)` to `(content: String, language: String?)`. Visibility stays `private`; the function name is unchanged. Two call sites in `MarkdownText.kt`'s block dispatcher updated — `CODE_FENCE` extracts the first `MarkdownTokenTypes.FENCE_LANG` child's text (trimmed; `null` if absent/blank); `CODE_BLOCK` (indented 4-space) passes `language = null`.
- Visual structure: outer `Surface(RoundedCornerShape(CodeBlockCornerRadius = 8.dp), color = surfaceContainer)` wrapping a `Box`. Inside the box: the scrollable code body (`Modifier.horizontalScroll(rememberScrollState())` + `Text(softWrap = false, fontFamily = Monospace)`) and, conditionally, a `Text(language, style = labelSmall, color = onSurfaceVariant)` aligned `Alignment.TopEnd` with `padding(horizontal = CodeBlockHorizontalPadding = 12.dp, vertical = CodeBlockLabelVerticalPadding = 4.dp)`. The label sits **outside** the `horizontalScroll` chain, so it stays anchored to the corner regardless of scroll position.
- **No outline border, no header strip with separator.** Ticket body pins flat-tile aesthetics on `surfaceContainer`; this deliberately diverges from Figma `16:45`–`16:50` which shows a `surface` + `outline-variant` border with a top-left label header. Recorded in [ADR 0003](../decisions/0003-syntax-highlighter-library.md).
- Languages supported per AC3:
  - `kotlin` / `kt` / `kts` → `SyntaxLanguage.KOTLIN`
  - `bash` / `sh` / `shell` / `zsh` → `SyntaxLanguage.SHELL`
  - `json` → `SyntaxLanguage.JAVASCRIPT` — the library has no dedicated JSON lexer, but JSON is a subset of JS object-literal syntax so strings, numbers, and punctuation tokenise correctly. The 3-line comment in `resolveSyntaxLanguage` documents the reason at the call site.
  - `markdown` / `md` → `null` (no Markdown lexer in the library) — falls through to plain monospace; the label still renders.
  - Anything else (`lua`, `rust`, `py`, …) → `null`, plain monospace, label still renders.
- Token → M3 slot mapping (the "no hardcoded hex" contract from AC6):
  - `structure.keywords` → `tertiary` + `FontWeight.Medium`
  - `structure.annotations` → `tertiary`
  - `structure.strings` → `secondary`
  - `structure.literals` → `primary`
  - `structure.comments` / `structure.multilineComments` → `onSurfaceVariant` + `FontStyle.Italic`
  - Punctuation / operators / default → inherited `LocalContentColor.current`
- Tokenisation memoised at two layers: `remember(language) { resolveSyntaxLanguage(language) }` (cheap string-to-enum), then `remember(content, syntaxLanguage) { if (syntaxLanguage == null) null else tokeniseCode(content, syntaxLanguage) }` (the library call). Tokeniser runs only when `(content, language)` changes — streaming (#132) re-tokenises per token batch which stays sub-millisecond for typical assistant code blocks.
- Defensive `runCatching { … }.getOrNull()` wraps the `Highlights.Builder().code(...).language(...).build().getCodeStructure()` call. On throw, returns `null` and the block falls through to plain monospace (same path as unknown-language fences). No logging, no toast — same posture as the link-no-op fallback in [#129](./129.md). The renderer is total; unknown languages, empty inputs, and library failures all degrade to "monospace on `surfaceContainer`" which is already the [#129](./129.md) baseline.
- Span ranges are coerced to `(0, content.length)` via `PhraseLocation.start.coerceIn(0, contentLength)` / `end.coerceIn(start, contentLength)` before `addStyle(...)`. Belt-and-suspenders against a future library version returning offsets that exceed the input length — `AnnotatedString.Builder.addStyle` throws `IndexOutOfBoundsException` on out-of-range spans, which would crash the assistant message. The coerce keeps the renderer total even if the library regresses on its offset contract.
- Preview surface: **extended the existing `MARKDOWN_PREVIEW_FIXTURE` in `MessageBubble.kt`** with three new fences (JSON, Bash, Markdown) alongside the pre-existing Kotlin fence. **No new `@Preview` functions, no new files** — `MessageBubbleMarkdownLightPreview` / `MessageBubbleMarkdownDarkPreview` automatically re-render the extended fixture and cover AC7 (Kotlin / JSON / Bash / Markdown in light + dark). The Markdown fence inside the fixture is the visual acceptance signal for the plain-monospace fallback path; the contrast against the three highlighted fences is itself the AC3 demonstration.
- AC6 ("no hardcoded hex") and AC2 ("no raw `.dp` / `.sp` literals") hold by construction: every colour reaches `MaterialTheme.colorScheme.<slot>` through `@Composable` lookups in `buildHighlightedCode`; every dimension is a named file-private `val` (`CodeBlockHorizontalPadding`, `CodeBlockVerticalPadding`, `CodeBlockLabelVerticalPadding`, `CodeBlockCornerRadius`); every text size is reached through `MaterialTheme.typography.bodyMedium` / `labelSmall`.
- AC4 ("Long lines scroll horizontally inside the block — no wrapping") is `Modifier.horizontalScroll(rememberScrollState())` + `Text(softWrap = false)`. Each `CodeBlock` invocation gets its own `rememberScrollState()` so independent fenced blocks scroll independently. No custom scrollbar — Compose's default fling-and-edge behaviour is sufficient for Phase 0.
- Architect-side security pass: PASS. Code-block content and the language label are rendered as plain `Text` — no execution path, no HTML parse, no path resolution. Library is reflection-free, no transitive AndroidX, pinned via `version.ref` (no SNAPSHOT, no version-range). Full reasoning in `docs/specs/architecture/130-code-block-rendering-syntax-highlighting.md` § Security.
- No new tests — same precedent as [#126](./126.md), [#128](./128.md), [#129](./129.md). Visual verification by `@Preview` rendering + `./gradlew assembleDebug` + `./gradlew lint` passing.

## Patterns established

- **Token-walking renderers own the colour mapping; the library produces structural spans only.** `dev.snipme:highlights` returns `PhraseLocation(start, end)` lists per category — strings, numbers, keywords, comments. The renderer reads `MaterialTheme.colorScheme.<slot>` and writes `SpanStyle(color = …)` per span. Pattern applies to any future "third-party tokeniser feeds a Compose `AnnotatedString`" component: never accept a library's colour palette; route every colour through `MaterialTheme.colorScheme` so theming (light/dark, dynamic colour, future high-contrast variants) works for free. The contract that hardens against AC6-style regressions is "the renderer never imports `Color`".
- **Coerce third-party span ranges before `addStyle`.** `AnnotatedString.Builder.addStyle(style, start, end)` throws `IndexOutOfBoundsException` on out-of-range offsets, which would crash the assistant message on a library bug. `applySpan` clamps both ends to `(0, contentLength)` and skips empty spans. Pattern applies to any future `AnnotatedString` builder that consumes externally-produced offsets (other syntax highlighters, search-highlight overlays, citation spans from the API). Cost is two `coerceIn` calls; benefit is the renderer can't crash on a library regression.
- **Two-layer `remember` for "string-to-enum, then enum-to-tokenised-output" pipelines.** `remember(language) { resolveSyntaxLanguage(language) }` is cheap but keys the second `remember(content, syntaxLanguage) { tokeniseCode(...) }` on the **resolved** language, so two fences with `"kotlin"` and `"kt"` info strings share the same cached tokenisation for identical bodies. Streaming-friendly: the per-batch re-tokenise costs are bounded to "content changed" not "language changed". Pattern applies to any future "parse stage → expensive-build stage" chain (e.g. compile-once + render-many for diagram blocks).
- **Library wire-up stays the smallest possible diff: one `[versions]` line, one `[libraries]` line, one `implementation(...)` line.** Same shape as [#129](./129.md). The `dev.snipme:highlights` choice was constrained by this rule — any candidate requiring proguard rules, KSP, or a Gradle plugin entry would have been re-evaluated. Pattern is reinforced ticket-over-ticket; future leaf libraries (e.g. `compose-shimmer`, `coil`) inherit the same three-line constraint.
- **Surface diverges from Figma when the ticket body and Figma disagree — the ticket body wins, the divergence is recorded.** Figma `16:45`–`16:50` shows `surface` + `outline-variant` border with a top-left label header strip. The ticket body pins `surfaceContainer` (matching the inline-code `CODE_SPAN` idiom from [#129](./129.md)) and a top-right overlay label with no separator. The architect spec recorded both deviations explicitly; the developer pass implemented the ticket-body version. Pattern: when ticket body and design source conflict, prefer the body, name the deviation in the architect spec + this ticket file, and leave the design-source reconciliation to a follow-up design call. Same posture as the [`MessageBubble`](../features/message-bubble.md) "Claude-style flat assistant" divergence shipped in [#128](./128.md).

## Lessons learned

- **`Highlights` does not expose a JSON lexer in 1.1.0.** The closest match is `SyntaxLanguage.JAVASCRIPT` — JSON's grammar is a strict subset of JS object literals, so strings, numbers, brackets, and colons tokenise correctly. The 3-line comment at the `"json" -> SyntaxLanguage.JAVASCRIPT` arm in `resolveSyntaxLanguage` documents the substitution at the call site so a future maintainer adding a real JSON lexer (or switching highlighter library) can find the swap point with `grep "JAVASCRIPT"` rather than re-deriving the rationale. Acceptance for AC3 is "the JSON fence in the preview renders with token colouring on strings and numbers" — preview-verified.
- **`Highlights` does not expose a Markdown lexer in 1.1.0 either.** `markdown` / `md` fences resolve to `null` and render as plain monospace. AC3 ("works for at least … Markdown") is interpreted as "renders correctly with the block styling — background, monospace, label, scroll — but without token colours." The visual signal in the preview is the Markdown fence rendering with the language label but without colouring; the contrast against the three highlighted fences is the AC3 demonstration. If a future ticket wants Markdown token colouring, the cheapest path is a tiny hand-rolled tokeniser inside `CodeBlock` (~40 lines: ATX headings, emphasis pairs, fence delimiters) rather than swapping libraries.
- **Language label overlays the code body without a separator strip — long first lines can sit under the label.** The label is anchored `Alignment.TopEnd` outside the `horizontalScroll` chain; if the first code line is wide enough to extend under the label area, glyphs and label text overlap visually. Acceptable for Phase 0 (preview-verifiable, no user complaint yet). If/when this becomes a complaint, the mitigations to consider: (a) add right-padding to the first code line equal to the label width, (b) tint the label background (e.g. `surfaceContainerHighest`), (c) reintroduce a top header strip with a divider (the Figma original). Don't pre-build any of those — pin the choice in a follow-up ticket against an observed failure mode.
- **`rememberScrollState()` inside the composable gives each `CodeBlock` instance its own independent scroll position.** Two fenced blocks in the same message scroll independently; recomposition with the same `(content, language)` preserves the scroll position. If a future ticket needs to reset scroll (e.g. when streaming reflows the block mid-stream), the right hook is keying the `rememberScrollState()` on a streaming-completion flag — not in scope here.
- **Compose's `horizontalScroll` does not render a visible scrollbar by default.** Long lines scroll on touch with edge-bounce feedback, but there's no static visual indicator of overflow. Acceptable for Phase 0; if users miss the affordance, a follow-up adds a thin track via a custom `Modifier` or a fading right-edge gradient. Not pre-built.
- **`runCatching { … }.getOrNull()` is the right shape for "library should be total but might regress" — the renderer is total even if the library throws.** `Highlights` is documented as total on valid `SyntaxLanguage` input, but the tokeniser walks model-controlled text (pathological unterminated strings, malformed escapes are exactly the inputs that trigger third-party-library bugs). The catch keeps the message readable when the highlighter regresses; the user-visible result degrades to plain monospace, which is already the [#129](./129.md) baseline. Same posture as the link-no-op fallback in [#129](./129.md).
