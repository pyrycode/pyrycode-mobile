# #212 — feat(ui/sheets): `WorkspacePickerSheet` stateless composable (Figma 20:2)

Lands [`WorkspacePickerSheet`](../features/workspace-picker-sheet.md), the first **sheet** composable in `ui/conversations/components/` (siblings are all rows + bubbles). Renders Figma `20:2`: a `"Choose workspace"` modal bottom sheet with a Recent section (caller-supplied paths in mono) and a single Other row (`"Create new folder under pyry-workspace…"`). Stateless contract per spec — no `ViewModel`, no repository binding, no host integration; the host that wires this sheet into the Channel List FAB / "Save as channel" flow, owns sheet visibility, and pairs it with the Create Folder follow-up dialog (Figma `19:44`) lands in #207. Consumes [`recentWorkspaces()`](../features/conversation-repository.md) from [#209](./209.md) and [`createWorkspaceFolder(name)`](../features/conversation-repository.md) from [#210](./210.md) only indirectly via host wiring.

See [`../features/workspace-picker-sheet.md`](../features/workspace-picker-sheet.md) for the evergreen contract.

- **`ModalBottomSheet` shell + `*Content` body split** is the load-bearing architectural move (carried verbatim from the architect spec). The public `WorkspacePickerSheet` wraps `WorkspacePickerSheetContent` inside a `ModalBottomSheet(onDismissRequest = onDismiss, sheetState = sheetState)`. Both are `internal`. The split exists because (a) the modal scrim + animation machinery don't render in the IDE preview pane, so previews target `WorkspacePickerSheetContent` directly with a `Surface(color = surfaceContainerLow)` wrap that simulates the sheet's default container colour; (b) `androidx.compose.ui.test.createComposeRule()` doesn't attach the modal host, so the five Compose UI tests also target `WorkspacePickerSheetContent` (same M3-sample pattern Material 3 docs use for sheet previews).
- **`sheetState` defaulted but exposed.** `sheetState: SheetState = rememberModalBottomSheetState(skipPartiallyExpanded = true)` lets the host (#207) drive animated-close via `sheetState.hide()` *before* invoking `onDismiss`; defaulting keeps the preview / one-off uses one-line. `skipPartiallyExpanded = true` because this is a short content list (≤ ~5 rows + the Other row), not a half-sheet — matches Figma's full-content layout.
- **Single `Column(fillMaxWidth)` body, five children:** `TitleRow(onClose = onDismiss)`, optional `SectionHeader("Recent")` + per-path `RecentRow`s gated on `recent.isNotEmpty()` (AC empty-state contract), `SectionHeader("Other")` always, `CreateNewRow(onClick = onCreateNew)`, `Spacer(height = 24.dp)`. No `LazyColumn` — the recents list is bounded (the host dedupes + caps via filtering), and a non-lazy `Column` keeps the layout stable inside the modal which has its own scroll machinery if content overflows.
- **`Modifier.clickable` attached unconditionally on every row.** Same idiom as [`ConnectionBanner`](./connection-banner.md) / [`ToolCallRow`](./tool-call-row.md): the M3 default ripple paints uniformly, the tap target is the whole row, no chevron / trailing affordance rendered. Leading folder icons set `contentDescription = null` because the adjacent `Text` carries the semantic content — a TalkBack user hears `"~/Workspace/Projects/pyrycode"`, not `"~/Workspace/Projects/pyrycode, folder icon"`.

## Patterns established

- **`ModalBottomSheet` shell + stateless `*Content` body split is the canonical M3 sheet shape for this codebase.** First sheet in `ui/conversations/components/`; the split serves three masters at once: (a) IDE previews can render the content directly because the modal scrim + animation machinery don't render in the preview pane, (b) `createComposeRule()`-style Compose UI tests can target the content without bringing up an `Activity`, and (c) the host gets `sheetState` exposed for animated-close control. **Rule for future sheets in this codebase:** the public composable is the `ModalBottomSheet(...)` shell that takes `onDismiss` + an exposed `sheetState`; the body lives in an `internal` sibling `*Content` composable with the same callback shape minus `sheetState`. Previews and tests target the `*Content`. Analogous to but distinct from the [`ToolCallRow`](./tool-call-row.md) / [`ConnectionBanner`](./connection-banner.md) "stateful-wrapper + stateless-content" split — there the inner exists because the outer carries `rememberSaveable` / a `when`-resolution that the preview can't drive deterministically; here the inner exists because the outer is structurally untestable from `createComposeRule()`.
- **Compose UI tests for new components in `ui/conversations/components/` live in `androidTest/`, not `test/`.** `androidx.compose.ui.test.junit4.createComposeRule()` requires the Android runtime; placing the file under `test/` (where the JUnit-on-JVM tests run) would fail to resolve `AndroidJUnit4` and the `compose.ui.test` artifacts. Same shape as the pre-existing `ConversationAvatarTest.kt`. Pattern for any future Compose-test addition in this package: file path is `app/src/androidTest/java/de/pyryco/mobile/ui/conversations/components/<Component>Test.kt`, runner is `AndroidJUnit4`, rule is `createComposeRule()`, content is the stateless inner (not the public modal / state shell), assertions use `hasText` / `hasContentDescription` lookups with `var invoked` lambda-capture for callback verification.
- **`internal` is the right visibility for both the modal shell and the content composable.** Same posture as `ThemePickerDialog` in `ui/settings/`. The host (#207) lives in a sibling package within the same module, so the visibility step from `private` (file-only) to `internal` (module-wide) is sufficient; `public` would leak the API surface outside the module. The content composable is `internal` rather than `private` for the same reason — the Compose UI tests live in `androidTest/`, which is its own source set but compiles against the module, so `private` would hide the symbol from the test. **Rule:** if a composable is targeted by an `androidTest/` Compose UI test, it must be at least `internal`; if it's also targeted by the host package, `internal` is still the right visibility — no need to escalate to `public` for cross-package use within the module.
- **Preview wrap simulates the runtime modal container colour explicitly.** `PyrycodeMobileTheme { Surface(color = surfaceContainerLow, contentColor = onSurface) { Column(padding(top = 12.dp)) { WorkspacePickerSheetContent(...) } } }` is the preview shape: the `Surface` paints the same `surfaceContainerLow` slot that `ModalBottomSheet` uses by default, and the leading 12dp top-padding substitutes for the drag-handle gap. Without the explicit `Surface` colour, the preview would render against the theme's default `surface` slot and the visual contrast with the runtime sheet would mislead reviewers. Pattern for any future `*Content`-style preview: pin the container colour to match what the runtime parent will paint.

## Lessons learned

- **`createComposeRule()` cannot render a `ModalBottomSheet`** — the modal requires an `Activity`-attached window for its scrim and animation machinery, and `createComposeRule()` (as opposed to `createAndroidComposeRule<ComponentActivity>()`) attaches to a synthetic `ComposeView` host. Pointing the five Compose UI tests at `WorkspacePickerSheet` (the modal shell) instead of `WorkspacePickerSheetContent` would have rendered nothing and the `hasText("Recent")` lookups would all `assertDoesNotExist`. The architect spec called this out by name (`"WorkspacePickerSheetContent (not the modal)"`) — the split exists precisely so the tests have something to attach to. **Lesson:** if a future agent thinks the `*Content` split is over-engineering and inlines the body back into the modal, they will silently break the test suite *and* the previews; the split is load-bearing, not cosmetic.
- **The trailing ellipsis on `"Create new folder under pyry-workspace…"` is the literal U+2026 character.** Typing `...` (three dots) compiles, renders visually similar, but fails an exact-equals match against the Figma string and against the `hasText("Create new folder under pyry-workspace…")` test selector. Same posture as [`ConnectionBanner`](./connection-banner.md)'s U+2014 em-dash in `"Offline — tap to retry"`. The Kotlin file is UTF-8, the IDE preserves the character on round-trip, `git diff` shows it as one character — no encoding hazard observed. **Pattern:** Figma-source copy strings with typographic punctuation (em-dashes, en-dashes, ellipses, smart quotes) carry the actual Unicode codepoint, not the ASCII multi-character approximation. The test that asserts on the literal copy is the regression artifact.
- **The four ViewModels / 5 screen tests / 11 anonymous `object : ConversationRepository` fakes were *not* touched by this ticket** even though it consumes the [`recentWorkspaces()`](./conversation-repository.md) data surface from [#209](./209.md) — because this ticket only renders what its caller passes in. The host (#207) is the seam that collects the flow, and *that* ticket will live with whatever wiring cascade falls out of `recentWorkspaces()` + `createWorkspaceFolder()` consumption. **Lesson:** the stateless-composable contract is exactly the lever that keeps the test-fake cascade off the UI slice — when the host owns the data collection, the renderer owns nothing but the visual contract. Pre-binding the renderer to the repository here would have pulled the cascade into this ticket without changing the eventual host's surface area.
- **`Icons.Outlined.Folder` and `Icons.Outlined.CreateNewFolder` are already imported via `material-icons-extended`** — added by [#131](./131.md) for `ToolCallRow`'s `Description` / `Terminal` / `Build` icons. No `gradle/libs.versions.toml` edit, no `app/build.gradle.kts` edit, no new dependency. The R8 tree-shaking that bounded the [#131](./131.md) release-APK cost to the icons actually referenced now extends by two more icons — same posture, no new bloat dimension. **Pattern:** before adding a new icon-library dependency, check whether `material-icons-extended` is already on the classpath; it covers ~7000 outlined / filled / rounded glyphs, and the additional R8-bounded production cost per icon is negligible against the alternative (vector drawable + `painterResource` lookup table).
