# #94 ‚Äî Archived Discussions screen + ViewModel + Settings entry

Builds the UI restoration surface for the archived state introduced in #93 / #96: a Settings ‚Üí "Archived discussions" entry navigates to a new screen that observes `ConversationFilter.Archived`, post-filters to `!isPromoted`, renders the rows with `Modifier.alpha(0.65f)`, and exposes `unarchive(id)` behind a long-press dropdown. No data-model changes; no 30-day auto-archive worker (separate ticket).

## Summary

- New `ArchivedDiscussionsViewModel` exposes `state: StateFlow<ArchivedDiscussionsUiState>` (`Loading` / `Empty` / `Loaded(discussions)` / `Error(message)`) built from `repository.observeConversations(ConversationFilter.Archived).map { it.filter { !isPromoted } }.catch { ‚Ä¶ }.stateIn(viewModelScope, WhileSubscribed(5_000L), Loading)`. The `!isPromoted` post-filter in the VM is deliberate: the repository's `Archived` filter (#93) returns archived rows regardless of `isPromoted`, but this screen only wants archived **discussions** ‚Äî keeping the data-layer filter generic and the projection screen-specific avoided fragmenting the repository interface with an `ArchivedDiscussions` variant for one consumer. Empty filtered list ‚Üí `Empty`; non-empty ‚Üí `Loaded`.
- `onEvent(ArchivedDiscussionsEvent)` handles two cases. `RestoreRequested(id)` fires `viewModelScope.launch { repository.unarchive(id) }` ‚Äî no optimistic UI bookkeeping; the row drops naturally on the next `MutableStateFlow` re-emission. `BackTapped` is a `Unit` arm in the VM so the screen can use a single uniform `onEvent` lambda; the actual `popBackStack()` lives in the NavHost. No `navigationEvents` channel ‚Äî restore stays on-screen, back is owned by the NavHost. Matches the spec verbatim.
- Stateless `ArchivedDiscussionsScreen(state, onEvent, modifier)` ‚Äî `Scaffold` + `TopAppBar` (title `R.string.archived_discussions_title`, back-arrow `IconButton` reusing `R.string.cd_back`) over a `when (state)` body. `Loading` / `Empty` / `Error` are centered text via a file-private `CenteredText(text, modifier)`; `Loaded` is a `LazyColumn` keyed by `conversation.id`. Each row is the file-private `ArchivedDiscussionRow` ‚Äî a `Box` wrapping the shared `ConversationRow` (with `lastMessage = null`, empty `onClick`, `Modifier.alpha(0.65f)`, `onLongClick = { menuExpanded = true }`) and a `DropdownMenu` with one `DropdownMenuItem` "Restore" that dispatches `RestoreRequested(conversation.id)` then dismisses. **Exactly one gesture (long-press), not also swipe** ‚Äî per the ticket's explicit "exactly one gesture, not both" clause. The 0.65 alpha is the same muted treatment `DiscussionListScreen` uses (the secondary-tier signal, lifted to here for the archived-tier signal).
- "Archive date" trailing slot: the existing `ConversationRow` trailing renders `formatRelativeTime(lastUsedAt)`. For an archived discussion that's "the last time it was used before being archived" ‚Äî close to the AC's "archive date" but not literally it. Shipped as-is for this slice; the `Conversation` model has no `archivedAt: Instant?` field today and adding one solely for this trailing slot would push scope past the AC. Follow-up: when the 30-day auto-archive worker ticket lands and starts stamping an `archivedAt`, switch this screen's trailing slot to format that field. Documented in the spec's ¬ß Open questions and called out below.
- "Last-message preview" AC item maps to the existing `ConversationRow` supporting-text slot, which is currently driven by `lastMessage: Message?` ‚Äî passed as `null` here because no per-`Conversation` last-message projection exists in Phase 0. The slot is wired so it'll fill automatically when that projection lands; no parallel row composable.
- Settings wiring: `SettingsScreen` gains a required `onOpenArchivedDiscussions: () -> Unit` parameter (no default ‚Äî navigation/event callbacks never carry defaults; established in #87 and #91). The pre-existing `Storage` ‚Üí `Archived conversations` row was rewritten in place: headline switches to `stringResource(R.string.archived_discussions_settings_row)` ("Archived discussions" ‚Äî verbatim from the ticket, since the row navigates to archived **discussions** only), the placeholder `supporting = "11 archived"` line is dropped (AC doesn't request a count; adding a live count would expand into another VM-StateFlow combinator), and `onClick = onOpenArchivedDiscussions`. Both `@Preview`s gain `onOpenArchivedDiscussions = {}`. `SettingsScreenTest` `setContent` blocks (`app/src/androidTest/.../ui/settings/SettingsScreenTest.kt`) propagate the new callback as `{}`; no new test methods ‚Äî the Storage row's wiring matches the project's existing precedent of leaving non-About row coverage to manual + VM unit tests.
- Nav graph: `Routes` gains `const val ARCHIVED_DISCUSSIONS = "archived_discussions"`. `composable(Routes.SETTINGS)` passes `onOpenArchivedDiscussions = { navController.navigate(Routes.ARCHIVED_DISCUSSIONS) }` to `SettingsScreen`. A new `composable(Routes.ARCHIVED_DISCUSSIONS)` block resolves `koinViewModel<ArchivedDiscussionsViewModel>()`, collects `vm.state` via `collectAsStateWithLifecycle()`, and forwards events with `BackTapped ‚Üí navController.popBackStack()` and `is RestoreRequested ‚Üí vm.onEvent(event)`. Mirrors the License-block shape.
- Koin: `viewModel { ArchivedDiscussionsViewModel(get()) }` appended to `appModule`. One constructor arg (the existing `ConversationRepository` singleton).
- Strings: four new ids in `app/src/main/res/values/strings.xml` ‚Äî `archived_discussions_title` ("Archived discussions"), `archived_discussions_empty` ("No archived discussions"), `archived_discussions_settings_row` ("Archived discussions"), `restore_action` ("Restore"). `cd_back` reused unchanged.

Package placement: under `ui/settings/` (alongside `SettingsScreen` and `LicenseScreen`), **not** `ui/conversations/list/`. Rationale per the spec: discoverability follows the entry point. The VM consumes `ConversationRepository` instead of the (settings-package) `AppPreferences`, which is fine ‚Äî `SettingsViewModel` consumes `AppPreferences`; settings-package ViewModels aren't constrained to settings-only dependencies.

## Patterns established

- **One feature, two new files (`<Feature>ViewModel.kt` + `<Feature>Screen.kt`) under the destination-aligned package, even when the VM contract is minimal.** `ArchivedDiscussionsViewModel` is tiny ‚Äî one stream, one launch ‚Äî but it still ships as a sibling file rather than as a nested object inside the screen. The cost is two ~50‚Äì80-line files instead of one; the upside is that the VM and screen always sit next to each other in IDE navigation, the test file maps 1:1 to the VM file, and there's never a "where does this VM live" question. Established by the discussion-list pair and re-confirmed here.
- **For a screen that filters a shared filter further, do the post-filter in the VM, not the data layer.** The repository's `ConversationFilter.Archived` returns archived rows of both tiers (#93's deliberate "an `Archived` filter that's still composable with `isPromoted` later"); the screen-side `.filter { !it.isPromoted }` inside the VM keeps the data-layer surface from accreting one variant per screen. Next slice that wants archived **channels** can read the same `Archived` filter with a different post-projection. Don't reach for a sibling `ConversationFilter.ArchivedDiscussions` variant just because there's currently one consumer.
- **Use a `Box`-wrapping pattern to attach a `DropdownMenu` to an existing shared row composable.** `ArchivedDiscussionRow` keeps `ConversationRow` untouched ‚Äî the `onLongClick` parameter (added in #25) plus a sibling `DropdownMenu` anchored to a wrapping `Box` is enough. Don't add a `trailing dropdown` slot to `ConversationRow` itself for one consumer. The same shape was used by `DiscussionListScreen`'s `DiscussionRow`; this is the second instance, so it's officially "the pattern" rather than "one screen does it this way."
- **Sole-event-channel screens hand the VM a `BackTapped` arm that does nothing.** `ArchivedDiscussionsEvent.BackTapped` is a `Unit` arm in the VM. The reason: it's cheaper to give the screen a uniform `onEvent: (Event) -> Unit` lambda than to thread a parallel `onBack: () -> Unit` callback. The NavHost intercepts `BackTapped` and routes to `popBackStack()` before falling through to the VM. Same shape as `DiscussionListEvent.BackTapped`; do this rather than `(state, onEvent, onBack)` for any future stateless secondary screen.

## Lessons learned

- **The repository `Archived` filter is `isPromoted`-agnostic by design (#93's choice), but a UI consumer that wants only one tier must post-filter in the VM, not assume the data layer scoped it.** Discovered when sketching the spec ‚Äî the obvious "subscribe to `Archived`, render the list" path would have included archived channels in the result if any existed. None do today (no production flow archives a channel), but the seed shape doesn't promise that forever. The tests `loaded_dropsPromotedArchivedFromList` and `empty_whenAllArchivedAreChannels` lock this in: any regression of the VM's post-filter (e.g. dropping the `!isPromoted` predicate) would fail loudly.
- **"Archive date" in the UI ‚âÝ `archivedAt` in the model ‚Äî yet.** `Conversation` has no `archivedAt` field; `archive(id)` flips a boolean (#93) without recording a timestamp. The trailing slot of `ConversationRow` renders `formatRelativeTime(lastUsedAt)`, which is "last used before being archived" ‚Äî a near-miss for the AC's "archive date" but the closest available signal without a model change. The right time to add `archivedAt: Instant?` is when the 30-day auto-archive worker lands and starts stamping it on the way in (the worker needs the timestamp to decide *which* records to evict, so it's the natural owner). Don't backfill the field here; this slice deliberately stays inside the AC's surface.
- **`ConversationRow.lastMessage = null` is the AC-defensible Phase-0 shape for the "last-message preview" requirement.** The slot is wired; only the data isn't. When `ConversationRepository` grows a per-`Conversation` last-message projection (currently the projection is per-conversation-id via `observeMessages`, which is the wrong shape for a list screen because it'd require N collectors), this screen fills automatically. Don't try to derive a preview from `observeMessages` here ‚Äî the cost is one collector per visible row and the cleanup-on-scroll surface is non-trivial.
- **Restore failure is a non-feature for this slice; do not add a snackbar.** The Fake's `unarchive(id)` only throws on unknown id, which `viewModelScope.launch { ‚Ä¶ }` swallows. Phase 4 will introduce real failure modes; the visible-error surface lands then. Defending a failure mode that hasn't been observed (CLAUDE.md's "evidence-based fix selection" rule) would add another `Channel<‚Ä¶>` for transient errors plus a `SnackbarHost` and a `LaunchedEffect`, all unused today.
- **Long-press dropdowns and `SwipeToDismissBox` are two different gestures, not one composed gesture; the ticket's "exactly one" wording is intentional.** `DiscussionListScreen` ships both for the save-as-channel action (#25), but that was a deliberate redundancy for a frequent destructive-feeling action ("save as channel" sounds reversible but isn't entirely ‚Äî the channel-name derivation is one-way). `RestoreRequested` is unambiguously reversible (re-archive lands when the worker ships, or via repository-direct in tests), and the AC says "long-press". A future change to swipe-only would be a one-line edit; a future change to both would be a one-line addition; either is cheaper than starting with both and arguing about which to drop later.
