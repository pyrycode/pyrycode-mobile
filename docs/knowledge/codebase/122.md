# #122 — Scanner Connecting: TopAppBar + spinner position

Re-aligns `ScannerConnectingScreen` to Figma node `32:20`: adds the standard `Pair with pyrycode` `TopAppBar` (back arrow + title) that the rest of the pairing flow already wears, repositions the spinner cluster above the vertical centre via a 316:372 weighted-spacer split, and corrects the headline's color role from `onSurface` to `onSurfaceVariant`. Walks back the #62 "top bar is a NavHost concern" rationale — the bar moves into the screen so callers don't have to re-host it once Phase 4 wires this destination in.

## Summary

- Replaces the prior `Surface → Column(Arrangement.Center)` cluster with `Surface → Column(systemBarsPadding) → TopAppBar + weighted body Column`. Mirrors `ScannerScreen.kt`'s pairing-flow chrome verbatim (transparent container, back-arrow `IconButton`, `titleLarge` title on `onSurface`).
- Spinner vertical position becomes proportional: `Spacer(weight = 316f)` above the cluster, `Spacer(weight = 372f)` below it — the 316:372 split is `(384 − 68)` against `(892 − 520)` from Figma 32:20, computed against the content area *below* the 68dp TopAppBar so the position survives different device heights (AC2).
- Intra-cluster spacing stays in fixed dp because Figma specifies pixel-accurate gaps between spinner, caption, and address: `Spacer(28.dp)` (spinner → caption, 460 − 432) and `Spacer(16.dp)` (caption → address, 500 − 484).
- New required `onBack: () -> Unit` parameter on the public composable (no default — matches `SettingsScreen` / `LicenseScreen` / `ArchivedDiscussionsScreen` precedent). Wired from the `navigationIcon` `IconButton` with `contentDescription = "Back"` (English literal, not `R.string.cd_back`).
- Headline color-role correction: `MaterialTheme.colorScheme.onSurface` → `onSurfaceVariant` on the `bodyLarge` "Connecting to your pyrycode server…" `Text`. AC3 contract. The `bodyMedium` Roboto-Mono server-address `Text` was already on `onSurfaceVariant` — unchanged.
- Light + dark `@Preview`s at 412×892 updated to pass `onBack = {}`; no other preview changes.

## Patterns established

- **Weighted spacers, not fixed dp, when a Figma frame specifies a proportional vertical position.** Figma's "y=384 within an 892-tall frame" target survives different device heights only if the layout splits the available area in the Figma ratio — not if it commits to absolute dp. The shape is `Column(weight = 1f)` containing `Spacer(weight = before)` + cluster + `Spacer(weight = after)`, with `before:after` computed in the Figma coordinate system. Reserve fixed dp for the gaps *inside* the cluster where Figma specifies pixel-accurate spacing — those are not proportional and chasing them with weights mis-spaces the cluster at non-892dp viewports.
- **Compute the ratio against the area below the TopAppBar, not against the full frame.** The TopAppBar consumes a fixed 64dp (≈68dp with content padding); subtract that from the Figma frame height before computing the split. Here: `(384 − 68) : (892 − 520) = 316 : 372`, not `384 : 508`. Otherwise the spinner drifts down by the bar height at every viewport.
- **`contentDescription` for the screen's own back-arrow stays as the English literal `"Back"` on these onboarding screens.** `LicenseScreen` (#91) routes through `R.string.cd_back` because its test resolves the literal via `InstrumentationRegistry.getString(...)` for the same reason it would localise. The pairing-flow screens (`ScannerScreen`, `ScannerConnectingScreen`) carry the literal inline — `ScannerScreen.kt` set this precedent in #60, so #122 matches it for cluster-internal consistency rather than reaching for the shared string id. Pick one or the other per nav-flow grouping; don't mix.

## Lessons learned

- **The "TopAppBar is a NavHost concern" rule from #62 was wrong.** #62 deliberately omitted the top bar with the rationale "future Phase 1.5 / Phase 4 catchup tickets in this nav graph: assume the top bar is host-level until the wiring ticket says otherwise. Don't re-debate it per screen." That rule got walked back here — the bar belongs *in* the screen because (a) the back-arrow + title are part of the visual identity the Figma frame specifies, not chrome the host overlays; (b) `ScannerScreen.kt` and `ScannerDeniedScreen.kt` already host their own bars (the only screens that don't are the centred-illustration ones, and even those Figma frames carry the bar); (c) "wire it later at the host" stranded the screen's back-navigation affordance through three release windows. The corrected rule: a `TopAppBar` shown in the screen's Figma frame goes in the screen, not the host — even when the screen is not yet wired into a NavHost. The `onBack: () -> Unit` parameter is the forward-compatibility hook; downstream tickets fill it in. (See also #61 Scanner Denied — that screen still omits the bar; it has its own back-navigation deferral note, but the precedent set here means a follow-up should reconsider.)
- **`@OptIn(ExperimentalMaterial3Api::class)` lives on the composable, not on the file.** `TopAppBar` is still flagged experimental in M3 ≥1.3; the annotation has to be on the composable that calls it (or one of its callers). Putting it on a `@file:OptIn(...)` would also work but reads as a project-wide pattern this codebase doesn't use — `ScannerScreen.kt` carries the annotation on the function, and #122 matches that shape.
- **Spinner-position pixel-asserts are brittle; visual-review against Figma carries the regression guard instead.** The added instrumented tests (`topBarTitle_renders`, `backButton_invokesOnBack`) intentionally don't assert spinner Y-coordinate or weighted-spacer ratios — those are visual-fidelity concerns that `@Preview` review against the Figma node covers without coupling the test to layout-tree internals. Same conclusion as #101 / #121 (no pixel-position or pixel-color snapshot tests).
