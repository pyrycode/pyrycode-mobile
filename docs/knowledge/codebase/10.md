# #10 — Seed channels with multi-session message histories

Fills the per-conversation `messages` storage shipped by #9 with realistic fixtures for the three seeded channels — each now has one historical session plus the existing current session, four `User`/`Assistant`-alternating messages per session. `observeMessages(<channel>)` therefore emits eight `MessageItem`s interleaved with one `SessionBoundary`; discussions stay at zero messages.

## Summary

- `seedChannel(...)` gained two defaulted parameters — `history: List<SeedSession>` and `currentMessages: List<SeedMessage>` — and now flattens both into the `ConversationRecord.messages` list (history sessions in list order, then current-session messages). Caller is responsible for chronological timestamps; the helper does not sort. `buildThreadItems` sorts at observe time and is the test-enforced authority.
- Two private companion-scoped fixture types: `SeedSession(sessionId, claudeSessionUuid, startedAt, endedAt, messages)` and `SeedMessage(role, content, timestamp)`. Authoring affordance only — the helper fills in `Message.id` (`"${sessionId}-msg-$index"`), `Message.sessionId`, and `isStreaming = false`. Tiny `ts(iso)` and `seedMsg(role, content, iso)` helpers keep the seed lists readable.
- Two sessions per channel (architect chose 2 of the AC-allowed 2–3 to keep the boundary-count test crisp). Historical session id is the existing current-session id plus a `-1` suffix (`seed-session-personal-1`, etc.); same convention for `claudeSessionUuid`. Historical `endedAt` is non-null and strictly precedes the current session's `startedAt`; current-session `endedAt` stays null.
- `Conversation.sessionHistory` for each channel is `[<historical-1>.sessionId, <current>.sessionId]`; `currentSessionId` is unchanged. `lastUsedAt` values are unchanged from #5 (the channel-list-ordering test reads those exact `Instant`s).
- `seedDiscussion` is untouched. `seed-discussion-a` and `seed-discussion-b` still ship with `messages = emptyList()` via `ConversationRecord`'s default.

## Patterns established

- **Authoring-time fixture DSL distinct from the runtime data model.** `SeedMessage` / `SeedSession` are *not* `Message` / `Session` with sensible defaults — they're a tighter shape (no `id`, no `sessionId`, no `isStreaming`) that the helper expands into the real model. Keeps the seed list call sites scannable as conversation transcripts, and means future fields on `Message` / `Session` only need a default in one place (the expander), not at every call site. New seed-data tickets in this repo should follow the same split.
- **`-N` suffix for derived historical session ids.** Historical sessions use `<current-session-id>-1`, leaving room for `-2` / `-3` if a later ticket extends the histories. Same convention applied to `claudeSessionUuid`. Don't switch to topic-based names (`seed-session-personal-lease`) — the numeric suffix is the contract anything else writing seed sessions should match.
- **Defaulted helper parameters keep callers terse.** `history` and `currentMessages` both default to `emptyList()`, so the `seedChannel` signature stays usable for a hypothetical empty-history channel without forcing every test or future seed to spell out two `emptyList()` arguments. This is the same shape #9 used for `initialMessages` on the constructor: optional extension surface via defaulted parameter, not a sibling helper.
- **Seed timestamps are chronologically dense, not artificially spaced.** Within each session, messages are 3–10 minutes apart; across sessions, the historical session ends days before the current session starts. The contiguous-per-session-in-time invariant is what makes `buildThreadItems`'s `sortedBy { it.timestamp }` produce exactly `sessions - 1` boundaries — drift here would silently produce extra (or fewer) boundaries. Future seed authors must keep `max(timestamp in session N) < min(timestamp in session N+1)`.

## Lessons learned

- **The `buildThreadItems` boundary-count invariant is `sessions - 1` only if message timestamps don't interleave across sessions.** A typo that put one current-session message at an earlier timestamp than the historical session's last message would, after sorting, split the run and emit *two* boundaries (one when entering the misplaced message, one when leaving). The boundary-count test catches this exactly. Worth remembering when later tickets edit seed timestamps or add a third session: keep per-session timestamp ranges disjoint, or expect the test to fail.
- **A `runtime` assertion in `seedChannel` would be redundant with the test surface.** Considered adding `require(history.all { it.endedAt <= sessionId-current.startedAt })` etc. — skipped. The four AC tests cover the same ground without coupling the helper to a debug-only check that nobody would re-run at production time. The architect's spec called this out and was right to skip; following the same instinct for similar "could be an assert" temptations in this repo.
- **Pre-existing tests written for an empty-message seed are load-bearing.** `observeMessages_knownConversationWithNoMessages_emitsEmpty` (#9) used `seed-channel-personal` specifically because that channel had no messages then. Touching the seed shape required redirecting the test to a discussion — not deleting it. The "known id, zero messages" branch (`buildThreadItems(emptyList())` → `emptyList()`) is semantically different from the "unknown id" early-return path and still needs coverage. Future tickets that further populate fixtures should likewise audit which tests pinned the pre-fixture state.
