# #86 — Persist `ThemeMode` preference + reactive `PyrycodeMobileTheme` wiring

Adds a `ThemeMode { SYSTEM, LIGHT, DARK }` enum backed by a string-typed DataStore key on `AppPreferences`, with a `Flow<ThemeMode>` getter and `suspend fun setThemeMode(...)` setter. The composition root collects the flow and resolves `darkTheme: Boolean` inside `@Composable` scope (`SYSTEM` keeps `isSystemInDarkTheme()`), then passes the boolean into the existing `PyrycodeMobileTheme(darkTheme: Boolean = …)` signature — no signature change. The Settings Theme row's hardcoded `"System"` subtitle is replaced by the flow value mapped through a file-private `ThemeMode.label()` to `"System default" / "Light" / "Dark"`; the row remains non-interactive in this slice. Picker dialog UX is the sibling ticket.

## Summary

- New `ThemeMode.kt` — single-line enum file in `de.pyryco.mobile.data.preferences`. No companion methods (label mapping lives at the Settings call site; resolution lives at the composition root).
- `AppPreferences.kt` — adds `themeMode: Flow<ThemeMode>` + `suspend fun setThemeMode(mode: ThemeMode)` mirroring the existing `pairedServerExists` shape. Storage key: `stringPreferencesKey("theme_mode")` holding `ThemeMode.name`. Default-on-absent and default-on-unparseable both fall through to `ThemeMode.SYSTEM` via `ThemeMode.entries.firstOrNull { it.name == stored } ?: ThemeMode.SYSTEM`.
- `MainActivity.kt` — `setContent` lambda hoists the `koinInject<AppPreferences>()` call above `PyrycodeMobileTheme(…)`, collects `themeMode` via `collectAsStateWithLifecycle(initialValue = ThemeMode.SYSTEM)`, resolves `darkTheme` with an exhaustive `when (themeMode)`, and passes it as `PyrycodeMobileTheme(darkTheme = darkTheme)`. The `composable(Routes.SETTINGS)` block independently `koinInject<AppPreferences>()`s + collects + passes `themeMode` into `SettingsScreen(themeMode = …)`. Yes, the flow is collected twice in the same activity — DataStore deduplicates, and drilling the value from the root through `PyryNavHost` into one `composable` would be more boilerplate than the duplicate collector is cost.
- `SettingsScreen.kt` — new `themeMode: ThemeMode = ThemeMode.SYSTEM` parameter (defaulted so existing two `@Preview`s + tests keep compiling untouched). The Appearance → Theme row's `supporting = "System"` becomes `supporting = themeMode.label()`. New file-private `ThemeMode.label()` extension returns the three contract strings the sibling picker ticket will reuse.
- `AppPreferencesTest.kt` — three new tests next to the existing two: `themeMode_defaultsToSystem`, `setThemeMode_roundTripsAllValues` (single function with an in-test loop over `ThemeMode.entries`), and `themeMode_unparseableStoredValue_fallsBackToSystem` (writes `"PURPLE"` to the underlying key directly via a re-declared `stringPreferencesKey("theme_mode")`, which references the same entry by name-equality without exposing the private key constant).

No changes to `Theme.kt`, `PyryApp.kt`, `AppModule.kt`, drawables, or `strings.xml`. The 18 `@Preview` call sites of `PyrycodeMobileTheme(…)` under `ui/` stay untouched because the function signature didn't move.

## Patterns established

- **Reactive `Flow<Enum>` preference, `name`-encoded, with a name-lookup fallback (not `valueOf`).** The pattern is `ThemeMode.entries.firstOrNull { it.name == stored } ?: ThemeMode.SYSTEM` rather than `runCatching { ThemeMode.valueOf(stored) }`. Both satisfy the AC, but `firstOrNull` has no throw-and-catch on the hot read path. Future enum-typed preferences (notification sound profile, default model, default effort — all enumerated in [Settings screen](../features/settings-screen.md)) should copy this exact shape: `stringPreferencesKey` storing `.name`, `entries.firstOrNull { it.name == stored } ?: <default>` on read, `mode.name` write. Ordinal-encoded keys are banned — see the spec's open-question resolution: enum reordering or removal silently corrupts ordinal-typed preferences across upgrades.
- **Caller computes `darkTheme: Boolean`; `PyrycodeMobileTheme`'s signature does not grow.** The composition root holds the `@Composable`-scope `isSystemInDarkTheme()` call (preserved on `SYSTEM`) and the exhaustive `when` over `ThemeMode`. Keeping `Theme.kt` unchanged means the 18 `@Preview` call sites under `ui/` (which pass an explicit `darkTheme = true/false` literal, never a `ThemeMode`) need no edits. The ticket body's "Either extend `PyrycodeMobileTheme`'s signature with `themeMode: ThemeMode`, or let the caller compute" Technical Note picks Option B here, and that's the precedent: theming primitives stay boolean-shaped at the API; preference-driven enums get resolved one layer up.
- **Activity-level collect of a global theme flow, sibling collects at consumer destinations.** The same `appPreferences.themeMode` flow is `collectAsStateWithLifecycle`-collected twice in `MainActivity.kt`: once at the top of `setContent` (drives `darkTheme`), once inside `composable(Routes.SETTINGS)` (drives the row subtitle). DataStore deduplicates the underlying read, both collectors are lifecycle-aware, and the `initialValue = ThemeMode.SYSTEM` matches the cold-flow default so no flash-of-wrong-theme can occur on first frame. The alternative — drilling the value through `PyryNavHost` to one destination — would route a globally-relevant value through navigation boilerplate. Future cross-cutting preferences (locale, dynamic-color toggle once Phase 3 wires it) should follow this shape: collect at the root for app-wide effects, collect again at any screen that displays the current value.
- **New `SettingsScreen` parameter lands *with* a default; the License row's "no default" precedent does not generalise.** `SettingsScreen.themeMode: ThemeMode = ThemeMode.SYSTEM` carries a default because the AC reserves the visible behavior change for the sibling picker ticket — the existing previews and tests should keep rendering "System default" without diff churn. Compare with `onOpenLicense: () -> Unit` in #91, which deliberately had no default because a silently-forgotten navigation lambda is worse than a compile error. The rule: navigation/event callbacks → no default; static display state with a meaningful safe value → default OK. The two-sentence justification is the same; the conclusion differs by parameter shape.
- **Direct-key write in the unparseable-fallback test.** `dataStore.edit { it[stringPreferencesKey("theme_mode")] = "PURPLE" }` re-declares the key by name inside the test rather than reaching for `internal` visibility on the production key constant. Two `stringPreferencesKey` instances with the same name resolve to the same on-disk entry, so the test exercises the production code path without widening the production API surface. Generalises to any future "stored value out of valid set" coverage on a key whose constant is `private`.

## Lessons learned

- **`collectAsStateWithLifecycle` is already on the classpath via `lifecycle-runtime-compose` — the [App preferences](../features/app-preferences.md) doc's "not in the version catalog today" caveat is stale.** That caveat predated #98 (or whichever ticket pulled in `androidx.lifecycle:lifecycle-runtime-compose`); by the time #86 landed the import resolved without a version-catalog edit. The feature doc will be corrected on the next pass.
- **Place the `koinInject<AppPreferences>()` call *above* `PyrycodeMobileTheme(…)`, not inside its content lambda.** The existing `Scaffold(...) { … val appPreferences = koinInject<AppPreferences>() … }` shape worked when only the inner `produceState` consumed it, but now `PyrycodeMobileTheme(darkTheme = darkTheme)` itself needs the resolved value — so the inject + collect both move up to the top of the `setContent` lambda. Re-declaring `koinInject` inside the `Scaffold` lambda would double-resolve the singleton; lifting it once at the `setContent` top closes that.
- **`initialValue = ThemeMode.SYSTEM` is correct precisely because `SYSTEM` is what the cold flow would emit first on a fresh DataStore.** Picking `ThemeMode.LIGHT` or `ThemeMode.DARK` as the initial value would cause a one-frame mis-themed flash on first launch (initial value renders, then the first real emit re-renders with the correct value). The architect's spec calls this out and the wiring honors it. Future flows backed by DataStore with a meaningful default should pass that same default as `initialValue` — never `null`, never a "loading" sentinel, when a same-shaped neutral value exists.
- **The "tests pass `themeMode = SYSTEM` implicitly" coverage is sufficient for this slice.** No new instrumented test was added for `SettingsScreen` because the row stays non-interactive — the existing `SettingsScreenTest` `setContent { SettingsScreen(onBack = {}, onOpenLicense = {}) }` blocks pick up the default and render `"System default"` without any test-file diff. The sibling picker ticket will be the one to add `ComposeTestRule` coverage for the subtitle reacting to a state change; this slice does not.
