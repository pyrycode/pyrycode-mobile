# #128 — feat(ui/thread): MessageBubble component (user + assistant variants)

First message-rendering primitive for the thread surface, slotting into the empty `LazyColumn(reverseLayout = true) { items(emptyList<Unit>()) { } }` body that landed in [#126](./126.md). Ships the component only; no `ThreadScreen` wiring, no `ThreadUiState` widening, no repository subscription — those are downstream slices (see § Links).

## Summary

- One new file: `app/src/main/java/de/pyryco/mobile/ui/conversations/components/MessageBubble.kt` (168 lines, including two paired Light/Dark previews). Zero edits to `ThreadScreen.kt`, zero edits to `ThreadViewModel.kt`, zero edits to `Message.kt` / `Role.kt`, zero `gradle/libs.versions.toml` entries, zero new strings.
- Public API is `@Composable fun MessageBubble(message: Message, modifier: Modifier = Modifier)`. Single `Message` parameter (not a pre-split `(text, isUser)` shape) because the eventual call site is `items(state.messages) { MessageBubble(it) }` — consumers don't pre-split the list by role.
- Role dispatch is a Kotlin `when` over `message.role` with three arms: `Role.User` → `UserMessageBubble`, `Role.Assistant` → `AssistantMessage`, **`Role.Tool` → `Unit` (renders nothing, intentional)**. No defensive `error("…")` throw, no debug placeholder `Text`. ThreadScreen will route `Role.Tool` to the tool-card composable from #131 before reaching this dispatch; the no-op arm is the contract until then. The `when` is exhaustive by compiler check, so a future fourth `Role` value forces an explicit decision at this site rather than slipping through a default arm.
- **User variant ships matched to Figma `16:23` pixels** — right-aligned `Surface(shape = UserBubbleShape, color = primaryContainer)` wrapping a `Text(style = bodyMedium, color = onPrimaryContainer)`. `widthIn(max = UserBubbleMaxWidth = 320.dp)` (4dp-grid-aligned approximation of Figma's `w-[330px]` on a 412dp canvas, ≈ 80% column width). Asymmetric `RoundedCornerShape(topStart = 20, topEnd = 20, bottomEnd = 6, bottomStart = 20)` — the tight bottom-right corner is the "tail" pointing toward the user side. Inner padding `14×12` (horizontal × vertical) lives on the `Text` inside the `Surface`, not on the `Surface` itself, so the bubble's painted background extends behind the padding band.
- **Assistant variant deliberately diverges from Figma** — flat `Box(Modifier.fillMaxWidth())` wrapping a left-aligned `Text(style = bodyMedium, color = onSurface, modifier = Modifier.fillMaxWidth())` with no `Surface`, no `Modifier.background(...)`, no shape, no inner padding. The Figma frame (`16:25`, `16:32`, etc.) shows a bubbled assistant on `surface-container-high`; the ticket body in both Context and AC3 explicitly overrides that to a Claude-style flat assistant. The architect spec calls this out as a deliberate design choice that supersedes the Figma — see "Figma-vs-AC conflict" note in `docs/specs/architecture/128-message-bubble-user-assistant-variants.md`. If a future ticket reconciles the Figma to match (or vice versa), that's its own design call, not a #128 follow-up.
- **AC4 spacing tokens — named file-private `val`s, not a `LocalSpacing` provider.** Five constants at the top of `MessageBubble.kt` (`MessageRowVerticalSpacing = 12.dp`, `UserBubbleMaxWidth = 320.dp`, `UserBubbleShape = RoundedCornerShape(...)`, `BubbleHorizontalPadding = 14.dp`, `BubbleVerticalPadding = 12.dp`). M3 doesn't ship a spacing scale, the codebase has no `LocalSpacing` ambient, and the prior #126 spec explicitly avoided introducing infra for placeholder work — naming the values inline satisfies the AC's spirit ("don't sprinkle arbitrary `.dp` literals") without committing to a provider that doesn't yet exist. Other files in `ui/conversations/components/` still use bare `.dp` literals (`ArchiveRow.kt:43`, `ConversationRow.kt:53`); **don't back-fill** the named-constants pattern into those files in this ticket.
- **Vertical rhythm lives on the component, not the consumer.** Each variant's outer `Row` / `Box` carries `Modifier.padding(bottom = MessageRowVerticalSpacing)`. Picked over the alternative (no padding inside `MessageBubble`; the eventual `LazyColumn` uses `verticalArrangement = Arrangement.spacedBy(MessageRowVerticalSpacing)`) because it keeps AC4 satisfied at the component level today, when no `LazyColumn` consumer is part of this ticket. The bottom-only direction (not symmetric `padding(vertical = 6.dp)`) avoids accumulating an extra leading gap above the topmost row; the trailing 12dp below the last message is acceptable visual padding before the (eventual) composer / status row.
- **Stability and recomposition: free.** `Message` is a data class with all stable fields (`String`, `Role` enum, `Instant`, `Boolean`); Compose's stability inference picks it up with no `@Stable` / `@Immutable` annotation. Identity-equal instances skip, structurally-equal instances skip via `equals`. The composable is pure (no `remember`, no `LaunchedEffect`, no coroutines), so reads of `MaterialTheme.colorScheme.*` and `MaterialTheme.typography.*` are the only recomposition triggers beyond `message` itself.
- **Previews pair Light + Dark with `uiMode = Configuration.UI_MODE_NIGHT_YES`**, both `widthDp = 412`. Both wrap a shared `MessageBubblePreviewSequence()` (file-private, four messages alternating User → Assistant → User → Assistant — the last assistant message is a long multi-sentence string to exercise multi-line wrap). The sequence is wrapped in `Surface { Column(Modifier.padding(horizontal = 16.dp)) { … } }` so the assistant variant's transparent background renders against the theme's surface color (matching the eventual `LazyColumn` runtime), and the user bubble has the 16dp column inset to pin against instead of the raw preview viewport edge. The `previewMessage(role, content)` helper fixes `id = "preview-${role.name}"`, `sessionId = "preview-session"`, `timestamp = Clock.System.now()`, `isStreaming = false`.
- **`Message.isStreaming` and `Message.timestamp` are ignored.** Streaming caret + animation lands in #132 without changing the public API (the role-arm composables internalise the change). Per-bubble or per-session-delimiter timestamp rendering is a separate concern handled elsewhere — this ticket renders the bubble shell only.
- **No tests added.** No unit tests (the composable is pure rendering with no logic beyond a `when` over `Role` — branch coverage is visible in the previews; building Robolectric + Compose-host or `ComposeTestRule` infra for a 100-LOC component is over-scope). No instrumented Compose UI tests (codebase has no `androidTest` infrastructure beyond existing fixtures — same precedent as #126, see `docs/knowledge/codebase/126.md:38`). Visual verification is via the two `@Preview`s rendering in Android Studio plus `./gradlew assembleDebug` + `./gradlew lint` passing. `./gradlew test` runs the existing suite green; no new tests.

## Patterns established

- **Named file-private spacing/shape constants are the project's first stand-in for an M3 spacing scale.** When a component needs more than one `.dp` literal whose name carries design intent (`UserBubbleMaxWidth`, `MessageRowVerticalSpacing`, `BubbleHorizontalPadding`), declare them as `private val` at the top of the file. Don't introduce a `LocalSpacing` `CompositionLocal` provider — the codebase has no ambient design-token plumbing and adding one to satisfy a single component is infrastructure for hypothetical future requirements. Other files keep their bare `.dp` literals until they grow a second naming pressure of their own; **do not back-fill** the named-constants pattern into `ArchiveRow.kt`, `ConversationRow.kt`, or the other existing component files. The first multi-component spacing pressure (the next ticket that needs the same value at two call sites in the same package, e.g. `MessageRowVerticalSpacing` shared with the session-boundary delimiter in #135) is when an `internal val` peer-file extraction becomes the right move — the same shape `formatRelativeTime`'s `RelativeTime.kt` peer file takes for the `ConversationRow` / `DiscussionPreviewRow` pair (see [`../features/discussion-preview-row.md`](../features/discussion-preview-row.md) § Shared `RelativeTime` helper).
- **Exhaustive `when` over `Role` with `-> Unit` arms for unimplemented roles, not defensive throws.** When a composable dispatches over an enum and one arm is owned by a not-yet-landed ticket (here: `Role.Tool` → #131), the right arm is `Role.Tool -> Unit`. No `error("Tool role not yet handled")` (would crash exactly the path the framework contract — ThreadScreen will route `Role.Tool` to a separate composable before reaching here — guarantees never happens), no debug placeholder `Text("[tool]")` (would leak into production builds where the contract isn't yet in force). Pair this with the absence of a default `else ->` arm: that keeps the Kotlin compiler enforcing exhaustiveness so a future fourth `Role` value (e.g. `Role.System`) is flagged at the dispatch site as a hard compile-time decision rather than silently absorbed. Pattern applies to any future `when (role)` / `when (kind)` dispatcher in `ui/conversations/components/` where a sibling ticket owns one of the arms.
- **Ship the component, not the wiring, when a single ticket's AC is component-shaped.** #128's AC list is pure component-shape (renders, variants, max-width, spacing, previews); it does not ask for `ThreadScreen` consumption. The architect spec calls out "scope contract: component only, no wiring" and explicitly forbids editing `ThreadScreen.kt`. Pattern applies to the rest of the `feat(ui/thread):` slice — #133 input bar, #134 connection banner, #135 session-boundary delimiter, #137/#138 empty states, #145 status row — each lands its composable in `ui/conversations/thread/` or `ui/conversations/components/`, and a separate "wire the X into ThreadScreen" ticket (or the next ticket that has data to drive it) does the consumption. Avoids cross-ticket merge churn on `ThreadScreen.kt` and keeps each PR diff bounded to the surface its AC describes.
- **Figma-vs-AC conflict resolution: AC body is authoritative; reconciliation is a downstream design call, not this PR's scope.** When the linked Figma frame and the ticket body's AC disagree (here: bubbled-assistant in Figma `16:25` vs flat-assistant in AC3), the AC wins and the architect spec records the precedence note ("Figma will be reconciled later — no follow-up ticket exists for the reconciliation yet"). If a code reviewer pushes back on the AC-divergent choice, the project rule is `needs-rework:po` not in-PR debate. Pattern applies to any future ticket where Figma drift > AC drift in the ticket lifecycle (Figma evolves continuously; ACs freeze at ticket creation). The PR author is not the right person to renegotiate the design — escalate.

## Lessons learned

- **AC4 ("M3 spacing tokens, not raw `.dp` literals") doesn't require a `LocalSpacing` provider — file-private named `val`s satisfy it.** The literal reading of the AC is hostile (M3 doesn't expose a spacing scale; there's no `MaterialTheme.spacing.small` to reach for), but the intent is "name the values so the design system relationship is legible". The architect spec resolved this by recommending named `val`s at file scope, mirroring how the project handles other one-off design tokens. No `Dp` extension functions, no `CompositionLocal`, no theme extension class. Worth pinning so the next ticket with the same AC text doesn't drift into building infrastructure: the answer is `private val Foo = 12.dp` until a second component in the same package needs the same value, then `internal val Foo` in a peer file.
- **`Surface(shape = …, color = …) { Text(modifier = Modifier.padding(...), ...) }` keeps the painted background under the padding band.** The reverse — `Surface(modifier = Modifier.padding(...), shape = …, color = …) { Text(...) }` — paints the bubble background inside the padded area, leaving the rounded corners floating in an unpainted gap. Caught in preview rendering; correct shape is padding on the inner `Text`, not on the outer `Surface`. Applies to any future M3 `Surface`-wrapped content where the surface's background is the visual affordance (here: the user bubble's `primaryContainer` fill).
- **`Box(Modifier.fillMaxWidth())` containing a `Text(modifier = Modifier.fillMaxWidth(), …)` is the right shape for "left-aligned full-width text" — `Row` works too but `Box` matches intent better.** Considered `Row(Modifier.fillMaxWidth())` for the assistant variant (symmetry with the user variant's `Row(Modifier.fillMaxWidth(), horizontalArrangement = Arrangement.End)`) but `Box` reads more honestly: there's one child, alignment is the default (top-start), there's no horizontal arrangement to negotiate. Both compile and render identically; the choice is readability.
- **The `previewMessage(role, content)` factory keeps preview noise contained.** Inlining the full `Message(id = ..., sessionId = ..., role = ..., content = ..., timestamp = ..., isStreaming = ...)` constructor at four call sites in `MessageBubblePreviewSequence` would dwarf the role/content signal with field repetition. The file-private factory takes only the two varying parameters and fixes the rest. Pattern applies to any future preview that exercises a data class with > 4 fields where most are fixed across variants — extract a `previewX(...)` helper in the same file rather than copy-pasting full constructors.
- **`kotlinx.datetime.Clock.System.now()` in `@Preview` composables is fine — previews aren't tested for determinism, and the timestamp isn't consumed by this ticket's render path.** Considered using a fixed `Instant.fromEpochMilliseconds(0)` for preview reproducibility, but the value never reaches `Text` or any visual surface in #128 (timestamps are rendered elsewhere, deferred). `Clock.System.now()` is the lowest-friction option and matches what `DiscussionPreviewRow.kt`'s previews do (`now = Clock.System.now()` then subtract `12.minutes`). When #132's streaming caret or a later timestamp-rendering ticket lands, the previews may need a fixed `Instant` to make the caret/timestamp visually predictable — that's a future-ticket concern.
