# #5 — Seed `FakeConversationRepository` with three channels

Phase 0 UI work needs fixture data; this ticket replaces the empty initial state delivered by #4 with three pre-populated channels and pushes the descending-by-`lastUsedAt` sort into the repository so every consumer gets it for free.

## Summary

- Three channel seeds (`Pyrycode Mobile`, `Joi Pilates`, `Personal`) populate `FakeConversationRepository`'s initial state, each with a distinct `cwd`, stable `currentSessionId`, and fixed `Instant.parse(...)` `lastUsedAt`. Discussion-tier seeds are explicitly out of scope and land in #6 alongside this list.
- Built via a private `companion object` (`SEED_RECORDS` / `buildSeedRecords()` / `seedChannel(...)`) so the data is evaluated once at class load. Seeds are inserted in a deliberately non-sorted source order so the AC test can't pass by accident.
- `observeConversations` gained a `.sortedByDescending { it.lastUsedAt }` projection inside the existing `state.map { ... }` block. The sort is filter-agnostic — both `Channels` and `Discussions` want most-recently-used first.

## Patterns established

- **Seed data lives in a `companion object`, not in the constructor.** Evaluated once at class load, signals intent ("fixture, not per-instance state"), and keeps the seed builder out of the class body where the mutators live. New fakes that need fixture data should follow the same `SEED_…` / `build…()` / per-row helper layout.
- **Deterministic seed identifiers, not `UUID.randomUUID()` / `Clock.System.now()`.** Seed ids carry a `seed-` prefix (`seed-channel-personal`, `seed-session-personal`, `seed-claude-personal`) so they're obviously fixture data when seen in logs. `lastUsedAt` and `startedAt` use `Instant.parse(...)` literals. Rationale: identical runs produce identical state — the AC test can compare names in a fixed order without flakiness, and the swap to the Phase 4 backend won't fight a moving timestamp.
- **Source order of seeds ≠ sorted order.** The three seeds are inserted Personal → Pyrycode Mobile → Joi Pilates, which is neither alphabetical nor `lastUsedAt`-descending. Future seed sets in this repo should keep that property so the sort projection is actually exercised.
- **Sort projection lives inside the repository, not the ViewModel.** The previous feature doc said ViewModels would sort once seeds existed; #5 walked that back. With one canonical sort order shared across every filter, hoisting it into `observeConversations` removes duplication from every future ViewModel and keeps the contract's "what consumers see" closer to the UI's expectations. If a future consumer needs an alternative order, it can re-sort the emission — but no consumer currently does.
- **Scope-narrowing pattern for tests against seeded state.** When initial state stops being empty, tests that previously did `assertEquals(listOf(created), emissions)` shift to `emissions.first { it.id == created.id }` (or `any { … }` / `none { … }`). For size assertions, the seed count becomes the new baseline (e.g. "channels go from 3 to 4"). Apply this whenever a future ticket adds seeds beneath an existing test suite.

## Lessons learned

- **Companion object can see the enclosing class's `private` nested types.** `ConversationRecord` stays `private` inside `FakeConversationRepository`; the companion's `seedChannel(...)` constructs it without any visibility bump. Worth knowing before reaching for `internal` to make a nested type accessible to its own companion — it's already accessible.
- **`Instant.parse(...)` runs at class load; mistyped literals fail fast.** No try/catch needed for static seed timestamps; the JVM will throw at first instance construction (i.e. before any test asserts). This is the right shape for fixture data and a reason not to wrap the call sites in defensive parsing.
- **The `Discussions` empty-list assertion is what #6 will flip.** Keeping it explicit (rather than collapsing it into the `Channels` assertion) makes the seam #6 will edit obvious from the test name and body. Same logic for the `Channels size == 3` baseline appearing in scope-narrowed tests: future seed additions should bump those numbers in one place per test, not bury the count inside a helper.
