# #167 — Welcome logo vector refresh (Figma 6:32 / mark 80:2)

Full rewrite of `ic_pyry_logo.xml` to track the post-#150 Figma re-author. The mark moved from node `9:2` (104 × 104, six stroked arm groups + a centre-cursor rectangle) to node `80:2` "pyrycode-mark 1" (92 × 104, **one filled continuous path** encoding the same six-pointed snowflake-with-cursor silhouette). The Android drawable mirrors the source 1:1: single `<path>` with `fillColor="#FFFFFF"`, viewport `91.002 × 103.812` at intrinsic `92dp × 104dp`. `WelcomeScreen.kt`'s `Icon` modifier switched from `Modifier.size(104.dp)` to `Modifier.size(width = 92.dp, height = 104.dp)` so the drawable's new 92:104 aspect ratio doesn't get inner-padded inside a 104×104 box (which would push the logo's left edge 6.5dp inboard of the Figma-locked 32dp column padding).

## Summary

- **Drawable** (`ic_pyry_logo.xml`): from `92` lines (six `<group>` arm wrappers + centre rectangle path + viewport `104×104`) down to `10` lines (single `<path>` + viewport `91.002 × 103.812` at `92dp × 104dp`). `pathData` is the verbatim `d=` attribute from the Figma SVG export of inner Group `80:3`. Path is one continuous filled outline (no stroked sub-paths, no `<group>` wrappers, no rotation symmetry to exploit) — the new Figma source is authored as a single filled glyph, not as the rotation-symmetric stroke composition #150 patched.
- **WelcomeScreen** (`WelcomeScreen.kt:75`): `Modifier.size(104.dp)` → `Modifier.size(width = 92.dp, height = 104.dp)`. Single-line edit. Without it, Compose `Icon`'s default `ContentScale.Fit` would render the new 92:104 drawable as ~91×104dp centred inside a 104×104 box — visually slightly inset but, worse, pushing the logo's left edge inward from the column's 32dp horizontal padding, breaking the Figma-locked left-edge alignment with the title text underneath.
- **`#FFFFFF` tinting-bed preserved** (the `Icon(tint = MaterialTheme.colorScheme.primary)` pathway is unchanged). The Figma SVG export's `fill="var(--fill-0, #7AB8E8)"` fallback was discarded — `Icon`'s `BlendMode.SrcIn` replaces per-pixel colour at composition time, so the XML fill is invisible. Same idiom as #150 / #57 and `ic_qr_scan_frame.xml`. No Kotlin `Color(0xFF…)` literal introduced (AC3 satisfied).
- **`WelcomeScreenTest` unchanged** — the four-method instrumented test matches on visible text strings only; the logo carries `contentDescription = null` and no matcher touches it. AC4 satisfied by construction with zero test edits.
- **`ChannelListScreen` not touched.** The drawable's second consumer (28dp `navigationIcon`) inherits the aspect-ratio change — under default `ContentScale.Fit` it now renders ~24.5×28dp inside its 28×28dp box, still aspect-correct and tinted correctly, just slightly narrower. Intentionally deferred (out of scope, ticket title scopes to Welcome). Flagged in the PR description for PO review; follow-up would set `Modifier.size(width = 25.dp, height = 28.dp)` at `ChannelListScreen.kt:83` if visual weight needs to be restored.

## Patterns established

- **When a Figma source switches encoding shape — stroke composition ↔ filled outline, multi-element ↔ single-path — rewrite the Android drawable to mirror the new shape rather than back-deriving the old shape from the new one.** The #150 architect spec's six-arm pivot/rotation table cannot be reused here because Figma now serves the glyph as one filled outline; reconstructing six rotated stroke arms by hand from the filled outline would be geometrically lossy and would diverge from the design source on the next edit. Direct 1:1 translation is both smaller (single-path drawable) and more durable.
- **Inline verbatim asset payloads (path data, gradient stops, palette tokens) into the architect spec rather than instructing the developer to re-fetch.** The spec body carries the full `pathData` string lifted from a known-good Figma SVG export. Same shape as the inline-design-tokens rule from #119 and prior mobile tickets — when a deterministic asset value belongs to the spec, the spec carries the value, not a "fetch it yourself" instruction. Reduces re-fetch tool-call cost and removes the class of bug where developer and architect resolve different Figma snapshots.
- **Match `Icon` bounding-box modifier to drawable intrinsic aspect ratio.** Compose `Icon`'s default `ContentScale.Fit` will inner-pad a non-square drawable inside a square `Modifier.size(N.dp)` box without warning — visually centred and aspect-correct, but the resulting render no longer respects the parent layout's edge alignment. When a drawable's intrinsic ratio is non-1:1, size the Icon at `Modifier.size(width = X.dp, height = Y.dp)` matching the intrinsic dimensions. Project precedent for any future non-square brand-mark drawable.

## Lessons learned

- **Figma sometimes serves Vector nodes as `image/svg+xml` from the design-context endpoint's asset URL, not as structured node JSON.** `mcp__plugin_figma_figma__get_design_context` on a single-Vector node returns a flattened `<img>` reference rather than a `Frame > Group > Vector` tree. The architect downloaded the SVG directly via `curl` against the asset URL (Content-Type confirmed `image/svg+xml`) and extracted the `<path d="…">` content from the raw markup. Future Figma-anchored vector tickets where the design-context payload looks "thin" should check the asset URL's Content-Type rather than assume the structured payload is the only path.
- **A drawable's intrinsic aspect-ratio change ripples to every `Icon(painter = painterResource(...))` consumer.** `ic_pyry_logo` has two call sites (`WelcomeScreen.kt:73` at 104dp, `ChannelListScreen.kt:83` at 28dp). Changing the drawable from 1:1 to 92:104 silently re-renders the nav-bar consumer at ~24.5×28dp instead of 28×28dp because Compose `Icon`'s default `ContentScale.Fit` preserves aspect ratio inside the bounding box. The change isn't broken — it's still aspect-correct and tinted — but it's a visual delta that wasn't part of the ticket scope. When touching a multi-consumer drawable, grep for `painterResource(R.drawable.<id>)` before merging and decide per-call-site whether the side-effect is acceptable; flag in the PR description so reviewers can decide whether a follow-up is needed.
- **Figma SVG `fill="var(--fill-0, #7AB8E8)"` fallbacks don't necessarily match the runtime variable binding.** The exported fallback `#7AB8E8` matches neither `primaryLight = #32628D` nor `primaryDark = #9DCBFC` from `Color.kt`. Possible causes: the logo's design-system fill is bound to a non-`Schemes/Primary` variable, the exporter rasterised a stale value, or the design intentionally shifted. **For tintable drawables, this is irrelevant** — the `Icon(tint = primary)` pathway overrides the XML fill via `BlendMode.SrcIn`, so the SVG fallback is invisible to the user. Don't propagate SVG fallback colours into the Android XML "for safety"; the `#FFFFFF` tinting-bed convention is the project idiom and discarding the fallback is correct. If reviewer-side Figma comparison reveals a tint mismatch in dark mode (Figma renders closer to `#7AB8E8` than Compose's `#9DCBFC`), that's a separate ticket about the `Icon(tint = …)` binding, not about the drawable.
- **The "patch arm `pathData` only, keep groups + rotations intact" approach from #150 doesn't generalise — it was a property of #150's Figma source, not a project-wide vector-edit pattern.** When the Figma re-author changes the encoding shape (stroke composition → filled glyph), the per-group surgical patch becomes inapplicable and the only honest translation is a full rewrite of the drawable. #150's `replace_all`-across-six-identical-strings pattern documented in [`codebase/150.md`](150.md#patterns-established) is specifically a pattern for *rotation-symmetric composite drawables*, not for vector drawables in general.
