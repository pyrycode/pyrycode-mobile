# #6 — Seed `FakeConversationRepository` with two discussions

Mirrors #5 on the unpromoted tier: appends two auto-named discussions to the existing seed list so the Phase 0 "Recent discussions" pill has fixture data without each call site invoking `createDiscussion` in setup. Purely additive on the production side; the rest is count bumps on tests #5 left flagged.

## Summary

- Two discussion seeds (`seed-discussion-a`, `seed-discussion-b`) join the existing `seeds` list in `buildSeedRecords()`. Both carry `name = null` (discussions are auto-named at render time), `isPromoted = false`, and the shared scratch sentinel `cwd = "~/.pyrycode/scratch"`.
- A new private `seedDiscussion(...)` helper sits beside `seedChannel(...)` in the companion. Two tiers, two helpers — chosen over parameterising `seedChannel` with `name: String?` / `isPromoted: Boolean` to keep call-site readability.
- Seeds are inserted in source order B → A, with A's `lastUsedAt` newer. `sortedByDescending` reorders them — same anti-coincidence pattern #5 established.
- Channels block untouched; `observeConversations(Channels)` still emits the same three seeds in the same order.

## Patterns established

- **Tier-per-helper for seed fixtures.** Channels get `seedChannel(...)`; discussions get `seedDiscussion(...)`. Each helper hard-codes the tier-defining fields (`isPromoted`, `name` nullability) so call sites carry no `(true, "Personal")` / `(false, null)` correlation. If a third tier ever appears in this repo, add a third helper rather than collapsing the first two into a parameterised one. The verb at the call site is the documentation.
- **Shared sentinel for shared cwd.** Both discussion seeds point at the literal string `"~/.pyrycode/scratch"`. This matches the conversations-model contract that discussions share scratch by definition; the string is a fixture marker, not a path Phase 0 ever opens. Future seed sets for tiers with a shared resource should follow the same "one literal, reused" approach instead of fabricating distinct paths that would imply isolation the model doesn't actually offer.

## Lessons learned

- **Source order matters because it doesn't.** Inserting B before A (despite A being newer) is the cheapest way to prove the sort projection is still doing work after this ticket — no extra assertion needed, the descending-timestamp test would fail if the projection were ever removed. Worth replicating for any future additive seed change: insert in the order that would fail a passively-sorted reader.
- **Count-bump tests are easier to write right than to read later.** Four tests had to move from `(3, 0)` to `(3, 2)` baselines. The architect's table in the spec pinned exact line numbers and old/new values; following it verbatim was faster than re-deriving the deltas. If a future seed-add lands without that table, copy the cross-reference style into the spec rather than chasing assertions one at a time.
