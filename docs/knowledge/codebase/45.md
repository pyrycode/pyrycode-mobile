# #45 — `ChannelListViewModel` + `UiState` (data path)

Establishes the channel-list data path before any UI lands. Adds a sealed `ChannelListUiState` (`Loading` / `Empty` / `Loaded(channels)` / `Error(message)`) and a `ChannelListViewModel` that converts `observeConversations(Channels)` into a hot `StateFlow` via `stateIn(WhileSubscribed(5_000))`. Also wires the first two Koin bindings — the pending `FakeConversationRepository` binding (deferred since #4) and the first `viewModel { }` line — closing the `// Bindings added by downstream tickets:` TODO block in `AppModule.kt`.

## Patterns established

- **Top-level `XxxUiState` sealed interface, one per screen, not nested.** `ChannelListUiState` is a sibling of `ChannelListViewModel` in the same file, not `ChannelListViewModel.UiState`. Composables import variants directly (`ChannelListUiState.Loaded`); nesting would force `ChannelListViewModel.UiState.Loaded` at every call site. Each future screen gets its own peer `<Screen>UiState` — the screen-name prefix carries the disambiguation. Same shape for `Event` types when they arrive.
- **`data object` for stateless variants, `data class` for payload variants.** Kotlin 2.0+ `data object` gives `toString` / `equals` for free and preserves singleton identity (`Loading === Loading`). `data class()` with no params would create distinct instances that compare unequal — wrong shape for state markers.
- **`stateIn(viewModelScope, WhileSubscribed(5_000), Loading)` is the project default for cold→hot ViewModel state.** The 5-second grace is the canonical Android cadence: survives configuration changes (rotation re-subscribes within ~ms) without re-collecting, but truly disposes when the user navigates away for >5s. `Lazily` leaks the upstream after the screen exits; `Eagerly` collects before the screen subscribes. The timeout lives as a `private companion object const` (`STOP_TIMEOUT_MILLIS`), not a magic literal — promote to a shared `UiDefaults` when a second VM needs the same value, not preemptively.
- **`.catch { }` between `.map { }` and `.stateIn(...)`.** `stateIn` is a terminal operator returning `StateFlow`, not a chainable `Flow` operator — `.catch` after it doesn't compile. Placing `catch` after the `map` lets one block intercept both upstream throwables and (hypothetical) map exceptions, surfacing them as a terminal `Error` state in the `StateFlow`.
- **Error message: `e.message` verbatim when non-blank, else literal fallback.** Use `if (raw.isNullOrBlank()) "Failed to load channels." else raw`. Don't fall through to `e.toString()` or `e::class.simpleName` — exception class names are implementation detail and leak through to user-facing UI badly (`java.net.UnknownHostException: example.com` is not what an end user should read).
- **VM constructor takes the interface, not the impl.** `ChannelListViewModel(repository: ConversationRepository)`. Koin's `bind ConversationRepository::class` resolves `get()` to the bound implementation. Phase 4's remote `ConversationRepositoryImpl` drops in behind the same bind line with zero VM-side change.
- **Repository bindings before VM bindings inside `module { }`.** Declaration order is irrelevant for resolution (Koin builds a graph) but matters for readability: singletons first (`DataStore` → `AppPreferences` → `FakeConversationRepository`), then `viewModel { }` lines. Matches the shape of canonical Koin examples and keeps the eye scanning top-to-bottom from "infrastructure" to "screens".
- **Per-test inline anonymous `ConversationRepository` stubs, not a shared `TestConversationRepository` base class.** Two helper functions at the bottom of the test file (`stubRepo`, `erroringRepo`), each ~10 lines of `object : ConversationRepository { ... TODO("not used") }`. No indirection, no separate test-fixtures file, no `mockk`. If a future test calls one of the `TODO` paths, the `NotImplementedError` failure points straight back to the same file.
- **`Dispatchers.setMain(UnconfinedTestDispatcher())` in `@Before`, `resetMain()` in `@After`.** Class-level `@OptIn(ExperimentalCoroutinesApi::class)` (the `setMain` / `UnconfinedTestDispatcher` APIs are flagged experimental). `UnconfinedTestDispatcher` propagates emissions eagerly so test bodies can read `vm.state.value` after `source.emit(...)` without explicit `runCurrent()`. `StandardTestDispatcher` would force `runCurrent()` between every line.
- **`MutableSharedFlow(replay = 0)` for emission-timing control in VM tests.** A `MutableStateFlow` forces an initial value at construction, which would obliterate the `Loading` observation window. `replay = 0` + an explicit `source.emit(...)` is the right primitive when the test needs to assert "Loading is visible before the source emits".

## Lessons learned

- **`WhileSubscribed(5_000)` only starts the upstream once `state` has a subscriber.** Without `launch { vm.state.collect { } }` in the test body, `source.emit(...)` has no listener and `vm.state.value` stays on `Loading` forever. The `initialState_isLoading` test deliberately *doesn't* subscribe — it relies on `stateIn`'s `initialValue` being immediately visible on `.value`, which is the one observation that doesn't need a hot collector. Every other test launches a no-op collector to keep the upstream hot.
- **`UnconfinedTestDispatcher` alone wasn't enough to make emissions observable — needed `advanceUntilIdle()` between `emit` and `assert`.** The spec called for "read a value immediately after emit", but in practice the `MutableSharedFlow` → `.map` → `.catch` → `.stateIn` pipeline has enough hops that the eager-but-not-synchronous propagation of `UnconfinedTestDispatcher` leaves a tick on the queue. `advanceUntilIdle()` between the launched collector + the `emit`, and again between the `emit` + the `assertEquals`, gives deterministic green tests. Pinning this for the next ViewModel-test ticket: if a test under `UnconfinedTestDispatcher` flakes between "set value, read value", reach for `advanceUntilIdle()` before reaching for `StandardTestDispatcher`.
- **Added a fifth test (`error_messageIsNonBlank_whenExceptionMessageIsNull`) beyond the spec's four.** The AC explicitly requires "non-blank message" for the Error state. The four-test minimum exercises the *happy* `e.message = "network down"` path; the null-message branch is what the AC's "non-blank" wording was written to defend. Tested via `RuntimeException(null as String?)` → upstream throws → `catch` evaluates `raw.isNullOrBlank()` → emits the fallback string. Worth the extra `@Test`; the fallback-string branch is otherwise untested.
- **Settled on `org.koin.core.module.dsl.viewModel` for the DSL import, not the spec's `org.koin.androidx.viewmodel.dsl.viewModel`.** Both resolve against the current `koin-bom = 4.0.4`, but the `androidx.viewmodel.dsl` form is in the process of being deprecated in favor of the `core.module.dsl` form — `core.module.dsl` is the multiplatform-safe path the Koin docs are migrating toward. No semantic difference today; future-proofs against the deprecation warning landing on a future BOM bump. Keep this import shape for subsequent `viewModel { }` lines.
- **`lifecycle-viewmodel-ktx` is needed explicitly at lifecycle 2.6.1.** `koin-androidx-compose` brings in `lifecycle-viewmodel-compose` transitively, which depends on plain `lifecycle-viewmodel` (no `-ktx`). `viewModelScope` lives in the `-ktx` artifact at this version. Compilation fails with `Unresolved reference: viewModelScope` without the explicit dependency. If a future lifecycle bump (≥2.9) merges `viewModelScope` into the base artifact, this `-ktx` line can be dropped — but that's a lifecycle-upgrade ticket's job, not opportunistic cleanup.
- **`map<List<Conversation>, ChannelListUiState> { ... }` needs explicit type parameters.** Without them, Kotlin's inference picks `Flow<ChannelListUiState.Empty>` (or `Loaded`, depending on which branch the compiler hits first) when the `if (channels.isEmpty()) Empty else Loaded(channels)` expression isn't widened to the sealed parent. The explicit `<List<Conversation>, ChannelListUiState>` lets the smart-cast resolve to the sealed interface so `stateIn(..., initialValue = Loading)` (also a `ChannelListUiState`) is assignment-compatible. An alternative is to annotate the `if` branches via `Loaded(channels) as ChannelListUiState` — explicit `map` type params are tidier.
