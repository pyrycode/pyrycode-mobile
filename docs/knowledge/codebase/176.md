# #176 — Archive · 2-tab segmented header with live counts (Figma 18:2)

The archive surface flips from "discussions-only `LazyColumn`" to a `SecondaryTabRow`-backed 2-tab header (Channels / Discussions) over the same shared [`ArchivedDiscussionRow`](../features/archived-discussions-screen.md). The VM now emits both archived tiers partitioned by `isPromoted` plus a `selectedTab` slot, so badge counts (`Channels (3)` / `Discussions (8)`) stay live without re-collection on tab switch. Body, restore gesture, and `unarchive(id)` wiring are unchanged — the archive-specific row + inline restore + snackbar is the follow-up child. Split from #125.

## Summary

- **Title flips from "Archived discussions" → "Archived".** `R.string.archived_discussions_title` removed and replaced by `R.string.archived_title` ("Archived" — Figma 18:2 single-word literal). Back-arrow `IconButton` + `BackTapped` event unchanged.
- **VM state shape rewritten.** Sealed `ArchivedDiscussionsUiState` loses `Empty` and gains a 3-field `Loaded(channels, discussions, selectedTab)`. The pre-#176 shape was `Loading | Empty | Loaded(discussions) | Error(message)`; the post-#176 shape is `Loading | Loaded(channels, discussions, selectedTab) | Error(message)`. AC §5's "tab header stays visible while one tier is empty" requires `Loaded` to render when either list is empty — the old top-level `Empty` couldn't carry the other tier's count, so per-tab emptiness is computed inside the body from `Loaded.<list>.isEmpty()`. New top-level `enum class ArchiveTab { Channels, Discussions }` declared next to the sealed interface in the same file.
- **Tab selection lives in the VM, not the composable.** A private `selectedTab = MutableStateFlow(ArchiveTab.Discussions)` is `combine`d with `repository.observeConversations(ConversationFilter.Archived)` and `stateIn`'d on `viewModelScope` (same `WhileSubscribed(5_000L)` grace as before). `TabSelected(tab)` event handler is `selectedTab.value = event.tab` — non-suspending, no `viewModelScope.launch`. Default tab is **Discussions** per AC §3 (preserves the pre-#176 landing surface; Figma 18:2 illustrates Channels-active for layout reference only, AC is authoritative).
- **`Discussions` default survives configuration changes; resets on a fresh nav back in.** `selectedTab` is a VM field, so it survives recomposition / rotation. Leaving the screen lets `WhileSubscribed(5_000L)` keep the VM warm for 5s; popping the back-stack within that grace window restores the previously-selected tab, popping after re-creates the VM with `Discussions` again. Acceptable per the spec's "State + concurrency model" — no `rememberSaveable` chosen so tests can assert tab-selection deterministically.
- **Body re-uses the existing `ArchivedDiscussionRow` unchanged for both tabs.** No new row composable, no archive-specific muted treatment beyond the pre-existing `Modifier.alpha(0.65f)`, no `SwipeToDismissBox`. Long-press `DropdownMenu` "Restore" stays as the sole gesture, dispatches `RestoreRequested(conversation.id)` for both archived channels and archived discussions. The archive-specific row + inline restore + snackbar is the follow-up child of #125.
- **`ArchivedDiscussionsScreen` keeps its name and signature.** No rename to `ArchiveScreen`; the spec called rename optional ("pick whichever minimises diff") and the developer chose keep-name because rename would have churned `MainActivity`, the Koin binding, and three test files for zero behavior gain. The screen's package (`ui/settings/`) is also unchanged — the discoverability rule "follow the entry point, not the entity" from #94 still holds since the sole entry point is still Settings.
- **`LoadedBody` private composable extracted.** New file-private `LoadedBody(state: Loaded, onEvent, modifier)` holds the tab row + per-tab list/empty branching. The top-level `ArchivedDiscussionsScreen`'s `when (state)` still has three arms (`Loading` / `Error` / `Loaded`) — the new structure lifts only the Loaded body, keeping the `Scaffold` and `TopAppBar` at the top level where the back-arrow event dispatch lives. `Loading` and `Error` render via the existing `CenteredText` helper exactly as before; `Error`'s copy still reads `"Couldn't load archived discussions: ${state.message}"` (intentionally retains "discussions" even though the title flipped — error-path copy refresh was explicitly out of scope per spec § Error handling).
- **`SecondaryTabRow` is the chosen primitive.** Chosen over `PrimaryTabRow` because Primary's M3 rounded-pill indicator does not match Figma 18:2's flat 2dp underline; `SecondaryTabRow`'s default indicator is already a 2dp `colorScheme.primary` underline, so the explicit `indicator =` override the spec sketched was dropped (the spec called this preferred — "If yes, prefer the default; no explicit `indicator =` override"). `selectedTabIndex = state.selectedTab.ordinal` works because `ArchiveTab.Channels.ordinal == 0` matches the first `Tab` child position; if the enum gains a third value in front of `Channels` later, the ordinal mapping breaks and the call site needs to swap to an explicit lookup. Documented as a follow-up risk in `archived-discussions-screen.md`, not enforced in code.
- **Five new strings + two removed.** Added: `archived_title` ("Archived"), `archived_tab_channels` ("Channels (%1$d)"), `archived_tab_discussions` ("Discussions (%1$d)"), `archived_empty_channels` ("No archived channels"), `archived_empty_discussions` ("No archived discussions"). Removed: `archived_discussions_title`, `archived_discussions_empty` — both had a single consumer in this screen. Kept: `archived_discussions_settings_row` and `archived_discussions_count_supporting` — those are Settings-side (`SettingsScreen.kt:195-202`) and out of scope for this slice. `cd_back` and `restore_action` reused unchanged.

No new collectors, no new route, no Koin change. The pre-existing `RestoreRequested → vm.onEvent(event)` and `BackTapped → navController.popBackStack()` arms are unchanged.

## Patterns established

- **UI-driven state slots live in the VM via `MutableStateFlow<X> + combine`, not in the composable via `rememberSaveable`.** The spec explicitly raised this as architect's-call ("can live in `rememberSaveable` in the composable or in the VM"); the chosen path puts tab selection in the VM so unit tests can assert `TabSelected(Channels)` → state-update deterministically without an instrumented Compose harness. Future tickets that introduce a similar UI-state slot (filter chips, sort order, expand/collapse) should default to the same shape: private `MutableStateFlow<X>` in the VM + `combine` with the data flow + `TabSelected(X)`-shaped event + arm in `onEvent` that sets `flow.value = event.x`. Justification: testability beats survives-process-death for ephemeral UI state in Phase 0 (no process-death scenario reachable on the fake-data path), and the VM's `WhileSubscribed(5_000L)` grace already covers the rotation case.
- **Sealed UI-state envelopes don't carry a top-level `Empty` when the screen renders a persistent chrome above the empty body.** Pre-#176, the sealed `Loading | Empty | Loaded | Error` shape meant the `Empty` branch couldn't carry the count needed for the tab header — so `Empty` collapsed into a `Loaded`-with-empty-lists. Rule: if the screen's chrome (header, tabs, FAB, segmented controls) needs to read **any** projection from the loaded state to render itself, fold the empty case into a `Loaded(<list>, …)` with `<list>.isEmpty()` checks in the body. Don't carry an envelope-level `Empty` whose payload would have to mirror `Loaded`'s shape. The `DiscussionListScreen` and `ChannelListScreen` shapes (`Loading | Empty | Loaded | Error`) stay because their empty surfaces render no count-dependent chrome — judge per screen.
- **`combine(dataFlow, uiStateFlow) { … }` is the canonical "VM holds both sides" shape.** When a VM blends one cold data source with one or more UI-driven state slots (selected tab, search query, sort order), the established pattern is `combine(repository.observe(…), selectedTab, sortOrder, …) { … }.catch { … }.stateIn(…)`. Tab switching is then a pure state mutation with no re-collection of the upstream — the `combine` re-emits with the new tab slot while the data side is cached at its last value. Established here for tabbed screens; `ChannelListViewModel`'s pre-existing `combine(channelsFlow, discussionsFlow)` is the data-only sibling. Both shapes are valid; pick `combine` whenever any VM input is a `Flow`.
- **`enum class <Tab>` for two-or-more-tab segmented headers, not `Boolean`.** Even with exactly two tabs today, `ArchiveTab { Channels, Discussions }` reads better at the call site (`state.selectedTab == ArchiveTab.Channels`) and survives future additions without rewriting the type. The `.ordinal` mapping to `SecondaryTabRow.selectedTabIndex` makes the enum the right primitive — `Boolean` would force a `if (channels) 0 else 1` translation at the call site. Don't pre-emptively grow the enum past two values for hypothetical future tiers; add when a real one lands.

## Lessons learned

- **`combine { ... } as ArchivedDiscussionsUiState`'s explicit cast inside the lambda is required, not stylistic.** The pre-#176 `.map { if (…) Empty else Loaded(…) }` returned `ArchivedDiscussionsUiState` because both arms of the `if` returned subtypes and Kotlin inferred the common supertype. The new `combine` body returns a single `Loaded(…)` expression, which Kotlin types as `Loaded` — narrower than `ArchivedDiscussionsUiState`. Without the `as ArchivedDiscussionsUiState` cast, the chained `.catch { e -> emit(ArchivedDiscussionsUiState.Error(…)) }` mismatches the upstream `Flow<Loaded>` type and the compiler rejects the `emit(Error(...))` call. The explicit cast widens the produced type back to the sealed parent so the catch's `emit(Error(...))` is type-correct. Documenting because the next VM that re-uses this `combine + .catch + stateIn` shape with a single-arm lambda will hit the same wall.
- **`MutableStateFlow.value = newValue` is non-suspending — `onEvent`'s `TabSelected` arm does not need `viewModelScope.launch`.** The `RestoreRequested` arm wraps `repository.unarchive(...)` in `viewModelScope.launch { … }` because `unarchive` is `suspend`; the `TabSelected` arm directly assigns `selectedTab.value = event.tab` because `MutableStateFlow.value` setter is not. Easy to copy-paste-and-launch the new arm following the pattern of the old; the launch would still work but advertises a suspend dependency that doesn't exist. The spec called this out explicitly ("synchronous; no `viewModelScope.launch` needed").
- **`SecondaryTabRow`'s default indicator already matches the Figma 18:2 spec, so the `indicator =` override is unnecessary.** Compose-bom default for `SecondaryTabRow` is a 2dp `colorScheme.primary` underline — exactly what AC §3 mandates. The architect's spec carried an explicit `SecondaryIndicator(Modifier.tabIndicatorOffset(...), color = MaterialTheme.colorScheme.primary, height = 2.dp)` sketch as a fallback in case the default didn't match; the developer verified the default matches against the current Compose-bom version, dropped the override, and shipped the 2-arg `SecondaryTabRow(selectedTabIndex = …) { Tab(…); Tab(…) }` call. Carry forward: when the spec drafts an explicit override with an "if the default already matches, drop it" instruction, verify against the actual Compose-bom version before shipping the override.
- **`AC §3 vs Figma reference frame disagreement on the default tab is normal; AC wins.** The Figma 18:2 frame illustrates Channels as the active tab — convenient for showing the layout — but AC §3 mandates Discussions as the default selected tab to preserve the pre-#176 landing surface. The spec called this out explicitly ("Figma is otherwise authoritative for visual treatment"); the VM encodes `Discussions` as the initial `MutableStateFlow` value. When the Figma frame and AC disagree on a stateful default, the AC is the contract.
