# #13 — Conditional `NavHost` start destination based on `pairedServerExists`

Replaces the hard-coded `startDestination = Routes.Welcome` with a runtime decision driven by `AppPreferences.pairedServerExists`. A returning user who has already paired skips Welcome and opens directly at `channel_list`; Welcome never enters the back stack for that path. The mechanism is a `produceState`-backed one-shot read of the DataStore flow with a neutral `Surface` placeholder shown while the first emit is in flight — no new dependency, no `runBlocking` on the main thread, no SplashScreen API.

## Summary

- `MainActivity.onCreate`'s `setContent` block now reads `AppPreferences` via `koinInject<AppPreferences>()` inside the `Scaffold` body, drives a `Boolean?` via `produceState<Boolean?>(initialValue = null, appPreferences) { value = appPreferences.pairedServerExists.first() }`, and `when`-branches: `null` → `Surface(fillMaxSize + padding(innerPadding)) {}`, otherwise → `PyryNavHost(startDestination = if (v) Routes.ChannelList else Routes.Welcome, modifier = Modifier.padding(innerPadding))`.
- `PyryNavHost`'s signature gains a leading `startDestination: String` parameter; the literal `startDestination = Routes.Welcome` inside its `NavHost { ... }` becomes the forwarded parameter. The route graph body (Welcome / Scanner / ChannelList / ConversationThread / Settings entries) is verbatim what #12 left in place.
- Four new imports in `MainActivity.kt`: `androidx.compose.material3.Surface`, `androidx.compose.runtime.getValue`, `androidx.compose.runtime.produceState`, `kotlinx.coroutines.flow.first`. No catalog or build-file edits — every dependency already on the classpath via #8 (`navigation-compose`), #11 (`AppPreferences` + Koin Compose accessor), and Compose Material 3 (`Surface`).
- No new files. No new tests. The existing `AppPreferencesTest` (default-false + round-trip) is the foundational coverage; the composition decision itself isn't meaningfully unit-testable without `ComposeTestRule` + Koin override scaffolding the project hasn't built yet. AC4 (`./gradlew assembleDebug`) is the mandatory automated signal; manual smoke on a device covers AC1–AC3.

## Files touched

- `app/src/main/java/de/pyryco/mobile/MainActivity.kt` — four new imports (lines 11, 15-16, 28); `setContent` body re-shaped to host the `produceState` + `when` block (lines 38-58); `PyryNavHost` signature gains `startDestination: String` (line 65); inner `NavHost` reads the forwarded parameter (line 71). ## Patterns established

- **One-shot DataStore-flag read at composition root via `produceState { value = …first() }`.** When an `Application`-scoped boolean must gate which composable composes *first* (rather than which one a screen reactively re-renders), the idiom is `val x: Boolean? by produceState<Boolean?>(initialValue = null, pref) { value = pref.flag.first() }` plus a `when (val v = x) { null -> placeholder; else -> realThing(v) }` switch. `produceState` makes the one-shot intent explicit at the call site and avoids the foot-gun of a reactive collector that would re-evaluate the predicate on every later flip (which `NavHost` then silently ignores because it captures `startDestination` at first composition only).
- **`collectAsStateWithLifecycle` is *not* the default for "read a DataStore flow in composition" in this project.** It would require adding `androidx.lifecycle:lifecycle-runtime-compose` to the catalog (only `lifecycle-runtime-ktx` is wired today), inflating dependency surface for a single use site. Prefer `produceState` for one-shot reads and reach for `collectAsStateWithLifecycle` only when the screen genuinely needs live updates *and* the artifact already needs to land for some other reason.
- **Neutral `Surface(fillMaxSize)` is the sanctioned placeholder while a composition-gating flag is loading.** Not `Box {}` (no background fill → exposes the window-manager background through a transparent flash), not a spinner (loading-state polish is out of scope), not the Android 12+ `core-splashscreen` API (separate dependency, separate ticket when a branded splash is wanted). `Surface` paints `?colorBackground` which matches the system splash background by default — the one or two frames of placeholder during DataStore's first read are visually indistinguishable from the splash continuing.
- **Capture `when (val v = paired)` so the smart-cast lands inside the branch.** The `else` branch passes `if (v) Routes.ChannelList else Routes.Welcome`; without the `val v =` capture the compiler can't smart-cast `paired` to `Boolean` (it's a delegated `Boolean?` property) and the expression breaks. Future similar gates should follow the same `when (val v = ...)` shape rather than `!!`-asserting.
- **`AppPreferences` is the right `produceState` key.** It's a Koin singleton (identity stable for the activity's lifetime, so the producer runs exactly once in practice); passing it as the key is defensive against a future refactor that makes `AppPreferences` swappable (e.g. a debug fake) — the produced state would correctly re-initialise. `Unit` would also work today but bakes in the singleton assumption; the explicit key documents it.

## Lessons learned

- **`NavHost.startDestination` is captured at first composition and is *not* reactive.** Calling `PyryNavHost(startDestination = X)` on first composition then later passing `startDestination = Y` does not rewrite the back stack — the NavController already has its own state. This is *why* this ticket gates the `NavHost` composition entirely on a non-null flag instead of letting `startDestination` flip later: a default-`false` + reactive-update shape would push Welcome onto the back stack for one frame and never reclaim it. The reactive-shape footgun is silent (no crash, no warning), so it's worth pinning here for the next person who reaches for `collectAsStateWithLifecycle` "because that's what the Compose docs use for Flows".
- **`Modifier.padding(innerPadding)` belongs on the placeholder too, not just on `PyryNavHost`.** First draft put the padding only on the `else` branch; the placeholder `Surface` then drew under the system bars while loading and the post-flag composition jolted down by the bar inset. Both branches paint into the `Scaffold`'s content slot — both need the inset padding, even when "both" includes a single-frame placeholder users won't consciously notice.
- **`runBlocking { …first() }` in `Application.onCreate` would also work — and is the right answer if a later ticket shows a measurable placeholder flicker.** The trade is documented in the spec's "Why not" section: zero placeholder time, but main-thread blocking during `Application.onCreate` is a documented ANR-variability risk on cold-start-degraded devices. We picked the non-blocking shape on equal-quality outcome. If real-device profiling ever shows the placeholder is visible long enough to register as a flash (DataStore first reads are typically < 50 ms on warm storage but cold-storage worst case is longer), revisit Option 1 — that's evidence-based escalation, not something to pre-engineer.
- **The `else` branch's `if (v) Routes.ChannelList else Routes.Welcome` reads naturally because of #8's destination-name choices.** If `Routes.Welcome` had been named `Routes.Onboarding`, the ternary would have read `if (v) ChannelList else Onboarding` and required a moment of "wait, paired = skip onboarding". Minor, but a data point on the durability of #8's literal route names: "what the user sees" beats "the conceptual stage" for predicate readability at branch points.
- **Don't add a `try { … } catch { value = false }` around the `first()` call** — silent fallback would mask DataStore corruption and treat a corrupted store as "fresh install", sending a real paired user back through Welcome on every launch. Per #11's "Error handling" section, the corruption case is reserved for a dedicated future ticket; until one is observed, propagating the throwable is the right behaviour.
