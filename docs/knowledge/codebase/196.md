# #196 — feat(data): `ConnectionState` model + stub source

Lands the data-layer seam for "is the client connected to the pyrycode server?". Three new files (`ConnectionState.kt`, `ConnectionStateSource.kt`, `FakeConnectionStateSource.kt`), one new Koin binding line, one new unit-test file. Pure data-layer slice — no Compose, no ViewModel, no UI integration. The banner that consumes this lands in [#197](https://github.com/pyrycode/pyrycode-mobile/issues/197).

See [`../features/connection-state.md`](../features/connection-state.md) for the evergreen contract.

## Summary

- **`ConnectionState` is a four-case sealed class.** Three `data object` cases (`Connected`, `Connecting`, `Offline`) plus one `data class Reconnecting(secondsRemaining: Int)` that carries the countdown the banner interpolates into `"Reconnecting in Ns"`. `Int` (not `kotlin.time.Duration`) because the UI consumer is a flat `String.format` — `Duration` would force the consumer to format. No `require(secondsRemaining >= 0)`; the fake is the sole producer in Phase 2 and constructor invariants for unobserved failure modes are pipeline-banned.
- **`ConnectionStateSource` is the load-bearing contract.** Two members: `fun observe(): Flow<ConnectionState>` and `suspend fun retry()`. Phase 4's walk-back swaps the Koin binding only — the interface does not change. `retry()` is kept on the interface despite being a no-op in Phase 2 because the ticket Technical Notes pin the interface as stable for #197; removing it now would force a churn rewrite.
- **`FakeConnectionStateSource` wraps one `MutableStateFlow<ConnectionState>` initialised to `Connected`.** `observe()` returns `state.asStateFlow()` widened to `Flow<ConnectionState>` so the interface signature doesn't leak `StateFlow`; `retry()` body is empty (one comment line); the public `emit(state)` mutator is the test/preview seam. `emit()` is **on the concrete class, not the interface** — production consumers depend on the interface via Koin so they cannot reach `emit()` by accident.
- **Koin binding is one line added immediately after the existing `FakeConversationRepository` binding.** Same `single { … } bind <Interface>::class` shape, singleton scope (Phase 4 will hold a long-lived WebSocket; the binding shape doesn't change). Two new imports.
- **Three unit tests in `FakeConnectionStateSourceTest`.** JUnit4 + `runBlocking` + `Flow.first()` per [`FakeConversationRepositoryTest`](../features/conversation-repository.md)'s idiom — no Turbine, no MockK, no `runTest`. Per-state `.first()` after each `emit(...)` is the simplification of the spec's "collect into list" plan; see § Lessons learned.

## Patterns established

- **Test/preview seam is a public mutator on the concrete fake, not on the interface.** `FakeConnectionStateSource.emit(state)` is `public` on the fake; `ConnectionStateSource` does not declare it. Production consumers resolve `ConnectionStateSource` via Koin and cannot reach `emit()`; tests construct `FakeConnectionStateSource()` directly and call `emit()` straight on the concrete reference. Pattern applies to any future fake that needs a test-only push affordance — keep it off the interface so production code can't accidentally drive state from the wrong layer. Same shape as [`FakeConversationRepository`](../features/conversation-repository.md)'s seed pipeline scaled down to one flow.
- **Interface stability over Phase-2 economy.** `retry()` is a no-op stub in Phase 2 yet remains on the interface because the consuming ticket (#197) is on the next slice. Pattern: when a downstream ticket is already named and its consumer signature is known, ship the contract complete even if the Phase-2 body is empty. Removing a no-op member now to add it back in two tickets' time is churn for negative value.
- **`asStateFlow()` widened to `Flow<T>` at the interface boundary.** `observe()` returns `Flow<ConnectionState>` while the fake's internal state is a `MutableStateFlow`. The widening lets Phase 4 pick a different flow shape (cold `flow { … }` over a WebSocket event stream) without changing the interface or any consumer. Pattern applies to any "fake exposes a hot `StateFlow`, real impl will be cold" seam — keep the interface at `Flow<T>` and don't leak the implementation's flow kind.

## Lessons learned

- **Per-state `.first()` after each `emit` is the right test shape for `MutableStateFlow`, not "collect into list".** The spec's primary plan ("collect `observe()` into a list while sequentially calling four `emit()`s, assert the collected list equals `[Connected, Connecting, Reconnecting(5), Offline, Connected]`") doesn't actually work against `MutableStateFlow.asStateFlow()`: the `StateFlow` collector is conflated — if four `emit()`s land before the collector resumes, intermediate values can be dropped and only the latest is observed. The spec called this out as a fallback option ("loop the four states, emit, assert"); the developer landed straight on the fallback and recorded the reason in an inline comment. **Pattern for future fake tests against `MutableStateFlow`:** prefer the per-state `.first()` loop unless the test is specifically about observing *transitions* via a buffered/replay flow, in which case the flow type itself needs to change. The AC's phrasing — "surfaces each in order via `observe()`" — is satisfied by the per-state shape.
- **Empty `retry()` body still needs a comment.** Without `// Phase 2: no-op stub. Phase 4 will trigger a reconnect attempt.` the empty function reads as a forgotten implementation. The one-line comment is the cheapest way to make "deliberately empty" visible to a future reader and to anyone running a "is this finished?" scan over the data layer. Same posture as the `Role.Tool -> Unit` deliberate-no-op pattern from [#128](./128.md), except here the no-op is a function body rather than a `when` arm.
- **Adding a new `single` binding cost two imports, not one.** `ConnectionStateSource` (the interface) **and** `FakeConnectionStateSource` (the concrete) both need importing in `AppModule.kt` because the `bind ConnectionStateSource::class` chain references the interface by class literal. Sibling pattern is the existing `FakeConversationRepository() } bind ConversationRepository::class` line — same two-import shape. Not a surprise after the fact, but easy to under-count when sizing a binding addition.
