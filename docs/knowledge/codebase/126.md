## Patterns established

- **Synchronous VMs use `MutableStateFlow(initial).asStateFlow()`, not `stateIn(...)`.** When a VM's state has no upstream cold flow to share â€” no `combine`, no async timing, no `WhileSubscribed` lifetime to negotiate â€” the construction shape is `MutableStateFlow(initial).asStateFlow()`. Reuse of the `StateFlow<UiState>` interface keeps the destination consumer (`val state by vm.state.collectAsStateWithLifecycle()`) identical to every other VM in the codebase, so the shape that lands here is a drop-in replacement when a later ticket widens to `stateIn(...)`. Reach for `stateIn(viewModelScope, WhileSubscribed(5_000L), Loading)` only when there's a real cold flow to share.
- **`data class UiState`, not `sealed interface UiState { Loaded }`, when there's no async surface.** A `sealed interface` with one variant prescribes more shape than the behaviour supports and forces every downstream test to pattern-match a single arm. The single-arm sealed interface is over-engineering by the project's "Don't design for hypothetical future requirements" rule. Widen to a sealed envelope when the first real `Loading` / `Error` distinction appears (likely #128 here).
- **Screens with no VM-owned event take direct `onBack: () -> Unit`, not `onEvent(BackTapped)`.** The canonical `(state, onEvent)` MVI shape is for screens whose VM owns at least one event. When the only event is a pure-navigation back-press (no VM-side side effect), the screen signature is `(state, onBack, modifier)` directly. Introducing a sealed `â€¦Event { BackTapped }` to hold one no-op arm is ceremony. When the first VM-owned event lands, the signature widens to `(state, onBack, onEvent)` *or* `BackTapped` folds into the then-introduced event sealed type â€” that's a design call for the ticket that introduces the second callback, not pre-paid here.
- **Koin `viewModel { â€¦ }` blocks consumed via `koinViewModel<T>()` auto-resolve `SavedStateHandle` via `get()`.** Inside a `NavBackStackEntry`-scoped `composable { }` block, `LocalViewModelStoreOwner` is the back-stack entry, which is both a `ViewModelStoreOwner` and a `SavedStateRegistryOwner`. Koin 4.0.4's `koin-androidx-compose` auto-provides the entry's pre-populated `SavedStateHandle` into the factory; no `koinNavViewModel<T>()` (KMP-compose artifact, not on classpath) and no `SavedStateViewModelFactory` plumbing is needed. The handle's `get<String>("argName")` returns the `navArgument` value. First demonstrated by `ThreadViewModel`; subsequent screens that need to read path arguments off the back-stack entry should use the same shape rather than parsing `backStackEntry.arguments?.getString(...)` at the destination.
- **Path-argument extraction moves into the VM via `SavedStateHandle`.** When a destination's screen has a real ViewModel that consumes the path argument, the extraction shape `savedStateHandle.get<String>("argName").orEmpty()` lives in the VM's constructor, not in the `composable { }` lambda. The inline `backStackEntry.arguments?.getString(...)` shape from #15 stays only for placeholder destinations that render `Text(...)` directly â€” once the destination grows a VM, the parse moves one layer in. Eliminates duplicate extraction at every consumer and gives `SavedStateHandle` survives-process-death semantics for free.

## Lessons learned

- **`koinViewModel<T>()` is the right call site for `SavedStateHandle`-consuming VMs; `koinNavViewModel<T>()` is the KMP-compose artifact that isn't on classpath.** Spent zero time debugging this thanks to the spec calling it out, but flag worth pinning: the navigation-aware variant lives in `koin-compose-viewmodel-navigation` (the Compose Multiplatform target), which the project doesn't depend on. The Android-Compose target (`koin-androidx-compose`, already on classpath since #46) gates `koinViewModel<T>()` on `LocalViewModelStoreOwner`, which Compose Navigation 2.9+ wires to the `NavBackStackEntry`. So `koinViewModel<ThreadViewModel>()` from inside a `composable { }` block resolves to a VM whose `SavedStateHandle` is the back-stack entry's â€” pre-populated with `navArgument` values, survives process death. If a future ticket reaches for `koinNavViewModel<T>()`, that's the signal someone added the KMP-navigation artifact to the catalog â€” flag it.
- **`SavedStateHandle` in `androidx.lifecycle:lifecycle-viewmodel-savedstate` is transitively present on `testImplementation`.** The architect's spec flagged a possible "missing artifact, may need explicit `testImplementation`" path; in practice the constructor `SavedStateHandle(initialState = mapOf(...))` resolved cleanly with no `libs.versions.toml` edit. The artifact rides in via `lifecycle-viewmodel-ktx` + `lifecycle-viewmodel-compose`, both of which are on classpath since the VM era began. Worth pinning so the next test-only artifact pre-flight doesn't get drafted twice.
- **Removing `androidx.compose.material3.Text` from `MainActivity.kt`'s imports needs an explicit grep.** The pre-#126 destination body's `Text("Conversation thread placeholder: $conversationId")` was the only `Text` consumer in `MainActivity.kt`; ktlint / Android Studio "Optimize imports" did not auto-flag the dead line during the edit (the project doesn't run an auto-remove-unused-imports hook). Caught by manual review pre-commit. Future ticket that replaces an inline `Text(...)` body with a composable call: grep for `import androidx.compose.material3.Text` and verify there's no remaining consumer in the file. A pre-commit hook for unused-imports is a tolerable follow-up (`spotless` + `removeUnusedImports`) but not in this ticket's scope.
- **`items(emptyList<Unit>()) { }` compiles cleanly; the architect's "may need to drop trailing `{ _ -> }`" worry was speculative.** The full form with an explicit empty lambda passes both `assembleDebug` and `lint` with no warnings. No need to fall back to `items(emptyList<Unit>())` (no-arg form would not compile â€” `items()` requires an `itemContent` lambda). Pin: the spec's "confirm at developer time which compiles cleanly" instruction resolved to "the explicit form is fine"; downstream tickets that introduce reverse-layout empty bodies should follow the same shape.
