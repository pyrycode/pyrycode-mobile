# #101 — Compose tests for Scanner state screens

Adds three instrumented Compose test classes — one per scanner state surface (`ScannerScreen`, `ScannerConnectingScreen`, `ScannerDeniedScreen`) — bringing the project's instrumented-test count to seven classes after #90 (`SettingsScreenTest`), #91 (`LicenseScreenTest`), #99 (`ChannelListScreenTest`), and #100 (`WelcomeScreenTest`). All three follow the architect's "structure only — assert visible elements render, assert click targets carry a click action; do not `performClick()`" scope: no callback-wiring coverage, no fakes, no MockK, no DI. No production-source, build-file, manifest, or strings changes.

## Summary

- New `app/src/androidTest/java/de/pyryco/mobile/ui/onboarding/ScannerScreenTest.kt` — three `@Test` methods: `topAppBar_rendersPairWithPyrycodeTitle` (exact match `"Pair with pyrycode"`), `hintCard_rendersPyryPairInstruction` (substring match `"pyry pair"`, covers the `buildAnnotatedString` body), `pasteCodeFallback_hasClickAction` (substring match `"Trouble scanning?"`, asserts click action).
- New `app/src/androidTest/java/de/pyryco/mobile/ui/onboarding/ScannerConnectingScreenTest.kt` — two `@Test` methods: `connectingMessage_renders` (substring match `"Connecting to your pyrycode server"` — substring deliberately avoids the U+2026 ellipsis), `serverAddress_rendersVerbatim` (exact match against the literal `"home.lan:7117"` argument). Single `serverAddress: String` parameter passed as `"home.lan:7117"` to match the screen's preview literal.
- New `app/src/androidTest/java/de/pyryco/mobile/ui/onboarding/ScannerDeniedScreenTest.kt` — three `@Test` methods: `heading_rendersCameraPermissionRequired` (exact match), `openSettingsButton_hasClickAction` (`"Open settings"` + `hasClickAction()`), `pasteCodeButton_hasClickAction` (`"Paste code instead"` + `hasClickAction()`).
- All three classes carry `@RunWith(AndroidJUnit4::class)`, a single `@get:Rule val composeTestRule = createComposeRule()`, and one `setContent { PyrycodeMobileTheme { <ScreenUnderTest>(...) } }` per `@Test`. Lambda args passed as `{}`.

## Files touched

- `app/src/androidTest/java/de/pyryco/mobile/ui/onboarding/ScannerScreenTest.kt` — new, 57 LOC.
- `app/src/androidTest/java/de/pyryco/mobile/ui/onboarding/ScannerConnectingScreenTest.kt` — new, 42 LOC.
- `app/src/androidTest/java/de/pyryco/mobile/ui/onboarding/ScannerDeniedScreenTest.kt` — new, 57 LOC.

No changes to `ScannerScreen.kt`, `ScannerConnectingScreen.kt`, `ScannerDeniedScreen.kt`, `MainActivity.kt`, navigation, DI, theme, strings, drawables, or `app/build.gradle.kts`.

## Patterns established

- **Structure-only test class for state surfaces with no callbacks (or callbacks intentionally left unasserted).** `ScannerConnectingScreen` has zero callbacks (single `serverAddress: String` arg); `ScannerScreen` and `ScannerDeniedScreen` have callbacks but this ticket's scope deliberately excludes click-wiring assertions. The resulting shape is `setContent { PyrycodeMobileTheme { … } }` → `composeTestRule.onNode(hasText(…)).assertExists()` for render checks and `.assert(hasClickAction())` for click-target checks — no `var <name>Count = 0` counters (the #100 pattern) and no `mutableListOf<Event>::add` (the #99 pattern). When future tickets revisit these screens to add wiring coverage, the `var`-counter pattern from #100 is the right next step for `ScannerScreen` / `ScannerDeniedScreen` (raw-callback signatures, ≤ 2 callbacks each).
- **`substring = true` is the right tool for `buildAnnotatedString` bodies and any text containing non-ASCII punctuation.** `ScannerScreen`'s hint card body assembles via `buildAnnotatedString { append("Run "); withStyle(...) { append("pyry pair") }; append(" on your pyrycode server to generate a QR code.") }` — the runtime text is the concatenated string, so the test asserts via `hasText("pyry pair", substring = true)` rather than chasing the full sentence. `ScannerConnectingScreen`'s headline ends in U+2026 (`…`); substring match on `"Connecting to your pyrycode server"` sidesteps any future copy tweak that swaps `…` for `...` or vice versa. Rule: substring for `AnnotatedString` content or copy with fragile glyphs; exact match for short standalone labels (`"Pair with pyrycode"`, `"Camera permission required"`, `"Open settings"`, `"Paste code instead"`, `"home.lan:7117"`).
- **`onNode(hasText(...)).assertExists()` over `onNodeWithText(...).assertIsDisplayed()` when the matcher needs `substring = true` or a non-text predicate.** `onNodeWithText` doesn't expose a substring flag in its primary signature, so the three substring assertions here go through `onNode(hasText(..., substring = true))` and pair with `.assertExists()`. The shape composes uniformly with the `hasClickAction()` matcher: `.onNode(hasText(...)).assert(hasClickAction())`. Pure exact-text-match assertions stay on `onNodeWithText(...).assertIsDisplayed()` (compare #100 `WelcomeScreenTest`); the two coexist deliberately and the choice is mechanical (substring or click-action predicate ⇒ `onNode + matcher`, otherwise `onNodeWithText`).

## Lessons learned

- **Three classes for three state surfaces, not one parameterised class.** The architect specified one file per screen; the developer kept that shape. Parameterising would have saved ~20 LOC of `@RunWith` / `@get:Rule` / package boilerplate but at the cost of crossing the three screens' independent test surfaces into a single fail-as-a-group reporting unit. Each screen lives in its own `.kt` file under `ui/onboarding/`; mirroring that 1:1 in `androidTest/.../ui/onboarding/` keeps "where's the test for X?" answerable from the filename. Same shape as #90's `SettingsScreenTest`, #91's `LicenseScreenTest`, #99's `ChannelListScreenTest`, and #100's `WelcomeScreenTest` — five precedents in a row, this ticket makes it eight test classes split across seven screens (one per).
- **`ScannerConnectingScreen` is the project's first instrumented test for a screen with no callbacks at all.** `serverAddress: String` is its only argument; no `() -> Unit` lambdas, no `onEvent`. Both `@Test` methods render the same composable with the same literal `"home.lan:7117"` and assert independent text fragments. No click-wiring assertions to skip — the spec called this out explicitly. The test pattern degenerates cleanly: the file is the project's shortest instrumented test class at 42 LOC. Any future no-callback state surface (a Phase 4 "connection failed" terminal screen, say) reaches for this same shape — no counters, no event list, just two render assertions and done.
- **Three of the scanner-screen feature docs carried "No `androidTest`, no unit test, no snapshot test — Previews are the visual-fidelity check" lines that are now stale.** `features/scanner-connecting-screen.md` and `features/scanner-denied-screen.md` both asserted under "Edge cases / limitations" that no instrumented test existed. Evergreen-not-append-only means those lines get rewritten, not kept as historical breadcrumbs. The documentation step of this ticket updates both. (`features/scanner-screen.md` happened not to carry such a line, so it gets the same INDEX-entry-style test note appended to its main body without an Edge-cases edit.) Heuristic for future test-only tickets: grep the relevant feature doc(s) for `androidTest`, `instrumented`, `snapshot`, `Preview…visual-fidelity` before writing the codebase notes — a stale negative claim is a worse failure mode than a stale positive one.
- **No build-file change required — the pattern has now held across five test-adding tickets.** `androidx.compose.ui.test.junit4` + `androidx.compose.ui.test.manifest` landed in `app/build.gradle.kts` during #90 and have been reused for #91, #99, #100, and now #101. Same heuristic as #100: grep `app/build.gradle.kts` for `androidx.compose.ui.test` before assuming a new instrumented test needs build deps. Solid until a test rule requires something genuinely new (Espresso interop for `Intent` capture, `createAndroidComposeRule<…>()` for an Activity-scoped surface, etc.).
