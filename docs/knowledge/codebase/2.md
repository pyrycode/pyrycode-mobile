# #2 — Conversation / Session / Message data classes

First ticket to create the `data/` package. Defines the foundational schema that the upcoming `ConversationRepository` and every UI tier consume.

## Summary

- New package `de/pyryco/mobile/data/model/` with three files: `Conversation.kt`, `Session.kt`, `Message.kt`.
- `Role` enum (`User`, `Assistant`, `Tool`) co-located in `Message.kt` — single consumer, three lines, no navigation hop.
- All timestamps use `kotlinx.datetime.Instant` (not `java.time.Instant`) so the data layer stays portable for a possible Compose Multiplatform walk-back.
- `kotlinx-datetime 0.6.2` wired through the version catalog (`libs.kotlinx.datetime`), declared once as `implementation` — on the test classpath transitively.
- No annotations on the data classes — `@Serializable` / `@JsonClass` are deferred to Phase 4 when the wire protocol lands.
- Paired JUnit 4 unit tests per type: equality + `hashCode`, `copy()` overrides one field and preserves the rest, `copy()` produces a new instance. `MessageTest` also pins `Role.entries` order to `[User, Assistant, Tool]`.

## Files touched

- `app/src/main/java/de/pyryco/mobile/data/model/Conversation.kt` — new, 13 LOC.
- `app/src/main/java/de/pyryco/mobile/data/model/Session.kt` — new, 11 LOC.
- `app/src/main/java/de/pyryco/mobile/data/model/Message.kt` — new, 14 LOC (incl. `Role`).
- `app/src/test/java/de/pyryco/mobile/data/model/{Conversation,Session,Message}Test.kt` — paired tests.
- `gradle/libs.versions.toml` — added `kotlinxDatetime = "0.6.2"` and `kotlinx-datetime` library entry.
- `app/build.gradle.kts` — added `implementation(libs.kotlinx.datetime)`.

## Patterns established

- **`data/model/` is the home for portable value types.** Future schema additions (e.g. `SessionBoundary` marker for the chronological message stream — see CLAUDE.md's "Conversations model") land here.
- **Always `kotlinx.datetime.Instant`, never `java.time.Instant`** in `data/`. The `Don't` section of CLAUDE.md treats Compose Multiplatform as a walk-back trigger; mixing `java.time` would silently couple the layer to the JVM.
- **One class per file, with a narrow exception for tiny tightly-bound enums.** `Role` lives in `Message.kt` because it has exactly one consumer; `Conversation` and `Session` get their own files for discoverability.
- **JUnit 4 + `org.junit.Test` / `org.junit.Assert.*`** is the unit-test idiom. JUnit 5, Kotest, Truth are not on the classpath — don't introduce them ad hoc.
- **Deterministic test fixtures via `Instant.fromEpochSeconds(0)`.** Avoid `Clock.System.now()` (non-deterministic) and `Instant.parse(...)` (adds a parse failure mode for no benefit).
- **Version-catalog accessor:** `libs.kotlinx.datetime` (dash becomes a dot in Gradle). Writing `libs.kotlinx-datetime` won't compile.

## Lessons learned

- `kotlinx-datetime` is a plain library — it does **not** require the `kotlin("plugin.serialization")` Gradle plugin. That plugin belongs to `kotlinx-serialization`, which is out of scope here.
- Declaring the dependency once as `implementation` is sufficient; `testImplementation` is redundant because `implementation` is on the test classpath transitively. Declaring it in both places is a maintenance trap.
