# #177 — Archive · archive-specific row with inline restore + snackbar (Figma 18:2)

The long-press `DropdownMenu` "Restore" gesture on archived rows is replaced with a purpose-built `ArchiveRow` (avatar + name + "Archived <relative>" subtitle + trailing 40dp icon-button) and a one-tap restore confirmed via a `Snackbar`. The snackbar message is delivered through a new `ArchivedDiscussionsEffect.RestoreSucceeded(displayName)` one-shot effect channel on the existing `ArchivedDiscussionsViewModel`. Closes the second follow-up child of #125 (after #176's tab header). The pre-#177 file-private `ArchivedDiscussionRow` (faded `ConversationRow` + long-press dropdown) is deleted entirely.

## Summary

- **New `ArchiveRow` composable** lives at `app/src/main/java/de/pyryco/mobile/ui/conversations/components/ArchiveRow.kt`. Signature `ArchiveRow(conversation: Conversation, displayName: String, onRestore: () -> Unit, modifier: Modifier = Modifier)`. Renders as a `Row(fillMaxWidth, padding(horizontal=16, vertical=12), Arrangement.spacedBy(12.dp))` with three children: shared 40dp [`ConversationAvatar`](../features/conversation-avatar.md), a weighted `Column` carrying `titleMedium` on `colorScheme.onSurface` plus a `bodySmall` subtitle on `colorScheme.onSurfaceVariant` at `Modifier.alpha(0.75f)`, and a trailing 40dp `IconButton` (default 48dp touch target overridden to 40dp via `Modifier.size(40.dp)` to match Figma) containing a 22dp restore icon on `colorScheme.onSurfaceVariant`. Name + subtitle both carry `maxLines = 1, overflow = TextOverflow.Ellipsis`. The row itself is not clickable; only the trailing icon-button is interactive.
- **`displayName` is hoisted as a parameter, not computed inside the row.** The caller (`LoadedBody` in `ArchivedDiscussionsScreen.kt`) computes the fallback once via a file-private `Conversation.displayName()` extension and passes the same string into `ArchiveRow`'s `displayName` argument, the `RestoreRequested(id, displayName)` payload, and (transitively) the snackbar text. Single source of truth for the fallback resolution; the row never re-derives it.
- **Restore icon is `Icons.Filled.Refresh`.** The spec sketched `Icons.Filled.Restore` with `Replay` / `History` / `Undo` as fallbacks and asked the developer to note any swap. The developer picked `Refresh` (the circular-arrow glyph in `material-icons-core`) as the closest visual match to the Figma 18:2 inline SVG — readable as "restore from archive" without needing `material-icons-extended` on the classpath (`Restore` ships there, not in `core`; `Refresh` is in `core` already). See § Lessons learned.
- **Subtitle reads `Archived <formatRelativeTime(lastUsedAt)>`.** `Conversation.lastUsedAt` is the timestamp source — a dedicated `archivedAt: Instant?` field is still out of scope (would require model + repository + auto-archive-worker changes well beyond this slice). The subtitle text is `stringResource(R.string.archived_relative_subtitle, formatRelativeTime(conversation.lastUsedAt))`. `formatRelativeTime` is `internal` at `ui/conversations/components/RelativeTime.kt` and `ArchiveRow` sits in the same package, so the call is direct (no import-visibility expansion needed).
- **`IconButton.contentDescription` interpolates the row's display name.** `stringResource(R.string.cd_restore_archive, displayName)` → e.g. `"Restore old-project-experiments"` or `"Restore Untitled discussion"`. The `displayName` argument propagates the same fallback the visible headline uses, so screen readers and visible label stay in lockstep even for nameless conversations.
- **`ArchivedDiscussionsScreen` swaps in `ArchiveRow` for both tabs.** `LoadedBody`'s `LazyColumn { items(items, key = { it.id }) { conversation -> ArchiveRow(...) } }` replaces the pre-#177 `ArchivedDiscussionRow(discussion, onRestore = { onEvent(RestoreRequested(it.id)) })` call site. Both archived channels and archived discussions go through the same row (same shape the pre-#177 `ArchivedDiscussionRow` had post-#176 — the type-name irrelevance of `Conversation` carries over). The pre-#177 `ArchivedDiscussionRow` private composable and its `ArchivedDiscussionsRowMenuPreview` preview are both deleted; the `Modifier.alpha(0.65f)` row-level dimming, the `combinedClickable` long-press wiring, and the `DropdownMenu` + `DropdownMenuItem` "Restore" all leave the file.
- **`ArchivedDiscussionsEvent.RestoreRequested` gains a `displayName: String` field.** Was `data class RestoreRequested(val conversationId: String)`; now `data class RestoreRequested(val conversationId: String, val displayName: String)`. The new field carries the resolved-fallback string from the row's call site into the VM, which forwards it into the `RestoreSucceeded` effect so the snackbar text doesn't need to re-derive the fallback after `unarchive` succeeds.
- **New `ArchivedDiscussionsEffect` sealed interface for one-shot UI signals.** Lives in `ArchivedDiscussionsViewModel.kt` next to `ArchivedDiscussionsEvent`; one variant today — `data class RestoreSucceeded(val displayName: String) : ArchivedDiscussionsEffect`. Exposed off the VM as `private val _effects = Channel<ArchivedDiscussionsEffect>(Channel.BUFFERED)` + `val effects: Flow<ArchivedDiscussionsEffect> = _effects.receiveAsFlow()`. First effect-channel in the codebase — see § Patterns established for the rule on event-vs-effect vs `navigationEvents`.
- **`onEvent`'s `RestoreRequested` branch wraps `unarchive` in `runCatching` and only sends the effect on success.** Spec drafted bare-throw semantics ("If `unarchive` throws, the effect is not sent — the exception propagates to the coroutine and is logged by the default uncaught-exception handler"); the developer swapped to `runCatching { repository.unarchive(event.conversationId) }.onSuccess { _effects.send(RestoreSucceeded(event.displayName)) }` to avoid crashing `viewModelScope`'s child on the Phase-4 Ktor failure path. The semantics for the snackbar (no emit on failure) are identical; the difference is the unhandled-exception behaviour — see § Lessons learned.
- **`ArchivedDiscussionsScreen` gained a fourth parameter `effects: Flow<ArchivedDiscussionsEffect> = emptyFlow()`.** Default keeps the three previews and any future preview/test call sites that don't care about effects from breaking. The route composable in `MainActivity` forwards `vm.effects` (single new line at `MainActivity.kt:234`). Inside the screen, a `LaunchedEffect(effects, snackbarHostState) { effects.collect { … } }` collects exactly once per `(effects, snackbarHostState)` pair — keys chosen to recompose if either reference changes; the channel's exactly-once delivery semantics mean recompositions while a collector exists won't replay stale snackbars.
- **`LocalResources.current` used instead of the spec's `LocalContext.current` + `context.getString(...)`.** Functionally equivalent for string lookup; `LocalResources` skips one accessor hop. No precedent for either pattern in the codebase before this slice; carry forward whichever a future ticket picks — both are valid.
- **`SnackbarHost` wired into `Scaffold.snackbarHost = { SnackbarHost(snackbarHostState) }`.** `snackbarHostState = remember { SnackbarHostState() }` lives at the top of `ArchivedDiscussionsScreen`, above the `LaunchedEffect` and above the `Scaffold`. First `SnackbarHost` in the codebase.
- **Strings shifted: three added, one deleted.** Added: `archived_relative_subtitle` = `"Archived %1$s"` (subtitle text under the name; `%1$s` is `formatRelativeTime`'s output), `restored_snackbar` = `"Restored %1$s"` (snackbar copy), `cd_restore_archive` = `"Restore %1$s"` (IconButton `contentDescription`). Deleted: `restore_action` = `"Restore"` (sole consumer was the now-deleted `DropdownMenuItem`; grep confirmed no other references). The five #176 strings (`archived_title`, `archived_tab_channels`, `archived_tab_discussions`, `archived_empty_channels`, `archived_empty_discussions`) and the two Settings-side strings (`archived_discussions_settings_row`, `archived_discussions_count_supporting`) are untouched.
- **The file-private `Conversation.displayName()` extension is duplicated, not refactored shared.** Mirrors `ConversationRow.kt`'s inline fallback (`name?.takeIf { it.isNotBlank() } ?: if (isPromoted) "Untitled channel" else "Untitled discussion"`) byte-for-byte. The spec explicitly forbade extracting a shared helper this slice — "**do not refactor `ConversationRow`**". Three call sites now hold the same fallback (`ConversationRow`, `DiscussionListViewModel.displayLabel`, `ArchivedDiscussionsScreen.displayName`); consolidation is a deliberate non-goal until a fourth call site appears or the strings change.
- **No new instrumented Compose tests.** The screen had none pre-#177 and #177 doesn't add any; the new `ArchiveRow` `@Preview`s (light + dark) provide visual coverage. The deleted `ArchivedDiscussionsRowMenuPreview` is the only preview-count regression — three `ArchivedDiscussionsScreen` previews + two `ArchiveRow` previews = five total post-#177 (vs four pre-#177, three of which were the screen previews unchanged).
- **No Compose-BOM bump, no `material-icons-extended` classpath addition.** `Icons.Filled.Refresh` is in `material-icons-core` (already on the classpath since #21).

## Patterns established

- **One-shot effect channels are `Channel<X>(Channel.BUFFERED) + receiveAsFlow()`, separate from `navigationEvents`.** This is the codebase's first non-nav one-shot effect. The pattern is: private `_effects = Channel<EffectType>(Channel.BUFFERED)`, public `effects: Flow<EffectType> = _effects.receiveAsFlow()`, collect in a `LaunchedEffect(effects, otherKey) { effects.collect { … } }` at the screen level (not the destination level — the screen owns the `SnackbarHostState`). When a screen has both navigation events and UI effects, expose two separate channels (`navigationEvents` for one-shot nav, `effects` for snackbar/toast/etc.); don't fold one into the other. Justification: nav events are routed at the NavHost destination, UI effects need a composable-scoped state holder (`SnackbarHostState`), so the collection sites differ structurally. The naming convention is `EffectType : ArchivedDiscussionsEffect` (singular `Effect` as the sealed-interface name, not `Effects` or `SideEffect`) — mirrors the established `ArchivedDiscussionsEvent` shape so the trio reads as `(UiState, Event, Effect)`.
- **Event payloads carry resolved display strings, not just ids.** `RestoreRequested(conversationId, displayName)` is the precedent: when an event's downstream side-effect needs both the database key and a display-string representation, plumb the resolved string through the event rather than re-resolving it at the side-effect site. Reasons: (a) the row composable that emits the event has already resolved the fallback (one source of truth); (b) the side-effect's display string can't accidentally diverge from the UI's display string; (c) the VM stays a thin router and doesn't need to look up the entity by id again to format the snackbar. Future events that fire a snackbar/toast naming the affected entity should follow this shape: include the display string in the event payload.
- **`runCatching { suspend-call }.onSuccess { post-success-effect }` is the canonical shape for "fire-and-forget repository side-effect with a UI confirmation."** Pre-#177 the codebase had no precedent — the `ArchivedDiscussionsViewModel.RestoreRequested` arm just called `repository.unarchive(id)` bare and relied on the upstream flow re-emission to update the UI. Post-#177 the same arm wraps in `runCatching` to gate the effect emission on success. Future tickets that introduce a similar shape (e.g. archive confirmation, message-send confirmation) should mirror this: launch on `viewModelScope`, wrap the suspend call in `runCatching`, post the success effect inside `.onSuccess { … }`. Don't add a `.onFailure { _effects.send(Failed(...)) }` branch until a real failure UI surface exists — premature error effects are dead code until the production Ktor remote arrives in Phase 4. The trade-off vs the spec's bare-throw is documented under § Lessons learned.

## Lessons learned

- **`Icons.Filled.Refresh` ships in `material-icons-core`; `Icons.Filled.Restore` only in `material-icons-extended`.** The spec recommended `Restore` and listed `Replay` / `History` / `Undo` as fallbacks. None of those four are in `core`; pulling them in requires `material-icons-extended` on the classpath, which is ~12 MB of bytecode for a single icon. `Icons.Filled.Refresh` (a circular-arrow glyph) is in `core` and reads as "restore" in context. Carry forward: when the spec recommends an icon constant, verify with `grep -r 'Icons.Filled.<Name>' app/src/main` against the current Compose-BOM that the constant is actually reachable — `material-icons-core` covers ~20 names; the rest are extended-only. Project rule from the [Settings screen](../features/settings-screen.md) doc ("`material-icons-extended` banned") is the upstream reason; this is its first archive-context application. If a future ticket genuinely needs `Restore`/`Replay`/etc., the cost is the extended dependency, not a `painterResource(R.drawable.ic_xxx)` workaround.
- **`runCatching { … }.onSuccess { … }` swallows the throw and short-circuits the effect emission in one line; the spec's bare-throw alternative leaks an uncaught exception into `viewModelScope`'s default handler.** Spec text: "If `unarchive` throws, the effect is not sent (no snackbar) — matches the AC's 'on successful restore'. The exception propagates to the coroutine and is logged by the default uncaught-exception handler". That's true today (Fake never throws) but Phase 4's Ktor implementation will throw on transient network failures — and `viewModelScope.launch { repository.unarchive(...).throws }` with no try/catch lets the throw propagate to the parent `CoroutineExceptionHandler`, which on Android logs at `Log.e` and on debug builds may crash via `StrictMode`. Wrapping in `runCatching` keeps the behavior identical for the snackbar (no emit on failure) while preventing the unhandled-exception path. The trade-off the spec didn't price in: swallowing the exception silently means the failure is not even logged. Future work that introduces a `RestoreFailed(...)` effect should re-derive the failure cause from `runCatching`'s `getOrElse` / `onFailure` branches; the inline comment in the `onEvent` arm pre-flags this.
- **`async { vm.effects.first() }` must be started *before* `vm.onEvent(...)` dispatches the event, but *after* the `state` collector + `advanceUntilIdle()` warm-up.** The order matters: the channel's `send` is buffered, but a collector that doesn't exist at `send` time and is only created later via `vm.effects.first()` after the dispatch+`advanceUntilIdle` works *because* `Channel(BUFFERED)` keeps the value until first take. So both orderings technically pass — but starting the `async` first makes the assertion's intent (the effect arrives once, in response to the event) read clearly in the test. The other ordering (dispatch → advance → first()) reads as "the channel happens to still hold the value" which is true but accidental. The `restoreRequested_emitsRestoreSucceededEffect_afterUnarchive` test uses the first ordering; carry the convention forward for future effect-channel tests.
- **`withTimeoutOrNull(100.milliseconds) { vm.effects.first() } == null` is the standard assertion for "no effect emitted."** In `runTest` with `UnconfinedTestDispatcher`, `advanceUntilIdle()` exhausts the queued coroutines synchronously, so a `vm.effects.first()` on a channel that received nothing would suspend forever. `withTimeoutOrNull` is the idiomatic escape; the 100ms wall-clock budget is generous (test-clock scheduling is synchronous post-`advanceUntilIdle`). Don't use bare `first()` for negative assertions; the test would deadlock the runner.
- **A `Flow<...> = emptyFlow()` default on a Composable parameter is the established way to keep previews and tests from caring about a hot stream.** `ArchivedDiscussionsScreen`'s new `effects: Flow<ArchivedDiscussionsEffect> = emptyFlow()` parameter follows this — the three pre-existing `@Preview`s don't pass it, so no preview rewrite. The alternative (require the parameter, supply `emptyFlow()` in every preview) is uglier and brittle. Carry forward: when adding a new hot-stream parameter to a Composable, default to `emptyFlow()` if the parameter is purely effectful (no rendered state depends on it).
- **`LocalResources.current` works and is one accessor shorter than `LocalContext.current.resources`.** Both pull from the same composition local under the hood. No reason to standardise either way until a second site appears — but the choice is invisible at the call site (`resources.getString(R.string.x, arg)` vs `context.getString(R.string.x, arg)` are byte-identical in bytecode).
