# #149 — Welcome pill CTA shape + hero top padding (Figma 6:32)

Two single-property edits in `WelcomeScreen.kt` to close the residual visual deltas flagged at the end of #57's code review against locked Figma node `6:32`. The "I already have pyrycode" primary CTA is now a fully-rounded pill, and the hero block's top padding matches the Figma `pt-[168px]` value.

## Summary

- Primary CTA `Button` gets `shape = RoundedCornerShape(28.dp)`. Combined with the existing `Modifier.height(56.dp)`, the silhouette is a 56dp-tall full-width pill (corner = height / 2), matching Figma node `6:41`'s `rounded-[28px]` instead of the M3-default `Shapes.medium` (~20dp).
- Hero `Column`'s `Modifier.padding(top = 136.dp)` becomes `padding(top = 168.dp)`, matching Figma node `6:32`'s `pt-[168px]`. Stays under `systemBarsPadding()` on the outer `Column` — the convention is to read Figma top values as "below the safe area," not as absolute distance from device top.
- Secondary `TextButton` deliberately gets **no** shape change. Figma node `6:43` also declares `rounded-[28px]`, but M3 text-button surfaces have no fill, so corner radius is visually irrelevant; adding a shape there would be dead code.
- One added import: `androidx.compose.foundation.shape.RoundedCornerShape`. No other imports change; the public `WelcomeScreen(onPaired, onSetup, modifier)` signature, both `@Preview` composables, drawables, and theme tokens are byte-identical.

## Patterns established

- **Figma top values are "below the safe area" in this codebase.** When a Figma node declares `pt-[Npx]` on the outer container, translate it to `Modifier.padding(top = N.dp)` *under* `systemBarsPadding()`, not to `N.dp − statusBarHeight` with inset math. Device-to-device status-bar variance is absorbed by `systemBarsPadding()`; the design number stays as written. If a future ticket reinterprets Figma top values as absolute-from-device-top, that's a screen-wide convention change worth its own ticket.
- **Pill buttons get an explicit `shape = RoundedCornerShape(height / 2)`.** For a single button this stays inline; don't introduce a shared `PillButton` composable or a `Shapes.pill` token until a second pill consumer arrives. Premature abstraction for one site is on the project's don't-do list.
- **`TextButton`s with no fill don't need a shape argument.** The corner radius of an invisible surface is invisible. Skip the no-op even when the Figma spec carries a matching `rounded-[…]` on the secondary CTA.
- **AC scope shape wins on a one-property fix.** The spec resolved an open question on the `systemBarsPadding()` interaction by picking the simpler reading; that's the right move for an XS ticket. Don't escalate "is the design value absolute or post-inset?" into an inset-math refactor inside a Figma-polish ticket.

## Lessons learned

- **`Button(shape = …)` vs `ButtonDefaults.shape` are distinct knobs.** The `shape` parameter overrides per-call; `ButtonDefaults.shape` is the project-wide default. For a single screen the parameter is the right surface — no need to fork `ButtonDefaults` or introduce a theme-level `Shapes` override.
- **`corner = height / 2` ⇒ pill, but only if the height is fixed.** The 56dp `Modifier.height(56.dp)` is what makes `RoundedCornerShape(28.dp)` resolve to a true pill rather than just "very rounded." If a future ticket removes the explicit height, the visual collapses; keep the two together.
- **Tests that match by visible text are robust to shape / spacing edits.** The four `WelcomeScreenTest` cases (`pairedCta_isDisplayed`, `setupCta_isDisplayed`, `tappingPairedCta_invokesOnPairedOnly`, `tappingSetupCta_invokesOnSetupOnly`) pass unchanged — no test edits needed for either property change. This is the design point of matching by text rather than by node geometry; preserve it.
