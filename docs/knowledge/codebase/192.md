# #192 — feat(data): `SessionBoundary` carries authored trigger reason + workspace path

Closes the gap [`#9`](./9.md) flagged in 2026-02: `FakeConversationRepository.buildThreadItems` hard-coded every derived `SessionBoundary` to `BoundaryReason.Clear` because no authored metadata was available. This ticket adds the seed-authoring channel (per-historical-session `nextBoundaryReason` + `nextWorkspaceCwd`), threads it into the projection, and assigns one variant per seeded channel so the three `BoundaryReason` values (`Clear`, `IdleEvict`, `WorkspaceChange`) are collectively exercised by the seed. Data-layer only — no UI changes; the consumer is [`#135`](https://github.com/pyrycode/pyrycode-mobile/issues/135) (delimiter component) downstream.

## Summary

- **One new field on `ThreadItem.SessionBoundary`: `workspaceCwd: String? = null`.** Documented invariant in KDoc: non-null iff `reason == BoundaryReason.WorkspaceChange`; `null` for `Clear` / `IdleEvict`. Default value is `null` so the **only** call site that constructs `SessionBoundary` directly outside the fake repo (the existing `observeMessages_messagesAcrossTwoSessions_emitsExactlyOneBoundary` test) keeps compiling unchanged. Invariant is asserted in tests, **not** enforced via `require(...)` — same posture as `Message.toolCall` (#191). DTOs in `data/` stay plain.
- **Field name chosen: `workspaceCwd`.** Precedent is `Conversation.cwd: String`; the `workspace` prefix disambiguates on a boundary type where "cwd" is otherwise ambiguous ("which session's cwd?"). The "new" semantics is implicit — a boundary is always a transition. The architect's spec offered `newWorkspaceCwd` as an equivalent option; `workspaceCwd` won on brevity.
- **Storage shape is a private map on `ConversationRecord`, keyed by successor session id.** `boundariesBySessionId: Map<String, AuthoredBoundary> = emptyMap()` (defaulted). `AuthoredBoundary(reason, workspaceCwd)` is a private supporting `data class` next to `ConversationRecord`. **Key choice is load-bearing**: keying by the *successor* session id (the session that "started for this reason") is forward-compatible with eager mutator emission — when `mintNewSession(reason, workspaceCwd)` lands later it writes `boundariesBySessionId[newSessionId] = AuthoredBoundary(...)` inside the same `state.update { ... }` block with no storage reshape. Keying by predecessor would have made that future shape awkward.
- **Seed-authoring channel: `SeedSession` gains `nextBoundaryReason: BoundaryReason? = null` and `nextWorkspaceCwd: String? = null`.** Both nullable so the `initialMessages` constructor path (which bypasses `seedChannel`) and any future history entries that intentionally lack metadata fall through to the default-`Clear`/`null` fallback in `buildThreadItems`. Semantics: `nextBoundaryReason` answers "why did the session immediately after this one begin?" — and the workspace path is `next`, not historical. Naming pair (`nextBoundaryReason` / `nextWorkspaceCwd`) chosen over `triggerForNextSession` / `workspaceForNextSession` for brevity; the architect's spec called both pairs acceptable.
- **`seedChannel` flattens authored seeds into the per-record map.** Walks `history` with `mapIndexedNotNull` pairing each `SeedSession` whose `nextBoundaryReason != null` with `orderedSessionIds[index + 1]` — where `orderedSessionIds = history.map { it.sessionId } + sessionId` (current session id is the tail). With the current shape (each channel has exactly one historical session), every authored entry pairs to the channel's current `sessionId`; the loop generalizes to multi-history channels without modification when seed data grows.
- **`buildThreadItems` signature gains `boundariesBySessionId`.** On a session-id delta, looks up `boundariesBySessionId[message.sessionId]`: if present uses `authored.reason` / `authored.workspaceCwd`; if absent falls back to `BoundaryReason.Clear` + `null`. `previousSessionId` / `newSessionId` / `occurredAt` are unchanged. `observeMessages` passes the map through after the nullable-record bail-out (changed from `?.messages ?: return@map emptyList()` to `records[id] ?: return@map emptyList()` + `buildThreadItems(record.messages, record.boundariesBySessionId)`).
- **Fallback path is the unchanged default, not an error path.** The `initialMessages` constructor route (used by `observeMessages_messagesAcrossTwoSessions_emitsExactlyOneBoundary` at L180) produces no authored entries; the test's `assertEquals(BoundaryReason.Clear, b.reason)` assertion continues to hold. Added `assertNull(b.workspaceCwd)` for symmetry — the spec called it optional but recommended; ships in this ticket.
- **Concrete seed assignments (one variant per channel):**
  - `seed-channel-personal` → `Clear` (no path). Story: `/clear` ends the first session.
  - `seed-channel-pyrycode-mobile` → `WorkspaceChange` with `"~/Workspace/pyrycode-mobile"`. Story: previous session ran elsewhere, switched into this channel's existing `cwd` to begin the next — path matches the conversation's own `cwd` so no other field needs touching.
  - `seed-channel-joi-pilates` → `IdleEvict` (no path). Story: re-uses the channel's existing sleeping-indicator posture (`currentSessionEndedAt = 2026-05-09T03:00:00Z`, seeded at the line that's now around L389) for end-to-end consistency — historical session ended idle, current session has gone idle again.
- **Four new tests in `FakeConversationRepositoryTest`.** Three per-channel scenarios (`seededPersonalChannel_boundary_isClear_withNullWorkspaceCwd`, `seededPyrycodeMobileChannel_boundary_isWorkspaceChange_withSeededPath`, `seededJoiPilatesChannel_boundary_isIdleEvict_withNullWorkspaceCwd`) plus an aggregate guard (`seededChannels_collectivelyExerciseAllBoundaryReasons`) asserting the set of reasons collected across the three channels equals `{Clear, IdleEvict, WorkspaceChange}`. The aggregate test exists specifically to fail loudly if a future seed edit silently collapses two channels onto the same reason — without it, two of the three single-channel tests could still pass after such a regression.
- **No interface widening; no mutator changes.** `ConversationRepository` is untouched. `startNewSession` / `changeWorkspace` still mint sessions without emitting authored boundaries; the deferred follow-up to lift that is unchanged in scope, but its target storage shape (`boundariesBySessionId[newSessionId] = AuthoredBoundary(...)`) is now in place and waiting.
- **`Session` / `Conversation` models are untouched.** Authored-boundary metadata is repository-private (`ConversationRecord` + `AuthoredBoundary`), not part of `data/model/`. Keeps the cross-package model portable per CLAUDE.md "Don't" rules (Compose Multiplatform walk-back) and keeps the diff narrow (no model-wide rebuild).

## Patterns established

- **Map-by-successor-session-id is the storage shape for authored boundary metadata.** Eager mutator emission (the deferred follow-up that #9 flagged and #192 declined to also do) writes `boundariesBySessionId[newSessionId] = AuthoredBoundary(reason, workspaceCwd)` inside the existing `state.update { ... }` block. No reshape needed; the key choice in this ticket is the forward-compatible one. If the follow-up reshapes the map (e.g. keyed by predecessor, or list-of-records), it's a regression on the affordance this ticket paid for — fix the follow-up, not the storage.
- **Repository-private metadata stays in `ConversationRecord` + a sibling private data class.** `AuthoredBoundary` lives next to `ConversationRecord` in `FakeConversationRepository.kt`, not in `data/model/`. The principle: anything not crossing package boundaries (UI-side consumers don't see `AuthoredBoundary` — they see `SessionBoundary.workspaceCwd` only) doesn't deserve a slot in `data/model/`. Keeps the cross-package model portable for the Compose Multiplatform walk-back per CLAUDE.md. Future repository-only metadata (e.g. per-conversation pagination cursors when Phase 4 lands) follows the same shape.
- **Defaulted-nullable new field on a public DTO + KDoc invariant + test assertions = the codebase pattern for "this combination of fields is constrained."** Three precedents already in place: `Conversation.isSleeping = false` (#20), `Conversation.archived = false` (#93), `Message.toolCall: ToolCall? = null` (#191). #192 adds a fourth (`SessionBoundary.workspaceCwd: String? = null`). All four share: (1) field is last in the constructor, (2) default value covers the "absent" case, (3) the cross-field invariant lives as KDoc, (4) tests assert both sides of the invariant, (5) no `require(...)` / `init { }`. Treat as default shape for any future DTO field that's only valid in some states.
- **Aggregate "covers all enum variants" guard test is a small, cheap regression net.** `seededChannels_collectivelyExerciseAllBoundaryReasons` doesn't replicate the per-channel tests — it specifically catches the failure mode where a future seed edit silently collapses two channels onto the same variant. Worth replicating any time the seed data is the canonical exercise surface for an enum the codebase only ever consumes (rather than produces) — e.g. if `ConversationFilter` ever needed a similar guard. Cost is ~12 lines; payoff is silent-regression detection.

## Lessons learned

- **`MutableStateFlow.map`'s nullable-record bail-out should land at the `record` level, not at `record.messages`, once the projection reads multiple record fields.** Pre-#192, `observeMessages` bailed out on `records[id]?.messages ?: return@map emptyList()` — convenient because `messages` was the only field touched. #192 now reads `record.boundariesBySessionId` as well, so the bail-out moved up one step (`records[id] ?: return@map emptyList()` + `buildThreadItems(record.messages, record.boundariesBySessionId)`). Worth keeping in mind for the next projection that grows a second field: rewrite the bail-out before adding the second field's pass-through, not after — otherwise you end up with two `?.` chains where one would do. Pre-emptive cleanup, half a line.
- **`mapIndexedNotNull` with `orderedSessionIds[index + 1]` cleanly pairs each `SeedSession` with its successor's id without needing `zipWithNext` or windowed iteration.** Tried/considered both — `windowed(2)` requires unwrapping a pair, `zipWithNext` makes the trailing-edge case (the last historical entry pairs to the *current* session id, not to itself) more awkward because the current session id isn't in `history`. The `orderedSessionIds = history.map { it.sessionId } + sessionId` lift is one cheap allocation that makes the indexed lookup obviously correct. Worth picking up for similar "pair each list element with its successor including a tail anchor" needs.
- **Don't paper over field-ordering hazards in `data class` extensions by passing **named** arguments at construction sites — keep the discipline of "new field is last + defaulted."** Could have inserted `workspaceCwd` after `reason` (the conceptually-related field) and updated the one direct construction site (the L180 test) to use named args. Rejected: the named-args-as-rescue pattern is fragile against future field insertions, and consistency with #20 / #93 / #191 (all of which appended) is more valuable than co-locating semantically-related fields. The KDoc already documents the `reason ↔ workspaceCwd` relationship; field ordering doesn't need to reinforce it.
