# #100 — Compose tests for `WelcomeScreen`

Adds the project's fourth instrumented Compose test class (after `SettingsScreenTest` / `LicenseScreenTest` from #90 / #91 and `ChannelListScreenTest` from #99), covering `WelcomeScreen`'s two CTAs and their callback wiring. The screen is a raw-callback Composable (`onPaired`, `onSetup` — no `(state, onEvent)` indirection), so the click-capture pattern degenerates from `mutableListOf<Event>::add` to plain `var pairedCount = 0` / `var setupCount = 0` counters. No production-source, build-file, manifest, or strings changes — pure test addition.

## Summary

- New `app/src/androidTest/java/de/pyryco/mobile/ui/onboarding/WelcomeScreenTest.kt` — four `@Test` methods, one assertion focus per test. Single `@get:Rule createComposeRule()`, `@RunWith(AndroidJUnit4::class)`, `PyrycodeMobileTheme { WelcomeScreen(onPaired = …, onSetup = …) }` wrap per test.
- Two display tests (`pairedCta_isDisplayed` / `setupCta_isDisplayed`) use no-op `{}` callbacks and assert `onNodeWithText(<label>).assertIsDisplayed()`.
- Two click tests (`tappingPairedCta_invokesOnPairedOnly` / `tappingSetupCta_invokesOnSetupOnly`) declare `var pairedCount = 0; var setupCount = 0` locally, attach increment lambdas as the callbacks, `performClick()` the matched node, and assert the canonical (1, 0) / (0, 1) tuple via two `assertEquals` calls — the symmetry catches "tap on one CTA fires both callbacks" regressions.
- Strings are hardcoded English literals (`"I already have pyrycode"` / `"Set up pyrycode first"`) — not `R.string.*` lookups — because the CTA labels themselves are hardcoded in `WelcomeScreen.kt` (no `strings.xml` entry exists for them).

## Files touched

- `app/src/androidTest/java/de/pyryco/mobile/ui/onboarding/WelcomeScreenTest.kt` — new, 89 LOC, four `@Test` methods:
  - `pairedCta_isDisplayed` — `onNodeWithText("I already have pyrycode").assertIsDisplayed()`.
  - `setupCta_isDisplayed` — `onNodeWithText("Set up pyrycode first").assertIsDisplayed()`.
  - `tappingPairedCta_invokesOnPairedOnly` — clicks "I already have pyrycode", asserts `pairedCount == 1 && setupCount == 0`.
  - `tappingSetupCta_invokesOnSetupOnly` — clicks "Set up pyrycode first", asserts `setupCount == 1 && pairedCount == 0`.

No changes to `WelcomeScreen.kt`, `MainActivity.kt`, navigation, DI, theme, strings, drawables, or other screens.

## Patterns established

- **Raw-callback screens use `var` counters; `(state, onEvent)` screens use `mutableListOf<Event>::add`.** `WelcomeScreen`'s public surface predates the project-wide MVI shape — it takes two `() -> Unit` callbacks directly rather than a sealed `Event` type routed through `onEvent`. The click-capture pattern degrades accordingly: a local `var <name>Count = 0` per callback, incremented in the lambda, asserted via `assertEquals(1, <name>Count)` and the symmetric `assertEquals(0, <other>Count)`. This is the right shape only because the screen has exactly two callbacks; any future raw-callback screen with three or more should reach for the `mutableListOf<…>::add` pattern from #99 (parameterised on a sealed type even if the source signature is `() -> Unit`s) to keep the assertion surface compact.
- **Hardcoded English literals are acceptable in tests when the screen itself hardcodes them.** `WelcomeScreenTest` asserts on `"I already have pyrycode"` directly, not `InstrumentationRegistry.getInstrumentation().targetContext.getString(R.string.<id>)`. The `LicenseScreenTest` / `ChannelListScreenTest` pattern of going through `R.string` is the right shape *when the production source uses `stringResource(R.string.…)`*, because then both sides reference the same id and a rename in `strings.xml` updates both. Here the production source carries the literal too (CTA labels live inline in `WelcomeScreen.kt`, no `strings.xml` entry); routing through a non-existent `R.string.*` would create a fake indirection that drifts the day someone adds a real string id. Rule: assert against whatever shape the source uses — literal-for-literal or id-for-id, never crossed.
- **One assertion focus per `@Test`, four `@Test`s for a two-CTA surface.** The architect's spec split "displayed" + "click wiring" into separate methods per CTA rather than collapsing both displays into one test and both clicks into another. Matches the `SettingsScreenTest` / `LicenseScreenTest` idiom; a failing display test reads "the `onSetup` CTA disappeared" rather than "something about the onboarding screen broke." Cost is four `setContent` blocks instead of two; benefit is per-test diagnostic clarity. Default for any 2–4-CTA screen: one `*_isDisplayed` per CTA + one `tapping*_invokes*Only` per CTA, no merging.

## Lessons learned

- **`WelcomeScreen`'s CTA labels are hardcoded literals, not `stringResource` lookups.** Confirmed via the spec's reference to `WelcomeScreen.kt:34-136` and verified the test author kept literals in step with the production source. If a future ticket extracts the labels to `strings.xml` it must update both `WelcomeScreen.kt` and `WelcomeScreenTest.kt` in the same change — the test won't auto-rebind. (The natural follow-up is an i18n-prep epic that drains every hardcoded English literal in `ui/`; until then, test literals shadow source literals.)
- **`createComposeRule()` is sufficient — no `Activity` dependency.** `WelcomeScreen` is a pure stateless Composable with no `LocalContext.current` reads, no `Intent` launches, no `Activity`-scoped lifecycle. `createComposeRule()` (not `createAndroidComposeRule<…>()`) keeps the test rule overhead minimal. The `Intent.ACTION_VIEW` dispatch for `onSetup` lives at the *call site* in `MainActivity.kt`, not inside the screen — passing `onSetup = { setupCount++ }` exercises the same surface as production without needing a real `Context`. This is the value the architect called out in `features/welcome-screen.md:36-44`: keeping `WelcomeScreen.kt` free of `NavController` / `Intent` references is what makes the test rule choice this simple.
- **No `performScrollTo()` needed — both CTAs render in the visible viewport at all test densities.** Unlike `SettingsScreen` (whose About rows sit below the fold on a 412dp viewport, per #90's lesson), `WelcomeScreen` uses `Column(SpaceBetween)` with the hero anchored to the top and the CTA stack to the bottom inside a `fillMaxSize` `Box` — both CTAs are always in the rendered frame. The architect explicitly called this out in the spec. Skipping `performScrollTo()` is the right call only because the layout guarantees viewport visibility; never skip it speculatively on a `verticalScroll` surface.
- **No build-file change required.** `androidx.compose.ui.test.junit4` + `…ui.test.manifest` landed in `app/build.gradle.kts` during #90 and were reused for #91, #99, and now #100. Same heuristic as #99: grep `app/build.gradle.kts` for `androidx.compose.ui.test` before assuming a new instrumented test needs build deps. The pattern of "first test wires the deps, subsequent tests freeride" has now held across four classes — solid until a test rule requires something genuinely new (e.g. Espresso interop for `Intent` capture).
