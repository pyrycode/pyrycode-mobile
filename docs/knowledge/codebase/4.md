# #4 — `FakeConversationRepository` (empty-state scaffold)

Phase 1 in-memory implementation of `ConversationRepository`. Returns empty state on first observation; the seed-data tickets (#5, #6) and message-history tickets (#9, #10) will populate it later behind the same interface. The Ktor-backed remote impl lands in Phase 4.

## Summary

- New file `app/src/main/java/de/pyryco/mobile/data/repository/FakeConversationRepository.kt` (155 LOC) plus JUnit 4 tests at `app/src/test/.../FakeConversationRepositoryTest.kt` (118 LOC).
- Single state-holder: `MutableStateFlow<Map<String, ConversationRecord>>`. `observeConversations` derives via `Flow.map`; mutators atomic via `Flow.update {}`.
- `observeMessages` returns `flowOf(emptyList())` for any id — real message storage and `SessionBoundary` interleaving are #9's problem; deliberately not wired to `state` to avoid misleading consumers.
- `changeWorkspace` and `startNewSession` share a private `mintNewSession(...)` helper — workspace change is a session-boundary reason per CLAUDE.md, so both paths mint a fresh `Session` and close out the previous one (`endedAt = now`).
- No DI wiring yet (no Koin module exists). The binding lands with the first ViewModel that consumes the fake.

## Patterns established

- **`ConversationRecord` private wrapper for fake storage.** `Conversation` only stores session **ids**; the fake needs full `Session` values so `startNewSession`/`changeWorkspace` can return them. The wrapper is `private` to the file — it is a storage detail, not a domain concept, and nothing else in the repo references it. Future fakes that need to carry per-entity satellite state should reach for the same shape.
- **Atomic `Flow.update {}`, no `Mutex`.** `MutableStateFlow.update {}` is compare-and-set with retry; no multi-step invariants in this class span two `update` calls, so an explicit `Mutex` would be cost without payoff. New mutators on this class should stay inside a single `update` block; if a flow ever needs to span two, that is the signal to revisit.
- **Unknown `conversationId` throws `IllegalArgumentException`.** Caller passed an id not from `observeConversations` — programmer error, not a runtime "not found" lookup. The interface doc (`features/conversation-repository.md`) originally said `NoSuchElementException`; the fake establishes the binding rule and the feature doc was updated to match. Phase 4's remote impl will distinguish "not found" vs network failure, but at that point we have a real not-found semantic to model.
- **`archive` does a separate membership check before `state.update { it - id }`.** The `Map.minus` operator is a no-op for missing keys, so without the explicit `if (id !in records) throw` line, `archive("nope")` would silently succeed — the same class of silent-failure bug `IllegalArgumentException` is meant to surface for the other mutators. Pattern: when a mutator's natural impl would no-op on a missing id, guard it explicitly.
- **JUnit 4 + `runBlocking` + `Flow.first()` for `StateFlow`-derived flows.** No `kotlinx-coroutines-test`. The fake's flows always have a current value (cold `Flow.map` over a `StateFlow`), so `first()` returns immediately. Save `runTest` / virtual time for the day a fake actually needs to coordinate `delay`.
- **`lateinit var` to escape values out of `update { ... }`.** `Flow.update`'s lambda returns the new state, not the entity to return to the caller. The mutator declares `lateinit var updated: Conversation` (or `newSession: Session`) before the `update` block and assigns inside. The update lambda is run on the successful CAS path only — under contention the block re-runs, reassigning the `lateinit` each time, which is benign because every retry sees the same input id and produces the same logical output.

## Lessons learned

- **`Conversation.id` is `currentSessionId` is `String`, not a session value.** Easy mis-read: the spec snippet calls it "`currentSessionId`" and the test asserts `assertNotEquals(created.currentSessionId, newSession.id)`. Both sides are `String`. Worth a glance at `data/model/Conversation.kt` before assuming the type of any id field on this class.
- **`flowOf(emptyList())` is a *cold* flow that emits once and completes** — `observeMessages` consumers that hold the subscription expecting future emissions will see completion at this stage. That is fine for #4 because no consumer of `observeMessages` exists yet; #9 swaps this for a `state`-derived hot stream. If a Phase 1 UI ticket lands ahead of #9 and binds to `observeMessages`, the binding will need to assume single-shot empty until #9 reworks the implementation.
- **Architect note: do not pre-factor.** The spec called out `mintNewSession` as "the only abstraction the file should introduce." Three other mutators have visibly similar `state.update { records -> records[id] ?: throw unknown(id); ... }` shapes — resist extracting them. Two call sites of an identical shape isn't a pattern; three is. (Same simplicity-first rule the pipeline already enforces.)
- **No new dependency this ticket.** Spec was explicit: tests use `runBlocking` from `kotlinx-coroutines-core` (already on classpath). `kotlinx-coroutines-test` is deferred until virtual-time / `delay` coordination is actually needed.
