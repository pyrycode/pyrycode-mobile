# #25 — 'Save as channel' affordances on discussion rows (long-press + swipe)

Two gesture entry points on each row of `DiscussionListScreen` — long-press opens a Material 3 `DropdownMenu` with a single "Save as channel…" item; right-to-left swipe reveals the same label on a trailing `secondaryContainer` background. Both gestures funnel into one new `DiscussionListEvent.SaveAsChannelRequested(conversationId)` variant that the VM swallows as `Unit` (`TODO(phase 2)` marker). The shared `ConversationRow` gains one optional parameter (`onLongClick: (() -> Unit)? = null`) — every other call site is untouched because the default is `null`.

## Summary

- Extended `sealed interface DiscussionListEvent` (in `DiscussionListScreen.kt`) with `data class SaveAsChannelRequested(val conversationId: String)`. Existing `RowTapped` and `BackTapped` arms are unchanged.
- `ConversationRow` gained `onLongClick: (() -> Unit)? = null` and switches its modifier chain between `Modifier.clickable(onClick)` (when `onLongClick == null`) and `Modifier.combinedClickable(onClick, onLongClick)` (when non-null). Annotated `@OptIn(ExperimentalFoundationApi::class)` for `combinedClickable`. All previously-existing call sites (`ChannelListScreen`, the two `ConversationRow.kt` `@Preview`s, the `DiscussionRowVsChannelRowPreview` in `DiscussionListScreen.kt`) compile unchanged.
- New private composable `DiscussionRow(discussion, onTap, onSaveAsChannel, menuInitiallyExpanded = false)` inside `DiscussionListScreen.kt` wraps every discussion row in: outer `SwipeToDismissBox` (swipe surface) → inner `Box` (long-press menu anchor) → `ConversationRow` + `DropdownMenu`. Replaces the previous direct `ConversationRow(...)` call in the `Loaded` branch's `items { }` lambda. The `Modifier.alpha(0.65f)` de-emphasis from #24 is now applied inside `DiscussionRow`'s inner `ConversationRow` call — unchanged behavior, just moved one layer inward.

## Patterns established

- **Optional `onLongClick: (() -> Unit)? = null` on shared row primitives.** When one call site of a shared component needs a long-press gesture and others don't, add a nullable lambda parameter that toggles the modifier chain between `clickable` and `combinedClickable`. The cost is one default-valued parameter and a two-branch `if`; the fan-out into existing call sites is zero. Reject the alternatives that look cleaner at first glance: a screen-level `Box { combinedClickable(...); ConversationRow(...) }` wrapper does **not** work because `ConversationRow`'s own inner `Modifier.clickable` consumes pointer events before the outer modifier sees them. Custom `pointerInput { awaitEachGesture { … } }` plumbing reimplements `combinedClickable` and is brittle. The optional-lambda pattern is the canonical Compose convention for gesture-bearing primitives and is what `Modifier.combinedClickable` was designed for. Apply the same shape if/when `ChannelListScreen` eventually needs its own long-press action (rename? archive?) — don't merge the two screens' gesture handlers into a shared wrapper until they share the same action.
- **Two-layer composition for gesture-bearing list rows: outer `SwipeToDismissBox`, inner `Box` as the dropdown anchor.** The `Box` is required as a separate node because `DropdownMenu` anchors to the nearest `Composable` parent — placing it as a sibling of `ConversationRow` inside a bare `SwipeToDismissBox.content` lambda would anchor it to the dismiss box's clipped surface and produce wrong positioning during a swipe-in-progress. The `Box { ConversationRow(...); DropdownMenu(...) }` keeps the menu positioned relative to the row's resting frame.
- **`confirmValueChange = { … false }` to make a swipe gesture purely a trigger.** When the swipe should fire an action but the underlying item must remain visible (no actual dismissal — common for Phase 0 stubs and undo-able destructive actions), invoke the side effect inside `confirmValueChange` and return `false`. The `SwipeToDismissBoxState` then springs back to `Settled` on its own. Don't try to drive the spring-back via `state.reset()` — that's a `suspend` call and `confirmValueChange` is synchronous. If a Phase 2 ticket adds an undo affordance, switch to returning `true` and manually `state.reset()` from a `LaunchedEffect` after the dialog opens.
- **`menuInitiallyExpanded: Boolean = false` parameter on row composables to make `DropdownMenu`s preview-visible.** The static `@Preview` system cannot drive runtime gestures, but it can pass a non-default value to a parameter that seeds `remember` state. Use this parameter (production callers omit it, accept the default `false`) instead of an interactive preview — interactive previews are heavyweight and don't render in the file's preview pane. The acceptance-criterion clause "interactive preview acceptable if static can't show the gesture" is permissive but not preferred; static-with-seed is cheaper.
- **No-op event arms remain in the sealed `when` with a `TODO(phase N)` line.** When a sealed event variant ships ahead of its handler (gesture surface in Phase 0, business logic in Phase 2), the VM arm is the literal token `Unit` with a `// TODO(phase 2): …` comment that names the eventual repository call and its parameters. Don't pre-build the suspend launch, don't pre-add the repository method to the interface, don't seed an analytics call. The `TODO` comment is the entire forward-looking surface; the rest is YAGNI. This convention extends the #24 pattern (`BackTapped -> Unit` for "composable handles it") to "no handler exists yet" — same shape, different reason for the no-op.

## Lessons learned

- **`combinedClickable` requires `@OptIn(ExperimentalFoundationApi::class)` even at Compose BOM 2026.02.01.** `Modifier.clickable` is stable; `Modifier.combinedClickable` is not. The annotation lives at the `@Composable` function level. If/when the parent component is annotated, prefer keeping the `@OptIn` local to the function that needs it — propagating experimental opt-ins upward is a slippery slope. The opt-in does **not** make the API likely to break in this BOM; it's an experimental tag on a stable-in-practice surface.
- **`Modifier.clickable` on the inner `ListItem` of `ConversationRow` shadows any outer pointer-input wrapper.** This is *why* the gesture parameter has to live on `ConversationRow` itself, not on a screen-side wrapper. If a future ticket tries to add a tap-gesture wrapper (e.g., for analytics) at the call site, the outer wrapper will not see clicks unless `ConversationRow`'s inner clickable is removed or that gesture is also threaded through the row's public surface. The straightforward way to add a new gesture-bearing surface is the same pattern as `onLongClick`: a new nullable lambda on `ConversationRow` that switches the modifier branch.
- **`SwipeToDismissBox` (M3) is the right primitive for one-direction swipes at this BOM.** Two known caveats. (1) The `backgroundContent` slot is only visible during an in-progress swipe — it is not a static decoration. The `Box(modifier = Modifier.fillMaxWidth().background(secondaryContainer))` surface is what the user sees *behind* the row as they pull it left; once the row springs back, the background is occluded again. Trailing-aligned `Text` is the right shape — leading-aligned text would only be visible at extreme swipe distances and looks broken. (2) `enableDismissFromStartToEnd = false` is necessary in addition to a `confirmValueChange` that rejects `StartToEnd` — without the `enable` flag, the user can still drag the row partway in the disallowed direction (it just snaps back), which reads as a buggy gesture. Set both flags explicitly.
- **`SwipeToDismissBoxValue.EndToStart` is right-to-left in LTR locales (the user's finger swipes from the right edge toward the left).** The naming reads backward — "end to start" sounds like the destination is the start edge, which it is. In LTR Android, end = right, start = left, so the gesture pulls left. Material 3 documents this direction as the trailing/destructive action (Gmail archive, system notification dismiss). The leading direction (`StartToEnd`, left-to-right finger drag) is conventionally a confirm/complete action and is reserved for a future "promote-without-naming" shortcut if it ever ships. Don't swap the directions on aesthetic grounds.
- **`remember(discussion.id) { mutableStateOf(...) }` is the correct keying for per-row UI state in a `LazyColumn`.** Without the `key` argument on `remember`, the state survives across recompositions correctly but does **not** reset if the list re-keys (e.g., the same composition slot is reused for a different `discussion.id` after a list re-emission). `LazyColumn`'s `items(..., key = { it.id })` does its own composition keying, so the `remember(discussion.id)` is belt-and-suspenders — but it's the belt that protects against the `LazyColumn` key being omitted by a future refactor. Cheap defensive idiom; keep it.
- **Anchoring `DropdownMenu` to a `Box` that wraps the row (not the row itself) is the working position.** `DropdownMenu` looks for the nearest composable parent's coordinates to position itself. Placing it inside the `ConversationRow` would couple positioning to whatever internal layout the row uses (currently `ListItem`); placing it as a sibling of `ConversationRow` inside a wrapping `Box` keeps it anchored to the row's outer frame. Tested by opening the menu while the row is mid-swipe — anchored to outer frame, so it travels with the row, which is the correct UX.
- **`Unit` as a single-token `when` arm trips no compiler warnings.** Idiomatic Kotlin for "explicit no-op branch in an exhaustive `when`". Don't replace with `{}` (empty block, looks like a forgotten body) or `return` (only valid in non-expression `when` and confuses the eye). Compare with the existing `DiscussionListEvent.BackTapped -> Unit` shape from #24 — same idiom, different reason.
