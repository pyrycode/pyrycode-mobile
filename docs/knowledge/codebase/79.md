# #79 — Adaptive icon resource scaffolding (placeholder)

Replaces the Android Studio template's default adaptive-icon scaffolding with a placeholder "P"-glyph icon whose background is externalised to a named color resource, so the eventual branding ticket is a one-drawable swap. Resource-only — no Kotlin, no manifest, no Gradle changes. Min-SDK 33 means the adaptive icon is the sole launcher icon source; the per-density `.webp` fallbacks are gone.

## Summary

- Adaptive-icon XMLs moved from `app/src/main/res/mipmap-anydpi/` to `app/src/main/res/mipmap-anydpi-v26/` (the qualifier `<adaptive-icon>` actually requires — it was introduced in API 26). The two XMLs' three-layer structure (`<background>` / `<foreground>` / `<monochrome>` all pointing at `@drawable/ic_launcher_{background,foreground}`) is unchanged; only the parent directory moved.
- `app/src/main/res/drawable/ic_launcher_background.xml` rewritten from a 170-line green-grid vector to a 10-line single-path solid fill whose `android:fillColor="@color/ic_launcher_background"` references the new color resource.
- `app/src/main/res/drawable/ic_launcher_foreground.xml` rewritten from the green Android-robot vector to an 11-line solid-white "P" glyph (`fillType="evenOdd"`, single path with the bowl cut out). Glyph sits in the centre-72 region of the 108-viewport, well inside the 66dp safe-zone circle. White fill is irrelevant in themed-icon mode (system renders monochrome layer as alpha silhouette) but gives high contrast in regular mode against the blue background.
- `app/src/main/res/values/colors.xml` gains `<color name="ic_launcher_background">#FF32628D</color>` with a comment explaining the duplication with `primaryLight` in `ui/theme/Color.kt` and the reason it can't be a theme attribute. The template-residue `purple_*` / `teal_*` / `black` / `white` entries are left untouched — they're not consumed by Compose code (`MaterialTheme` reads from `Color.kt`), so removing them is out of scope.
- 10 raster `ic_launcher*.webp` files deleted across the five density buckets (`mipmap-{m,h,xh,xxh,xxxh}dpi/`); the empty parent directories are removed from the filesystem too. The adaptive icon in `mipmap-anydpi-v26/` is the sole launcher icon source on min-SDK 33.

## Files touched

- `app/src/main/res/mipmap-anydpi-v26/ic_launcher.xml` — moved from `mipmap-anydpi/`, content unchanged.
- `app/src/main/res/mipmap-anydpi-v26/ic_launcher_round.xml` — moved from `mipmap-anydpi/`, content unchanged.
- `app/src/main/res/drawable/ic_launcher_background.xml` — full rewrite, 10 lines.
- `app/src/main/res/drawable/ic_launcher_foreground.xml` — full rewrite, 11 lines. Path data: `M30,18 H60 a18,18 0 0 1 0,36 H42 V90 H30 Z M42,28 H58 a8,8 0 0 1 0,16 H42 Z` (outer "P" shape with the bowl as an evenOdd cutout).
- `app/src/main/res/values/colors.xml:10-17` — new `ic_launcher_background` color + duplication-warning comment.
- `app/src/main/res/mipmap-mdpi/`, `mipmap-hdpi/`, `mipmap-xhdpi/`, `mipmap-xxhdpi/`, `mipmap-xxxhdpi/` — directories deleted (each contained only the two now-removed `.webp` files).
- `app/src/main/res/mipmap-anydpi/` — directory deleted (now empty after the v26 move).

No manifest changes. `android:icon="@mipmap/ic_launcher"` + `android:roundIcon="@mipmap/ic_launcher_round"` continue to resolve transparently through Android's resource fallback chain to `mipmap-anydpi-v26/`.

## Patterns established

- **Adaptive-icon XMLs live in `mipmap-anydpi-v26/`, never `mipmap-anydpi/`.** The `-v26` qualifier is the documented way to declare "this resource requires API 26 or higher" (which is when `<adaptive-icon>` shipped). `mipmap-anydpi/` happens to work today only because we're min-SDK 33 and the resource resolver finds it as the only candidate — a hypothetical API-25 device would crash trying to parse the tag. When the branding ticket lands, drop the new drawables into the same `-v26` directory.
- **Launcher icon background goes through `@color/ic_launcher_background`, never an inline hex literal in the drawable.** Single edit point for branding. The color resource also acts as the documentation anchor for the cross-file duplication with `ui/theme/Color.kt` (`primaryLight`) — the comment in `values/colors.xml` is the load-bearing pointer that the next branding ticket must update both locations.
- **Don't try to reference theme attributes (`?attr/colorPrimary`) in launcher drawables.** Launchers render the icon outside the app process, so theme attributes don't resolve. Use a static `@color` resource. This is the only resource in the project where the "duplicate the M3 seed" pattern is the correct choice instead of binding through the theme — flagged in the colors.xml comment for the same reason.
- **Reuse the foreground vector as the monochrome layer.** The system renders the monochrome layer as a colour-stripped silhouette, so whatever fill the foreground uses is ignored in themed-icon mode; the alpha shape is what matters. Keeps the adaptive-icon XML to three lines of layer references and the drawable count to two. Only split into a separate monochrome vector if the foreground silhouette genuinely doesn't read at the launcher's themed-icon scale.

## Lessons learned

- **The per-density `mipmap-*dpi/` raster fallbacks are dead weight on min-SDK 26+ — and definitely on min-SDK 33.** Android Studio's project template seeds them by default (with green-robot `.webp` placeholders) because the IDE supports older min-SDKs. On min-SDK 33 they're never read: every supported device renders the adaptive icon from `mipmap-anydpi-v26/`. Deleting them shrinks the APK and removes a swap surface for the branding ticket (one drawable to replace, not eleven). If a future ticket lowers min-SDK below 26, the fallbacks would need to come back as PNGs/WebPs — but that walk-back is out of scope.
- **`fillType="evenOdd"` is the right tool for "letter P with a bowl cut out" in a single path.** The alternative — two separate `<path>` elements with the bowl as a hole — works but doubles the path count and complicates the monochrome story. `evenOdd` keeps the glyph one path with the outer "P" outline traced first and the bowl interior traced as a closed sub-path; the renderer XORs the fill state at each path boundary so the bowl reads as transparent. Same trick generalises to any "letterform with an enclosed counter" (B, D, O, R, etc.) for future placeholder icons.
- **Adaptive-icon safe zone is 66dp inside a 108dp viewport — but target ~72×72 in the centre for breathing room.** The 66dp circle is what the system *guarantees* survives any launcher mask (square, circle, squircle, teardrop, etc.); art that touches the safe-zone edge feels cramped at launcher size. The placeholder "P" path data (`M30,18 ... V90 H30`) spans roughly 30×72 in viewport units, sitting comfortably inside the inner circle even after the launcher applies its mask. No `<aapt:attr>` gradient blocks, no `strokeColor` decorations, single fill — adaptive icons render at tiny sizes and any decoration smaller than ~2dp at launcher scale will alias into noise.
- **Lint stays quiet about `IconMissingDensityFolder` / `IconLauncherShape` once the adaptive icon supersedes the rasters.** During the WIP state where the XMLs were already moved but the `.webp` files were still present, lint flagged the inconsistency (rasters in density buckets but no adaptive icon in `mipmap-anydpi/`). Once the rasters are gone *and* the adaptive icon is in the right qualifier, lint is silent. If a future ticket sees a new icon-related lint warning, treat it as a spec gap rather than reaching for `tools:ignore` — the warnings here are usually pointing at a real fallback chain problem.
