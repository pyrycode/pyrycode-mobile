# #191 — feat(data): structured tool-call payload for tool messages

Lands the data-layer shape `Role.Tool` messages need so [`ToolCallRow`](https://github.com/pyrycode/pyrycode-mobile/issues/131) (#131) can render a collapsed summary + expandable details view without parsing free-form strings. One new `data class ToolCall(toolName, input, output)` co-located with [`Message`](../features/data-model.md), one new nullable field `toolCall: ToolCall? = null` on `Message`, and one realistic seeded tool message in `seed-channel-pyrycode-mobile`'s current session. No UI rendering — `MessageBubble.kt`'s `Role.Tool -> Unit` no-op arm is unchanged; #131 replaces it.

## Summary

- **Nullable field, not sealed hierarchy.** The ticket body handed the architect the shape choice and pre-quantified the cost of the sealed-hierarchy alternative (≥16 `seedMsg` / `Message(...)` call sites, 5 files, two test files — trips PO's >3-file and >10-call-site red lines). The architect picked nullable-field at S size: 2 production files + 1 test file + ~30 LOC + zero forced consumer edits (the default value absorbs every existing positional and named `Message(...)` call site). See § Why nullable, not sealed for the trade-off and the consequence for #131.
- **`ToolCall` is three non-null `String`s.** `toolName`, `input`, `output` — exactly the three pieces the ticket names. **No** `output: String?` for "tool call in flight, no response yet" — that's a Phase 4 streaming concern; modelling it now would defend a failure mode that hasn't been observed.
- **`toolCall: ToolCall? = null` as the LAST field on `Message`, defaulted.** Load-bearing:
  - All existing positional `Message(...)` call sites continue to type-check unchanged — notably `FakeConversationRepositoryTest.observeMessages_messagesAcrossTwoSessions_emitsExactlyOneBoundary` which uses six positional args. The default fills the new slot.
  - All existing named `Message(name = …)` call sites in `seedChannel`, `seedDiscussion`, `sendMessage`, and `previewMessage` in `MessageBubble.kt` continue to type-check unchanged — none pass `toolCall`, so the default applies.
  - `data class` auto-generated `copy()`, `equals`, `hashCode`, and `toString` include the new field uniformly. `MessageTest` is not modified — its fixture has `toolCall = null` on both sides of every comparison.
- **Invariant `toolCall != null iff role == Role.Tool` is documented, not enforced.** One-line KDoc on the field (`/** Non-null iff [role] is [Role.Tool]. */`) — no `init { }` block, no `require(...)`. Reasoning: `FakeConversationRepository` is the sole `Message` constructor entry point in Phase 0; there is no external producer that could violate the invariant, so a runtime check would defend an unobserved failure mode. If Phase 4's remote impl introduces a path where the invariant could drift, add the check then.
- **Seed-helper pipe-through is mechanical.** Private `SeedMessage` data class and `seedMsg(...)` factory both gain `val toolCall: ToolCall? = null` as the last field/param; three `Message(...)` constructions inside `seedChannel` (historical + current) and `seedDiscussion` propagate `toolCall = msg.toolCall` as the last named arg. All other fields untouched.
- **One seeded tool message in `seed-channel-pyrycode-mobile`.** Inserted in the current session between the `2026-05-10T08:58:00Z` user message and the `2026-05-10T09:00:00Z` streaming assistant message, at `2026-05-10T08:59:00Z` — preserves chronological order, doesn't disturb the single existing session boundary. `toolName = "Read"`, `input = "app/src/main/java/de/pyryco/mobile/ui/conversations/thread/ThreadScreen.kt"`, `output` is a small Kotlin `@Composable fun ThreadScreen(...)` excerpt (multi-line `"""…""".trimIndent()`). One message is sufficient for AC3 — more tool messages, tool messages in other channels, and tool messages in historical sessions are all #131's call.
- **`Message.content` slot on the tool message stays sensible.** Set to `"Read app/src/main/java/de/pyryco/mobile/ui/conversations/thread/ThreadScreen.kt"` — a short summary string. #131's renderer may or may not use it; the seed still reads coherently if a future consumer projects only `content`.
- **One new test in `FakeConversationRepositoryTest`.** `observeMessages_emitsSeededToolMessage_withStructuredPayload` collects `repo.observeMessages("seed-channel-pyrycode-mobile").first()`, filters to `Role.Tool` `ThreadItem.MessageItem`s, asserts exactly one, then `assertEquals` on each of `toolCall.toolName` / `toolCall.input` / `toolCall.output` against the same string literals the seed uses (the test is a round-trip on the structured payload, not a render check). Existing tests are not modified — none assert on tool-message content, and the new seed message doesn't disturb session-boundary count or chronology.
- **No interface widening, no stub cascade.** `ConversationRepository` is untouched, so the `TODO("not used")` cascade pattern (#161, #187) does **not** apply here. No `object : ConversationRepository { … }` test stubs need editing.
- **`MessageBubble.kt` is not edited.** The `Role.Tool -> Unit` arm at line 56 keeps the `when (message.role)` exhaustive over the unchanged `Role` enum; the build stays green without touching the file. Under the nullable-field shape, the existing `when` arm does **not** become a compile-time exhaustiveness error — see § Why nullable, not sealed.

No new imports — existing `runBlocking`, `assertEquals`, `assertNotNull`, `Role`, `ThreadItem` cover the new assertions.

## Patterns established

None new — this ticket is a pure shape addition with no new wiring idiom. The nullable-defaulted-last-field idiom matches the precedent already in place on `Conversation` for `isSleeping: Boolean = false` (#20) and `archived: Boolean = false` (#93): each new field lands at the **end** of the data class with a default so positional and named call sites stay source-compatible across the codebase. Treat that as the default shape for any future `Message` / `Conversation` / `Session` field that's nullable or has an unambiguous "absent" value.

## Lessons learned

None new — the implementation matched the spec exactly. Two reminders worth restating against the next data-shape addition:

- **Defaults at the END of the data class are the source-compat lever.** The reason this ticket stays at 3 files is that the new field is last in the constructor *and* defaulted. Inserting `toolCall` mid-constructor (between, say, `role` and `content`) would have broken `FakeConversationRepositoryTest.observeMessages_messagesAcrossTwoSessions_emitsExactlyOneBoundary`'s six-positional-arg call (the new value would have landed in the `content` slot). Mid-constructor data-class fields are a force-multiplier for cascading test edits; always append.
- **Documented invariants beat runtime-enforced invariants in Phase 0.** The spec explicitly chose the KDoc form over an `init { }` guard because the only `Message` producer is `FakeConversationRepository`. The Evidence-Based Fix Selection principle (CLAUDE.md) makes the call: defend observed failure modes, not speculative ones. When the Phase 4 remote impl lands, the call site that constructs `Message` from wire data is the right place to enforce the invariant — not the data-class constructor in `data/model/`, which would still be redundant against a well-behaved local producer.

## Why nullable, not sealed (and what this means for #131)

The ticket body explicitly handed the shape choice to the architect, naming both alternatives:

> "either type-level via sealed/sibling type, or a non-null `toolCall` field on `Message`."

And it pre-sized the trade-off (S size only if nullable; sealed promotes to >S with a forced split). The architect chose nullable. The consequence for the downstream consumer is non-trivial and worth surfacing here:

**The ticket body claimed:** "#131 expects the existing `Role.Tool -> Unit` no-op arm in `MessageBubble.kt:56` to become a compile-time exhaustiveness error once this ticket lands."

**That doesn't happen under nullable-field.** The `Role` enum is unchanged. The `when (message.role) { Role.User -> …; Role.Assistant -> …; Role.Tool -> Unit }` arm remains a valid (and reachable) branch. #131 will replace the `Role.Tool -> Unit` body with a real `ToolCallRow(message.toolCall!!)` call (or via a helper like `requireToolCall(message): ToolCall`), but the **shape** of `MessageBubble.kt`'s dispatch doesn't change — three `when` arms over the same three `Role` values.

If, when #131 runs, the consuming agent finds the lack of type-level discrimination genuinely painful (rather than stylistically unfortunate), they can route back through PO to split out a sealed-hierarchy promotion ticket. That decision is theirs, not this ticket's — by deferring the type-shape decision, this ticket avoids pre-paying a refactor cost on speculation.

Promoting later (when concrete pain is observed) costs the same as promoting now would have: `Message` becomes a `sealed interface` with `Message.UserOrAssistant(role, content, …)` and `Message.Tool(toolCall, …)` variants, every `seedMsg` / `Message(...)` call site picks the right variant, and `MessageBubble.kt`'s `when` switches from `message.role` to `message`. The `toolCall: ToolCall?` field added here becomes the payload of the `Tool` variant. Nothing wasted; the deferred refactor is the same refactor.
