# #139 — ThreadScreen TopAppBar (back + conversation name + overflow)

Replaces the file-private placeholder `ThreadTopBar` introduced in [#126](./126.md) with the real top bar from Figma node `16:8`. Surfaces the conversation's `displayName` instead of the raw `conversationId` and exposes lambda hooks (`onTitleClick`, `onOverflowClick`) for the downstream rename dialog ([#141]) and context-aware overflow menu ([#140]) to wire — defaulted to `{}` no-ops in this ticket so `MainActivity` stays untouched.

## Summary

- New stateless `@Composable fun ThreadTopAppBar(title, onBack, onTitleClick, onOverflowClick, modifier)` in its own file (`ui/conversations/thread/ThreadTopAppBar.kt`). Public visibility (the spec called this out as deliberate so a future preview can pin the bar shape independently). `TopAppBar` defaults are kept — no `colors = TopAppBarDefaults.topAppBarColors(...)` override (`colorScheme.surface` already matches Figma `Schemes/surface`), no `scrollBehavior`, no `windowInsets` override. The `Text` title carries a `Modifier.clickable(onClick = onTitleClick).semantics { role = Role.Button }` so the ripple aligns with the visible text bounds and TalkBack announces a button role.
- `ThreadUiState` widens to `data class ThreadUiState(conversationId: String, displayName: String)`. `displayName` is **always a `String`, never `null`** — the fallback chain (`conv?.displayName() ?: conversationId`) collapses to the raw id when the lookup misses, so the `Text(title)` composable never needs a null check.
- `ThreadViewModel` widens to `(savedStateHandle, repository: ConversationRepository)` and switches its `state: StateFlow` from `MutableStateFlow(initial).asStateFlow()` to a `stateIn(viewModelScope, WhileSubscribed(5_000), initialValue = ThreadUiState(id, displayName = id))`-materialised flow derived from `repository.observeConversations(ConversationFilter.All).map { list -> ThreadUiState(id, list.firstOrNull { it.id == id }?.displayName() ?: id) }`. The placeholder `displayName = conversationId` initial value matches what the screen shows before the upstream's first emission lands; the fake's `MutableStateFlow.map` chain emits synchronously, so in practice the placeholder is invisible.
- `private fun Conversation.displayName(): String` is **re-declared at file scope** in `ThreadViewModel.kt` (same shape as the existing private extension in `ArchivedDiscussionsScreen.kt:164-166`). The decision to **not extract** to a shared helper is intentional: extraction would push the spec to a 5-`.kt`-file diff (busting the S-budget), and the canonical "extract on third use" threshold isn't crossed yet. The third caller is most likely [#140]'s overflow menu's "Rename …" label, which is the ticket that extracts.
- `ThreadScreen` signature gains two **defaulted** lambdas (`onTitleClick: () -> Unit = {}`, `onOverflowClick: () -> Unit = {}`) so the destination block in `MainActivity.kt` keeps its existing two-argument call. The file-private `ThreadTopBar` placeholder is deleted; eight now-unused imports (`Icons`, `AutoMirrored.Filled.ArrowBack`, `Icon`, `IconButton`, `Text`, `TopAppBar`, `stringResource`, `de.pyryco.mobile.R`) are removed. Both `@Preview`s widen the `ThreadUiState(...)` constructor to add `displayName = "kitchenclaw refactor"` against a `seed-channel-personal` id.
- Koin binding one-line edit: `viewModel { ThreadViewModel(get(), get()) }`. The second `get()` resolves to the `FakeConversationRepository` singleton already bound at `AppModule.kt:28`. No new dependencies, no `gradle/libs.versions.toml` edit.
- New string `cd_more_actions = "More actions"` in `values/strings.xml`. `cd_back` is reused. Naming follows the existing `cd_*` content-description convention; deliberately generic so a non-thread overflow can reuse it later.
- `ThreadViewModelTest` rewrites the three pre-existing tests to use `Dispatchers.setMain(UnconfinedTestDispatcher())` / `resetMain()` and `runTest { launch { vm.state.collect {} }; advanceUntilIdle(); … }` — same idiom as `ChannelListViewModelTest`. Seven tests total (initial-value placeholder, channel happy path, discussion fallback, channel fallback via a stub repo, missing-conversation fallback, **rename re-emit**, absent-key empty string).
- `MainActivity.kt` is **not modified**. The destination block at `MainActivity.kt:193-203` keeps its two-argument call to `ThreadScreen(state, onBack)`; `onTitleClick` / `onOverflowClick` fall through to their `{}` defaults until [#141] / [#140] wire real handlers.

## Patterns established

- **Defaulted `() -> Unit = {}` lambdas at the screen signature, not hoisted lambdas at `MainActivity`.** When a ticket lands the entry-point hook for a downstream feature (here: rename + overflow menu), keep the hook caller's site narrow — defaulted lambdas on the screen signature let the placeholder no-op live *in* the screen rather than being wired through every consumer. `MainActivity` stays untouched until the downstream ticket has a real handler. This is the cheapest path that satisfies "wired to no-op stubs in `ThreadScreen`" literally. The first established case in `ChannelListScreen`'s optional `onEvent` default is the precedent; this ticket pins the shape for thread-area chrome.
- **Synchronous → `stateIn(WhileSubscribed)` is an in-place upgrade, not a rewrite.** When a previously-synchronous VM (`MutableStateFlow(initial).asStateFlow()` from [#126](./126.md)) grows its first cold-flow upstream, switching to `stateIn(viewModelScope, WhileSubscribed(5_000), initialValue = <pre-flow snapshot>)` keeps the `StateFlow<UiState>` public surface byte-identical. The destination consumer (`val state by vm.state.collectAsStateWithLifecycle()`) does not change. Pick `initialValue` to match what the screen would show before the upstream's first emission — for this VM, the `displayName = conversationId` fallback is the same value the post-emission flow renders when the id is missing, so the placeholder is the steady-state under any path.
- **Re-declare the private `Conversation.displayName()` extension at file scope rather than extracting to a shared helper — until the third caller arrives.** The same `name?.takeIf { it.isNotBlank() } ?: if (isPromoted) "Untitled channel" else "Untitled discussion"` shape now lives in two places (`ArchivedDiscussionsScreen.kt:164-166`, `ThreadViewModel.kt`). Two occurrences is below the canonical extract-on-third-use threshold, and the spec's S-budget self-check ruled extraction out. When the third caller appears (likely [#140]'s overflow menu's rename label), that ticket extracts to `internal fun` in a sibling under `ui/conversations/` and deletes both copies. Flagged in INDEX.md and in [`features/thread-screen.md`](../features/thread-screen.md#display-name-fallback) so a later editor doesn't lose the breadcrumb.
- **`firstOrNull { it.id == conversationId }` over the full `observeConversations(All)` list is the cheapest plumbing path for surfacing per-conversation projections in a ViewModel.** Three plumbing options were considered (route nav arg, new `observeConversation(id)` interface method, filter-in-VM); option (c) wins on file count (4 `.kt` files vs. 5+) and gets free re-emission on `rename` / `archive` / `unarchive` / any other repo mutation by virtue of sharing the existing `observeConversations` flow. The `O(n)` scan is acceptable at Phase 0 fake-cardinality (~5 records). When the Phase 4 Ktor-backed impl lands and the per-emission scan has a real cost, a follow-up introduces `observeConversation(id): Flow<Conversation?>` as the option-(b) endpoint.
- **`ConversationFilter.All` for ViewModels that consume one specific conversation, not `Channels` / `Discussions`.** The thread screen is reachable from channels, discussions, and (in future) the archive screen + deep links. Restricting to non-archived would silently break the archive entry path. Same reasoning as `FakeConversationRepository`'s `All -> true` branch — pass everything, filter by id in the consumer.

## Lessons learned

- **`stateIn(initialValue = …)` reads synchronously *before* any subscription, but the synchronous read returns `initialValue`, not the upstream's first emission.** The `state_initialValue_isConversationIdPlaceholderBeforeSubscription` test pins this: with no `launch { vm.state.collect {} }`, `vm.state.value` is the `ThreadUiState(id, displayName = id)` placeholder — the upstream `map { … }` has not been subscribed to. The very next test (`state_resolvedTitle_isChannelNameForSeededChannel`) needs the `runTest { launch collector; advanceUntilIdle() }` triple to materialise the resolved title. Worth pinning: a `vm.state.value` assertion at construction time in a `stateIn(WhileSubscribed)` VM tests the *initial value*, not the upstream behaviour. Reviewers seeing a test that only asserts `state.value` post-construction should expect to find a sibling test that exercises the subscription.
- **`Dispatchers.setMain(UnconfinedTestDispatcher())` is required for `stateIn(viewModelScope, …)` to publish emissions synchronously in tests.** Pre-#139 the VM was synchronous (`MutableStateFlow(initial).asStateFlow()`), so plain JUnit tests with no main-dispatcher setup worked. Post-#139 the VM's `viewModelScope` defaults to `Dispatchers.Main.immediate`, and without a test override `viewModelScope.launch { … }` (which `stateIn` triggers internally) deadlocks under `runTest`. The fix is the standard `@Before setMain(UnconfinedTestDispatcher()) / @After resetMain()` pair from `ChannelListViewModelTest:1-60`. Pin: any VM that grows its first cold-flow upstream needs its tests upgraded to the same scaffold simultaneously.
- **Free rename re-emission through option (c) means [#141] does not need to touch `ThreadViewModel`.** When the rename dialog lands, `repository.rename(conversationId, newName)` mutates the fake's `MutableStateFlow<State>`; the upstream `observeConversations(All)` re-emits a list with the renamed conversation; the `.map { … }` recomputes `displayName`; `stateIn` publishes; `collectAsStateWithLifecycle()` triggers recomposition. The `state_displayName_reemitsOnRename` test pins this contract. The ticket-body Out-of-Scope item "Plumbing a live rename back into the title without re-navigating — covered when #141 lands" is actually **already wired** by option (c); [#141]'s only TopAppBar-related work is wiring the dialog's success path to `repository.rename(...)`. Flag for [#141]'s architect: do not duplicate the projection in the rename dialog's VM — the existing flow already feeds the title.
- **Removing unused imports needs an explicit grep — Android Studio "Optimize imports" was not run in this edit.** Eight imports were removed from `ThreadScreen.kt` (`Icons`, `AutoMirrored.Filled.ArrowBack`, `Icon`, `IconButton`, `Text`, `TopAppBar`, `stringResource`, `de.pyryco.mobile.R`); ktlint flagged none of them during the edit because the project doesn't run a `removeUnusedImports` spotless rule. Caught by the developer reading the diff before commit — same lesson as [#126](./126.md)'s `MainActivity.kt` `Text` import removal. Pin: when replacing an inline composable with a sibling-file composable, grep the file for every import that previously backed the inlined call and verify no remaining consumer. A pre-commit `spotless` `removeUnusedImports` rule is a tolerable follow-up (still not in scope here).
- **`androidx.compose.foundation.clickable` on `Text` (not on a wrapping `Row`) gives a tighter ripple bound aligned to text width.** The spec called this out as a deliberate tradeoff for the rename entry: wrapping the title slot in a `Row` with `clickable` makes the ripple span the full title-slot column (which feels sloppy in narrow titles), while `clickable` on `Text` aligns with the visible text bounds plus the small Material ripple padding. Confirmed visually by the M3 default ripple radius — TalkBack still announces the button role via the chained `.semantics { role = Role.Button }` modifier. If [#141]'s UX wants "tap anywhere in the bar to rename", this composable widens the clickable region to a wrapping `Row` — but that's that ticket's call, not this one's.
