# #81 — R8 / ProGuard rules baseline + `assembleRelease` green

Flips the `release` build type to `isMinifyEnabled = true` + `isShrinkResources = true`, wires the implicit debug `signingConfig` so the shrunk APK is installable, and replaces the stock-template `app/proguard-rules.pro` with a documented, per-library rule baseline. The only materially required rule across the currently-wired dependency set is a single `-dontwarn` for the coroutines debug agent — every other section is intentionally empty because the library ships sufficient consumer-rules at the pinned version. Release-keystore signing config and Phase 4 serialization-model rules are explicitly deferred.

## Summary

- `app/build.gradle.kts:24-34` `buildTypes.release` block: `isMinifyEnabled = false` → `true`, added `isShrinkResources = true`, added `signingConfig = signingConfigs.getByName("debug")` (reuses AGP's implicit debug config — *not* a new `signingConfigs { ... }` block, which the ticket forbids). The existing `proguardFiles(getDefaultProguardFile("proguard-android-optimize.txt"), "proguard-rules.pro")` line is unchanged.
- `app/proguard-rules.pro` rewritten from the Android Studio template comment block to a header + four library sections. Header states the policy ("only rules sourced from each library's official documentation, on the version pinned in `gradle/libs.versions.toml`"; `-dontwarn` entries must name the library and the reason; empty sections are intentional and mean "consumer-rules suffice — re-test after every version bump").
- Library sections: **Koin 4.0.4** (empty body, comment explains the runtime DSL + no `koin-annotations`/KSP means no rules are needed; `ChannelListViewModel` / `DiscussionListViewModel` survive R8 because `koinViewModel<T>()` materializes the type token at call sites the compiler already keeps), **Kotlin Coroutines 1.10.2** (single rule: `-dontwarn kotlinx.coroutines.debug.AgentPremain`; comment names the lib + reason), **AndroidX DataStore Preferences 1.1.7** (empty), **Lifecycle / ViewModel / Navigation Compose / Compose BoM / kotlinx-datetime** (single combined empty section, comment notes Navigation Compose registers destinations by string route — no `-keep` needed for screen composables).
- Verified end-to-end on an emulator: `assembleRelease` succeeds with no R8 warnings / missing-class errors, paired-flow advances `Welcome → Scanner → ChannelList`, channel + recent-discussion rows render with fake data intact, tapping into a `ConversationThread` works.
- **Release APK size baseline: 2,402,922 bytes (2.3 MB)**, signed with the debug keystore. First numeric baseline in the repo; future tickets that meaningfully grow this should be flagged in their PR body.

## Patterns established

- **`proguard-rules.pro` is organised by library, with a header per section naming the artifact + the version it was authored against.** Empty sections are kept (not deleted) so the next reader can immediately see "we considered this lib and decided no rules were needed at version X" instead of having to re-derive that. When a dep version bumps, the section header is the cue to re-verify against the new version's docs.
- **Every R8 rule must trace to an official source — the library's README / docs site on the pinned version tag, or `context7` if the README is thin.** No copy-pasted Stack Overflow rules. No "I think we'll need this." If a rule is needed but no official source documents it, that's a signal it's not needed; drop it and let the build tell you.
- **`-dontwarn` is only acceptable with a comment naming the library and the missing class.** The single live entry (`-dontwarn kotlinx.coroutines.debug.AgentPremain`) is the template — name the lib, name the class, one-sentence reason. Bare `-dontwarn` lines are a code-review reject (and AC #3 treats unexplained warnings as failures).
- **Release-keystore signing is *not* a `signingConfigs { create("release") { ... } }` block yet — it's `signingConfigs.getByName("debug")`.** This reuses AGP's auto-generated debug config so the APK is installable on an emulator without introducing real keystore material to source control. The Phase 5 release-keystore ticket is the right place to add a real `signingConfigs` block; until then, "signed with debug keystore" is the intended Phase 0 state.
- **APK size baselines belong in the PR body, not in code or a tracked file.** Future tickets that materially grow the shrunk APK (new dep, real network client in Phase 4, etc.) should note the delta in their PR body against this baseline (2,402,922 bytes). No size-tracking file in the repo until there's a need.

## Lessons learned

- **For Koin 4.x with the explicit-constructor DSL, no R8 rules are needed.** This is the surprising one — historic Koin documentation (1.x / 2.x era) prescribes a sizeable rule set, and a lot of internet copy-paste still references those. Koin 4.x is mostly inline; the runtime reflective surface is `KClass`-based lookup of types Koin itself doesn't need to keep, and the `viewModel { ChannelListViewModel(get()) }` factory lambda materializes the VM constructor at the call site `koinViewModel<ChannelListViewModel>()`. The compiler already keeps both sides. If `koin-annotations` + KSP get wired later, that recalculates — re-read the version-pinned Koin docs at that point.
- **Coroutines' debug agent (`kotlinx.coroutines.debug.AgentPremain`) is a tooling-only entry point absent from the release classpath.** R8 will emit a missing-class warning unless suppressed. `-dontwarn` is correct here (not `-keep`) — the class is never instantiated at runtime in release; we just don't want R8 failing the build over a class that legitimately isn't on the classpath. The rule-of-thumb: missing-class warning *without* reflective lookup = `-dontwarn`; missing-class warning *with* reflective lookup = `-keep` + investigate why a release-classpath class went missing.
- **Compose, Navigation Compose, Lifecycle, DataStore Preferences, and kotlinx-datetime all ship sufficient consumer-rules at the versions pinned in `libs.versions.toml`.** Navigation Compose registers destinations by string route (`composable("channels") { ... }`) rather than by reflective class lookup, so no `-keep` is needed for screen composables; the build proves this — the channel list renders post-shrink. If a future ticket adds a navigation library that uses reflective destination registration (or a serialization-based deep-link parser), that calculus changes.
- **`signingConfig = signingConfigs.getByName("debug")` reuses the implicit debug config; declaring `signingConfigs { create("debug") { ... } }` would create a *second* config and break installation.** AGP auto-generates the debug config; you only need a `signingConfigs { }` block for *new* configs (i.e. the release one in Phase 5).
- **`isShrinkResources = true` requires `isMinifyEnabled = true`.** Gradle hard-fails the configuration phase otherwise. Both flags landed in the same diff; if a future debugging session needs to disable R8 to bisect a runtime issue, flip both off together.
- **Instrumented tests run against the debug build by default and don't exercise R8.** Reading "all tests pass" is not evidence the release APK works — the binding test for this ticket is manual emulator verification of the post-shrink APK. Wiring `testBuildType = "release"` has consequences for the rest of the instrumented suite (it changes which build the existing `SettingsScreenTest` runs against) and was out of scope.
