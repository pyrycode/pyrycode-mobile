# #131 — feat(ui/thread): tool-call collapsed/expanded component

Lands [`ToolCallRow`](../features/tool-call-row.md), the stateless row primitive that renders `Role.Tool` messages in the conversation thread. Replaces the deliberate `Role.Tool -> Unit` no-op that [#128](./128.md) shipped in `MessageBubble`'s dispatcher with a real route into `ToolCallRow`. Collapsed by default with a two-tone `ToolName · primary-arg` summary and a leading icon; tap-to-toggle reveals an "Input" / "Output" pair. Code-ish output (multi-line or > 80 chars) re-uses [`CodeBlock`](../features/markdown-text.md#code-blocks) from `MarkdownText.kt`, widened from `private` to `internal` so the cross-file reuse is one keyword change instead of a duplicated rounded-monospace surface. New `material-icons-extended` dependency for `Description` / `Terminal` / `Build` icons (`Edit` was already in `material-icons-core`).

## Summary

- **One new file:** `ui/conversations/components/ToolCallRow.kt` — public `@Composable fun ToolCallRow(toolCall: ToolCall, modifier: Modifier = Modifier)`, a private stateless `ToolCallRowContent(toolCall, expanded, onToggle, modifier)`, three private helpers (`iconForTool`, `primaryArg`, `isCodeIsh`, plus the `@Composable buildSummaryAnnotated`), and the preview block. Mirrors the [stateful-wrapper + stateless-content split](./184.md#patterns-established) shipped by [#184](./184.md) for `StreamingAssistantBody` / `StreamingAssistantBodyView` — previews call the stateless `ToolCallRowContent` directly with pinned `expanded = false` / `expanded = true` snapshots, so the matrix renders both states without a coroutine or saved-state harness.
- **Two modified Kotlin files:**
  - `MessageBubble.kt:55` — `Role.Tool -> Unit` is replaced with `Role.Tool -> message.toolCall?.let { ToolCallRow(toolCall = it, modifier = modifier) }`. The null-safe unwrap absorbs the documented [data-class invariant](../features/data-model.md) (`toolCall` non-null iff `role == Role.Tool`) silently — no defensive throw, identical posture to the pre-#131 no-op. **Routing decision: dispatch via `MessageBubble`, not `ThreadScreen`.** The architect spec considered both; the `?.let` shape was picked because `MessageBubble` already owns the `Role`-fanout and lifting Tool-routing one level up would force every future consumer to repeat the unwrap.
  - `MarkdownText.kt:238` — `private fun CodeBlock(` widened to `internal fun CodeBlock(`. **One keyword change, no body edit.** The two existing in-file call sites (the `CODE_FENCE` and `CODE_BLOCK` arms) compile unchanged because `internal` is a superset of `private`-within-file. `ToolCallRow.kt`'s `ExpandedSection` calls `CodeBlock(content = content, language = null)` for code-ish output — `language = null` keeps the no-label monospace path so no syntax highlighting fires for tool output in #131.
- **Two modified build configs:**
  - `gradle/libs.versions.toml:43` — one new `[libraries]` line `androidx-compose-material-icons-extended = { group = "androidx.compose.material", name = "material-icons-extended" }`. Inherits the Compose BOM, no `[versions]` entry needed (same shape as the existing `material-icons-core` line).
  - `app/build.gradle.kts:84` — one new `implementation(libs.androidx.compose.material.icons.extended)` line in the `dependencies { }` block, alphabetically adjacent to the existing core-icons import.
- **State model.** Toggle state is composable-local via `var expanded by rememberSaveable { mutableStateOf(false) }` inside the public `ToolCallRow`. Survives configuration changes (rotation, dark-mode toggle, process death + restore) by serialising the `Boolean` to the saved-instance-state bundle. **No `ViewModel` coupling, no `key` parameter** — the default `rememberSaveable` key (composable call-site identity) suffices because the eventual `LazyColumn` item slot keys on `Message.id`, so each `ToolCallRow` instance has stable identity per message. The stateless inner `ToolCallRowContent` is pure rendering: no `remember`, no `LaunchedEffect`, no flows.
- **Visual contract.** Outer `Row(Modifier.fillMaxWidth().padding(bottom = MessageRowVerticalSpacing))` wraps a `Surface(RoundedCornerShape(ToolCallCornerRadius = 12.dp), color = surfaceContainerHigh, modifier = Modifier.fillMaxWidth().clickable(onClick = onToggle))` with inner `Column(padding(horizontal = ToolCallHorizontalPadding = 12.dp, vertical = ToolCallVerticalPadding = 8.dp))`. The tile stretches edge-to-edge of the column (no `widthIn` cap, unlike the [user bubble's `320.dp`](../features/message-bubble.md)) — content-driven width and a wider tap target. **`Modifier.clickable` carries the default M3 ripple**; no custom indication, no `onClickLabel`, no `Role.Button` semantics.
- **Three deliberate Figma deviations (recorded by the architect spec).**
  1. **Background.** Figma `16:28`–`16:31` uses `surface-container` + 1dp `outline-variant` border. AC3 pins **`surfaceContainerHigh` with no border**. The flat `surfaceContainerHigh` slot matches the assistant variant's tonal slot from [#128](./128.md), so within the thread `Role.User` bubble (`primaryContainer`), tool card (`surfaceContainerHigh`), and the eventual session delimiter (transparent) form a three-step contrast ladder that's legible without an outline stroke.
  2. **Leading icon.** Figma has no icon. AC5 mandates one — 18dp `Icon` on the leading side of the collapsed-row content, tinted `MaterialTheme.colorScheme.onSurfaceVariant`, `contentDescription = null` (the adjacent `Text` carries the semantic content; a TalkBack user hears `"Read · path/to/file.kt"`, not `"Read · path/to/file.kt, document icon"`).
  3. **Expand affordance.** Figma shows the card always-collapsed. AC1+AC2 add the tap-to-toggle expanded state. **No chevron / caret rendered** — the entire surface is the click target. A chevron is design-for-hypothetical-future; the tap target is the whole row.
- **Two-tone summary preserved from Figma.** `buildSummaryAnnotated(toolCall)` (`@Composable`, builds an `AnnotatedString` of the form `"$toolName · $primaryArg"`):
  - Tool name in `MaterialTheme.colorScheme.tertiary` + `FontFamily.Monospace` (matches Figma's `Roboto Mono Regular`).
  - Separator `" · "` and arg in `MaterialTheme.colorScheme.onSurfaceVariant` (no monospace switch — proportional, matches Figma's `body-small`).
  - The `Text` renders with `MaterialTheme.typography.bodyMedium`, `maxLines = 1`, `overflow = TextOverflow.Ellipsis`, and `Modifier.weight(1f)` so the primary-arg ellipsises on long single-line inputs (AC4).
- **Primary-arg derivation is one line.** `private fun primaryArg(toolCall: ToolCall): String = toolCall.input.trim()`. Given the current [`ToolCall(toolName, input, output)`](../features/data-model.md) shape from [#127](./127.md) (a flat `String input` carrying the path for `Read` / `Edit` or the command for `Bash`), the AC's three cases all reduce to "use `input` directly". **No JSON parser, no regex, no first-param branching** — the data model doesn't expose a structured input field. If [#191](./191.md)'s eventual structured payload replaces `input: String` with `Map<String, JsonElement>`, this helper gains real branching; for #131, the trivial body is the contract.
- **Icon mapping, case-insensitive on the trimmed tool name.** `iconForTool(toolName.trim().lowercase())` over a four-arm `when`:

  | Tool name (matched lower-cased) | Icon | Library |
  |---|---|---|
  | `"read"` | `Icons.Outlined.Description` | `material-icons-extended` |
  | `"edit"` | `Icons.Outlined.Edit` | `material-icons-core` |
  | `"bash"` | `Icons.Outlined.Terminal` | `material-icons-extended` |
  | _else_ | `Icons.Outlined.Build` (wrench) | `material-icons-extended` |

  The seeded `Read` fixture in [`FakeConversationRepository`](../features/conversation-repository.md) uses `toolName = "Read"` (PascalCase, matching Claude Code's tool names); future wire-format tool calls may arrive in lowercase. Canonicalising at the comparison point is cheaper than canonicalising the data. **Outlined family**, not filled — passive metadata, not interactive controls; visual weight reads better against the `surfaceContainerHigh` tile.
- **Expanded body.** `Column(padding(top = ToolCallExpandedTopPadding = 8.dp), verticalArrangement = Arrangement.spacedBy(ToolCallExpandedGap = 8.dp))` rendering two `ExpandedSection(label, content)` instances: `"Input"` then `"Output"`. Each section is `Column(verticalArrangement = Arrangement.spacedBy(ToolCallExpandedGap / 2))` of a `labelSmall` caption (in `onSurfaceVariant`) + the body. Body dispatch:
  - **Code-ish (`content.contains('\n') || content.length > CODE_ISH_LENGTH_THRESHOLD = 80`):** calls `CodeBlock(content = content, language = null)` — inherits #130's rounded `surfaceContainer` rectangle, monospace `bodyMedium`, horizontal scroll, and the no-language-label path.
  - **Plain (else):** `Text(content, style = bodyMedium.copy(fontFamily = FontFamily.Monospace), color = onSurface)` with **no surface wrapper**. The AC's "plain monospace text on `surface`" reads as "no nested coloured tile, just the screen surface behind the text" — the text inherits the parent's padding context (already inside the tool-call surface's `12×8` padding).
- **Code-ish heuristic is applied per-section, independently.** Input vs output dispatch separately: `Read`'s input (a path) is short + single-line → plain `Text`; `Read`'s output (file contents) is almost always multi-line → `CodeBlock`. `Bash`'s input (a command) is usually short + single-line → plain; output varies. The `80`-char threshold matches typical terminal-column conventions and aligns roughly with the `bodyMedium` monospace wrap point on a 412dp canvas.

A future ticket that introduces `Spacing.kt` or a `LocalSpacing` `CompositionLocal` provider can collapse the two declarations then. See [Lessons learned](#lessons-learned).
- **Preview matrix: two `@Preview`s, one per theme, each rendering all six (tool × state) cells in a vertical `Column`.** `ToolCallRowLightPreview` and `ToolCallRowDarkPreview` (the latter with `uiMode = Configuration.UI_MODE_NIGHT_YES`) both call into `ToolCallRowPreviewMatrix()`, which invokes `ToolCallRowContent` directly with `expanded = false` / `expanded = true` for `Read`, `Edit`, `Bash` fixtures. ## Patterns established

- **Cross-file reuse of an in-file Compose composable widens visibility by one step (`private` → `internal`), never by two (`private` → `public`).** `CodeBlock` is a module-internal implementation detail of `ui/conversations/components/`; consumers outside the module (none today, but Phase 4 adds Ktor-network code in `data/`) have no use for it. `internal` is the smallest visibility widening that satisfies the cross-file reuse. Pattern applies to any future "this private helper turned out to be useful from a sibling file" case: prefer the keyword change to either (a) duplicating the implementation (loses the single source of truth) or (b) hoisting to `public` (leaks the API surface). The widening is a non-breaking refactor — `internal` is a superset of `private`-within-file at every call site.
- **Routing of role-dispatched message variants stays inside the `MessageBubble` `when`, not lifted to `ThreadScreen`.** [#128](./128.md) shipped the dispatch contract with a deliberate `Role.Tool -> Unit`; [#131](./131.md) realises that contract by replacing the arm body with `message.toolCall?.let { ToolCallRow(...) }`. Considered alternative: filter / route `Role.Tool` upstream in `ThreadScreen` so the dispatcher only sees `User` / `Assistant`. Rejected because `MessageBubble` already owns the `Role`-fanout, and pulling Tool-routing one level up would force every future consumer of `Message` to repeat the null-unwrap. Pattern: keep variant-routing co-located with the discriminator's dispatcher; lift only when the dispatch needs to fan out *across* multiple parent surfaces.
- **Null-safe `?.let` absorbs documented data-class invariants without a defensive throw.** The [`Message`](../features/data-model.md) invariant says `toolCall` is non-null iff `role == Role.Tool` — a data-layer guarantee, not a UI assumption. `Role.Tool -> message.toolCall?.let { ToolCallRow(...) }` silently renders nothing if a bug ever produces `Role.Tool` with `toolCall = null`, identical to the pre-#131 no-op. **No `!!`, no `requireNotNull`, no `error("...")`.** Pattern applies to any future "downstream UI consumes a documented data-class invariant" case: prefer the silent no-op fallback over a defensive crash; if the invariant ever breaks in production, the diagnosis is "tool message renders blank" — a visible bug, but not a crash. Same posture as the `Role.Tool -> Unit` it replaces.
- **Stateful wrapper + stateless content split for previewability of toggle / animation state.** `ToolCallRow` (stateful, `rememberSaveable`) → `ToolCallRowContent` (stateless, takes `(expanded, onToggle)`). Previews call `ToolCallRowContent` directly with pinned `expanded = false` and `expanded = true` snapshots, so the matrix renders both states without driving the toggle through Android Studio's preview runtime. Same split as [#184](./184.md)'s `StreamingAssistantBody` / `StreamingAssistantBodyView`. Pattern applies to any future composable whose UX is "tap to flip a boolean and re-render": split the state shell from the view, drive the preview from the view side with pinned values.
- **Preview fixtures for visual-only components live as file-private `val`s next to the `@Preview`, not in `FakeConversationRepository`.** The three `Preview*ToolCall` constants exist solely for the preview matrix. Adding them to the seed pipeline would (a) couple preview cosmetics to the data-layer fake, and (b) leak preview-only payloads into the runtime app, since the channel-list / thread screens iterate the same seeds. Pattern applies to any future "I need a fixture for a `@Preview` but the data layer doesn't seed one of this shape" case: declare it file-privately, name it `Preview<Variant><Type>`, keep the data-layer seeds focused on runtime UX.
- **`material-icons-extended` is the right artifact for "one or two additional icons beyond the core set"; vector drawables are the right hedge for "we need to trim later".** Adding the extended library cost three icons (`Description`, `Terminal`, `Build`) and ~3–4 MB of debug-APK bloat. R8 / minify-release strips unused icons via tree-shaking when `minifyEnabled true` is on, so the release-APK cost is bounded to the three icons actually referenced. Pattern applies to any future icon-bump need: try `material-icons-extended` first (one-line dependency, no resource files, BOM-managed), fall back to inline `res/drawable/*.xml` only when (a) a profiler / size budget flags the bloat, or (b) the icon doesn't exist in the M3 set. **Don't pre-inline.**

## Lessons learned

- **`internal` is the right visibility step for cross-file Compose composables; the keyword change is the entire diff.** Widening `MarkdownText.kt`'s `CodeBlock` from `private` to `internal` was a one-keyword edit with zero call-site churn — the two existing in-file callers (`MarkdownText.kt:125`, `MarkdownText.kt:132`) continued to compile because `internal` is a superset of `private`-within-file at every Kotlin call site. The alternative considered (re-implement the rounded-monospace surface inside `ToolCallRow.kt`) would have duplicated ~8 lines of `Surface(shape = RoundedCornerShape(8.dp), color = surfaceContainer) { Text(monospace, softWrap = false, horizontalScroll) }` boilerplate that would visually drift from [#130](./130.md)'s contract. Future tickets that need to reuse a private Compose composable across files should default to the keyword change; the single source of truth wins.
- **The `MessageRowVerticalSpacing = 12.dp` duplication between `MessageBubble.kt` and `ToolCallRow.kt` is intentional, not technical debt to chase.** Promoting it to an `internal val` shared constant would touch every component in `ui/conversations/components/` that carries its own variant of the per-row rhythm, and the size-S budget for this ticket doesn't include a cross-component refactor. The right trigger for extracting these to a `Spacing.kt` peer file (or a `LocalSpacing` `CompositionLocal` provider) is the *third* component declaring a `MessageRowVerticalSpacing` — at that point the rule-of-three justifies the abstraction. **Don't pre-pay** for the abstraction when two duplications still satisfy YAGNI.
- **`material-icons-extended` ships ~7000 icons; the debug-APK bloat is real but acceptable for Phase 0.** Adding the dependency for three icons (`Description`, `Terminal`, `Build`) inflates the debug build by ~3–4 MB uncompressed. R8 tree-shaking removes the unreferenced 6997 icons from release builds, so the production size cost is bounded to "the three glyphs we actually call". The Phase-0 debug-build cost is the price of avoiding a vector-drawable maintenance burden (three `res/drawable/ic_tool_*.xml` files + a `painterResource` lookup table). If a future ticket needs to trim, the swap path is clear: inline the three vectors, drop the dependency, edit the four-arm `when` in `iconForTool` to use `painterResource` instead of `imageVector`. **Don't pre-build the swap.**
- **`Icons.Outlined.*` reads better than `Icons.Filled.*` against `surfaceContainerHigh`; the call is preview-verifiable, not theoretical.** The architect spec called the choice as preferable-but-swappable. The shipped previews confirm outlined reads as the right visual weight — passive metadata, not an interactive control. If a future ticket lands a darker / higher-contrast tonal slot for the tool card, re-evaluate against the filled family at that point; the import path is the same (`androidx.compose.material.icons.filled.<Name>`), so the swap is mechanical.
- **`rememberSaveable { mutableStateOf(false) }` is the right primitive for "toggle state survives configuration changes but not list-item disposal".** Rotation, dark-mode toggle, and process death + restore all preserve `expanded`; scrolling the row off-screen in the eventual `LazyColumn` (and disposing the item) resets `expanded` to `false`. The Phase-0 trade-off is acceptable — users haven't complained about losing expansion state on scroll-away because nothing scrolls yet. If users do start complaining, the fix is hoisting `expanded` into the `ViewModel` keyed on `Message.id` (a `Map<MessageId, Boolean>`), not switching to a different `remember*` variant. **Don't pre-hoist** for a problem that hasn't been observed. Same posture as [#184](./184.md)'s `revealedLength` lifetime trade-off.
- **A flat `String input` data model collapses the AC's three-way primary-arg branching to a one-line helper.** AC4 names three cases (`Read`/`Edit` → file path, `Bash` → command, else → first input param's value), but [`ToolCall(input: String)`](../features/data-model.md) from [#127](./127.md) carries the path-or-command directly as a flat string. The helper is `primaryArg(toolCall) = toolCall.input.trim()` — no parser, no regex, no branch table. **Don't pre-bake** a JSON parser or first-param lookup for a structured-input field that doesn't exist on the data model. If [#191](./191.md) replaces `input: String` with a typed `Map<String, JsonElement>` or similar, this helper gains real branching; the public `ToolCallRow` API stays unchanged.
