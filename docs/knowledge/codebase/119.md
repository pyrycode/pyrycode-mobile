# #119 — Warning color slot on `MaterialTheme.colorScheme`

Adds a `warning` semantic color to the theme primitive via the Compose extension pattern (`CompositionLocal` + extension property on `ColorScheme`), since Material 3's `ColorScheme` is a `final` class with fixed slots and cannot accept new constructor params. Foundational only — no call site consumes `MaterialTheme.colorScheme.warning` yet; the existing raw `#D8B85A` literals (Phase 1.5 audit) stay put until downstream tickets #145 / #146 migrate them.

## Summary

- New `WarningColors.kt` owns the small `@Immutable data class WarningColors(val warning: Color)`, the `internal LocalWarningColors: ProvidableCompositionLocal<WarningColors>` (created via `staticCompositionLocalOf { error("WarningColors not provided. Wrap content in PyrycodeMobileTheme.") }`), and the single extension property `val ColorScheme.warning: Color @Composable @ReadOnlyComposable get() = LocalWarningColors.current.warning`.
- `Color.kt` gains six `warning*` `val` declarations (`warningLight`, `warningLightMediumContrast`, `warningLightHighContrast`, `warningDark`, `warningDarkMediumContrast`, `warningDarkHighContrast`), each placed one line after its matching `onErrorContainer*` line so the file's mode-grouped ordering is preserved.
- `Theme.kt` gains six `private val` `WarningColors(...)` instances right after the last `*ColorScheme` definition, then extends the `PyrycodeMobileTheme` `when` block into a destructured `val (colorScheme, warningColors) = when { … }` and wraps the existing `MaterialTheme(...)` call in `CompositionLocalProvider(LocalWarningColors provides warningColors) { … }`. Dynamic-color (wallpaper) branches fall back to `darkWarningColors` / `lightWarningColors` — the wallpaper-derived `ColorScheme` carries no warning hint.
- `WarningColorSlotTest.kt` (instrumented, `androidTest`) — two-method test under `createComposeRule()` asserts the captured `MaterialTheme.colorScheme.warning` equals `warningLight` in `PyrycodeMobileTheme(darkTheme = false, dynamicColor = false)` and `warningDark` in `PyrycodeMobileTheme(darkTheme = true, …)`. The `CaptureWarning` private composable writes the resolved color into an outer `var captured: Color?` via a plain lambda — no `SideEffect` needed since the property read sits inside the composition.

No changes to `MainActivity.kt`, `PyryApp.kt`, `AppModule.kt`, drawables, or `strings.xml`. The 18+ `@Preview` call sites of `PyrycodeMobileTheme(...)` keep working — the function signature is unchanged.

## Patterns established

- **Custom color slots go through `CompositionLocal` + extension property on `ColorScheme`, not via a parallel `LocalCustomColors.current.foo` read.** The project's first non-Material color slot. The extension property `val ColorScheme.warning: Color` preserves `MaterialTheme.colorScheme.<token>` as the call-site surface, which means downstream consumers don't have to learn a second access pattern. Future custom semantic colors (e.g. brand `success`, `info`) should follow the same shape: small `@Immutable data class XColors(...)`, `internal val LocalXColors = staticCompositionLocalOf { error("...") }`, `val ColorScheme.x: Color @Composable @ReadOnlyComposable get() = LocalXColors.current.x`, and a sibling provider inside `PyrycodeMobileTheme`. The `internal` modifier on the local + the `error(...)` default + `@ReadOnlyComposable` on the getter are all load-bearing and should be copied.
- **`staticCompositionLocalOf` for theme-scope values, not `compositionLocalOf`.** Theme tokens change at most once per process (when the user flips a theme preference), so reads should not be tracked by the recomposition machinery. Same choice Material 3 makes for `LocalAbsoluteTonalElevation` and the rest of its theme locals. `compositionLocalOf` would invalidate every reader on every change — wasted bookkeeping for a value that effectively never changes mid-screen.
- **Fail-fast default in `staticCompositionLocalOf { error(...) }`, with the error message naming the fix.** A silent `Color.Unspecified` fallback would ship invisible UI to production; an exception at first read crashes loudly in dev/test with `"WarningColors not provided. Wrap content in PyrycodeMobileTheme."` — the exact remediation. Future custom-color locals should error the same way; the alternative ("provide a neutral default") looks safer but masks "forgot to wrap in theme" bugs that only surface visually.
- **Destructured `when` over `Pair<ColorScheme, WarningColors>` keeps the two selections atomic.** `val (colorScheme, warningColors) = when { … darkTheme -> darkScheme to darkWarningColors … }` ensures the warning palette can never get out of sync with the color scheme. The alternative — two parallel `when` blocks — would invite future maintenance drift when a new branch is added to one and not the other. Apply the same shape to any future per-mode primitive that pairs with the active `ColorScheme`.
- **Compose-extension-property tests live in `androidTest`, not `test`.** Resolving an extension property that goes through `@Composable @ReadOnlyComposable get()` requires a live composition; the project has no Robolectric. Mark the test `@RunWith(AndroidJUnit4::class)`, use `createComposeRule()` (not `createAndroidComposeRule<...>()` — there's no activity to bind), and capture the resolved value into an outer `var` via a private `@Composable` capture helper. The AC's word "unit test" maps to an instrumented Compose test in this codebase — call this out in the PR description so reviewers don't expect a `src/test/` file. Same convention every prior screen Compose test followed (`WelcomeScreenTest`, `ScannerScreenTest`, `ChannelListScreenTest`).
- **AC values + Figma source link beat MCP lookups for foundational color tickets.** The ticket body listed the six hex values explicitly with an authored-on date (2026-05-15) and the Figma variable ID for verification; the developer used those literals directly. The spec's "read via `mcp__plugin_figma_figma__get_variable_defs` before writing" instruction is best-effort — if MCP access is flaky or the variable is freshly created (e.g. the same day as the ticket), the in-body table is the canonical source. Future foundational design-token tickets should mirror this shape: pre-seed the values in the ticket body so the dev-side write is deterministic.

## Lessons learned

- **AC enumerates the deliverable, spec drafts the extension surface — when they diverge, follow the AC.** The architect's spec drafted a four-field `WarningColors(val warning, val onWarning, val warningContainer, val onWarningContainer)` and called for 24 color vals (warning + on + container + onContainer × 6 modes); the AC only enumerated six warning-tone vals and named one extension property (`ColorScheme.warning`). The developer shipped the minimal-AC shape — one field, six vals, one extension property. The reasoning generalises: the spec's wider draft is "the shape this could grow into," not "the shape that has to ship today." Adding the three companion slots before a consumer needs them would be three more API surfaces to keep in sync with no caller to validate against. The pattern doc in [`features/warning-color.md`](../features/warning-color.md) names this as a known extension point and the path to take when the first 4-slot consumer arrives.
- **AC naming (`Warning80` / `WarningMC80` / `Warning30`) was rejected in favour of the local file convention.** `Color.kt` already uses mode-suffix names (`errorLight`, `errorDark`, `errorLightMediumContrast`, etc.) — not M3 tone-number names. Adopting `Warning80` would have made one block stylistically inconsistent with every other block in the same file. The right call when an AC's naming clashes with an existing file's convention is to match the file, not the AC; the AC's intent is to enumerate which six tokens to add, and the names are reviewable suggestion-grade detail. Worth re-reading the surrounding file before transcribing AC names verbatim.
- **`createComposeRule()` is sufficient for theme-extension tests — no `createAndroidComposeRule<Activity>()` needed.** The two prior `androidTest` patterns in this repo (`WelcomeScreenTest` / `ScannerScreenTest`, etc.) use `createAndroidComposeRule<...>` because they exercise screens that wire into the activity's resources. A theme-resolution test only needs a composition root, and `createComposeRule()` is the lighter primitive. Future tests that only assert on what's reachable from inside `setContent { … }` (no `LocalContext.current` consumers, no string-resource lookups) should reach for `createComposeRule()`; the activity-bound variant is reserved for tests that need `R.string.*` or `BuildConfig.*`.
- **Capturing the value of a `@ReadOnlyComposable` extension property in a test is a plain lambda call.** `CaptureWarning { captured = it }` with `onColor(MaterialTheme.colorScheme.warning)` inside a `@Composable` capture body works because the read happens during composition; no `SideEffect`, no `LaunchedEffect`, no `remember` needed. The architect's spec suggested `SideEffect` as an option — in practice the plain-lambda shape is the simpler one and the resolved value is captured before the test rule advances past the first frame. Apply this pattern to any future "what does this theme local resolve to" assertion.
- **The four un-routed `WarningColors` instances are intentional, not dead code.** `mediumContrastLightWarningColors`, `highContrastLightWarningColors`, `mediumContrastDarkWarningColors`, `highContrastDarkWarningColors` exist in `Theme.kt` but are never selected by the current `when` block — the same shape the existing `*ColorScheme` medium/high-contrast instances live in. Whichever ticket eventually wires the accessibility-contrast level through `PyrycodeMobileTheme` (currently no ticket filed) will be the one to extend the `when` to pick them. Don't delete them as part of an unrelated cleanup pass.
