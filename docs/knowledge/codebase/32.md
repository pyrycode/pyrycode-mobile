# #32 ‚Äî Koin DI scaffold

Stands up the Koin DI framework so downstream tickets can append bindings to a known place. No app behaviour change ‚Äî the `appModule` ships empty.

## Summary

- `koin-bom 4.0.4` + `koin-androidx-compose` added to the version catalog. `koin-androidx-compose` has no `version.ref`; the BOM pins it (same pattern as the Compose BOM ‚Üî `androidx-compose-material3`).
- New `PyryApp : Application` at the package root calls `startKoin { androidContext(this@PyryApp); modules(appModule) }` in `onCreate`. Registered via `android:name=".PyryApp"` on `<application>`.
- New top-level `val appModule = module { }` at `de/pyryco/mobile/di/AppModule.kt`. Body is a three-line comment pointing at the tickets that fill it (#11 AppPreferences, #4 FakeConversationRepository, upcoming ViewModel tickets). The comment is transient ‚Äî whichever of #11/#4 lands second should delete it.
- `MainActivity` is **not** touched. Koin initialises synchronously in `Application.onCreate` before any activity is created, so `by inject()` / `koinViewModel()` from composables is safe by Android lifecycle guarantee.

## Files touched

- `gradle/libs.versions.toml` ‚Äî `koinBom = "4.0.4"` in `[versions]`; `koin-bom` (BOM) + `koin-androidx-compose` (no version, pinned by BOM) in `[libraries]`.
- `app/build.gradle.kts:44,53` ‚Äî `implementation(platform(libs.koin.bom))` next to the Compose BOM line; `implementation(libs.koin.androidx.compose)` between the `androidx-*` block and `kotlinx-*` (alphabetic by alias root).
- `app/src/main/AndroidManifest.xml:6` ‚Äî `android:name=".PyryApp"` as the first attribute on `<application>`.
- `app/src/main/java/de/pyryco/mobile/PyryApp.kt` ‚Äî new, 16 lines.
- `app/src/main/java/de/pyryco/mobile/di/AppModule.kt` ‚Äî new, 10 lines (mostly comment).

## Patterns established

- **Single `appModule` until ~30 bindings.** Don't pre-split into `dataModule` / `uiModule` / etc. The first split is justified when the file gets unwieldy or when a feature module is extracted from `app/`; until then, every `single { ... }` / `viewModel { ... }` line appends to `AppModule.kt`.
- **`modules(appModule)` is a single varargs slot, not a list-builder.** When the second module is introduced, the call becomes `modules(appModule, otherModule)` ‚Äî no `listOf(...)` wrapping, no `module { includes(...) }`. Keep the composition root literal.
- **No Koin logger in Phase 0.** Default `EmptyLogger` is enough. Adding `androidLogger(Level.INFO)` is deferred until a real binding-misconfiguration bug surfaces (evidence-based ‚Äî don't ship a defense for an unobserved failure mode).
- **No `koin-test` / `verify()` until a binding needs verifying.** An empty `appModule` is a no-op to verify. The first ticket to add a real binding (likely #11) is the place to bring in `koin-test`. This ticket adds zero test dependencies.
- **`startKoin` is unwrapped ‚Äî no `try/catch`.** If it throws, the app cannot start; the process crash is the correct surface, and the stack trace is the debugging signal. No fallback path makes sense at the composition root.

## Lessons learned

- **`koin-androidx-compose` transitively pulls in `koin-android` and `koin-core`.** Do not add either explicitly to the catalog ‚Äî that creates two version-managed entries for libraries the BOM already pins, and risks drift when the BOM bumps. One alias is all this codebase needs.
- **Koin BOM `4.0.x` is the right pin for Kotlin 2.2.10 + AGP 9.2.1.** If a future bump trips a Kotlin-compiler-version mismatch or an unresolved BOM, bump *up* to the latest stable `4.0.x` patch (`./gradlew --refresh-dependencies assembleDebug`). Do **not** downgrade to `3.5.x` ‚Äî it predates the Kotlin 2.2 alignment, same trap noted for `navigation-compose` in #8.
- **`android:name` goes first on `<application>` by convention.** Lint accepts either order, but AGP-generated manifests and reviewers expect `android:name` as the first attribute. The other attributes (`android:allowBackup`, `android:theme`, ‚Ä¶) keep their existing order.
