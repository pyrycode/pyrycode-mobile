# #46 — `ChannelListScreen` (LazyColumn + tap nav)

First stateless UI consumer of a ViewModel in the project, and the first place a sealed `Event` type lands. Adds `ChannelListScreen` + `ChannelListEvent` next to `ChannelListViewModel` (#45), replaces the `Text("Channel list placeholder")` body at the `channel_list` NavHost destination with a real `koinViewModel<…>()` + `collectAsStateWithLifecycle()` collection block, and translates `RowTapped` into `navController.navigate("conversation_thread/$id")` inline at the destination.

## Patterns established

- **Sealed `Event` interface lives in the screen file, not the VM file, when the VM doesn't consume events.** `ChannelListEvent` sits at the top of `ChannelListScreen.kt` alongside the composable that produces it (`ConversationRow.onClick → onEvent(RowTapped(...))`). The mirror precedent from #45 keeps `ChannelListUiState` in `ChannelListViewModel.kt` because the VM is its producer. When a future ticket adds a VM-side reducer (`fun onEvent(event: ChannelListEvent)`), the event type moves into `ChannelListViewModel.kt` to join the producer-side. The principle is: sealed type lives next to its producer.
- **Stateless composable shape: `@Composable fun XxxScreen(state: XxxUiState, onEvent: (XxxEvent) -> Unit, modifier: Modifier = Modifier)`.** No `viewModel()`, no `koinViewModel()`, no `NavController`, no `LocalContext.current` inside the screen body — those resolve at the `composable(...)` destination block. `modifier: Modifier = Modifier` is the canonical trailing parameter even when the current caller doesn't pass anything, matching the Compose API surface convention.
- **`when (state)` on the sealed `UiState` is exhaustive — relied on for forward safety.** When a fifth `ChannelListUiState` variant lands (e.g. `Stale` in Phase 4), the compiler points at this `when` block. Don't add an `else ->` branch; the missing-branch error is the feature.
- **`LazyColumn(items = state.channels, key = { it.id })` — key is non-optional for identity-based diffing.** Without `key`, reorders (e.g. when a channel's `lastUsedAt` bumps it up the list) recompose every row below the moved item. `Conversation.id` is the stable identity per the data-model contract; same key shape lifts to any future list of `Conversation` (recent discussions drilldown, thread message list).
- **VM resolution + state collection lives at the `composable(...)` destination, not in the screen.** Canonical shape: `val vm = koinViewModel<XxxViewModel>(); val state by vm.state.collectAsStateWithLifecycle(); XxxScreen(state = state, onEvent = { ... })`. The destination is the right scope because every `composable { ... }` block is a `NavBackStackEntry`-keyed scope — the VM lifetime ties to the back-stack entry, surviving configuration changes (rotation) and tearing down on pop.
- **`collectAsStateWithLifecycle()`, not `collectAsState()`, is the default for screen-side state collection.** The lifecycle-aware variant pauses upstream collection when the lifecycle drops below `STARTED`, which is what makes the VM's `WhileSubscribed(5_000)` actually save work when the screen is backgrounded. Plain `collectAsState` keeps collecting even when the screen is invisible.
- **Inline `when (event)` dispatch in the destination's `onEvent` lambda for navigation-only events.** No `private fun onChannelListEvent(event: ChannelListEvent)` extraction; the inline form is shorter and keeps the navigation target visible at the destination. Lift to a helper only when event-translation grows non-trivial (a future ticket that adds 3+ events, or events that need ambient state beyond `navController`).
- **Concrete navigation targets for parameterized routes are built inline at the call site.** `navController.navigate("conversation_thread/${event.conversationId}")` — no `Routes.conversationThread(id)` helper. Matches the navigation feature doc's existing rule (helpers appear when a second caller does).
- **Placeholder UI for `Loading` / `Empty` / `Error` is a single centred `Text` inside `Box(Modifier.fillMaxSize(), contentAlignment = Alignment.Center)`.** AC permits a single `Text(...)` — centring it is a basic layout decision, not a "visual" the AC forbids (no spinner, no illustration, no error icon). Throwaway code; #23 replaces with real empty-state UI.
- **`@Preview` covers the `Loaded` branch only, with 3 inline `Conversation` instances varying `name` / `cwd` / `lastUsedAt`.** One uses `DefaultScratchCwd` to verify the workspace-label-omission path through `ConversationRow` from #17. `Loading` / `Empty` / `Error` are not previewed — they're throwaways being replaced in #23, and the time-to-preview cost outweighs the value.

## Lessons learned

- **`androidx.lifecycle.compose.collectAsStateWithLifecycle` lives in its own artifact (`lifecycle-runtime-compose`), not in `lifecycle-runtime-ktx` and not in `lifecycle-viewmodel-compose`.** `koin-androidx-compose:4.0.4` brings `lifecycle-viewmodel-compose` transitively (enough for `koinViewModel<…>()`), and `lifecycle-runtime-ktx` was already on the classpath since the project's first ticket — but neither pulls `lifecycle-runtime-compose`. Without the explicit `implementation(libs.androidx.lifecycle.runtime.compose)`, the import doesn't resolve and `assembleDebug` fails. Pinning for future ViewModel-consuming screens: any new screen that uses `collectAsStateWithLifecycle` is safe once this dependency is on the classpath; don't re-add it. Three lifecycle artifacts are now on the classpath (`-ktx` for `LifecycleOwner`, `-viewmodel-ktx` for `viewModelScope`, `-runtime-compose` for `collectAsStateWithLifecycle`); they must stay on the same version-ref (`lifecycleRuntimeKtx = "2.6.1"`) since they share `LifecycleOwner` ABIs.
- **`koinViewModel<T>()` and `koinInject<T>()` are different imports from different packages.** `koinViewModel` lives in `org.koin.androidx.compose`; `koinInject` lives in `org.koin.compose`. `MainActivity.kt` already imported `koinInject` (used by the `scanner` block for `AppPreferences`); adding `koinViewModel` is a second import, not a rename. Don't try to resolve a `ViewModel` via `koinInject<XxxViewModel>()` — it bypasses the `viewModelStoreOwner` wiring that ties the VM's lifetime to the back-stack entry, and the VM survives navigation in surprising ways.
- **Compose Navigation 2.9+ auto-wires `LocalViewModelStoreOwner` to the current `NavBackStackEntry`.** Bare `koinViewModel<ChannelListViewModel>()` inside a `composable(...)` block resolves to the entry-scoped store with no explicit `viewModelStoreOwner = backStackEntry` argument. Confirmed against the pinned `navigationCompose = "2.9.5"`. If a future ticket introduces nested navigation graphs, the explicit form may be needed — until then, the bare call is right.
- **`ConversationRow.lastMessage` is nullable; `null` produces no subtitle.** Verified by re-reading `ConversationRow.kt`: the supporting-content branch is `lastMessage?.let { ... }`. Passing `lastMessage = null` for every row in `ChannelListScreen` is safe and renders cleanly — the row collapses to a single-line title with the trailing relative-time stamp. Per-row last-message data is not part of `ChannelListUiState.Loaded` (the VM projection from #45 carries `List<Conversation>` only); a future ticket may extend `Loaded` to carry a `Map<String, Message?>` or a `data class ChannelRow(val conversation, val lastMessage)`, but that's a deliberate deferral, not a missed feature.
- **The `WhileSubscribed(5_000)` upstream-grace from #45 starts paying off here.** With the screen now collecting via `collectAsStateWithLifecycle`, navigating to the conversation thread and immediately back keeps the upstream repository flow hot (subscribers briefly drop to zero but the 5s grace bridges the gap). Confirmed observable in the preview / device run: returning to the channel list does not re-show `Loading`. The 5s figure is a project default now — don't change it per-screen without reason.
- **Per the navigation feature doc, screen composables are `NavController`-free; the destination block translates events into nav calls.** The temptation when wiring a single navigation event was to pass `onRowTap: (String) -> Unit` directly instead of routing through `ChannelListEvent`. Resisted: CLAUDE.md mandates the sealed `Event` shape on every screen taking user input, and the channel list is the first place that shape lands. Future tickets adding a second event (retry, archive, long-press menu) will plug into the same `onEvent: (ChannelListEvent) -> Unit` parameter with zero shape change.
