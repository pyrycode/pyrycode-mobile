# #163 — Settings copy fixes + License row non-clickable

Three Settings copy / treatment mismatches against locked Figma `17:2` corrected: Theme subtitle `"System default"` → `"System"`, wallpaper-colors row headline `"Use wallpaper colors"` → `"Use Material You dynamic color"`, License row reduced to a text-only `SettingsRow(headline = "License: MIT")` with no chevron and no row click. The fourth edit deletes the now-orphaned `LicenseScreen.kt` + `Routes.LICENSE` + `assets/LICENSE` + `R.string.license_title` + `LicenseScreenTest.kt`. Split from #124.

## Summary

- **Theme subtitle.** Single-line edit to the `internal ThemeMode.label()` extension at the bottom of `SettingsScreen.kt`: the `SYSTEM` arm switches to `"System"`. `LIGHT` / `DARK` arms unchanged. Since #87 widened the extension to `internal`, the same change flows to both the row subtitle and `ThemePickerDialog`'s `SYSTEM` radio row from one edit.
- **Wallpaper-colors headline.** Pure resource edit — `app/src/main/res/values/strings.xml`'s `settings_use_wallpaper_colors` value flipped to `"Use Material You dynamic color"`. The composable already reads `stringResource(R.string.settings_use_wallpaper_colors)`; no Kotlin change. The disabled-pre-S `settings_use_wallpaper_colors_unsupported` ("Requires Android 12 or newer") stays as-is — not part of the locked Figma, and structurally unreachable in production today (Min SDK 33 > S = 31).
- **License row non-clickable.** The row at `SettingsScreen.kt:222` collapsed from `SettingsRow(headline = "License: MIT", trailing = { ChevronIcon() }, onClick = onOpenLicense)` to `SettingsRow(headline = "License: MIT")`. No new parameter on `SettingsRow` — the existing default `onClick: (() -> Unit)? = null` already produces a non-clickable row (the `Modifier.clickable` branch is skipped when `onClick` is null). The `onOpenLicense: () -> Unit` parameter was dropped from `SettingsScreen`'s signature; both `@Preview` call sites and both `SettingsScreenTest.kt` `setContent` blocks lost the `onOpenLicense = {}` argument.
- **Orphan deletion (one ticket, one commit).** After Edit 3, `onOpenLicense` was the only consumer of `Routes.LICENSE`, which was the only consumer of `LicenseScreen`, which was the only consumer of `assets/LICENSE` + `R.string.license_title`. All became dead in the same commit, and all five were deleted: `LicenseScreen.kt`, `LicenseScreenTest.kt`, `assets/LICENSE`, the `composable(Routes.LICENSE) { ... }` block + `LicenseScreen` import + `Routes.LICENSE` constant in `MainActivity.kt`, and the `license_title` string. The `assets/` directory was left in place (empty-directory removal is out-of-scope churn).
- **New unit-test file `SettingsLabelsTest.kt`.** Pure-Kotlin JUnit at `app/src/test/java/de/pyryco/mobile/ui/settings/SettingsLabelsTest.kt` with three `@Test` methods locking each `ThemeMode.label()` return value (`"System"` / `"Light"` / `"Dark"`). The `internal` visibility on `label()` keeps the test in the same module. The SYSTEM-case test is the load-bearing one (AC 1 regression guard); the LIGHT/DARK tests are cheap regression-guards-by-symmetry, not AC clauses.
- **Instrumented test extension.** Two new `@Test` methods appended to `SettingsScreenTest.kt`: `wallpaperColorsRow_rendersMaterialYouLabel` (`onNode(hasText("Use Material You dynamic color")).performScrollTo().assertExists()`) and `licenseRow_hasNoClickAction` (`onNodeWithText("License: MIT").performScrollTo().assert(hasClickAction().not())`). The spec also drafted a trailing-icon-absence assertion via `onChildren().assertCountEquals(1)` against `ListItem`'s internal structure; the developer correctly dropped it as the secondary regression-guard the spec marked it as — the click-action assertion alone covers the load-bearing half of AC 3.

## Patterns established

- **Delete orphaned routes + screens + assets + strings in the same commit as the entry-point edit that orphans them.** Spec-driven: once the entry point disappears, the route → screen → asset → string graph collapses transitively, and all five layers are dead at the same instant. Deleting them in one commit (vs. a follow-up "cleanup" ticket) keeps the dead-code window at zero — no transient interval where Android Lint's `UnusedResources` / `UnusedSymbol` checks would fire, no follow-up ticket grooming overhead, no temptation to leave the screen "just in case" against a hypothetical future entry point. Restoring from git history is trivial if a future entry point is ever wired. Generalised rule: if a deletion would render a sibling layer dead, delete that sibling too in the same commit unless an alternative caller is known to be landing imminently.
- **No new `isClickable` parameter on `SettingsRow` — use the existing `onClick: (() -> Unit)? = null` default.** Reduced-spec hygiene. The non-clickable License row reads as `SettingsRow(headline = "License: MIT")` because the `SettingsRow` API was already shaped so that "no `onClick`" means "no `Modifier.clickable` and no ripple" (verified at the `SettingsRow` body's `if (onClick != null) Modifier.clickable(onClick) else Modifier` branch). Introducing an `isClickable: Boolean = true` parameter would have been an `if`-flag duplicating the lambda-null signal — same negative-space pattern as a `nullable` parameter and a sibling `is<Field>Set` boolean, which the M3 component conventions don't carry. When a row affordance disappears, drop the argument, don't add a new parameter to disable it.
- **One commit per "match Figma" sweep, even when the corrections span three independent rows.** #163 bundled Theme subtitle + wallpaper-colors headline + License-row treatment + dead-route deletion into one commit because all four changes share the same justification ("Figma `17:2` is the source of truth"), the same risk profile (presentation-only, no state-shape), and the same review surface (`SettingsScreen.kt` + `strings.xml` + a small test-file diff). The alternative — three separate tickets per row — would have multiplied review overhead by 3× with no isolation benefit. The bundling rule has a counter-example: #149 (pill polish) + #150 (logo path repair) stayed separate tickets even though both targeted the Welcome screen, because the second was a defect repair against shipped art rather than a Figma-locked copy edit. Bundle by *kind of change*, not by *screen of change*.

## Lessons learned

- **`SettingsRow`'s existing API made the License-row edit a one-line collapse.** The spec correctly verified at `SettingsScreen.kt:248` that a `null` `onClick` produces a non-clickable row, and that `trailing = null` is the parameter default. Both invariants survived the M3 4.x `ListItem` migration that landed in earlier tickets, so the License row collapsed from a 4-line block to a single named-argument line with no API surface added. The lesson is the diligence: when an AC asks for "row X loses behaviour Y", grep `SettingsRow.kt`'s default-parameter values *before* drafting a new parameter — a default-shaped API often already covers the reduced spec by passing nothing.
- **The pre-#163 doc claim "label string diverges from Figma `17:2`" was technically true at write-time and stale by ticket-end.** `settings-screen.md` line 46 prior to this ticket said the wallpaper-colors headline diverged from Figma because the user-facing copy had been clarified to `"Use wallpaper colors"` since the Figma was drawn. #163 reversed that policy — the Figma is now the source of truth (matching the wording the ticket asked for). When updating that doc, the divergence note must come out, not just the literal string. Feature docs that document deliberate copy divergences carry a longer half-life of stale risk than docs that document inherent code shape; treat the divergence note itself as the changeable unit when the policy flips.
- **Spec scope vs. AC: when in doubt, follow AC.** The architect spec drafted a secondary "no chevron icon under this row" Compose assertion using `onChildren().assertCountEquals(1)` as a regression guard, but flagged it as "drop if brittle against `ListItem`'s internal structure". The developer correctly shipped the click-action assertion only — the spec's note that "click-action assertion alone satisfies AC 3" was the load-bearing guidance. When spec drafts a richer assertion than AC requires and flags it as droppable, the developer should drop it. The richer assertion buys negligible defect-detection lift and binds the test to M3 internal node structure that can churn under library upgrades. The lesson generalises: a Compose test that asserts on M3's child-node count is a maintenance liability — keep tests pinned to call-site behaviour (`hasClickAction`, `hasText`, `hasContentDescription`), not to library internals.
