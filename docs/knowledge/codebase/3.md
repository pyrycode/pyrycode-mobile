# #3 — `ConversationRepository` interface

Contract-only ticket. Defines the data-layer seam every Phase 1 UI tier binds to and that the Phase 4 Ktor-backed implementation will satisfy. No `Fake*` or `Remote*` class lands here.

## Summary

- New package `de/pyryco/mobile/data/repository/` with a single file `ConversationRepository.kt` (64 LOC).
- One file holds the interface, the `ConversationFilter` enum, the `ThreadItem` sealed interface (`MessageItem` + `SessionBoundary` variants), and the `BoundaryReason` enum — all referenced by the interface's signatures, no independent consumers, same colocation rule that put `Role` in `Message.kt`.
- Stream-shaped reads return cold `Flow<List<…>>`; mutators are `suspend` one-shots returning the affected entity (`archive` returns `Unit` — no useful handle).
- IDs stay raw `String` to match the shipped #2 model; value-class promotion is deferred to a separate model-wide pass.
- `kotlinx-coroutines-core 1.10.2` pinned via the version catalog and declared as `implementation` on `app/`.

## Patterns established

- **One file per logical contract, even when it contains several declarations.** When an interface, its filter enum, its element wrapper, and that wrapper's classification enum are all referenced by the interface's own signatures and have no independent consumers, they live in one file. Same rule as `Role` in `Message.kt` (#2); contrast with `Conversation` / `Session` getting their own files because each is a top-level entity.
- **`sealed interface` over `sealed class`** when all variants are `data class`es with no shared state. Lighter parent, still exhaustive `when`.
- **`MessageItem` wraps `Message` rather than `Message` implementing the marker interface.** Keeps the layer direction one-way — `data/model/` stays free of `data/repository/` parents.
- **Mutators throw, they don't `Result`-wrap.** Not-found ids → `NoSuchElementException`; already-promoted → `IllegalStateException`; blank name → `IllegalArgumentException`. The single happy path simplifies every UI call site; recovery use cases will add a `getConversation(id): Conversation?` query when one actually exists.
- **Contract is dispatcher-agnostic.** No `flowOn(...)` in the interface; each impl picks. Imposing a dispatcher in the contract would force every implementation into the same threading model.
- **`data/` pins its own coroutines version.** Even though Compose / lifecycle already pull coroutines onto the runtime classpath, the data layer declares `kotlinx-coroutines-core` explicitly so a transitive BOM bump can't silently swap the version under the contract.

## Lessons learned

- **`kotlinx-coroutines-core`, not `kotlinx-coroutines-android`.** The data layer must stay JVM-portable for the Compose Multiplatform walk-back trigger named in CLAUDE.md → "Don't". The `-android` artifact adds the `Main` dispatcher and is JVM/Android-only; the UI layer already pulls it in transitively via Compose. `data/` only needs `-core`.
- **`Flow` on a `main/` interface needs a compile-classpath dep, even if a transitive runtime dep already exposes it.** Returning `Flow<…>` from a public signature forces `kotlinx-coroutines-core` onto `compileClasspath`. Verifying with `assembleDebug` is the cheap check.
- **Version-catalog accessor dot rule (re-confirmed from #2):** `libs.kotlinx.coroutines.core`, not `libs.kotlinx-coroutines-core`. Each `-` in the alias becomes a `.` in Gradle.
- **Value-class promotion is a model-wide refactor, not an interface-level decision.** The ticket body's Technical Notes suggested `ConversationId` / `SessionId` value classes, but #2 shipped raw `String` ids on `Conversation` and `Session`. Introducing value classes only on the repository surface would have left `Conversation.id: String` inconsistent with `repository.archive(id: ConversationId)`. Architect flagged it as a follow-up; the contract uses `String` end-to-end.
