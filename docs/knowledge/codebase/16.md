# #16 ‚Äî Settings placeholder route in NavHost

Adds a fourth, unparameterized destination ‚Äî `settings` ‚Äî to the NavHost so the future Channel List TopAppBar's gear icon (separate ticket) has a real route to call `navigate(...)` on. The destination renders `Text("Settings placeholder")` plus a `TextButton("Back")` that pops the back stack; Phase 3 replaces the whole `composable(...)` block wholesale.

## Summary

- One new route constant in `MainActivity`'s `private object Routes`: `Settings = "settings"`. Plain unparameterized string ‚Äî same shape as `Welcome` / `ChannelList`.
- One new `composable(Routes.Settings) { ... }` block inside `PyryNavHost` that calls a private `SettingsPlaceholder(onBack = { navController.popBackStack() })`. The `popBackStack()` return value is intentionally discarded ‚Äî Settings cannot be the start destination under the graph as registered, and no caller landing here as the start exists in this wave.
- One new private top-level `@Composable SettingsPlaceholder(onBack: () -> Unit)` factored out of the `composable { ... }` block. The factoring crosses a threshold #15 didn't: two children (text + back affordance) plus an `onBack` callback to wire ‚Äî reads better as a tiny named composable than an inline `Column { ... }`.
- Two new imports: `androidx.compose.foundation.layout.Column`, `androidx.compose.material3.TextButton`. No catalog or build-file edits.

## Files touched

- `app/src/main/java/de/pyryco/mobile/MainActivity.kt` ‚Äî `Column` + `TextButton` imports (lines 7, 12), new `composable(Routes.Settings) { ... }` block inside `PyryNavHost` (lines 65-67), new `SettingsPlaceholder` Composable (lines 71-79), `Settings` constant in `Routes` (line 85).

## Patterns established

- **Interactive placeholders factor into a named private Composable taking `() -> Unit` callbacks; bare-`Text` placeholders stay inline.** The break point is "more than one child or a callback to wire." `composable(Routes.ChannelList) { Text("Channel list placeholder") }` stays inline; `SettingsPlaceholder(onBack = ...)` doesn't. Don't add a `Modifier` parameter to throwaway placeholder composables ‚Äî the outer `Scaffold` in `MainActivity` already applies `innerPadding` at the `NavHost` boundary, and the placeholder gets rewritten in Phase 3 anyway.
- **Back affordances use `TextButton("Back")` until `material-icons-core` is on the classpath.** The ticket AC explicitly allows "equivalent Material 3 back-button pattern" alongside the `IconButton + Icons.AutoMirrored.Filled.ArrowBack` example. Adding a `material-icons-core` catalog entry + dependency just so a throwaway placeholder can render a glyph is wasted scope; defer to the first ticket that ships durable UI needing icons (almost certainly the Channel List TopAppBar ticket ‚Äî gear, plus likely hamburger / search). That ticket should also sweep any surviving `TextButton("Back")` placeholders.
- **`navController.popBackStack()`'s `Boolean` return is discarded at the registration site for placeholder destinations.** Don't wrap it in `if (!navController.popBackStack()) { ... }` defensiveness ‚Äî under the current graph there's no path where the placeholder is the start destination. If a future caller can make Settings the start, the fallback is that caller's problem to handle.
- **The placeholder Composable stays `NavController`-free** ‚Äî it takes `onBack: () -> Unit` and the `composable { ... }` block supplies `{ navController.popBackStack() }`. This keeps the placeholder liftable into a `@Preview` or a test harness without dragging Navigation along, even though no preview or test is written for this ticket. Same shape Welcome already uses (`onPaired` / `onSetup` callbacks).

## Lessons learned

None ‚Äî the registration shape and the `TextButton`-over-`IconButton` decision both fell out of the spec without surprises during implementation. The `material-icons-core` classpath check (`./gradlew app:dependencies --configuration debugRuntimeClasspath | grep material-icons` returning nothing, plus `androidx.compose.material:material:1.10.4` only bundling private `androidx.compose.material.internal.Icons`) was done at architect time; it'd be the first thing to re-verify when the TopAppBar ticket adds the dependency, but nothing in this implementation surfaced new failure modes.
