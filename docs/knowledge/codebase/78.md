# #78 — Minimal promote-discussion confirmation dialog

Wires `DiscussionListEvent.SaveAsChannelRequested` (the gesture surface that shipped in #25 as a VM `Unit` stub) to a Material 3 `AlertDialog`. Confirm calls `repository.promote(conversationId, derivedName, workspace = null)`; cancel/scrim/back dismiss without mutating state. The dialog *is* the promotion flow — there is no separate name-entry or workspace-picker UI in this ticket. Those land in a follow-up when their Figma is ready.

## Summary

- `DiscussionListEvent` (in `DiscussionListScreen.kt`) gained two new `data object` variants: `PromoteConfirmed` and `PromoteCancelled`. No payload — the VM owns the pending state and already knows which discussion is in flight; routing the id back through the event would let the screen and VM disagree.
- `DiscussionListUiState.Loaded` gained `val pendingPromotion: PendingPromotion? = null` with a `null` default. The default preserves every positional `Loaded(discussions)` call site (the existing `loaded_containsOnlyUnpromoted_inSourceOrder` VM test, the three `@Preview`s, and the `DiscussionListScreenTest` fixtures all keep working unchanged).
- New top-level `data class PendingPromotion(val conversationId: String, val sourceName: String?)` next to `DiscussionListUiState` in `DiscussionListViewModel.kt`. `sourceName` is the verbatim `Conversation.name` snapshotted at request time; the dialog body and the post-promote channel name both derive from it via two file-scope helpers.
- VM replaced its `observeConversations(...).map { … }` projection with `combine(observeConversations(...), pendingPromotion) { discussions, pending → … }`, where `pendingPromotion` is a private `MutableStateFlow<PendingPromotion?>(null)`. The combine projects four arms — `Loading` (initial), `Empty` (empty list, regardless of pending), `Loaded(discussions, null)`, `Loaded(discussions, pending)`. When the upstream collapses to empty while a dialog is open, the combine body side-effects `pendingPromotion.value = null` inline (documented with a one-line comment) so a later non-empty emission cannot revive the stale dialog.
- Three new VM seams: `openPromotionDialog(id)` reads `state.value`, returns early if it isn't `Loaded` or the id isn't in `discussions`, and otherwise sets `pendingPromotion.value = PendingPromotion(id, match.name)`. `confirmPromotion()` snapshots `pendingPromotion.value` into a local, clears `pendingPromotion.value = null` **before** the suspend launch (deterministic dismissal + prevents double-tap re-entry), then `viewModelScope.launch { repository.promote(snapshot.conversationId, derivedChannelName(snapshot.sourceName), workspace = null) }`. `PromoteCancelled` is a one-liner: `pendingPromotion.value = null`.
- Two file-scope `internal` helpers at the bottom of `DiscussionListViewModel.kt`: `derivedChannelName(sourceName: String?) = sourceName?.takeIf { it.isNotBlank() } ?: "Untitled channel"` (what gets persisted) and `displayLabel(sourceName: String?) = sourceName?.takeIf { it.isNotBlank() } ?: "Untitled discussion"` (what the dialog body reads). Two distinct strings on purpose — the dialog identifier matches `ConversationRow`'s fallback so the user sees the same label they long-pressed; the post-promote name reflects the new tier.
- `DiscussionListScreen.kt` adds a private `PromotionConfirmationDialog(pending, onConfirm, onDismiss)` composable rendered as an overlay **after** the `Scaffold { … }` block when `(state as? Loaded)?.pendingPromotion != null`. Standard M3 `AlertDialog` — title, body, confirm `TextButton`, dismiss `TextButton`, `onDismissRequest = onDismiss` (M3 default routes both scrim taps and back-press through this). The dialog is wired through the existing `(state, onEvent)` shape; no `MutableState` is hoisted into the screen.
- `MainActivity`'s `PyryNavHost` destination block gained two `when` arms — `DiscussionListEvent.PromoteConfirmed -> vm.onEvent(event)` and `DiscussionListEvent.PromoteCancelled -> vm.onEvent(event)` — alongside the existing `SaveAsChannelRequested` forward. The `RowTapped` and `BackTapped` arms are unchanged.
- Four new strings in `values/strings.xml`: `promote_dialog_title` ("Save as channel?"), `promote_dialog_body` (`Save \"%1$s\" as a channel?` — Android-escaped quotes around a positional placeholder fed `displayLabel(pending.sourceName)`), `promote_dialog_confirm` ("Save as channel"), `promote_dialog_cancel` ("Cancel"). English-only, non-plural — matches the rest of the codebase.
- VM unit tests rewrote the prior `saveAsChannelRequested_isNoOp_inPhase0` into seven new cases (see `DiscussionListViewModelTest.kt`): pending-state set on a valid id, no-op on an unknown id, cancel clears, confirm calls `repository.promote` with the right name across non-blank / null / whitespace `sourceName`, no-op on confirm without pending state, and the empty-upstream-clears-pending edge. The hand-rolled `recordingRepo` captures `(id, name, workspace)` triples into a `mutableListOf<PromoteCall>` and `TODO("not used")`s every other repository method — same conventions as the prior `stubRepo`.
- New instrumented test file `app/src/androidTest/.../ui/conversations/list/DiscussionListScreenTest.kt` — the project's first Compose UI test for this screen. Six cases: dialog appears / does-not-appear, confirm and cancel emit the right events, dialog body renders `sourceName` when present and `"Untitled discussion"` when null. Drives the screen directly via `state` + `events::add` (no VM, no Koin) — mirrors `ChannelListScreenTest`'s conventions verbatim.

## Patterns established

- **`combine(upstreamFlow, MutableStateFlow<UiSubstate>)` is the project pattern for "transient UI sub-state that needs to compose with the repository stream".** Reach for it whenever a dialog, expanded-row state, or other UI-only fact needs to live next to repository-driven content inside the same `StateFlow<UiState>`. Don't lift the sub-state into a parallel `StateFlow` exposed alongside `state` — UIs would then have to collect twice and reason about ordering. Don't store it as `var ... by mutableStateOf(...)` in the composable either — that breaks the (state, onEvent) MVI shape and loses the deterministic ordering across the `viewModelScope.launch` boundary. The pattern naturally generalizes for the next dialog/picker that needs the same shape (rename channel, archive confirm, workspace picker).
- **Side-effect inside `combine` body to invalidate stale UI sub-state on upstream collapse.** When a piece of UI state is meaningful only in the presence of a particular upstream shape (here: a dialog about a discussion only makes sense while that discussion is in the list), reset the sub-state's `MutableStateFlow` inline from the combine body when the upstream shape changes. The alternative — a separate `.onEach { … }` collector — costs more cognitive load and an extra coroutine; the inline `if (pending != null) pendingPromotion.value = null` is one line and reads in place. Annotate with a one-line comment explaining the invariant. This is the right shape for "data dependency between two pieces of state" without adding a third flow.
- **Two helpers, two strings, when a UI label and a persisted value diverge.** When the same `String?` source produces two different fallbacks depending on consumer (here: `"Untitled discussion"` for the dialog body, `"Untitled channel"` for the persisted `Conversation.name`), declare two `internal` file-scope helpers and route each consumer through one. Don't collapse them into a single helper with a parameter (`fallback: String`) — the call sites become parameterless and the intent is clearer with two named functions. Don't inline the `?: "literal"` at the call sites either — keeping the fallback logic in one place lets tests pin the exact strings.
- **Clear UI sub-state *before* the suspend launch in confirm handlers.** When a confirm action both dismisses a dialog and starts a suspend operation, set the dismissal state first and `viewModelScope.launch` after. This makes the UI update synchronously (no stale dialog during the suspend point) and a deterministic re-entry guard against double-taps (a second `PromoteConfirmed` arrives to a `null` snapshot and short-circuits). Don't reverse the order — `viewModelScope.launch { repository.X(); pendingPromotion.value = null }` opens a window where a fast double-tap fires `X` twice.
- **`data object` for confirm/cancel pairs; `data class` (with the entity id) for the open-request.** The open-request event carries the id from the gesture surface up to the VM (that's where the truth comes from); the confirm/cancel events have no payload because the VM already owns the truth (`pendingPromotion.value`). Resist the symmetry temptation of putting an id on `PromoteConfirmed` "to make it self-describing" — that gives the screen a way to disagree with the VM about which discussion is being promoted.
- **`stringResource` with a `%1$s` placeholder fed by a UI helper.** When a dialog body interpolates a user-visible identifier, route the identifier through a named helper (`displayLabel`) at the composable boundary rather than passing the raw `Conversation.name?` into `stringResource(..., name)`. The helper carries the fallback string; the resource carries the framing. This keeps the resource file free of fallback noise and pins fallback choice in code (where tests can lock it).

## Lessons learned

- **`combine` is *not* a drop-in for `map` when seeding a `stateIn` with an initial value — but the project's existing shape handles it.** `combine` does not emit until *every* upstream has produced at least one value. The upstream `observeConversations` is hot-shared but cold on the first subscriber; `pendingPromotion: MutableStateFlow` has the initial `null`. Until the first repository emission lands, the combined flow produces nothing — but `stateIn(initialValue = Loading)` covers exactly that gap, so subscribers see `Loading` until the first combine emission. If a future flow lacks a `stateIn` seed, swap to `pendingPromotion.flatMapLatest { pending → upstream.map { … } }` or give the upstream an `onStart { emit(emptyList()) }` seed.
- **The combine body's inline `also { … }` arm vs. a leading `if (pending != null) pendingPromotion.value = null` line.** The architect's sketch used `Empty.also { if (pending != null) pendingPromotion.value = null }`; we shipped the leading-`if` form because the side effect happens *whether or not* the assignment is the projection's return value. Both compile, both work — the leading-`if` form reads better because the side effect is separated from the return expression. If a third reset condition lands later, this is the shape that scales (keep the projection at the bottom of the `when` arm; pile the resets above).
- **`(state as? DiscussionListUiState.Loaded)?.pendingPromotion` is the right read shape for "render an overlay only in one variant".** A `when (state) { is Loaded -> if (state.pendingPromotion != null) Dialog(...) else Unit; else -> Unit }` block is more verbose and adds a sealed `else -> Unit` arm that the compiler would not otherwise force. Stick with the safe-cast smart-elvis form whenever the overlay condition is a single nullable on a single variant.
- **`MutableStateFlow<T?>` with `null` initial value is the right primitive for "transient UI state that is sometimes absent".** Not a `Channel` (the dialog is a *state*, not an *event* — rotation needs to preserve it), not a `SharedFlow` with `replay = 1` (asymmetric — clearing requires emitting `null`, which has the same shape as setting). The combination of `MutableStateFlow<T?>.value` synchronous writes + `combine(...)` automatic re-projection on each write is exactly what's needed.
- **`viewModelScope.launch { repository.promote(...) }` is fire-and-forget and that's correct here.** The promote completion is *not* awaited before clearing the dialog. The dialog clears first (synchronous), the launch completes in the background, and the next `observeConversations` emission drops the now-promoted discussion from the discussion list (it's now a channel). No spinner, no in-flight state, no `isLoading` flag. If the user backs out of the screen between confirm and emission, the launch still completes — they confirmed; the conversation should still promote. Phase 4 adds its own dispatcher switching inside `RemoteConversationRepository`.
- **No try/catch around `repository.promote(...)`.** The ticket scope is an AC-level confirmation flow; an error surface (banner, snackbar, error UiState) would expand scope and pre-build infrastructure the rest of the app does not yet have. The `Fake` never fails for a valid id, and stale-id failures are out of scope (the `openPromotionDialog` guard filters them at request time, and a race between the long-press and the dialog opening that promotes a now-archived discussion would surface as an uncaught coroutine exception — acceptable for Phase 0). The next ticket that needs error-surfacing should introduce a project-wide pattern.
- **AC #6(d) "scrim/back dismiss without calling promote" is satisfied by construction.** The M3 `AlertDialog` defaults route both scrim taps and back-press through `onDismissRequest`; passing `onDismiss = { onEvent(PromoteCancelled) }` to that slot means both gestures emit `PromoteCancelled`, which the VM handles as `pendingPromotion.value = null` with no repository call. The composable-level test asserts the cancel-button → `PromoteCancelled` wire; the VM-level test asserts `PromoteCancelled` does not call `promote`. Together they satisfy the AC without a dedicated scrim-tap test (Compose's `AlertDialog` does not expose a stable test tag for the scrim, and an `Espresso.pressBack()` from a `createComposeRule()`-only test is awkward at this Compose BOM).
- **`Conversation.name` is the only "preview" available.** The ticket body suggested "preview text, truncated; falling back to 'Untitled channel'". `DiscussionListScreen` passes `lastMessage = null` to `ConversationRow` (no per-conversation message preview in Phase 0), so there is no message-derived preview to feed the dialog. Using `Conversation.name` as both the dialog identifier and the post-promote channel name is the simplest faithful reading of the ticket — and it matches what the user sees on the row (`ConversationRow.displayName` uses the same fallback shape).
