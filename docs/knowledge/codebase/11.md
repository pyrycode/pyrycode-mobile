# #11 — DataStore-backed `AppPreferences`

First real binding in the Koin `appModule` (scaffolded by #32). Adds a shared `DataStore<Preferences>` singleton and a thin typed wrapper exposing one app-level boolean (`pairedServerExists`) for downstream consumers.

## Summary

- `androidx.datastore:datastore-preferences` `1.1.7` added to the catalog and the `app` module.
- New `AppPreferences` wrapper at `data/preferences/AppPreferences.kt`. Takes a `DataStore<Preferences>` in its constructor (not a `Context`), so unit tests can supply a tmp-file-backed store without Robolectric.
- `appModule` gains two `single { ... }` bindings: the `DataStore<Preferences>` (built from `androidContext().preferencesDataStoreFile("app_prefs")`) and the `AppPreferences` wrapper that consumes it.
- One unit test (`AppPreferencesTest`) exercises the default and the round-trip against a real DataStore over a `TemporaryFolder` file.

## Patterns established

- **`data/preferences/` is the home for key-value app state.** Separate from `data/repository/` (conversation-domain data). Phase 3 Settings will add siblings here (`ThemePreferences`, notification toggles) using the same shape: one private `*PreferencesKey` constant, one `val xxx: Flow<T>`, one `suspend fun setXxx(...)`. No abstraction over preferences classes until ~5 keys (rule of three).
- **Preference wrappers take `DataStore<Preferences>`, not `Context`.** The Android binding lives in `AppModule.kt`; the wrapper itself is portable Kotlin and stays Compose-Multiplatform-friendly. This is also the testability seam — the unit test injects a tmp-file-backed store directly.
- **One `DataStore<Preferences>` singleton per on-disk file.** DataStore enforces a one-instance-per-file invariant at runtime, so the `DataStore<Preferences>` gets its own `single { }` (separate from the wrapper) so Phase 3 Settings can inject the same instance instead of constructing a second one against `app_prefs`.
- **The on-disk filename `app_prefs` is part of the contract.** Resolves to `<filesDir>/datastore/app_prefs.preferences_pb`. Renaming it later orphans every user's stored state — treat as locked from this ticket onward.
- **Preferences flavor, not Proto.** Stack Decision locks this; the typed-key API (`booleanPreferencesKey`, `intPreferencesKey`, …) is the only sanctioned path. Don't reach for `stringPreferencesKey` + `.toBoolean()` shortcuts.
- **No `Result` / `.catch { }` / error wrapping in preference wrappers.** DataStore handles `IOException`s in its own retry layer; surfacing a `Result<T>` is theatre. If on-disk corruption ever surfaces in the field, the right fix is a `ReplaceFileCorruptionHandler` at the `PreferenceDataStoreFactory.create` site — evidence-based, deferred until observed.
- **Default-on-miss with elvis, not a seed write.** `prefs[KEY] ?: false` avoids a cold-start write on first launch.
- **Unit tests use a real `DataStore` over `TemporaryFolder` + plain `runBlocking`.** No Robolectric, no `kotlinx-coroutines-test`, no MockK. The pattern matches `FakeConversationRepositoryTest` and is the project's preferred testing shape (fakes/real components over mocks). Each test owns its `CoroutineScope` and cancels it in `@After` so DataStore's internal coroutines don't leak between tests.

## Lessons learned

- **`PreferenceDataStoreFactory.create` is the DI-friendly entry point — not `Context.preferencesDataStore(name = ...)`.** The delegate creates a hidden `Context`-extension singleton that Koin can't see and forces callers to pass a `Context` through. Using the factory directly inside `single { ... }` keeps construction injectable.
- **`produceFile = { ... }` is called lazily.** Inside the test, calling `tmp.newFile(...)` inside the lambda (rather than at `@Before` time) means the file is created only when the DataStore first reads or writes — defensive but not load-bearing. Doing it eagerly also works.
- **Don't override `scope` on `PreferenceDataStoreFactory.create` in production code.** The default is `CoroutineScope(Dispatchers.IO + SupervisorJob())`, which is correct for app-scoped state. Passing `viewModelScope` would tear DataStore down on screen exit; passing `GlobalScope` is what the default already gives you. The test is the *only* place that overrides `scope`, because tests need a scope they can cancel in `@After`.
- **The transient `AppModule.kt` comment from #32 is being consumed line-by-line.** #11 removed the `#11 — AppPreferences (DataStore)` line and left the `#4` and "upcoming UI tickets" lines behind. When #4 lands it drops its line; whichever ViewModel ticket lands last deletes the whole block.
