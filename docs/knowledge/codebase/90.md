# #90 ‚Äî Settings: Version row wired to BuildConfig + Open-source row launches browser

Wires two previously-dead rows in `SettingsScreen`'s About section: the Version row now renders from generated `BuildConfig` instead of hardcoded literals, and the Open-source row dispatches an `ACTION_VIEW` `Intent` at `https://github.com/pyrycode/pyrycode-mobile` via `LocalContext`. Locked Figma row treatment from node `17:2` preserved ‚Äî dynamic data substitutes into the existing `headline` / `supporting` slots; the row shape is unchanged. License row is intentionally untouched (sibling ticket owns its viewer). First `createComposeRule()` instrumented test in the project lands here.

## Summary

- `app/build.gradle.kts:37-40` ‚Äî added `buildConfig = true` to `buildFeatures { }` so AGP 8.x generates `de.pyryco.mobile.BuildConfig`. `defaultConfig` already carried `versionCode = 1` / `versionName = "1.0"` since #2; only the `buildFeatures` toggle was missing.
- `SettingsScreen.kt` Version row: `headline = "Version ${BuildConfig.VERSION_NAME}"` + `supporting = "build ${BuildConfig.VERSION_CODE}"`. Reads inline at composition time ‚Äî `BuildConfig.*` are compile-time `const val`s, no `remember`, no recomposition trigger.
- `SettingsScreen.kt` Open-source row: `onClick = { context.startActivity(Intent(Intent.ACTION_VIEW, Uri.parse(SOURCE_REPO_URL))) }`. `val context = LocalContext.current` captured once at the top of the `Scaffold` content lambda (next to the existing three `remember`'d switch states). URL extracted to a file-private top-level `const val SOURCE_REPO_URL` at the bottom of the file (single use, but the test asserts on text containing `BuildConfig.VERSION_NAME` and the row label `"Open source"` ‚Äî keeping the URL out of the row literal makes the eventual second consumer / move trivial).
- New `app/src/androidTest/java/de/pyryco/mobile/ui/settings/SettingsScreenTest.kt` ‚Äî two `@Test` methods using `createComposeRule()` and `PyrycodeMobileTheme { SettingsScreen(onBack = {}) }`. `versionRow_rendersBuildConfigVersionName` scrolls to and asserts existence of a node matching `hasText("Version ${BuildConfig.VERSION_NAME}", substring = true)` ‚Äî substring + live `BuildConfig` read so the test stays correct when `versionName` is bumped. `openSourceRow_hasClickAction` scrolls to `hasText("Open source", substring = true)` and asserts `hasClickAction()`. No `Intent`-launch verification per AC.
- `startActivity` call is unguarded ‚Äî no `try`/`catch` for `ActivityNotFoundException`. Min SDK 33 + a default-browser-resolvable `https://` URI makes the failure mode irrelevant on supported hardware.

## Patterns established

- **`LocalContext.current` captured at the composable body's top for one-shot external-URL launches.** Inside `SettingsScreen`'s `Scaffold` content lambda ‚Äî adjacent to the existing `remember { mutableStateOf(...) }` switch states ‚Äî read `val context = LocalContext.current` once and close over it inside row `onClick` lambdas that need to dispatch `Intent.ACTION_VIEW`. Don't thread an `ActivityResultLauncher` through composable parameters, don't pull a context from a child composable's scope, don't pass it as a `SettingsScreen` parameter ‚Äî the lambda-capture is the project-wide shape for one-shot `startActivity` calls from a stateless screen. Generalises to any future Settings / About / Help row that opens a URL.
- **`Intent.ACTION_VIEW` + `Uri.parse(...)` is unguarded for `https://` URIs on min SDK 33+.** No `try`/`catch` around `context.startActivity(...)`. The theoretical `ActivityNotFoundException` requires a device with zero browsers installed; on supported hardware it never fires, and adding the catch creates an empty-branch maintenance hazard. If a follow-up ever needs a custom-tab / in-app web view fallback it lands in its own ticket ‚Äî wrapping the unconditional path now would be premature.
- **`BuildConfig.*` read inline at composition time, no `remember`, no constants in source.** `const val` reads are compile-time stable. Inlining `BuildConfig.VERSION_NAME` / `VERSION_CODE` directly into the `headline` / `supporting` string templates (rather than pulling them through `val name = BuildConfig.VERSION_NAME` at the top) keeps the wiring obvious at the call site and matches the "single-file diff for Phase 3 data wiring" precedent established in #64.
- **File-private `const val` for a single-use external URL when a test should ignore the literal.** `SOURCE_REPO_URL` lives at the bottom of `SettingsScreen.kt` even though it's used exactly once. The motivation isn't reuse ‚Äî it's that the test asserts on the row's *label* text (`"Open source"`), not the URL string; pulling the URL out of the row's `onClick` lambda body keeps the literal off the row's text-rendered surface so a `hasText` matcher can never accidentally key on it. If a second consumer ever needs the same URL it just imports the const ‚Äî same shape, no rewrite.
- **First `createComposeRule()` instrumented test pattern.** `SettingsScreenTest.kt` is the project's first Compose instrumented test. Shape: single `@get:Rule val composeTestRule = createComposeRule()`, `composeTestRule.setContent { PyrycodeMobileTheme { <Screen>(...) } }`, find-by-`hasText(substring = true)` ‚Üí `performScrollTo()` ‚Üí assertion. Wrap in `PyrycodeMobileTheme` so `MaterialTheme.colorScheme` / `typography` resolve; use `createComposeRule()` (not `createAndroidComposeRule<‚Ä¶>()`) when nothing launches an `Activity` (the Intent dispatch is not exercised by the test).

## Lessons learned

- **AGP 8.x's `buildFeatures.buildConfig` defaults to `false`.** Without the explicit `buildConfig = true` toggle, `BuildConfig` is not generated and `BuildConfig.VERSION_NAME` won't resolve at compile time ‚Äî even though `defaultConfig.versionName` / `versionCode` are set. AGP 7.x generated `BuildConfig` unconditionally; the 8.x default flipped to opt-in. Heuristic: any future ticket that needs `BuildConfig` reads must check this flag is on, not just that `versionName` exists in `defaultConfig`.
- **`performScrollTo()` is required before asserting `assertIsDisplayed` / `assertExists` on the About rows.** `SettingsScreen`'s body is a `Column(Modifier.verticalScroll(...))` and the About section sits at the bottom; on a default 412dp-wide test viewport the About rows render off-screen at composition. A naked `onNode(hasText(...)).assertExists()` happens to pass (semantics tree captures the off-screen node), but `assertIsDisplayed()` doesn't, and matcher debug output mis-reports the missing-from-tree-vs-off-screen distinction. The robust shape is `onNode(...).performScrollTo().assertExists()` (or `.assert(hasClickAction())`) every time. Generalises to any `verticalScroll` screen whose target row may sit below the viewport.
- **`ListItem`'s merged-descendant semantics carry the row's `clickable` `OnClick` on the same node that holds the headline `Text`.** No need to navigate the semantics tree from the text node up to the `clickable` ancestor ‚Äî `onNode(hasText("Open source", substring = true)).assert(hasClickAction())` resolves cleanly because M3 `ListItem` merges its descendant semantics by default. If a future row ever needs the unmerged tree (e.g. to assert on a child node's properties), use `useUnmergedTree = true` on the finder.
- **The Version-row substring matcher reads `BuildConfig.VERSION_NAME` directly, not a hardcoded `"Version 1.0"` literal.** The first draft of the test pinned `"Version 1.0"`; this would silently break the moment `versionName` bumps to `"1.1"` and force a doc/test edit in lockstep with every release. Reading the matcher from `BuildConfig.VERSION_NAME` itself means the test only fails if the *wiring* breaks (e.g. the row reverts to a constant), which is the actual invariant. Same principle applies to any test that asserts on data that *originates* in the system-under-test.
