# #24 — Discussion list drilldown screen

Standalone destination for the unpromoted (discussion) tier of conversations: `DiscussionListScreen` + `DiscussionListViewModel`, both colocated next to their channel-list siblings in `ui/conversations/list/`. Same MVI shape as the channel list, same row component, visually de-emphasized via an outer `Modifier.alpha(0.65f)` at the call site. Mounted at the `discussions` route in `PyryNavHost`; the entry point that links here (the "Recent discussions (N) →" pill on the channel list) is owned by #26 and is not wired yet.

## Summary

- `DiscussionListViewModel` mirrors `ChannelListViewModel` structurally: same `stateIn(WhileSubscribed(5_000), initial = Loading)` projection of `observeConversations(...)` into a sealed `UiState` (`Loading` / `Empty` / `Loaded` / `Error`), same `Channel<…>(BUFFERED) → receiveAsFlow()` one-shot navigation seam. Differences: filters on `ConversationFilter.Discussions`, error fallback string is `"Failed to load discussions."`, and there is no suspend-shaped event arm — the only consumed event is `RowTapped` (sends `ToThread(id)`); `BackTapped` is `Unit`.
- `DiscussionListScreen` mirrors `ChannelListScreen`'s `Scaffold + when (state)` skeleton with a `LazyColumn` of `ConversationRow`s in `Loaded`. The TopAppBar carries a `navigationIcon = ArrowBack` (`Icons.AutoMirrored.Filled.ArrowBack`) instead of a settings gear, no FAB, and no trailing actions. Each row in the `LazyColumn` is wrapped with `modifier = Modifier.alpha(0.65f)` at the call site — the de-emphasis mechanism for the secondary tier. `ConversationRow` itself was **not** modified.
- New `discussions` route in `PyryNavHost`. Destination block runs the standard pattern: `koinViewModel<DiscussionListViewModel>()` + `collectAsStateWithLifecycle()`, `LaunchedEffect(vm) { vm.navigationEvents.collect { … } }` translating `DiscussionListNavigation.ToThread` into `navController.navigate("conversation_thread/$id")`, and an inline `onEvent` lambda that *also* navigates on `RowTapped` and `popBackStack`s on `BackTapped`. The dual nav wiring is deliberate — see "Patterns established".
- Koin: one new `viewModel { DiscussionListViewModel(get()) }` line in `AppModule.kt` under the existing `ChannelListViewModel` registration. No new singletons; the repository binding is reused.
- Strings: three new entries in `values/strings.xml` — `discussion_list_title` ("Recent discussions"), `discussion_list_empty` ("No discussions yet"), `cd_back` ("Back"). `cd_back` is the first reusable back-button content description; future screens with an `ArrowBack` icon should reuse it.
- Tests: `DiscussionListViewModelTest.kt` covers all six scenarios from the spec (initial Loading, Loaded passthrough, empty → `Empty`, `RowTapped` → navigation event, error message passthrough, null-message fallback). Reuses the channel-list test conventions verbatim — `UnconfinedTestDispatcher` + `Dispatchers.setMain`, hand-rolled `stubRepo` with `MutableSharedFlow<List<Conversation>>(replay = 0)`, `async { vm.navigationEvents.first() }` launched before the trigger.
- `./gradlew assembleDebug` and `./gradlew test` pass.

## Patterns established

- **De-emphasis via outer `Modifier.alpha(...)` at the call site, not a parameter on the row component.** When the *same* shared composable needs to render with reduced visual weight at a *single* call site, the cheapest path is `Modifier.alpha(...)` on the outer modifier. Adding a `deEmphasized: Boolean = false` parameter to `ConversationRow` would have forced every existing call site (both channel-list `Loaded` and the two `ConversationRow.kt` `@Preview`s) to pass `deEmphasized = false`, with zero current benefit. The alpha approach also survives Material 3 ripples correctly — the ripple renders on the un-alpha'd `ListItem` content, while the outer alpha dims everything uniformly. Use **0.65** as the secondary-tier dim factor; values < 0.5 read as disabled rather than de-emphasized.
- **Dual navigation wiring (composable-side `navController.navigate` AND VM-side one-shot channel) for nav events the test asserts on.** The spec's AC required "tapping a row emits a navigation event whose target is `conversation_thread/{id}` ... asserted via the ViewModel's navigation flow in a unit test", so the VM must `send` to a `navigationChannel` on `RowTapped`. The production navigation path runs *directly* in the destination block's `onEvent` lambda via `navController.navigate(...)` (same as channel-list `RowTapped`), so a user tap never depends on the VM-side path. Both wires emit; the test observes the VM-side one. If a future ticket consolidates navigation through the VM, the composable-side branch comes out then. Don't optimize the redundancy away now — the test contract is explicit and the cost is one extra `send`.
- **No-op event arms stay in the sealed type, dispatched as `Unit` in the reducer.** `DiscussionListEvent.BackTapped` is handled by `navController.popBackStack()` at the destination; the VM's `onEvent` includes a `DiscussionListEvent.BackTapped -> Unit` arm purely so the sealed `when` compiles exhaustively. Keep the event in the sealed type even when the VM does nothing with it — it documents intent for future analytics/tests, and `-> Unit` is one line.
- **`cd_back` is the project-wide back-button content description.** First screen with a back arrow; the resource key is intentionally screen-agnostic. Future drilldown screens reuse `R.string.cd_back` rather than introducing `cd_<screen>_back`. This sits alongside the per-screen user-copy convention (`channel_list_empty`, `discussion_list_empty`, `discussion_list_title`) established by #23 — `cd_*` for content descriptions, `<screen>_<role>` for visible copy.

## Lessons learned

- **The `Modifier.alpha(0.65f)` placement matters for `ListItem` ripples.** Applying alpha as the *outer* modifier of `ConversationRow` (which internally uses `ListItem`) preserves the ripple feedback at full opacity within the row's clickable region — Material 3 ripples render against the un-alpha'd inner surface. Applying alpha *inside* `ConversationRow` (e.g. on the `ListItem`'s color slots) would have flattened the ripple as well. The visual contract is "dimmed content, normal interaction"; alpha-from-outside delivers that without ceremony.
- **`Modifier.alpha` does not block hit-testing.** Tapping an alpha-0.65 row still fires the row's `onClick`. This is the correct behavior (discussions are interactive, just secondary), but worth knowing if a future ticket wants a truly-disabled visual — that needs `enabled = false` on the `ListItem` (Material 3 propagates a disabled color scheme through `LocalContentColor`), not alpha.
- **`@Preview` is the entire visual-regression contract.** No instrumented test, no screenshot diffing. The spec's AC enforces a side-by-side preview (`DiscussionRowVsChannelRowPreview`) precisely so a future change to `ConversationRow` that breaks the de-emphasis is observable in Android Studio. Don't move the `Column { channelRow; discussionRow }` preview out of `DiscussionListScreen.kt` — it lives with the file that owns the alpha call site.
- **The "workspace label is absent on discussion rows" AC is already satisfied by `ConversationRow`.** `ConversationRow` calls `condenseWorkspace(conversation.cwd)` which returns `null` for `DefaultScratchCwd`; the `WorkspaceChip` is gated on a non-null result. Discussions seeded by the fake use `DefaultScratchCwd`, so the chip simply doesn't render. The AC exists as a regression tripwire: if a future change to `ConversationRow` or `condenseWorkspace` re-introduces a label for the default scratch cwd, the `DiscussionListScreenLoadedPreview` would surface it visually. No code in this ticket enforces the rule — the preview *is* the enforcement.
- **The VM emits a navigation event the production path doesn't actually consume.** It would be tempting to skip the `viewModelScope.launch { navigationChannel.send(...) }` on `RowTapped` and route taps only through the composable-side `navController.navigate(...)`. Don't — AC #4 demands the VM-observable path. The dual wiring reads as "redundant" but is testable; collapsing to a single wire would force the test to either spin up a `TestNavHostController` (out of scope, #8 deferred that) or assert on `navController` interactions (brittle). The right shape is two wires, both emit, test observes the VM one.
