# #14 — Welcome `onSetup` → external browser

Replaces the `TODO(#14)` placeholder in the `composable(Routes.Welcome) { ... }` block with an `Intent.ACTION_VIEW` launch at `https://pyryco.de/setup`. Closes the second half of the original Welcome-CTA wiring; the `onPaired → Scanner` half shipped in #12. `WelcomeScreen.kt` itself is untouched — `onSetup: () -> Unit` stays a hoisted, `Context`-free callback.

## Summary

- One file changed: `app/src/main/java/de/pyryco/mobile/MainActivity.kt` (+9 / −1).
- New `private const val SetupUrl = "https://pyryco.de/setup"` at file scope, placed between `SettingsPlaceholder` and `private object Routes`. Not inside `Routes` — `Routes` is the route-name namespace, a URL isn't a route.
- Inside `composable(Routes.Welcome) { ... }`: `val context = LocalContext.current` captured at the top of the block; the `onSetup` lambda body is `context.startActivity(Intent(Intent.ACTION_VIEW, Uri.parse(SetupUrl)))`.
- Three new imports: `android.content.Intent`, `android.net.Uri`, `androidx.compose.ui.platform.LocalContext`. No new Gradle dependencies; all three are already on the classpath (Android SDK + Compose UI).
- No `try` / `catch` around `startActivity`. The `ActivityNotFoundException` edge case is explicitly deferred (ticket's "Out of Scope" + project [[Evidence-Based Fix Selection]] principle) — stock Android 13+ resolves `https` `ACTION_VIEW` unconditionally. Wrap-and-toast becomes a local edit at this same site if the failure is ever observed.
- No new tests. The change is one SDK call wired to a callback; existing `assembleDebug` + `lint` cover compile / preview / static checks, and the `WelcomeScreen` previews still pass `onSetup = {}` no-ops (no preview-only branching needed).

## Files touched

- `app/src/main/java/de/pyryco/mobile/MainActivity.kt` — three new imports (lines 3-4, 21), `LocalContext.current` capture + `onSetup` body (lines 77-88), new `SetupUrl` constant (line 131).

## Patterns established

- **Per-destination `LocalContext.current` capture for one-shot `Context.startActivity` calls.** Read `LocalContext.current` at the top of the `composable(...)` block, not inside the callback lambda — `CompositionLocal` reads belong in composable code. The captured reference flows into the lambda closure; the `Intent` is constructed lazily on tap. No `remember { ... }` wrapping: `LocalContext.current` is stable for the lifetime of the composition and re-reading on recomposition is a no-op. Same shape as `koinInject<X>()` + `rememberCoroutineScope()` in the `Routes.Scanner` block (#12).
- **File-scope `private const val` for one-off URL constants, next to `Routes`, not inside it.** Greppable, no separate `Urls` namespace, no new singleton. If a second URL appears, the right move is *still* a sibling `const val`, not premature grouping — wait until there are three or there's a non-trivial reason (env switching, build-flavor variants) to extract.
- **`Uri.parse(SetupUrl)`, not `SetupUrl.toUri()`.** Both work (`androidx.core:core-ktx` ships `.toUri()` and is already on the classpath), but `Uri.parse` is the universally recognized idiom and avoids importing an extension whose only job is to call the same parse method.
- **Defer `ActivityNotFoundException` hardening for `https` `ACTION_VIEW` on min-SDK 33.** Ship the defense when the failure is observed. No `<queries>` manifest declaration is required either — apps targeting API 30+ don't need `<queries>` to *launch* an intent they don't introspect; package-visibility restrictions only bite when enumerating or pre-checking resolvers in code.
