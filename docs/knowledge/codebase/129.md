# #129 — feat(ui/thread): markdown rendering in assistant messages

Re-skins the `Role.Assistant` arm of `MessageBubble` so its `content` string is rendered as CommonMark markdown instead of plain text. User-variant rendering, host container, alignment, and the file-private spacing constants in `MessageBubble.kt` are all unchanged. Ships a new reusable `MarkdownText` composable that owns the AST walk; the user variant continues to render plain `Text(...)` per AC2.

## Summary

- One new file: `app/src/main/java/de/pyryco/mobile/ui/conversations/components/MarkdownText.kt` (332 lines including imports, six file-private composables, the inline-builder helper, six file-private spacing constants, and the link-safety predicate).
- Three edits to existing files: `MessageBubble.kt` (swap the inner `Text(...)` for `MarkdownText(...)` inside a `CompositionLocalProvider(LocalContentColor provides onSurface)`, plus a new pair of markdown-specific previews), `gradle/libs.versions.toml` (one `[versions]` line + one `[libraries]` line), `app/build.gradle.kts` (one `implementation(libs.jetbrains.markdown)` line in the `dependencies { }` block).
- Library is **`org.jetbrains:markdown` 0.7.3** — pure-Kotlin CommonMark parser, no Android dependencies, no transitive AndroidX. The Compose rendering side is hand-rolled in this codebase, not delegated to `dev.jeziellago:compose-markdown` (which was the ticket-suggested "preferred" library but is a Markwon-on-`AndroidView` wrapper — the rationale for rejecting it is recorded in [ADR 0002](../decisions/0002-markdown-renderer-library.md)).
- `MarkdownText(markdown, modifier)` has a deliberately minimal signature — **no `style`, no `color`, no `onLinkClick`**. Text styles resolve from `MaterialTheme.typography.<slot>` by element kind (h1 → `headlineSmall`, h2 → `titleLarge`, h3 → `titleMedium`, body / lists / blockquotes / code blocks → `bodyMedium`, with `FontFamily.Monospace` overlay on code). Text colour inherits from `LocalContentColor.current`; the `AssistantMessage` consumer supplies `MaterialTheme.colorScheme.onSurface` via `CompositionLocalProvider(LocalContentColor provides …) { MarkdownText(...) }`. Links route through `LocalUriHandler.current.openUri(url)` directly — no consumer-supplied tap callback.
- Element set covered (matches AC3 + AC4): ATX headings h1–h3, bold (`STRONG`), italic (`EMPH`), inline code (`CODE_SPAN`), ordered list, unordered list, blockquote (italic body, `outlineVariant` 4dp bar), inline link (`INLINE_LINK`), fenced code block (`CODE_FENCE`), indented code block (`CODE_BLOCK`). Unsupported AST kinds (tables, HTML blocks/inline, horizontal rules, strikethrough, task lists, images, autolinks beyond `[text](url)`) hit the `else -> Text(node.getTextInNode(source).trim(), style = bodyMedium)` fallback in `MarkdownBlock` / `appendInline` — the message is never blank, the visual degrades to readable raw source.
- Fenced code blocks render **flat** — `Surface(shape = RoundedCornerShape(CodeBlockCornerRadius = 8.dp), color = surfaceContainer)` wrapping a `Text(monospace, padded 12×8)`. **No language label, no outline border, no syntax highlighting** — all three land in #130. The fence's info string (e.g. `kotlin` after the opening fence) is discarded.
- Link safety: file-private `isSafeLinkScheme(url): Boolean` allowlists `http` / `https` / `mailto` after a `.lowercase()` of the prefix before the first `:`. The `LinkInteractionListener` consults the predicate before calling `uriHandler.openUri(url)`. Rejected URLs no-op silently — the underlined span remains drawn, the tap does nothing. The label `security-sensitive` on the ticket is satisfied here, not deferred. Architect re-run verdict: **PASS**.
- Parse is wrapped in `remember(markdown) { MarkdownParser(MarkdownFlavour).buildMarkdownTreeFromString(markdown) }` — AST is rebuilt only when the source string changes. `MarkdownFlavour` is a file-private `val` of `CommonMarkFlavourDescriptor()` so flavour selection is one line away from `GFMFlavourDescriptor()` if Claude API responses turn out to lean on GFM (tables, strikethrough, task lists). Not switched preemptively.
- `MarkdownText` is a pure stateless composable — no `LaunchedEffect`, no coroutines, no flows. Parsing is synchronous and CPU-only; typical assistant messages parse in well under a millisecond. The `remember` is defensive against pathological multi-KB inputs, not a hot-path optimisation.
- Six file-private spacing constants at the top of `MarkdownText.kt` (`ParagraphSpacing = 8.dp`, `ListItemIndent = 8.dp`, `BlockquoteBarWidth = 4.dp`, `BlockquoteContentIndent = 12.dp`, `CodeBlockCornerRadius = 8.dp`, `CodeBlockPadding = PaddingValues(horizontal = 12.dp, vertical = 8.dp)`). Same shape as [`MessageBubble.kt`](../features/message-bubble.md)'s constants — local-until-shared. **No `.sp` literal anywhere in this file**; AC5 holds by construction.
- Preview pair added to `MessageBubble.kt` (not to `MarkdownText.kt`): `MessageBubbleMarkdownLightPreview` and `MessageBubbleMarkdownDarkPreview`, both `widthDp = 412`, both render an assistant-variant `MessageBubble` with a file-private `MARKDOWN_PREVIEW_FIXTURE` constant that exercises every AC element (h1/h2/h3, bold, italic, inline code, link to `https://pyryco.de`, unordered list, ordered list, blockquote, fenced Kotlin code block). The pre-existing `MessageBubbleLightPreview` / `MessageBubbleDarkPreview` pair continues to render the four-message User → Assistant sequence; those previews now also exercise the no-markdown-syntax path through `MarkdownText`. Satisfies AC6.
- No new tests — same precedent as #126 and #128. Codebase has no Compose UI test infrastructure (no `ComposeTestRule`, no Robolectric); building it for a pure-rendering 332-line component is over-scope. Visual verification is by `@Preview` rendering plus `./gradlew assembleDebug` + `./gradlew lint` passing. The architect's spec calls this precedent out as load-bearing.

## Patterns established

- **Library wire-up is the smallest possible diff: one `[versions]` line, one `[libraries]` line, one `implementation(...)` line — and nothing else.** No KSP, no kapt, no proguard rules, no Gradle plugin entry. When a candidate library would require any of those, that's a signal worth questioning the choice — the JetBrains markdown parser was a clean fit because it's pure Kotlin and reflection-free. Pattern applies to future leaf-library additions (`compose-shimmer`, `coil`, similar): if the wire-up grows beyond the three lines, surface that cost in the architect spec rather than absorbing it silently in the dev pass.
- **Renderer components that need M3 typography per-element resolve via `MaterialTheme.typography.<slot>`, not via a caller-supplied `style` parameter.** The `MarkdownText` signature deliberately omits `style: TextStyle`, because the renderer's job is to map element kinds (h1, body, code) to slots (`headlineSmall`, `bodyMedium` + monospace overlay). Exposing `style` would invite consumers to override one slot and accidentally re-tint everything else. The same posture applies to any future "render this structured content per-element" composable (tool-card body in #131, status-row in #145): own the typography map inside the component, expose only the input shape and `modifier`.
- **Text colour inherits via `CompositionLocalProvider(LocalContentColor provides …)` at the consumer call site, not via a `color: Color` parameter on the renderer.** Keeps the renderer reusable across surfaces (assistant message on `surface` → `onSurface`; future system message on `surfaceVariant` → `onSurfaceVariant`; future tool card on `primaryContainer` → `onPrimaryContainer`) without growing its signature. The consumer carries the colour, the renderer reads `LocalContentColor.current` through `Text`'s default-color path.
- **Security-sensitive labels resolve at the architect step with a deterministic mitigation in the spec, not as a "future TODO" in the codebase.** The `security-sensitive` label on #129 turned into the `isSafeLinkScheme` allowlist + the `LinkInteractionListener` guard before the developer started. Pattern applies to every future ticket carrying the label: the architect's pass must produce a concrete mitigation (or a documented "no surface here" finding), not a `// TODO: validate link URLs` shaped comment.
- **Allowlist over denylist for URL schemes.** The renderer accepts `http` / `https` / `mailto` and silently no-ops everything else. A denylist of `intent://` / `javascript:` / `file://` is fragile against case tricks (`InTeNt://`), whitespace tricks (`\tjavascript:`), and future scheme additions. Allowlist enforced at the `.lowercase()` scheme prefix is robust by construction. Pattern applies to any future "accept-or-reject" predicate over input strings — keep the accept set explicit, default-deny the rest.
- **Fallback for unsupported AST kinds is `Text(node.getTextInNode(source).trim(), style = bodyMedium)`, not `error("unsupported")`.** The renderer is total because the parser is total: any input string produces an AST. Throwing on an unsupported node kind would crash on the first table or HTML block Claude emits. Rendering the raw source as a `bodyMedium` paragraph degrades gracefully (the message is never blank), and the developer doesn't have to enumerate every `MarkdownElementTypes` and `MarkdownTokenTypes` variant exhaustively. Pattern applies to any future AST-walking renderer: the `else ->` arm IS the contract, not a stub.

## Lessons learned

- **`CompositionLocalProvider(LocalContentColor provides …)` is the right shape for surfaces that own a colour contract and host a coloured renderer.** Considered passing `color = MaterialTheme.colorScheme.onSurface` into a hypothetical `MarkdownText(markdown, color, modifier)` parameter; the `CompositionLocalProvider` approach is one line longer at the call site but keeps the renderer signature minimal and means future colour-aware spans inside the renderer (e.g. tool-card destructive accents) read the same ambient. Compose's `Text` already picks up `LocalContentColor` as its default — no extra plumbing inside the renderer.
- **`org.jetbrains:markdown`'s AST tokens require both `MarkdownElementTypes` (block kinds, composite inline kinds like `EMPH` / `STRONG` / `CODE_SPAN` / `INLINE_LINK`) and `MarkdownTokenTypes` (leaf tokens like `EOL`, `WHITE_SPACE`, `LBRACKET`, `LIST_BULLET`, `ATX_HEADER`).** A `when` over only `MarkdownElementTypes` misses the leaf-token leaves of the tree; a `when` over only `MarkdownTokenTypes` misses the composite elements. The renderer uses both, and the `MarkdownBlock` / `appendInline` `when`-arms reference values from both enums (`MarkdownTokenTypes.CODE_FENCE_CONTENT`, `MarkdownTokenTypes.CODE_LINE`, `MarkdownTokenTypes.EOL`, `MarkdownElementTypes.PARAGRAPH`, etc.). Pin: when extending the renderer (e.g. adding strikethrough), check both enum sources for the right type constant.
- **`Row(modifier = Modifier.height(IntrinsicSize.Min))` is the right shape for "blockquote bar fills the height of the quoted content".** The bar `Box(width = 4.dp, fillMaxHeight, background = outlineVariant)` needs a parent that gives `fillMaxHeight` a finite bound — without `IntrinsicSize.Min`, the row's intrinsic height is unbounded and `fillMaxHeight` collapses to 0. `IntrinsicSize.Min` measures the row's children to their minimum intrinsic height, then the bar `fillMaxHeight`'s to that. Same idiom is the right starting point for any future "vertical separator beside variable-height content" pattern.
- **Heading inline content needs explicit token filtering — `ATX_HEADER` (the `#` marker), `WHITE_SPACE` (the space after), and `EOL` (the trailing newline) are sibling tokens of the heading's `TEXT` content, not stripped by the parser.** `HeadingBlock` filters those three out before walking the remainder through `appendInline`. Without the filter, the `#` marker renders as literal text in the heading. Same shape applies if a future ticket adds setext headings (lines underlined with `===` / `---`) — they're a different AST shape and would need their own filter.
- **`./gradlew assembleDebug` + `./gradlew lint` is the full automated verification surface for a Compose-only renderer with no logic worth unit-testing.** Project has no Compose UI test infrastructure (same as #126, #128); building it for a 100-LOC component is over-scope. The two preview pairs (`MessageBubbleLightPreview` / `MessageBubbleDarkPreview` exercising plain-text-through-`MarkdownText`, and `MessageBubbleMarkdownLightPreview` / `MessageBubbleMarkdownDarkPreview` exercising every AC element) are the visual verification surface in Android Studio. The architect spec called this out as load-bearing precedent — pin so future rendering tickets don't drift into building infrastructure for a single component.
