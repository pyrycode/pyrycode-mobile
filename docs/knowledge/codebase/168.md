# #168 — Welcome logo M3 elevation per Figma 6:32

Single-modifier addition. `WelcomeScreen.kt`'s logo `Icon` gains `Modifier.shadow(elevation = 6.dp, shape = RectangleShape, clip = false)` chained **before** `.size(width = 92.dp, height = 104.dp)` so the framework synthesises the two-tier (key + ambient) M3 Elevation Level 3 drop-shadow under the snowflake glyph. Mirrors the post-#167 Figma re-author at node `6:32` that bound `M3/Elevation Light/3` to the Hero group (node `6:33`) — scoped here to the logo `Icon` only, since on the dark canonical surface (`#101418`) only the bright tinted glyph has enough luminance contrast for the shadow to read at all.

## Summary

- **`Modifier.shadow(...)` over `Surface(tonalElevation = ...)` or a custom `drawBehind` blur.** The Figma effect is purely a `DROP_SHADOW` (two layers — key `#0000004D` offset (0,1) blur 3, ambient `#00000026` offset (0,4) blur 8 spread 3) with no tonal change to the surface. M3's `tonalElevation` lifts the surface *colour* via `surfaceTint` overlay and does not draw a shadow; the correct M3 token for shadow rendering is `Modifier.shadow` (equivalently `Surface(shadowElevation = ...)`). `Modifier.shadow` is the smaller edit — no new composable layer, no `color = Color.Transparent` workaround to suppress a Surface fill, no spurious click/semantic surface. A hand-rolled `drawBehind { ... BlurMaskFilter ... }` would re-derive what `Modifier.shadow(elevation = 6.dp)` already gives us (M3 Level 3 == 6.dp shadow elevation; the framework synthesises both layers from the single dp value).
- **Modifier order: `.shadow(...)` before `.size(...)`.** Compose modifiers apply outside-in for layout; placing `.shadow(...)` first wraps the size modifier so the shadow draws in the area the size modifier reserves. Putting `.size(...)` first would size the Icon and then attempt to draw the shadow over a zero-size wrapper.
- **`shape = RectangleShape, clip = false`.** The shadow's outline is the Icon's 92 × 104 dp bounding rectangle, **not** the snowflake's alpha-derived silhouette (see "Lessons learned" → Compose shadow ≠ CSS `drop-shadow`). `clip = false` is the documented "shadow-only, no clipping intent" convention; default would be `clip = (elevation > 0.dp)` == `true`, which is a no-op here since `RectangleShape` already matches the Icon's pixel bounds, but explicit `false` keeps a future `ContentScale` change from clipping the Icon by accident.
- **No `ambientColor` / `spotColor` override.** Defaults to Compose's `DefaultShadowColor` (`Color.Black`), matching Figma's `#000000`-with-alpha shadow base. AC3 ("no hardcoded `Color(0xFF…)` for shadow tint — uses theme tokens") is satisfied by construction: no per-call hex literal is introduced, and M3's `ColorScheme` exposes `outline` / `outlineVariant` / `scrim` but **no `shadow` token** — there is no theme token *to* use. Framework default is the canonical answer.
- **Two new imports, alphabetised into the existing draw and graphics blocks.** `androidx.compose.ui.draw.shadow` (right after `drawBehind`) and `androidx.compose.ui.graphics.RectangleShape` (alongside `Brush` / `Color`). +9 / −1 LOC, one file.
- **Scope: logo `Icon` only, not the Hero group.** Figma binds the effect on Hero (`6:33`), but on dark surface the text glyphs (`onSurface = #E0E2E8` / `onSurfaceVariant = #C2C7CF`) are too light-grey on too-near-black for a black-with-alpha shadow to be visible underneath them. Per-glyph `TextStyle(shadow = Shadow(...))` on the three `Text` composables would be the faithful translation under text, but is out of scope; the Hero-level `Modifier.shadow` would produce one rectangular shadow under the entire ~280 dp Hero column — visually heavy and unlike Figma's per-element CSS filter. Same scope discipline as #167's "out-of-scope second caller" rule.

## Patterns established

- **For Material 3 elevation effects sourced from Figma, default to `Modifier.shadow(elevation = <m3-level-dp>, shape = <bbox-shape>, clip = false)`.** The framework maps a single dp value to the M3 two-tier (key + ambient) shadow recipe — Compose's renderer does what Figma's `M3/Elevation Light/<N>` effect describes, without per-layer parameters. M3 dp lookup: Level 1 = 1.dp, Level 2 = 3.dp, Level 3 = 6.dp, Level 4 = 8.dp, Level 5 = 12.dp (`m3.material.io/styles/elevation/tokens`). Reach for `Surface(tonalElevation = ...)` only when the Figma effect is a *tonal* lift (no `DROP_SHADOW` layer, just a `surfaceTint` overlay), and reach for custom `graphicsLayer { renderEffect = BlurEffect(...); shadowElevation = ... }` only when the bounding-box rectangle shadow is judged visibly wrong against the design source.
- **Modifier chain order for shadowed sized elements: `.shadow(...)` before `.size(...)`.** Compose layout modifiers compose outside-in — the *first* modifier in the chain wraps the rest. Putting `.shadow(...)` first means the shadow is drawn around the area that `.size(...)` reserves; reversing the order draws the shadow over a wrapper of unspecified size. Project precedent for any future `Modifier.shadow + Modifier.size` pairing.
- **AC "no hardcoded `Color(0xFF…)` for X — uses theme tokens" reads as a ban on per-call-site hex literals, not a requirement to invent a new `MaterialTheme` slot.** When M3's `ColorScheme` exposes no token for the role in question (here: `shadow`), the framework default (e.g. `DefaultShadowColor`, `Color.Black`, `Color.Transparent`) satisfies the AC by construction. Same shape as #150's drawable-XML-literals-are-spec-permitted carve-out and #167's `#FFFFFF`-tinting-bed precedent — the AC's intent is "no surprise per-call colour decisions", not "every colour must be a `ColorScheme` field".

## Lessons learned

- **Compose `Modifier.shadow(elevation, shape)` traces the `shape` parameter's geometry, *not* the modified element's alpha channel.** Figma's CSS `drop-shadow(...)` filter traces the rendered alpha, so the Figma preview shows a snowflake-shaped shadow tracing the glyph silhouette; Compose's preview shows a rectangle the size of the Icon's bounding box. The three available silhouette-tracing alternatives in Compose are all disproportionate to a one-modifier ticket: (1) build a `GenericShape` from the drawable's `pathData` — requires parsing the 6 KB SVG path string per recomposition; (2) bake the shadow into the drawable XML — Android Vector Drawable has no blur primitive, so this requires API-31+ `RenderEffect` and the blur is then static and not lifted from any M3 token; (3) `graphicsLayer { renderEffect = BlurEffect(...); shadowElevation = ... }` with a custom clip path — same `GenericShape` problem as (1) with added complexity. The visual delta vs. Figma is tolerable on dark surface (rectangular shadow under a 92 × 104 dp box reads as a barely-perceptible soft glow at M3 L3 contrast on `#101418`) but more visible on light surface (`#F8F9FF`) where the rectangle outline reads more sharply; if a future reviewer judges the difference unacceptable, the follow-up is `graphicsLayer { renderEffect = BlurEffect(...) }` or a baked-in drawable shadow, both weeks of detail work for a Figma-faithful silhouette shadow. Don't reach for those without observed need.
- **M3 `ColorScheme` has no `shadow` token.** Reading `androidx.compose.material3.ColorScheme`'s field list directly confirms `outline`, `outlineVariant`, and `scrim` exist but no `shadow`. The Compose `Modifier.shadow` default colour is the constant `DefaultShadowColor` (`Color.Black`), which matches Figma's `#000000`-with-alpha shadow base by construction. Future tickets that want a `MaterialTheme.colorScheme.shadow`-shaped extension should follow the [warning color slot](../features/warning-color.md) pattern from #119 (`@Immutable data class ShadowColors` + private `CompositionLocal` + `@ReadOnlyComposable` extension property + provider inside `PyrycodeMobileTheme`); but don't add it until a real call site needs to vary shadow colour by theme, since the framework default is already correct on every surface this project ships today.
- **`tonalElevation` is not a drop-shadow.** M3's `Surface(tonalElevation = N.dp)` lifts the surface *colour* by compositing a `surfaceTint` overlay; it does **not** draw a shadow. If a Figma effect carries `DROP_SHADOW` layers, `tonalElevation` will produce no visible result (especially on a transparent Surface — the tint composites against the parent fill, which here is the radial-gradient backdrop, and the lift is invisible). For drop-shadow Figma effects use `Modifier.shadow` or `Surface(shadowElevation = ...)`; for tonal-only effects use `Surface(tonalElevation = ...)`. The ticket-body's option 2 ("wrapping in `Surface(tonalElevation = ...)`") was a misread of M3 terminology — the spec corrected it before the developer touched the code.
- **Figma binding `M3/Elevation Light/<N>` on a dark theme surface is intentional, not a copy-paste mistake.** The "Light" suffix in the M3 elevation token family refers to the *light-source* model (a small key light + a larger ambient light from above) that the Material spec uses to derive the two-layer shadow recipe, not the *theme* (light vs. dark). The same `M3/Elevation Light/3` token applies on dark surfaces — the shadow is just less visible against near-black. Don't reach for a `M3/Elevation Dark/...` token; that's not a thing in the M3 spec, and Figma's variable browser will not offer it.
