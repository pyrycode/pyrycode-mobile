# #15 ‚Äî ConversationThread placeholder route in NavHost

Adds the first parameterized destination to the NavHost: `conversation_thread/{conversationId}`. Same shape as the existing `channel_list` placeholder ‚Äî bare `Text(...)` body ‚Äî but the rendered string interpolates the received `conversationId` to prove the path argument round-trips. Downstream channel-list-row tap wiring (later in this wave) gets a real `navigate(...)` target without needing to wait on the Phase 2 thread UI.

## Summary

- One new route constant in `MainActivity`'s `private object Routes`: `ConversationThread = "conversation_thread/{conversationId}"`. The `{name}` placeholder lives inside the constant ‚Äî it's the literal pattern `composable(...)` requires; no "base + helper" split.
- One new `composable(...)` block inside `PyryNavHost`, declared with an explicit `arguments = listOf(navArgument("conversationId") { type = NavType.StringType })`. `NavType.StringType` is the default, but the explicit declaration documents the argument's type at the call site and makes a future type swap a one-token edit.
- Body extracts the id via `backStackEntry.arguments?.getString("conversationId").orEmpty()` and renders `Text("Conversation thread placeholder: $conversationId")`. No `Surface` / `Scaffold` / `Modifier` ‚Äî outer `Scaffold` in `MainActivity` owns chrome.
- Two new imports: `androidx.navigation.NavType`, `androidx.navigation.navArgument`. Both ship with the already-classpath `navigation-compose 2.9.5`; no catalog or build-file edits.
- Type-safe `@Serializable` routes deliberately **not** adopted. #8 deferred the migration to "the first parameterized route lands"; this ticket is that route, and the answer remains stay-on-strings. Partially migrating one route while `Routes.Welcome` / `Routes.ChannelList` stay as `String` constants is worse than either end-state ‚Äî a full migration is its own future ticket.

## Files touched

- `app/src/main/java/de/pyryco/mobile/MainActivity.kt` ‚Äî added `NavType` + `navArgument` imports, the new `composable(Routes.ConversationThread, arguments = ...)` block inside `PyryNavHost` (lines 56-62), and the `ConversationThread` constant in `Routes` (line 69).

## Patterns established

- **Parameterized routes declare arguments explicitly even when `NavType.StringType` is the default.** `arguments = listOf(navArgument(name) { type = NavType.StringType })`. The redundancy is intentional ‚Äî it documents the type at the call site and isolates the swap point if the underlying id type ever changes.
- **Path-argument extraction uses `backStackEntry.arguments?.getString(name).orEmpty()`.** The `?.` is for the type system; Compose Navigation guarantees presence before composition for required path arguments declared in the route pattern. `.orEmpty()` keeps the local as `String`, not `String?`, so call-site interpolation reads cleanly. Don't replace with `requireNotNull(...)` or `?: error(...)` ‚Äî the framework contract is the source of truth.
- **Placeholder destinations render `Text("<screen> placeholder: $arg")`** ‚Äî prefixed with the screen name (or analog) so the placeholder origin is obvious to anyone who lands on it during downstream debugging. Pure `Text(arg)` is too easy to mistake for production content.
- **The route constant string contains the literal `{name}` placeholder**, not a base form plus a builder. The graph DSL consumes the literal; callers building concrete targets write `"conversation_thread/$id"` inline. Lift to a `Routes.conversationThread(id)` helper only when a second or third caller appears.

## Lessons learned

None worth memorializing ‚Äî the registration shape follows the Compose Navigation idiom directly and matched the spec without surprises.
