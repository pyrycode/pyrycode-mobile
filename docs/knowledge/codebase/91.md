# #91 — Settings: in-app License viewer screen behind About License row

Adds a generic M3 secondary screen `LicenseScreen` that renders the MIT license text bundled at `app/src/main/assets/LICENSE`, reached from `SettingsScreen`'s About-section License row. The asset is a byte-identical copy of the repo-root `LICENSE` so the rendered content cannot drift from what shipped in the APK. No `ViewModel`, no repository, no coroutines — synchronous `AssetManager` read cached inside `remember(LICENSE_ASSET_NAME)`.

## Patterns established

- **One-shot synchronous asset read inside `remember(KEY) { ... }`, no `ViewModel`, no coroutines.** For tiny build-packaged assets (≤ a few KB of prose / config) used by a single stateless screen, read via `LocalContext.current.assets.open(name).bufferedReader().use { it.readText() }` directly inside a keyed `remember` block. The composable stays stateless, the read happens once per composition entry, and the cached `String` survives recomposition. Don't introduce a `ViewModel` + `StateFlow` for a sub-millisecond pure read — boilerplate-to-value ratio is wrong. Don't move the read off the main thread either; `AssetManager.open` for a ~1 KB asset is well below a single frame budget. The pattern generalises to any future "render this bundled text/config file" screen (open-source-licences list, third-party-notices page, sample-data fixtures used by previews).
- **Key `remember(...)` on the asset name even when it's a compile-time constant.** Use `remember(LICENSE_ASSET_NAME) { ... }`, not bare `remember { ... }`. The key never changes in practice (the asset name is a `const val`), but spelling out the cache invariant — "re-read iff the asset key changes" — makes the lifecycle explicit at the call site and matches the AC's "do not re-read on every recomposition" wording. Documentation by code, not behaviour.
- **No try/catch around build-packaged asset reads.** `assets.open(...)` failure on a checked-in asset is a packaging defect, not a runtime user condition. Let `IOException` propagate. Per the project's boundary-validation rule, build-packaged assets are not a system boundary — only validate at user input and external APIs.
- **New screen parameters land without defaults; existing call sites are updated together.** `SettingsScreen.onOpenLicense` is `() -> Unit` with no `= {}` default. The reasoning: a default value lets a call site silently forget to wire the navigation lambda, and "silently navigates nowhere" is a worse failure mode than "won't compile". Five call sites — one production, two previews, two tests — all updated in the same diff. Under the 10-call-site fan-out red line; bundle the parameter add with its call-site sweep rather than landing in two tickets.
- **Test-runtime string resolution from `R.string.*` for content-description matchers.** `LicenseScreenTest.backArrow_isPresent` resolves `R.string.cd_back` via `InstrumentationRegistry.getInstrumentation().targetContext.getString(R.string.cd_back)` rather than hardcoding the English literal. The matcher tracks the resource, not a frozen translation, so a future localization or wording tweak doesn't break the test. Same shape as the substring matcher in `SettingsScreenTest` reading `BuildConfig.VERSION_NAME` directly — assert on the source of truth, not a copy of it.

## Lessons learned

- **`createComposeRule()` (not `createAndroidComposeRule<…>()`) is sufficient for `LocalContext.current.assets` reads in an instrumented test.** `setContent { LicenseScreen(...) }` resolves `LocalContext.current` to a host `Context` provided by the Compose test infrastructure, whose `AssetManager` correctly resolves `assets/LICENSE` from the test APK. No `ActivityScenario` plumbing needed for asset-read-only tests; reach for `createAndroidComposeRule` only when the test itself needs to dispatch an `Activity`-scoped side effect (e.g. asserting that an `Intent` launch resolves to a specific Activity — which `SettingsScreenTest` deliberately doesn't do per its AC).
- **The androidTest substring matcher picks `"Permission is hereby granted"` over `"MIT License"`.** Both are stable across the byte-identical-copy guarantee. `"MIT License"` is only the first line and is a substring of strings other test fixtures might add later (a future "third-party notices" screen listing libraries' MIT headers). The mid-document fragment is a tighter regression guard against accidental edits to the asset. Generalises: when a substring matcher is your regression guard against asset drift, pick a phrase from the body, not the title.
- **`remember(asset_name)` rather than a `LaunchedEffect(asset_name) { mutableStateOf(...) }` shape.** The first instinct for "read I/O once per screen entry" is often `LaunchedEffect` + a coroutine, but `LaunchedEffect` cannot return a value into composition — you'd need an extra `var state by remember { mutableStateOf<String?>(null) }` and have to handle the null/loading interval at render time. For a synchronous read that completes well under a frame budget, the cleaner shape is `remember(key) { syncRead() }` and let composition block briefly. Reserve `LaunchedEffect` for genuinely-asynchronous work whose suspension you cannot eliminate.
