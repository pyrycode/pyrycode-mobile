# #187 — feat(data): sendMessage on ConversationRepository (interface + Fake impl)

Adds the missing message-append mutator to [`ConversationRepository`](../features/conversation-repository.md). One new `suspend fun sendMessage(conversationId, text): Message` on the interface; one new `state.update { ... }` block in `FakeConversationRepository` that mints a `Message(role = Role.User, sessionId = currentSessionId, timestamp = Clock.System.now(), isStreaming = false)`, appends it to the conversation's `messages`, and bumps the conversation's `lastUsedAt` to the same `now` so observers see a consistent pair. No UI wiring — the thread input bar that consumes this lands in a follow-up ticket. Phase 4's Ktor-backed remote impl will preserve the same surface (route through the WS bridge under the hood).

## Summary

- **Interface addition.** One new method dropped after `changeWorkspace` on `ConversationRepository`:
  ```kotlin
  suspend fun sendMessage(conversationId: String, text: String): Message
  ```
  KDoc names the user-authored / current-session / `IllegalArgumentException` contract and explicitly disclaims trim / blank-rejection (that's a UI-layer concern; the caller is expected to have validated `text.isNotBlank()`).
- **`FakeConversationRepository.sendMessage` shape mirrors `rename` / `mintNewSession`.** `Clock.System.now()` is captured once at the top into a local `now`; one atomic `state.update { records -> … }` block looks up `records[conversationId] ?: throw unknown(conversationId)`, mints the `Message` with `id = UUID.randomUUID().toString()`, returns the updated map (`record.copy(conversation = record.conversation.copy(lastUsedAt = now), messages = record.messages + appended)`); the constructed `Message` escapes the lambda via a `lateinit var appended: Message` and is returned to the caller. Same `lateinit`-escape idiom every other return-the-mutated-entity mutator uses.
- **`now` is captured once before `state.update` and reused for both `Message.timestamp` and `Conversation.lastUsedAt`.** Observers downstream of either projection see the same `Instant` — `observeLastMessage(id).first().timestamp == observeConversations(...).first { it.id == id }.lastUsedAt` holds by construction, not by lucky timing. Tests assert on the equality directly.
- **ID strategy: `UUID.randomUUID().toString()`.** Matches the runtime-minted id pattern `createDiscussion` and `mintNewSession` already use. The seed-side `"$sessionId-msg-$index"` shape is a convenience for static seeds, not a runtime contract — picking the UUID variant keeps the runtime path uniform across the three mutators that mint ids today. If Phase 4's remote impl flips the contract to "server returns the persisted Message with its assigned id" the call site changes but the interface signature doesn't have to.
- **No new model types.** `Message`, `Role`, `ConversationRecord` are unchanged. No edits to `unknown(id)`, no new helpers, no companion-object additions. Net production-code delta: ~30 lines.
- **No coroutine launch, no dispatcher choice.** `suspend` modifier exists only because the interface mutators are `suspend` and the future remote impl will be I/O-bound; the Phase 0 fake does no actual I/O. `MutableStateFlow.update`'s built-in CAS serialises the mutation — no `Mutex`, no `withContext(Dispatchers.IO)`.
- **Cascade in test stubs: 10 object literals across 4 files.** Adding a method to the interface forces an override on every anonymous `object : ConversationRepository { … }` stub. Each gets one mechanical addition matching the established pattern:
  ```kotlin
  override suspend fun sendMessage(
      conversationId: String,
      text: String,
  ): Message = TODO("not used")
  ```
  Sites: `SettingsViewModelTest.kt` (1), `ChannelListViewModelTest.kt` (2), `DiscussionListViewModelTest.kt` (3 — `stubRepo`, `erroringRepo`, `RecordingRepo` class), `ArchivedDiscussionsViewModelTest.kt` (4 — `stubRepo`, `erroringRepo`, `RecordingRepo` class, plus the one inline in `state_unsubscribesUpstream_whenScopeCancelled` since the test holds an anonymous repo). None of the ViewModels under test in this ticket consume `sendMessage` yet — the overrides are pure compile-fix landmines that document "this VM doesn't reach this path."
- **Four unit tests in `FakeConversationRepositoryTest`** cover the AC matrix end-to-end via the established `runBlocking { … } + .first()` idiom against the live `FakeConversationRepository()`:
  - `sendMessage_appendsMessageAtTailOfObserveMessages` — sends "hello" into `seed-discussion-a` (4 seeded messages, current session); asserts the `observeMessages(id)` tail is a `ThreadItem.MessageItem` wrapping the returned `Message`, its size grew by exactly one (no spurious `SessionBoundary` because the new message is in the current session), and the wrapped fields (`content == "hello"`, `role == Role.User`, `isStreaming == false`, `sessionId == currentSessionId`) match.
  - `sendMessage_observeLastMessage_reEmitsTheNewMessage` — same setup; asserts `repo.observeLastMessage(id).first()` equals the returned `Message`.
  - `sendMessage_updatesLastUsedAt_andReEmitsViaObserveConversations` — picks `seed-discussion-b` (the earlier of the two discussion timestamps, so promoting it must reorder the discussion list); reads `lastUsedAt` before, calls `sendMessage`, reads after; asserts `updated.lastUsedAt > before`, `updated.lastUsedAt == sent.timestamp`, and `discussions.first().id == id` (post-send the conversation sorts first by `lastUsedAt`-descending — the sort projection in `observeConversations` handles the reorder without any extra work in this mutator).
  - `sendMessage_onUnknownId_throws` — mirrors `promote_onUnknownId_throws` exactly (`try { runBlocking { repo.sendMessage("nope", "hi") } ; assertTrue("expected …", false) } catch (_: IllegalArgumentException) { }`).
- **No clock injection.** The third test asserts a *strict increase* in `lastUsedAt` (`updated.lastUsedAt > before` plus the `Instant` equality), not a specific instant — `Clock.System.now()` is fine, no `TestClock` plumbing needed. The seed `lastUsedAt` for `seed-discussion-b` is `2026-04-23T08:30:00Z` (well in the past) so the strict-increase assertion holds against wall-clock now without flake risk.

## Patterns established

None new — this ticket consolidates rather than invents. The append-message mutator follows the existing `rename` / `mintNewSession` template verbatim:

1. Capture `val now = Clock.System.now()` once at the top.
2. Use `lateinit var <entity>: <Type>` to escape the freshly-minted entity out of the `state.update { … }` lambda.
3. Throw via `unknown(id)` on missing record.
4. Update `lastUsedAt` to the same `now` if the operation counts as "usage" (`sendMessage` does; `archive` / `unarchive` don't).

The `TODO("not used")`-per-stub cascade established in [#161](./161.md) for `observeLastMessage` is reused here for the second time. The two-tier rule from #161 (real default value for stubs whose VMs *could* reach the new method; `TODO("not used")` for those that can't) collapses to one tier in this ticket — no current VM consumes `sendMessage`, so every stub gets `TODO`.

## Lessons learned

None new — the implementation matched the spec's predicted shape exactly. Two latent reminders worth restating against the next mutator that lands:

- **The `lateinit var` escape from `state.update { … }` is correct, not lazy.** `MutableStateFlow.update`'s lambda can re-execute under contention (the standard CAS retry shape), so any logic with observable side effects inside the lambda would be unsafe. `Message` construction is pure data-class instantiation with no side effects, so re-running it on a CAS-retry is fine — `appended` gets overwritten with an `equals`-identical value on each retry. The runtime-minted UUID would not be `equals`-identical across retries, but no observer outside the lambda sees the discarded UUIDs (they don't enter `records`), so the externally-visible behaviour is still "one mutation, one minted message, one observable change."
- **`./gradlew test` is the full verification surface for this ticket.** No instrumented tests (no `ComposeTestRule`, no `androidTest` infra needed), no `assembleDebug` requirement beyond the standard compile-clean check. Adding the interface method without the cascade in the 10 stub literals breaks `compileDebugUnitTestKotlin` — the compile error surfaces every missing override and serves as the build-time forcing function for finding all the stubs. No `Find Usages` sweep needed.
