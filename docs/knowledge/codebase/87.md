# #87 — Settings theme picker dialog (System / Light / Dark)

Wires the previously inert Theme row in `SettingsScreen` to a Material 3 single-choice `AlertDialog` and hoists the read+write of `ThemeMode` out of the Settings NavHost destination into a new `SettingsViewModel`. Confirming the dialog calls `vm.onSelectTheme(mode)` → `AppPreferences.setThemeMode(mode)`; the same flow is also collected at the `setContent` root for `PyrycodeMobileTheme(darkTheme = …)` (#86), so a single write fans out to both surfaces with no extra wiring. Material You dynamic color stays out of scope.

## Summary

- New `SettingsViewModel.kt` — first ViewModel in `ui/settings/`. Constructor-injected `AppPreferences`; exposes `themeMode: StateFlow<ThemeMode>` via `stateIn(viewModelScope, WhileSubscribed(5_000L), ThemeMode.SYSTEM)` and `fun onSelectTheme(mode)` that fire-and-forgets a `viewModelScope.launch { appPreferences.setThemeMode(mode) }`. No sealed `SettingsState` / `SettingsEvent` — the ticket explicitly defers the full MVI envelope until more rows leave the placeholder phase, so a single hot `StateFlow<ThemeMode>` + a single typed callback is the public surface. Mirrors `ChannelListViewModel`'s `stateIn` shape exactly (`STOP_TIMEOUT_MILLIS = 5_000L` lives in a `private companion object`).
- New `ThemePickerDialog.kt` — `internal` composable in the same package as `SettingsScreen`. M3 `AlertDialog` (`title = stringResource(R.string.settings_theme_dialog_title)`) over a `Column(Modifier.selectableGroup())` of three `Row`s (one per `ThemeMode.entries`); each row uses `Modifier.selectable(selected, role = Role.RadioButton, onClick = { pending = mode })` + a `RadioButton(selected, onClick = null)` (the row owns the gesture, the radio is decorative-but-rendered) + 8.dp `Spacer` + `Text(mode.label(), style = bodyLarge)`. OK / Cancel `TextButton`s use `android.R.string.{ok,cancel}` (no app-side string id needed). Pending selection: `var pending by remember(selected) { mutableStateOf(selected) }` — keying on `selected` resets the radio when the parent reopens the dialog after the persisted value changed in the background.
- `SettingsScreen.kt` rewiring — required parameters `themeMode: ThemeMode` and `onSelectTheme: (ThemeMode) -> Unit` move ahead of `onBack` / `onOpenLicense`; the prior `themeMode = ThemeMode.SYSTEM` default from #86 is dropped. New local `var showThemeDialog by remember { mutableStateOf(false) }` inside the `Scaffold` content lambda alongside the three placeholder switches; the Theme row's `onClick = {}` becomes `onClick = { showThemeDialog = true }`; an `if (showThemeDialog) ThemePickerDialog(selected = themeMode, onConfirm = { mode -> onSelectTheme(mode); showThemeDialog = false }, onDismiss = { showThemeDialog = false })` block sits above the body `Column`. The previously file-private `ThemeMode.label()` extension is widened to `internal` so `ThemePickerDialog.kt` in the same package imports the same definition (no second `when`).
- `MainActivity.kt` Settings route — drops the inline `koinInject<AppPreferences>()` + `themeMode` collect from #86; replaces with `val vm = koinViewModel<SettingsViewModel>()` + `val themeMode by vm.themeMode.collectAsStateWithLifecycle()` and forwards `onSelectTheme = vm::onSelectTheme`. Note: `vm.themeMode` is a `StateFlow`, so the no-arg `collectAsStateWithLifecycle()` overload is used (it picks up `vm.themeMode.value` as initial); the route doesn't need to redeclare `initialValue = ThemeMode.SYSTEM`. The `koinInject` import at the top of `MainActivity.kt` stays — it's still consumed by the Scanner route and the `setContent` root-level `appPreferences` collection that drives `PyrycodeMobileTheme(darkTheme = …)`.
- `AppModule.kt` — single line added: `viewModel { SettingsViewModel(get()) }` alongside `ChannelListViewModel` / `DiscussionListViewModel`. `get()` resolves the already-registered `single { AppPreferences(get()) }`.
- `strings.xml` — single `settings_theme_dialog_title` string ("Theme"). The three radio labels deliberately stay in `ThemeMode.label()` rather than fanning into `strings.xml` (one source of truth for the row subtitle and the dialog options).
- `SettingsViewModelTest.kt` — six methods using the `AppPreferencesTest` rig (real `PreferenceDataStoreFactory` over `TemporaryFolder` + real `AppPreferences`, no fake) and the `ChannelListViewModelTest` dispatcher rhythm (`Dispatchers.setMain(UnconfinedTestDispatcher())` + `runTest(dispatcher) { … advanceUntilIdle() … }`). Covers fresh-default emit, pre-write mirror on construction, per-`ThemeMode` persistence (LIGHT / DARK / SYSTEM round-trip from DARK), and flow re-emission across two consecutive `onSelectTheme` calls. Each scenario that asserts `vm.themeMode.value` `launch`es a no-op collector first to keep the `WhileSubscribed(5_000L)` `stateIn` upstream subscribed during the test (otherwise `.value` would stay at the `initialValue = ThemeMode.SYSTEM`).
- `SettingsScreenTest.kt` (existing instrumented file from #90) — both `setContent` blocks updated to pass `themeMode = ThemeMode.SYSTEM, onSelectTheme = {}` explicitly because the parameter no longer carries a default. Test scope (Version row + Open-source row) is unchanged.

## Patterns established

- **Defer the sealed `UiState` / `Event` envelope until the third row earns it.** `SettingsViewModel` exposes a single `StateFlow<ThemeMode>` and a single `fun onSelectTheme(mode)` rather than the `StateFlow<SettingsState> + Channel<SettingsEvent>` shape used by [`ChannelListViewModel`](../features/channel-list-viewmodel.md) / [`DiscussionListViewModel`](../features/discussion-list-viewmodel.md). The ticket body authorises this explicitly ("Full sealed `SettingsState` / `SettingsEvent` types are NOT required at this slice's scope") because Settings still has exactly one row backed by real state. The next persisted Settings preference (notification sound, default model, default effort — all enumerated in [Settings screen](../features/settings-screen.md)) is the trigger to lift `themeMode` into a sealed `SettingsState` with a per-field reducer; until then, adding a second hot `StateFlow<X>` + `fun onSelectX(...)` pair is the cheaper move and matches the precedent set here.
- **Dialog open-state is local UI state; selection is hoisted; pending choice is dialog-local.** Three concentric scopes:
  - `var showThemeDialog by remember { mutableStateOf(false) }` lives in the screen composable — it does not survive process death deliberately (re-entering Settings should land closed, not on a stale open dialog).
  - The persisted `themeMode` lives in the ViewModel; `onSelectTheme` is the single write callback into it.
  - `var pending by remember(selected) { mutableStateOf(selected) }` lives inside `ThemePickerDialog` — keyed on `selected` so a background change to the persisted value resets the pending radio when the dialog reopens. Cancel / `onDismissRequest` discard `pending` by composition teardown; OK calls `onConfirm(pending)`. Generalises to any future M3 single-choice dialog (notification-sound picker, default-effort picker): hoist the *persisted* value to the VM, keep *open-state* local, keep *pending choice* dialog-local with `remember(selected)`.
- **OK / Cancel over tap-to-commit-and-dismiss for an app-wide visible side effect.** AC #2's "confirming" / "cancelling" wording is the in-spec justification, but the durable rule is: when a dialog write triggers an immediately-visible global re-render (here: the entire app's color scheme), require the explicit confirm step. Two-button M3 `AlertDialog` is the right shape; tap-to-commit is reserved for selections whose effect is local to the dialog's parent screen.
- **`android.R.string.ok` / `android.R.string.cancel` for the dialog button labels.** No new app-side string ids; the platform's localised OK / Cancel strings cover every locale the app will ship in. Future M3 confirm/cancel dialogs should follow — only introduce a custom `R.string.*` when the action verb is meaningfully more specific than "OK" (e.g. `promote_dialog_confirm` = "Save as channel").
- **`internal` widening (not move) when a same-package neighbour needs a private extension.** `ThemeMode.label()` started life as `private` at the bottom of `SettingsScreen.kt` (#86); this slice promotes it to `internal` so `ThemePickerDialog.kt` imports it from the same `de.pyryco.mobile.ui.settings` package without any code motion or visibility change at runtime. The function does **not** move into `data/preferences/` — these are UI strings and `data/` is the [Compose Multiplatform walk-back boundary](../decisions/0001-kotlinx-datetime-for-data-layer.md) per `CLAUDE.md`. Generalises: when a private helper grows a second same-package caller, widen to `internal` first; only move to a new file when a *cross-package* caller appears.
- **No-arg `collectAsStateWithLifecycle()` when collecting a `StateFlow` from a NavHost destination.** `vm.themeMode` is a `StateFlow<ThemeMode>` (the `stateIn(...)` return type), so the destination uses `vm.themeMode.collectAsStateWithLifecycle()` with no `initialValue` — the overload picks up `StateFlow.value`. Compare with #86's root-level `appPreferences.themeMode.collectAsStateWithLifecycle(initialValue = ThemeMode.SYSTEM)` which collects a *cold* `Flow<ThemeMode>` and must supply a same-shaped neutral default. The rule: cold flow → pass an explicit neutral `initialValue`; hot `StateFlow` → no `initialValue` argument.
- **Real `AppPreferences` over a fake in `SettingsViewModelTest`.** The test rig from `AppPreferencesTest` (`TemporaryFolder` + `PreferenceDataStoreFactory.create(scope = backgroundScope, produceFile = { tmp.newFile("app_prefs.preferences_pb") })`) is the right rig because the VM is thin (`stateIn` + a one-line setter forward) and the round-trip behaviour *is* the contract. Adding a `FakeAppPreferences` interface for one test class would invert the cost. Future thin VMs that wrap a single `AppPreferences` key should follow — instantiate a real `AppPreferences` over a `TemporaryFolder`-backed `DataStore` rather than introducing a fake.
- **Launch a no-op collector before asserting `StateFlow.value` in `runTest`.** `WhileSubscribed(5_000L)` means the upstream `appPreferences.themeMode` only emits when there's a subscriber; without the `val collector = launch { vm.themeMode.collect { } }; advanceUntilIdle()` warm-up, `vm.themeMode.value` stays at the `initialValue = ThemeMode.SYSTEM` and the test passes for the wrong reason on the "fresh datastore" path *and* fails on the "pre-written DARK" path. Write-then-`first()` tests (`assertEquals(LIGHT, prefs.themeMode.first())`) don't need it — they go through `prefs` directly, not through the VM's `stateIn` projection. Generalises to every `WhileSubscribed`-backed `StateFlow` test in this codebase.

## Lessons learned

- **`stateIn` initial value vs. flow first emission can mask test failures silently.** With `started = WhileSubscribed(5_000L)` and `initialValue = ThemeMode.SYSTEM`, `vm.themeMode.value` returns `SYSTEM` *until* a subscriber appears and the upstream's first emission lands. A test that constructs the VM, immediately reads `vm.themeMode.value`, and asserts `SYSTEM` will pass — but it isn't asserting what the test name suggests; it's asserting the `initialValue` literal. The fix used here (launch a collector + `advanceUntilIdle()` before the assertion) is the durable shape; the trap is that the SYSTEM-default path looks fine without it.
- **Parameter order matters for the previews and tests when the default disappears.** Dropping the `themeMode = ThemeMode.SYSTEM` default from `SettingsScreen` (#86) forced the new required `themeMode` and `onSelectTheme` parameters to live *before* `onBack` / `onOpenLicense` (Kotlin requires required-before-default-before-trailing-lambda); both `@Preview` composables and the two `SettingsScreenTest` `setContent` blocks had to pass all four explicitly. The compiler caught it cleanly, but it's the kind of churn worth budgeting for whenever a "defaulted display state" parameter from a prior slice gets promoted to "wired event surface" — every preview + every test built on the prior shape needs an edit.
- **`ThemeMode.entries` is the right iterator for the dialog rows.** Using `ThemeMode.values()` would have worked identically at runtime, but `entries` is the post-Kotlin-1.9 idiom (immutable list, no defensive copy on each call) and matches what `AppPreferences.themeMode` already uses for the unparseable-fallback path. New enums in this codebase should use `.entries` consistently.
- **Test names that document the contract beat test names that document the implementation.** `themeMode_flowReEmits_afterOnSelectTheme` is named for AC #5's "exposed flow re-emits after persistence" wording, not for the `stateIn` machinery underneath. The test setup (collector + double `onSelectTheme` + assert `.value` reaches each) is incidental; the contract being asserted is the AC. Future VM tests should follow.
