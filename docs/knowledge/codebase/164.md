# #164 — Settings · Archived discussions row shows live count

The Storage section's "Archived discussions" row in `SettingsScreen` gains a Figma-17:2-mandated `"N archived"` supporting line driven by a new `SettingsViewModel.archivedDiscussionCount: StateFlow<Int>`. The flow mirrors `ArchivedDiscussionsViewModel`'s `Archived` + `!isPromoted` filter exactly so the row's count and the destination screen's list cannot disagree. Split from #124.

## Summary

- **`SettingsViewModel` gains a second constructor argument and a third public `StateFlow`.** Signature becomes `class SettingsViewModel(private val appPreferences: AppPreferences, conversationRepository: ConversationRepository)`. The repo is **not** stored on `this` (no `private val`) because the only consumer is the field-initializer block that wires `archivedDiscussionCount`; promoting it to a property would advertise an instance dependency that doesn't exist. `archivedDiscussionCount: StateFlow<Int>` is built as `conversationRepository.observeConversations(ConversationFilter.Archived).map { it.count { c -> !c.isPromoted } }.catch { emit(0) }.stateIn(viewModelScope, WhileSubscribed(STOP_TIMEOUT_MILLIS), initialValue = 0)` — same `STOP_TIMEOUT_MILLIS = 5_000L` companion constant the prior two flows share, same `WhileSubscribed`/`stateIn` shape, same `initialValue = 0` that matches Figma's empty-state literal ("0 archived"). No sealed `SettingsUiState` envelope — the existing VM exposes parallel primitive flows by design (`themeMode` / `useWallpaperColors`); a third flow follows the established rule that the MVI envelope arrives on the *third real persisted preference*, which is unaffected by adding a derived projection from the data layer.
- **`.catch { emit(0) }` is upstream of `stateIn`.** Deliberate divergence from `ArchivedDiscussionsViewModel`, which surfaces `Error(message)` for the same upstream because the list **is** the screen's primary content. Settings' supporting text is read-only metadata about the archived-discussions screen; a thrown upstream collapses to "0 archived" rather than tearing down the flow or blanking the row. The architect's spec called this out explicitly and the developer followed; the precedent is "supportive-metadata projections swallow errors, primary-content projections surface them."
- **Mirror the filter, don't generalise it.** `count { !it.isPromoted }` is duplicated from `ArchivedDiscussionsViewModel.state`'s `filter { !it.isPromoted }`, not factored into a shared helper. Two callers is below the project's three-caller threshold for extracting a helper, and the `Archived` filter being `isPromoted`-agnostic at the repository layer is deliberate (#93's design) so it composes for a future archived-channels screen — a shared helper would lock in an assumption the repo layer specifically left open.
- **`SettingsScreen` gains a required `archivedDiscussionCount: Int` parameter.** Positioned third — after `themeMode` + `useWallpaperColors` and before `onSelectTheme` — so state values stay grouped before event callbacks, matching the ordering convention #89 set. No default, same rule the file already enforces for every value that drives a visible affordance. The Storage row's `supporting = stringResource(R.string.archived_discussions_count_supporting, archivedDiscussionCount)` is the only body change; the row stays clickable via the unchanged `onClick = onOpenArchivedDiscussions` regardless of count, satisfying Figma 17:2's "row stays present, reads '0 archived'" empty-state.
- **New string resource: `archived_discussions_count_supporting` → `"%1$d archived"`.** Plurals deferred per spec — the codebase has no plurals precedent (`"0 plugins"` elsewhere is hardcoded), English `"0 / 1 / 11 archived"` are all grammatical without inflection, and switching to a `<plurals>` resource later is mechanical and isolated to this one string. YAGNI.
- **`MainActivity` collects the new flow at `composable(Routes.SETTINGS)`.** One added line `val archivedDiscussionCount by vm.archivedDiscussionCount.collectAsStateWithLifecycle()` and one added named-arg pass-through; no-arg `collectAsStateWithLifecycle()` because `vm.archivedDiscussionCount` is already a `StateFlow` (same idiom as the two sibling collectors at this destination).
- **`AppModule.kt`'s `viewModel { SettingsViewModel(get()) }` becomes `viewModel { SettingsViewModel(get(), get()) }`.** Koin resolves the second `get()` to the existing `ConversationRepository` binding on line 27 (the same `FakeConversationRepository` instance `ChannelListViewModel` / `DiscussionListViewModel` / `ArchivedDiscussionsViewModel` share). No new binding required.
- **Tests use a hand-rolled `ConversationRepository` stub, not the real `FakeConversationRepository`.** The architect's spec offered both options; the developer picked the stub because it gives the tests precise control over what the `Archived` flow emits and when (`MutableSharedFlow<List<Conversation>>(replay = 0)` + `source.emit(...)` between `advanceUntilIdle()` calls), and lets one test capture the requested `ConversationFilter` value to verify the VM passes `Archived` rather than `All`. The Fake's `archive(id)` / `unarchive(id)` chain would test the data layer too; the stub keeps the VM contract isolated. The pre-existing nine tests gain a small `makeVm(prefs, repo = stubRepo())` helper so the prior call sites stay one-arg — the architect's option (b), elegant over the call-site cascade.
- **One new instrumented Compose test method.** `archivedDiscussionsRow_rendersSupportingTextWithCount` sets content with `archivedDiscussionCount = 11`, `performScrollTo()`s the row, asserts a node with `hasText("11 archived", substring = true)`. One representative non-zero count per AC #4; value transitions are unit-test territory. The four pre-existing test methods each gained an `archivedDiscussionCount = 0` argument to keep compiling.

## Patterns established

- **Supportive-metadata projections swallow upstream errors; primary-content projections surface them.** Two coexisting consumers of the same `observeConversations(ConversationFilter.Archived)` flow now demonstrate the rule: `ArchivedDiscussionsViewModel` surfaces `Error(message)` because the archived-discussions list IS the screen's content; `SettingsViewModel.archivedDiscussionCount` swallows the throw with `.catch { emit(0) }` because the Settings row is a metadata pointer to that other screen. When adding a new derived projection on top of an existing repository flow, ask "is this projection the primary content of its surface?" — if no, `.catch { emit(<neutral>) }`. Future supportive-metadata flows (count badges, last-updated timestamps, "N pending" lines) should follow.
- **`STOP_TIMEOUT_MILLIS = 5_000L` is the single shared sharing-grace constant for a multi-flow VM.** `SettingsViewModel` now uses it across three `stateIn` calls. When a VM grows multiple `WhileSubscribed`-shaped `StateFlow`s, declare the constant once in `private companion object` and reference it from every call site — don't inline `5_000L` per flow. Same number, one tuning knob; the precedent (one constant per VM) was set in #87 and reinforced by #89; this is the third flow that follows it.
- **Drop-the-`private val`-when-only-the-initializer-uses-it.** `conversationRepository` is a constructor parameter, not a property, because no method body needs it — only the field initializer for `archivedDiscussionCount` reads it once. If a future event handler ever needs to call `conversationRepository.archive(...)` from `SettingsViewModel`, promote it to `private val` then. Don't pre-promote in anticipation; the constructor-parameter form correctly advertises "the repo is consumed at construction time and not retained" to readers.

## Lessons learned

- **A hand-rolled `ConversationRepository` stub lets one test verify the requested `ConversationFilter` value cheaply.** The `archivedDiscussionCount_passesArchivedFilter` test captures the `ConversationFilter` argument into a `mutableListOf<ConversationFilter>` and asserts it equals `[ConversationFilter.Archived]`. With the real `FakeConversationRepository`, this assertion would require either inspecting internals or carefully constructing a scenario where the wrong filter would yield a visibly different count — both more fragile than a one-line `captured.add(filter)` on the way through. The spec mentioned this as a "use the fake" / "use a stub" judgment call; for filter-value verification, the stub wins. Same shape any future "is the VM asking the repo the right question?" test should reach for.
- **No need for an `archivedDiscussionCount` `@Preview` permutation alongside the existing light/dark pair.** The two `@Preview`s simply pass `archivedDiscussionCount = 11` (Figma's reference frame value); a separate "0 archived" preview was considered and dropped — the supporting-text format string is trivial and the empty-state behavior (row stays present) is identical at the Compose level, so a second preview would burn pane real-estate without exercising any new visual contract. The instrumented test pins the non-zero render at `11`; the unit test covers `0`/transitions. Two layers of testing is enough.
- **Aliased imports in the test file's import block need the same spotless treatment #162 documented.** This file uses no aliases (the data-model `Role` enum doesn't appear here), but the addition of new imports (`ConversationFilter`, `ConversationRepository`, `kotlinx.coroutines.flow.MutableSharedFlow`, `kotlinx.coroutines.Flow`, `Message`, `Session`, `ThreadItem`, `DEFAULT_SCRATCH_CWD`, `Conversation`) was non-trivial and `./gradlew spotlessApply` was the right first checkpoint. Carry forward from [`162.md`](./162.md): when growing an import block, run `spotlessApply` once before committing rather than relying on local IDE ordering.
