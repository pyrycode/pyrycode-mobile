# #165 — Settings · Version row supporting text shows Git short SHA

The About-section Version row's supporting line in `SettingsScreen` flips from the static `BuildConfig.VERSION_CODE` (the integer `versionCode = 1`, identical across every build) to a build-time Git short SHA injected via Gradle as `BuildConfig.GIT_SHA`. A Settings screenshot is now mappable back to an exact commit; a build outside a git checkout still succeeds with the literal string `"unknown"`. Split from #124.

## Summary

- **`app/build.gradle.kts` gains a parameterless `GitShaValueSource: ValueSource<String, ValueSourceParameters.None>`** declared between the `plugins { ... }` block and `android { ... }`. The class injects `ExecOperations` via `@get:Inject abstract val execOperations: ExecOperations` and implements `obtain()` as `try { execOperations.exec { commandLine("git", "rev-parse", "--short", "HEAD"); standardOutput = stdout; isIgnoreExitValue = true } ... } catch (_: Throwable) { "unknown" }`. The `try` body captures stdout into a `ByteArrayOutputStream`, decodes via `Charsets.UTF_8`, trims, and returns the SHA only when `result.exitValue == 0 && sha.isNotEmpty()` — every other path (non-zero exit, blank stdout, `ProcessException`, `ExecException`, missing-binary `IOException`) collapses to `"unknown"`. `catch (_: Throwable)` is deliberate: subtypes of Gradle's process exceptions don't share a common `Exception` ancestor that catches every observed failure mode at this seam, and this is build-time fallback code, not application logic where a broad catch would mask real bugs.
- **`buildConfigField` injected at the end of `defaultConfig`.** Two new lines after `versionName = "1.0"`: `val gitSha = providers.of(GitShaValueSource::class.java) {}` and `buildConfigField("String", "GIT_SHA", "\"${gitSha.get()}\"")`. The embedded `\"` quotes around `${gitSha.get()}` are mandatory — `buildConfigField` writes the value verbatim into the generated `BuildConfig.java`, so for a `String`-typed field the third arg must already be a Java string literal (quoted). A bare `gitSha.get()` would emit `public static final String GIT_SHA = a8f3c2d;` and fail to compile.
- **Configuration-cache compliance is the load-bearing AC.** `providers.of(ValueSource::class.java)` is the canonical Gradle pattern that lets a configuration-time process read survive the cache: the value is captured once and the cache is invalidated only when the `ValueSource`'s declared inputs change (here: none — every invocation re-runs `obtain()` at configuration-cache restore time, which is what we want for "rebuild → new SHA"). A naive `"git rev-parse".execute()` or a direct `Runtime.exec(...)` at the `defaultConfig` body would invalidate the cache on every invocation, failing AC #4. AGP 9.2.1 (verified via `gradle/libs.versions.toml`) supports the pattern natively — no new dependency.
- **`SettingsScreen.kt:213` is the only UI change.** `supporting = "build ${BuildConfig.VERSION_CODE}"` becomes `supporting = "build ${BuildConfig.GIT_SHA}"`. The headline on line 212 (`"Version ${BuildConfig.VERSION_NAME}"`) is unchanged per the ticket body — semver and SHA serve different diagnostic purposes and both stay. `BuildConfig` is already imported in the file (the #90 wiring established it).
- **One new instrumented Compose test method.** `versionRow_rendersSupportingTextWithGitSha` (placed directly after `versionRow_rendersBuildConfigVersionName` at lines 22–42) follows the same shape: `setContent { PyrycodeMobileTheme { SettingsScreen(...) } }` with default params, then `composeTestRule.onNode(hasText("build ${BuildConfig.GIT_SHA}", substring = true)).performScrollTo().assertExists()`. The matcher reads `BuildConfig.GIT_SHA` itself, not a hardcoded SHA — this is the correct invariant for both dev-machine builds (real 7-char SHA) and CI tarball builds (`"unknown"`); the test verifies the binding between the build-time field and the rendered text regardless of value.
- **No unit tests added.** The Gradle build script is implicitly exercised by every build; the Compose UI uses a `String` constant — there's nothing for `./gradlew test` to cover that the instrumented test or the build itself doesn't.

## Files touched

## Patterns established

- **Build-time process reads go through `ValueSource`, never `Runtime.exec` / `providers.exec` at configuration time.** Project's first configuration-time external-process read; the `ValueSource` pattern is the canonical Gradle answer for "run a process at configuration time without invalidating the configuration cache". Future tickets that need to inject build metadata derived from a process (current branch, latest tag, dirty-tree marker, container hash) declare a new sibling `*ValueSource` class with the same shape — `abstract class … : ValueSource<String, ValueSourceParameters.None>` + `@get:Inject abstract val execOperations: ExecOperations` + `try { ... } catch (_: Throwable) { <neutral default> }` — and register the `buildConfigField` from a one-line `providers.of(...).get()` call inside `defaultConfig`. Don't extract a shared helper yet — two sibling `*ValueSource` classes will share <10 lines of mechanical structure, and the second one's exact failure-mode neutral default ("unknown" / "" / "dirty" / something else) is a per-source judgement call that a helper would over-constrain.
- **`catch (_: Throwable)` at the Gradle-build seam is correct; the same shape at the application seam would be a smell.** Configuration-time process reads can fail in shapes Gradle doesn't fully document (subprocess `IOException`, `BuildException`, `ProcessException`, `ExecException`, `IllegalStateException` from a corrupted `.git/HEAD`, anything else). The seam is "the build either succeeds with a known SHA or succeeds with `unknown`" — the catch's only job is to make sure no shape escapes to fail the build. Inside the app, `catch (_: Throwable)` would mask real bugs and is banned. Future build-time fallbacks follow this rule: broad catch is fine at the build seam; narrow catch (or none) is the default everywhere else.
- **Build-time-derived `BuildConfig.*` `String` constants quote the value through `\"`.** `buildConfigField("String", "<name>", "\"${value}\"")` is the canonical shape for any `String` field. The third arg is a Java string literal, not a value — the docs quote this but it's easy to miss until the build breaks. If a future ticket needs to inject an `Int` / `Long` / `Boolean` instead, the quotes are dropped (`buildConfigField("Int", "FOO", "$value")`).

## Lessons learned

- **`buildConfigField` writes its third argument verbatim into `BuildConfig.java`, so quoting matters more than it looks.** Easy to write `buildConfigField("String", "GIT_SHA", gitSha.get())` and have the build fail with a confusing `error: cannot find symbol` on the SHA characters — the generated file becomes `public static final String GIT_SHA = a8f3c2d;` and `a8f3c2d` is parsed as an identifier. The fix is the embedded `\"` escape around `${gitSha.get()}`; the AGP error message doesn't surface the underlying generated-code error clearly, so any future "BuildConfig field doesn't compile" symptom should check the `buildConfigField` quoting first.
- **`isIgnoreExitValue = true` is necessary even with the `try`/`catch` wrapper.** `ExecOperations.exec { }` throws on non-zero exit by default, and `git rev-parse --short HEAD` exits with `128` when invoked outside a git checkout. The `try { } catch (_: Throwable) { "unknown" }` would catch the throw and produce the right answer, but `isIgnoreExitValue = true` lets the empty-stdout path produce `"unknown"` without going through the catch — keeping the catch reserved for "the binary isn't even on PATH" / "the subprocess infrastructure threw" shapes. Both shapes converge on `"unknown"`, but separating them in source makes the "what error mode are we handling here?" question answerable from the code.
- **No `--configuration-cache` re-run gate in CI yet.** AC #4 was verified manually before the PR. A future ticket could add a Gradle smoke test or wire `--configuration-cache` into the build's regular invocation, but this ticket didn't — observed-failure threshold not met, and the cost (~CI minutes, build complexity) outweighs the protection for a one-line build script change that doesn't regress easily. If a future build-script ticket breaks the cache, this is the second observation that flips the gate from "manual" to "automated".
- **`BuildConfig.GIT_SHA` is `"unknown"` in instrumented test runs unless the test APK build itself runs inside the git checkout** — which it does on dev machines and CI agents that clone the repo, but wouldn't on a hypothetical "build the APK in a clean room, ship the artifact, run instrumented tests against the artifact" setup. The test assertion `hasText("build ${BuildConfig.GIT_SHA}", substring = true)` reads `BuildConfig.GIT_SHA` from the test APK's own `BuildConfig`, so it always matches whatever the screen renders — same APK, same constant. Not a problem today; worth flagging if a future test infrastructure ticket separates the build environment from the test environment.
