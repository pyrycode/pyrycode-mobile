# #89 — Settings "Use wallpaper colors" switch row

Replaces the local-only `materialYou` placeholder in the Appearance section with a VM-backed, SDK-gated row that mirrors and writes `AppPreferences.useWallpaperColors`. The Material You / dynamic-color switch becomes the second piece of real Settings state (after the Theme row from #87): tapping it persists through DataStore and the composition-root collector from #88 repaints `PyrycodeMobileTheme(dynamicColor = …)` live, with no extra wiring at the screen. On Android < 12 (unreachable in production today — Min SDK 33 > S = 31, but implemented per AC) the switch renders disabled with helper text "Requires Android 12 or newer".

## Summary

- `SettingsViewModel.kt` — extended in line with the existing `themeMode` / `onSelectTheme` pair: new `useWallpaperColors: StateFlow<Boolean>` via `appPreferences.useWallpaperColors.stateIn(viewModelScope, WhileSubscribed(STOP_TIMEOUT_MILLIS), initialValue = false)` and `fun onToggleUseWallpaperColors(enabled: Boolean) { viewModelScope.launch { appPreferences.setUseWallpaperColors(enabled) } }`. Same `STOP_TIMEOUT_MILLIS = 5_000L` companion constant, no new tuning knob. No sealed `SettingsState` / `SettingsEvent` envelope — still deferred per #87's pattern; two `StateFlow<X>` + `fun onSelectX(...)` pairs is the cheaper move until a third real preference lands.
- `SettingsScreen.kt` rewiring — adds `useWallpaperColors: Boolean` + `onToggleUseWallpaperColors: (Boolean) -> Unit` to the signature, between the existing `themeMode` and `onSelectTheme` parameters (state values together, then event callbacks together). Inside the `Scaffold` content lambda the prior `var materialYou by remember { mutableStateOf(true) }` is deleted outright and the placeholder `SettingsRow(headline = "Use Material You dynamic color", trailing = { Switch(materialYou, …) })` is replaced with a `SettingsRow` whose `headline = stringResource(R.string.settings_use_wallpaper_colors)`, conditional `supporting = stringResource(R.string.settings_use_wallpaper_colors_unsupported)` on the disabled-pre-S branch, and a trailing `Switch(checked = useWallpaperColors, onCheckedChange = onToggleUseWallpaperColors, enabled = dynamicColorSupported)`. SDK gate `val dynamicColorSupported = Build.VERSION.SDK_INT >= Build.VERSION_CODES.S` evaluated once at composable build (no `remember` — process-lifetime constant). Both `@Preview` composables now pass `useWallpaperColors = false, onToggleUseWallpaperColors = {}` explicitly. The other two local-state switches (`defaultYolo`, `pushNotifications`) are untouched — only the `materialYou` row was in scope.
- `MainActivity.kt` — Settings route gains `val useWallpaperColors by vm.useWallpaperColors.collectAsStateWithLifecycle()` immediately after the existing `themeMode` collect, and forwards both new params into `SettingsScreen(...)`. The root-level `appPreferences.useWallpaperColors` collector from #88 (the one that drives `PyrycodeMobileTheme(dynamicColor = …)` at `setContent`) is **untouched** — the VM's collector is an independent second subscriber to the same DataStore flow, and one write fans out to both.
- `strings.xml` — two new ids: `settings_use_wallpaper_colors` ("Use wallpaper colors") and `settings_use_wallpaper_colors_unsupported` ("Requires Android 12 or newer"). The replaced hard-coded `"Use Material You dynamic color"` literal is gone; the other switch rows on this screen still use hard-coded strings — extracting them is out of scope.
- `SettingsViewModelTest.kt` — three new tests next to the existing six, using the same `TemporaryFolder` + real `AppPreferences` rig: `useWallpaperColors_initialState_emitsFalse_whenNoStoredValue`, `useWallpaperColors_initialState_mirrorsPersistedTrue`, and `onToggleUseWallpaperColors_persistsAndFlowReEmits` (the third covers both the AC's "toggling persists via the fake" and "exposed flow re-emits after persistence" jointly via a `false → true → false` walk — same compression as #88's round-trip-plus-re-emit collapse).
- `SettingsScreenTest.kt` (instrumented) — both `setContent` blocks gained explicit `useWallpaperColors = false` and `onToggleUseWallpaperColors = {}` arguments to keep the file compiling against the new required parameters. **The developer commit (4089bd2) shipped without this update**, and `compileDebugAndroidTestKotlin` failed with two pairs of "No value passed for parameter" errors; the documentation pass caught it and patched the two call sites — see Lessons learned below. Existing test scope (Version row + Open-source row) is unchanged.

No new files. No new types. The row composable is **not** extracted — it stays inline among its `SettingsRow(...)` siblings, matching the existing per-row inline convention.

## Patterns established

- **Add a second `StateFlow<X>` + `onSelectX(...)` pair to `SettingsViewModel` before reaching for the sealed envelope.** #87 set the rule that the MVI envelope would arrive on the *third* real persisted preference; this slice exercises the "growth by one more pair" half of that contract. The VM now has two parallel `stateIn(WhileSubscribed(5_000L))` projections and two fire-and-forget setter forwards. Adding a third pair is still cheaper than lifting both into a sealed `SettingsState`; the next persisted Settings preference (notification sound / default model / default effort — see [Settings screen](../features/settings-screen.md)) is the trigger to lift. Stay disciplined and don't pre-MVI on a count of two.
- **Two upstream collectors on the same DataStore flow — composition root + VM — both live, no coordination needed.** `appPreferences.useWallpaperColors` is now subscribed twice: once at `MainActivity.setContent` (drives `PyrycodeMobileTheme(dynamicColor = …)`, #88's wiring) and once via `SettingsViewModel.useWallpaperColors` (drives the switch's `checked` state). DataStore broadcasts to every collector, so one `setUseWallpaperColors(...)` write reaches both surfaces from the single emission. This is the load-bearing fact behind AC #2's "updates immediately without a process restart" — already true after #88, this slice adds the second subscriber on the VM side. Generalises to every future preference that needs both a global-theme effect *and* a Settings-row reflection: don't try to share a single collector, don't thread the value through navigation arguments — just subscribe twice.
- **SDK gate evaluated at composable build, not memoized.** `val dynamicColorSupported = Build.VERSION.SDK_INT >= Build.VERSION_CODES.S` lives near the top of the `SettingsScreen` body with no `remember` wrapper. `Build.VERSION.SDK_INT` is a process-lifetime constant and the comparison is a single integer compare; wrapping it in `remember { }` would add a slot-table entry for zero benefit. Future SDK gates inside composables follow this shape — plain `val` at the top of the body, no caching.
- **Disabled `Switch` with `enabled = false` over a short-circuited `onCheckedChange`.** The disabled-pre-S branch passes `onCheckedChange = onToggleUseWallpaperColors` unconditionally and gates purely on `enabled = dynamicColorSupported`. M3's `Switch` ignores taps when `enabled = false`, applies the disabled colors automatically, and never invokes the callback — so the callback's presence is harmless. The alternative (`onCheckedChange = if (dynamicColorSupported) onToggleUseWallpaperColors else null`) is fine too but adds a branch where M3 already owns the gate. Pick the simpler shape.
- **Parameter ordering: state values together, then event callbacks together, then nav callbacks, then `modifier`.** Adding `useWallpaperColors` *next to* `themeMode` (both state values) and `onToggleUseWallpaperColors` *next to* `onSelectTheme` (both event callbacks) keeps related concerns visually grouped in the signature. The Compose stdlib uses the same shape (`Switch(checked, onCheckedChange, modifier, enabled, …)`). When the third real preference lands, the (state, event) pair should slot in adjacent to the existing two — not appended at the end.

## Lessons learned

- **The disabled-on-< S branch is structurally unreachable today.** `CLAUDE.md` pins Min SDK 33 (Android 13), which is two API levels above S (Android 12 = API 31). No production device will ever render the disabled state of this switch. The ticket body asked for it anyway and #75's broader Material You UX work informs why: the disabled-with-helper-text affordance is a deliberate discoverability surface for the (future) downstream walk-back to a lower Min SDK or for a release-vs-debug Min SDK split. Implementing it costs ~5 lines and adds an `R.string.settings_use_wallpaper_colors_unsupported` resource; don't argue it away as dead code, but also don't write a Compose test for it (faking `Build.VERSION.SDK_INT` from a test runner exercises Robolectric or reflection, not the production behaviour — net negative).
- **The placeholder `materialYou` switch had `mutableStateOf(true)` as its initial value; the real preference defaults to `false`.** A user upgrading from a pre-#89 build will see the switch jump from "on" (the prior cosmetic stub) to "off" (the persisted default from #88). This is correct — the placeholder was never persisted, so there's nothing to migrate, and the AC explicitly says "Default is OFF, so the brand palette renders unless the user opts in." Worth noting only so a future regression report ("the Material You switch turned itself off after the update!") doesn't trigger a bisect. The answer is "yes, intentionally."
- **No Compose UI test for the row itself.** The repo's instrumented Compose surface for Settings (`SettingsScreenTest.kt`) covers Version + Open-source rows; #87 added no row-level coverage when it wired the Theme picker dialog, and this slice continues that — the VM tests carry the contract that matters (`StateFlow.value` mirrors persisted state, `onToggleUseWallpaperColors` writes through, flow re-emits). Compose-side fidelity rides on the two `@Preview`s plus a manual emulator smoke test (toggle the switch, watch `PyrycodeMobileTheme` repaint). When a row-level instrumented test does land, it should drive the screen through state + lambda directly (the `ChannelListScreenTest`/`DiscussionListScreenTest` shape) — no VM, no Koin.
- **Adding a required parameter to `SettingsScreen` breaks the instrumented test compile — and code review didn't catch it.** The developer remembered to update the two `@Preview` composables in `SettingsScreen.kt` (Kotlin compiler catches `@Preview` mismatches at `compileDebugKotlin`) but missed the two `setContent` blocks in `app/src/androidTest/.../SettingsScreenTest.kt` (caught only at `compileDebugAndroidTestKotlin`, a separate Gradle task that the developer and reviewer apparently didn't run). The documentation pass caught it and patched the two call sites. **Lesson:** any change to a `public` screen signature with required parameters must run *both* `./gradlew compileDebugKotlin` *and* `./gradlew compileDebugAndroidTestKotlin` (or just `./gradlew assembleDebugAndroidTest`) before the PR is opened. The repo's `./gradlew test` target also doesn't pull in the androidTest source set, so VM tests passing is not sufficient. Same trap as #87's "parameter order matters for the previews and tests when the default disappears" lesson, but the androidTest source set adds a second compile target the local build feedback loop has to cover. Worth adding to a pre-PR check list for every future screen-signature change.
- **The label was clarified from "Use Material You dynamic color" (Figma) to "Use wallpaper colors" (ticket body).** The architect's spec called this out as a label divergence — Figma node `17:2` shows the older copy that the placeholder happened to mirror. Follow the ticket body, not the Figma string, when there's a written mismatch. Visual layout (row height, leading body-large label, trailing switch, no subtitle in the enabled state) still follows Figma exactly. If the future cycles produce a designer-driven re-rename, the entry point is `R.string.settings_use_wallpaper_colors` now — single string id, no scattered literals.
