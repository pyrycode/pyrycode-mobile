# #12 — Stub Scanner screen + route

Adds a tap-anywhere stub Scanner screen between Welcome and the Channel List, closing the onboarding navigation loop before real CameraX + ML Kit pairing lands in Phase 4. The new `scanner` destination is the first to (a) wire a Koin singleton directly inside a `composable(...)` block, (b) launch a coroutine for a `suspend` side-effect from a destination block, and (c) use a `popUpTo { inclusive = true }` back-stack trim on a forward navigation.

## Summary

- New file `app/src/main/java/de/pyryco/mobile/ui/onboarding/ScannerScreen.kt`. Stateless `@Composable fun ScannerScreen(onTap: () -> Unit)`. Composition: outer `Surface(color = background, fillMaxSize, pointerInput { detectTapGestures { onTap() } })` over an inner `Column(systemBarsPadding, padding 24/32, Center/Center)` with a square `Box` viewport (`aspectRatio(1f)`, `RoundedCornerShape(16.dp)`, `surfaceVariant` fill) containing a centered `0.8f` × `2.dp` `primary`-colored scan-line `Box`, then `Spacer(24.dp)` + `Text("Tap to pair", titleMedium, onBackground)`. Two `@Preview` functions (Light / Dark at 412×892, dark uses `Configuration.UI_MODE_NIGHT_YES`).
- One new route constant in `MainActivity`'s `private object Routes`: `Scanner = "scanner"`, placed between `Welcome` and `ChannelList` (cosmetic — matches user flow order).
- One new `composable(Routes.Scanner) { ... }` block inside `PyryNavHost`. Resolves `appPreferences` via `koinInject<AppPreferences>()` (from `org.koin.compose`), remembers a `CoroutineScope` via `rememberCoroutineScope()`, and passes an `onTap` lambda that `scope.launch`es `appPreferences.setPairedServerExists(true)` then `navController.navigate(Routes.ChannelList) { popUpTo(Routes.Scanner) { inclusive = true }; launchSingleTop = true }`.
- Welcome's `onPaired` callback re-wired from `navController.navigate(Routes.ChannelList)` to `navController.navigate(Routes.Scanner)`. `WelcomeScreen.kt` itself is not edited — only the destination wired into its callback at the `composable(Routes.Welcome) { ... }` site.
- Five new imports in `MainActivity.kt`: `androidx.compose.runtime.rememberCoroutineScope`, `de.pyryco.mobile.data.preferences.AppPreferences`, `de.pyryco.mobile.ui.onboarding.ScannerScreen`, `kotlinx.coroutines.launch`, `org.koin.compose.koinInject`. No catalog or build-file edits — every dependency was already on the classpath via #8 (`navigation-compose`) + #11 (`AppPreferences` Koin binding + `koin-androidx-compose`).
- No ViewModel. No `UiState` / `Event` sealed types. No tests beyond `assembleDebug` + preview smoke. Matches the established stub precedent (#15, #16, `WelcomeScreen`).

## Patterns established

- **Destination-block-scoped Koin + `rememberCoroutineScope`, not a ViewModel, for stub screens with a one-shot `suspend` side-effect.** Stub destinations with no observable state, no `UiState` to expose, and no lifecycle past a single `suspend` write don't need a `ScannerViewModel`. The pattern is: `val x = koinInject<X>(); val scope = rememberCoroutineScope(); MyScreen(onAction = { scope.launch { x.doThing(); navController.navigate(...) } })`. The CLAUDE.md MVI rule ("Hoist state to the ViewModel; UI receives `state: UiState` and `onEvent: (Event) -> Unit`") applies to screens that *have* state to manage — it isn't a mandate to wrap every stub in scaffolding that Phase 4 will throw away.
- **Sequential `suspend` write then `navigate(...)`, not fire-and-forget.** When a destination both mutates `AppPreferences` and navigates away, await the write before navigating. Once #13 lands its `pairedServerExists` collector at the start-destination predicate, fire-and-forget would race the next composition. Cost is one `await` on a millisecond-scale DataStore write; benefit is no race window.
- **`popUpTo(<self>) { inclusive = true }` is the idiom for "remove this stub from the back stack on success."** `popUpTo(<self>)` without `inclusive = true` is a no-op when the route is already the top of the stack — a subtle footgun. Pair with `launchSingleTop = true` to guard against double-taps stacking duplicate destinations during in-flight coroutines (a real failure mode when the success side-effect is async).
- **Tap-anywhere uses `Modifier.pointerInput(Unit) { detectTapGestures(onTap = { onTap() }) }`, not `Modifier.clickable { onTap() }`.** `clickable` applies a Material ripple to the entire screen. `detectTapGestures` gives the tap callback with no visual feedback — correct for a "fake camera accepting any tap" affordance where a full-screen ripple would fight the metaphor. The `clickable(indication = null, interactionSource = remember { MutableInteractionSource() })` variant works but reads as "clickable, minus what makes clickable useful" and needs two more imports.
- **Tap detection on the outer edge-to-edge `Surface`; `systemBarsPadding()` on the inner `Column`.** `enableEdgeToEdge()` makes content draw under system bars. The `Surface` fills the full window (so taps in the inset region are caught), the inner `Column` carries `systemBarsPadding()` (so the viewport + caption don't clip into the status bar). Don't swap them — putting the `pointerInput` on the inner `Column` makes the inset region a dead zone, which a real user with edge-to-edge insets would notice.
- **Re-wiring an existing callback's destination is a one-line edit at the registration site, never inside the screen Composable.** `WelcomeScreen.kt`'s `onPaired: () -> Unit` is untouched; the only thing that changed is what `composable(Routes.Welcome) { ... }` passes as `onPaired` in `PyryNavHost`. The callback name `onPaired` stays accurate (the button still means "user asserts they have pyrycode") — don't rename to `onContinue` or similar just because the destination changed.

## Lessons learned

- **`koinInject` is in `org.koin.compose`, not `org.koin.androidx.compose`.** The `io.insert-koin:koin-androidx-compose` artifact transitively brings `koin-compose`, and the Compose accessor lives in the latter package. `org.koin.androidx.compose.koinInject` does exist as a deprecated alias in older versions but the canonical import is `org.koin.compose.koinInject`. If autocomplete suggests both, pick the `org.koin.compose` one.
- **`pointerInput`'s key matters for closure capture.** `pointerInput(Unit)` captures the initial `onTap` closure for the lifetime of the gesture detector; if `onTap` could meaningfully change across recompositions (e.g. picking up new state), the key should be `onTap` or a more specific dependency. For this ticket the `onTap` lambda wraps a stable `scope.launch { ... }` body with no recomposable-state captures, so `Unit` is correct. Don't pre-emptively change to `pointerInput(onTap)` — the recompositions that would create a new lambda don't change the lambda's *behavior*, and re-keying restarts the gesture detector unnecessarily. Revisit only if a real stale-callback bug shows up (it shouldn't for this destination's shape).
- **`rememberCoroutineScope()` is the right scope for a stub's one-shot side-effect even though it cancels on destination-exit.** The cancellation race (write completes, then composition leaves before `navigate(...)` runs) is observable in theory but requires the user navigating away within a millisecond of tapping — and the cancellation only happens *because* the user already navigated away. `GlobalScope` would survive the cancellation but is forbidden by convention and leaks across navigation. `viewModelScope` is unavailable (no VM by design). `rememberCoroutineScope` is correct.
- **`./gradlew assembleDebug` + preview smoke is the bar for stub UI work in this codebase, by precedent.** CLAUDE.md says "test-first" as a project convention; the in-flight wave (#7, #15, #16, this ticket) has consistently shipped UI-only stubs with no `ComposeTestRule` test or `runTest` coverage, on the grounds that the AC is satisfied by the build passing + the previews rendering, and the whole stub gets replaced in a future phase. If the next durable UI ticket (e.g. real Channel List binding to `ConversationRepository`) ships without tests, that's the moment to push back — but for stubs the precedent stands.
