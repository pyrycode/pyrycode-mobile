# #9 — `FakeConversationRepository`: message storage + SessionBoundary interleaving

Replaces the `observeMessages` stub (`flowOf(emptyList())`) with real per-conversation in-memory storage and a derived projection that interleaves `ThreadItem.SessionBoundary` markers between consecutive messages whose `sessionId` differs. Mechanism only — seeded message *content* is #10.

## Summary

- `ConversationRecord` gains `messages: List<Message> = emptyList()` (third, defaulted field). Storage lives in the same `MutableStateFlow<Map<String, ConversationRecord>>` as conversation metadata — single source of truth, atomic `state.update {}` for any future mutator. No sibling map.
- `FakeConversationRepository` gains a constructor parameter `initialMessages: Map<String, List<Message>> = emptyMap()`. At construction it overlays the provided messages onto matching seeded records via `SEED_RECORDS.mapValues { ... }`; ids absent from `SEED_RECORDS` are silently dropped. Production callers pass nothing; tests use it to set up boundary scenarios. No test-only mutator.
- `observeMessages(conversationId)` is now `state.map { records -> records[id]?.messages ?: return@map emptyList(); buildThreadItems(messages) }`. Re-emits on every `state.update {}` as a property of `MutableStateFlow.map` — no separate emission tracking. Unknown id and known-with-no-messages collapse to the same empty-list path.
- `buildThreadItems(messages)` is a private linear pass: sort by `Message.timestamp` ascending, then walk the sorted list emitting a `SessionBoundary` whenever the current message's `sessionId` differs from the previous. Sort cost is `O(n log n)` per emission; fine for fixture sizes.
- Boundary fields: `reason = BoundaryReason.Clear` for every derived boundary; `occurredAt = message.timestamp` of the *new* session's first message (not the previous session's last).
- All five existing seeds (`seed-channel-*` ×3, `seed-discussion-*` ×2) keep `messages = emptyList()` by virtue of the default; no `seedChannel` / `seedDiscussion` / `buildSeedRecords` line changes.

## Patterns established

- **Test-time fixture injection via defaulted constructor parameter, not an `internal fun seedX(...)`.** The constructor parameter is read once at construction and never re-touches internal state, which avoids creating a runtime mutator that anything in the same module could call. Future stateful fakes that need scenario-specific data should follow the same shape: defaulted constructor parameter overlaid onto the production seed at init, not a post-construction mutator.
- **Derived projection over `state.map { ... }` is the canonical observer shape.** `observeMessages` now mirrors `observeConversations` — both are cold flows derived from the single `MutableStateFlow<Map<String, ConversationRecord>>`. Future observers added to this repository should follow the same shape rather than holding a sibling state holder, even if the data feels "different enough" to deserve one.
- **Observation tolerates unknown ids; mutation does not.** `observeMessages("unknown")` returns an empty list (a stale id from a recomposing screen should not crash), while `promote`/`rename`/`archive`/`startNewSession`/`changeWorkspace` still throw `IllegalArgumentException` on unknown id. The asymmetry is deliberate. Future observers added here should default to tolerant; future mutators to strict.
- **Derive boundaries from session-id deltas in the projection until there's a live consumer for richer metadata.** Eager boundary emission (writing a `SessionBoundary` at `startNewSession` / `changeWorkspace` time with the *true* `BoundaryReason` and *true* `occurredAt`) is the obvious next step — and explicitly out of scope here. Don't preempt it. When the thread screen lands and needs distinguishable reasons, lift the metadata into `Session` or `ConversationRecord` and thread it through `buildThreadItems`; until then, derived `Clear` is honest about the inference being lossy.

## Lessons learned

- **Defaulted record fields silently spare every existing seed builder.** Adding `messages: List<Message> = emptyList()` as the *third* field on `ConversationRecord` left every positional `ConversationRecord(conversation, mapOf(sessionId to session))` call site (`seedChannel`, `seedDiscussion`, `createDiscussion`, `mintNewSession`) untouched. Worth keeping in mind for future record-shape extensions: append + default beats inserting in the middle or making the new field non-defaulted. A non-defaulted new field would have forced edits in five call sites for no behavioural gain.
- **`MutableStateFlow.map` re-emits on every state mutation, not only on mutations that affect the projected slice.** Both `observeConversations` and now `observeMessages` re-emit whenever *any* part of `state` changes, including changes to a different conversation's record. No `.distinctUntilChanged()` is applied — consistent with `observeConversations`. If a future ticket adds a chatty mutator (e.g. per-keystroke streaming token append) the over-emission could matter; the fix at that point is a subtype-aware dedupe in the projection, not a sibling state holder.
- **Linear loop reads more honestly than `fold`/`zipWithNext` for "emit X between consecutive elements that differ in field F".** Tried/considered the chained-collection forms; the imperative loop kept the "boundary emitted between elements" structure visible at the top level. Worth picking up for future inter-element transformations in this repo.
