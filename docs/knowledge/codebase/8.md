# #8 — Navigation Compose: Welcome → ChannelList placeholder

Turns the merged `WelcomeScreen` (#7) into the actual app entrypoint. Adds a single-activity `NavHost` rooted at `welcome` with a `channel_list` placeholder destination, replacing the AGP-template `Greeting("Android")` body in `MainActivity`.

## Summary

- `navigation-compose 2.9.5` added as its own `[versions]` pin (Compose BOM does not cover the `androidx.navigation` artifact group) with one `[libraries]` alias.
- `MainActivity` now hosts a private `PyryNavHost` Composable. `Greeting` and `GreetingPreview` deleted.
- Routes pinned as `private object Routes { const val Welcome = "welcome"; const val ChannelList = "channel_list" }`. String constants, not type-safe `@Serializable` routes — destinations have no arguments yet.
- `onPaired` navigates to `channel_list`; `onSetup` is an inert no-op with a `TODO(#14)` comment.
- `channel_list` route body is literally `Text("Channel list placeholder")` — outer `Scaffold` in `MainActivity` provides the chrome.

## Files touched

- `gradle/libs.versions.toml` — added `navigationCompose = "2.9.5"` and `androidx-navigation-compose` alias.
- `app/build.gradle.kts:51` — single `implementation(libs.androidx.navigation.compose)` line, alphabetically placed.
- `app/src/main/java/de/pyryco/mobile/MainActivity.kt` — rewrite. `PyryNavHost` at lines 33-55, `Routes` object at lines 57-60.

## Patterns established

- **Single-activity NavHost lives in `MainActivity` via a private `PyryNavHost` Composable.** Keeps `setContent { ... }` at one level of abstraction (theme → scaffold → nav host) and isolates `rememberNavController()` inside a composition context. Future routes are added as `composable(Routes.X) { ... }` blocks inside `PyryNavHost`.
- **Route strings live in a `private object Routes`** colocated with `MainActivity`. Never inline `composable("welcome")` or `navigate("channel_list")` at call sites. When the first parameterized route lands (e.g. `thread/{conversationId}`), revisit whether to migrate to `@Serializable` type-safe routes — that decision is deferred, not foreclosed.
- **Stateless screen Composables receive navigation as `() -> Unit` callbacks**, wired by the NavHost — not a `NavController` parameter. This is what #7 set up; #8 confirms it works. Future screens follow the same shape.
- **Outer `Scaffold` in `MainActivity` owns system-bar insets via `innerPadding` → `Modifier.padding(innerPadding)` on the NavHost.** Individual screens may still apply their own `systemBarsPadding()` (as `WelcomeScreen` does); the double-padding is harmless. Don't refactor `WelcomeScreen` to drop its inset handling — its signature is fixed.
- **Back-stack policy is default `navigate(route)`.** No `popUpTo`, no `launchSingleTop`, no `inclusive`. #13 will revisit when the conditional start destination lands.

## Lessons learned

- **Compose BOM does not cover `androidx.navigation`.** It's a separate artifact group, so it needs its own `[versions]` entry. Don't assume BOM coverage based on the `androidx.compose.*` prefix — the BOM only covers what JetBrains publishes under the Compose umbrella.
- **`navigation-compose` 2.9.x is the right pin for Kotlin 2.2.x + Compose BOM 2026.02.x.** Do not downgrade to 2.8.x — it predates the Kotlin 2.2 toolchain alignment and will surface as a compiler-version mismatch at build time.
- **No navigation instrumentation test was written, by design.** `TestNavHostController` / `ComposeTestRule` setup is non-trivial and the ticket explicitly deferred it. The unused `androidx-compose-ui-test-junit4` catalog entry stays unused until a later ticket needs it.
