# #161 — surface recent-discussion last-message in ChannelListUiState

Data-layer-only widening. Adds `ConversationRepository.observeLastMessage(id): Flow<Message?>`, extends `ChannelListUiState.{Empty, Loaded}` with `recentDiscussionLastMessages: Map<String, Message> = emptyMap()`, and seeds `seed-discussion-a` with a 4-message history so the non-null path is exercised end-to-end. Zero composable signatures change, zero rendered UI changes — the UI consumer slice is sibling #162 (`addBlockedBy`-linked).

## Summary

- **New repository method.** `fun observeLastMessage(conversationId: String): Flow<Message?>` dropped next to `observeMessages` on the `ConversationRepository` interface. Cold flow, emits `Message?` keyed on `state` re-emissions, returns `null` for unknown ids and for known ids with no messages (same tolerant-observation / strict-mutation rule as `observeMessages`). Fake implementation is a one-liner: `state.map { records -> records[id]?.messages?.maxByOrNull { it.timestamp } }`.
- **UiState gains a parallel map, not a composite.** `recentDiscussionLastMessages: Map<String, Message> = emptyMap()` lands on both `ChannelListUiState.Empty` and `ChannelListUiState.Loaded` (Loading + Error untouched). Key = conversation id of a recent discussion; absent key = "no last message" — `Map<String, Message?>` with null values was rejected as an anti-pattern. Architect picked parallel-map over a composite `RecentDiscussion(conversation, lastMessage)` because the AC explicitly forbids any composable-signature change — a composite would force `RecentDiscussionsSection(discussions: List<Conversation>, …)` to change its parameter type, and that edit belongs in #162.
- **Default-value affordance.** The `= emptyMap()` default on both data-class fields is load-bearing: without it, every existing `Empty(...)` / `Loaded(...)` construction site (6 in `ChannelListScreen.kt` previews, 6 in `ChannelListViewModelTest`, 2 in `ChannelListScreenTest`) would need an explicit `recentDiscussionLastMessages = emptyMap()` argument. With the default, none of those 14 sites change. Data-class equality semantics still hold: the VM emits with `emptyMap()` (because the stub `observeLastMessage` returns `flowOf(null)` ⇒ `mapNotNull` filters it out) ⇒ matches expected `Empty/Loaded(...)` literals that omit the new arg.
- **VM gains a third combined input via `flatMapLatest`.** `combine(channelsFlow, discussionsFlow) { … }` widens to `combine(channelsFlow, discussionsFlow, lastMessagesFlow) { … }`. `lastMessagesFlow` is derived from a `recentIdsFlow = discussionsFlow.map { it.take(3).map(Conversation::id) }.distinctUntilChanged()`, then `flatMapLatest { ids -> if (ids.isEmpty()) flowOf(emptyMap()) else combine(ids.map { id -> observeLastMessage(id).map { msg -> id to msg } }) { pairs -> pairs.mapNotNull { (id, msg) -> msg?.let { id to it } }.toMap() } }`. The `mapNotNull` keeps the map non-nullable-valued (the "absent key = null last message" contract); `distinctUntilChanged` prevents re-subscription churn when discussions re-emit without changing the recent-3; `flatMapLatest` cancels obsolete per-row subscriptions when the recent slice shifts so no stale entry leaks. The new method needed `@OptIn(ExperimentalCoroutinesApi::class)` on the `state` property.
- **`Loading` window unchanged.** With `combine` now waiting for three inputs, the `loadingPersists_untilBothFlowsEmit` test still passes because the stub's `observeLastMessage` returns `flowOf(null)` ⇒ `lastMessagesFlow` resolves synchronously the same tick the discussions flow emits. No new error pathway either — the existing `.catch { e -> emit(Error(...)) }` covers throws from any of the three sources.
- **Seed data: one discussion gets a history.** `seedDiscussion(...)`'s signature gains `messages: List<SeedMessage> = emptyList()` mirroring `seedChannel`'s `currentMessages`. `seed-discussion-a` gets 4 alternating `User`/`Assistant` messages spanning 12 minutes, last timestamp anchored to the seed's `lastUsedAt` = `2026-05-11T14:00:00Z` (test asserts on this exact value). `seed-discussion-b` and `seed-discussion-archived` stay empty so the null path is also reachable.
- **One existing test narrowed, three added.** `seededDiscussions_remainEmpty` renamed to `seededDiscussions_remainEmpty_exceptDiscussionA` and its id list shortened to `["seed-discussion-b", "seed-discussion-archived"]` — the prior message about discussion A is now false. The empty-thread-by-default test (`observeMessages_emptyByDefault`) was also rewired from `"seed-discussion-a"` to `"seed-discussion-b"` since A now carries messages. Three new tests in `FakeConversationRepositoryTest` cover the null-on-empty, null-on-unknown, and most-recent-by-timestamp paths; one new test in `ChannelListViewModelTest` (`recentDiscussionLastMessages_populatedFromFake_endToEnd`) drives the real fake through the real VM and asserts both the populated `seed-discussion-a` entry and the absence of `seed-discussion-b` from the map.
- **Stub repos in three VM test files get the new override.** `ChannelListViewModelTest.stubRepo` returns `flowOf(null)` (load-bearing — keeps every existing stub-backed test producing the default empty map, no other test edits required). `ChannelListViewModelTest.erroringRepo`, plus the `DiscussionListViewModelTest` and `ArchivedDiscussionsViewModelTest` anonymous stubs, get `TODO("not used")` — those VMs don't call `observeLastMessage`, so the override is unreachable. Mechanical interface widening.

## Patterns established

- **Per-row hot subscription via `flatMapLatest(distinctUntilChanged(idsFlow))` is the bounded-fan-out pattern.** When a UiState slot wants per-element side data (here: last-message per recent discussion) and the parent slice is bounded (here: `RECENT_DISCUSSIONS_LIMIT = 3`), build a `recentIdsFlow` from the parent, `distinctUntilChanged` it, then `flatMapLatest { ids -> combine(ids.map { perRowFlow(it) }) { … } }`. The `distinctUntilChanged` is non-optional — without it, every parent re-emission tears down and rebuilds the inner `combine`, churning subscriptions even when the id set hasn't changed. The `flatMapLatest` is non-optional — without it, obsolete per-row subscriptions leak when the slice shifts. Use this shape whenever the per-row data needs its own cold flow rather than a snapshot read; if the per-row data is a static lookup, prefer a single `combine(parentFlow, lookupFlow)` instead. A batched `observeLastMessages(ids: Set<String>)` was considered and rejected as Phase 1 over-engineering — the bound is 3.
- **Parallel `Map<id, T>` with default `emptyMap()` beats composite types when the AC forbids consumer-signature changes.** Two contestants for the UiState shape were considered: composite `data class RecentDiscussion(conversation: Conversation, lastMessage: Message?)` vs parallel `Map<String, Message>` next to the existing `List<Conversation>`. Parallel-map won because the AC explicitly bans composable-signature changes — a composite would force `RecentDiscussionsSection`'s parameter type to change, and that edit belongs in the sibling ticket #162. Default `emptyMap()` is the second half of the pattern: without it, the field would have cascaded into 14 construction sites across previews + tests; with it, only the VM emission sites and the one new test touch the field, and data-class equality keeps existing assertions green. Generalised rule: when widening a UiState data class with a field that has a sensible "no data" identity (`emptyMap()`, `emptyList()`, `null` only for nullable references), provide a default — call sites that don't care won't notice. Skip the default only when forcing every site to think about the new field is the actual goal (e.g. a new mandatory invariant).
- **Interface widening cascades to test-only stubs as `TODO("not used")` overrides.** Two-tier rule: stubs whose VMs *can* exercise the new method (here: `ChannelListViewModelTest.stubRepo`) get a real default return value (`flowOf(null)`) so existing tests stay green by default. Stubs whose VMs cannot reach the new method (`erroringRepo` because it terminates at Error; `DiscussionListViewModelTest` + `ArchivedDiscussionsViewModelTest` because those VMs don't subscribe) get `TODO("not used")`. The `TODO` documents intent ("if this fires, the VM grew an unexpected subscription path") and turns unreachable code into a fail-loud landmine if someone later widens those VMs' surface. Future interface widenings should follow the same two-tier mechanical sweep.

## Lessons learned

- **Anchoring a seed timestamp to the seed's `lastUsedAt` makes the test assertion deterministic without coupling to message content.** The 4 seeded messages on `seed-discussion-a` span `13:48`/`13:52`/`13:58`/`14:00` UTC; the test asserts only on the last message's timestamp == `2026-05-11T14:00:00Z`. Anchoring the seed's `lastUsedAt` to the *last* message's timestamp (rather than some arbitrary post-message moment) means the test reads the contract — "most-recent by timestamp" — without depending on the specific content strings or the number of messages. Future seeds with histories should preserve this invariant: `seed.lastUsedAt == seed.messages.maxByOrNull { it.timestamp }!!.timestamp`. The ordering shape (User → Assistant alternating) matches `seedChannel`'s convention; content was chosen as a plausible scratch-thread snippet but isn't asserted on.
- **`@OptIn(ExperimentalCoroutinesApi::class)` on the property (not the file) keeps the experimental surface visible at the use site.** `flatMapLatest` carries the `ExperimentalCoroutinesApi` marker in `kotlinx-coroutines 1.10.2`. Placing the opt-in on the `state` property (the only consumer in this file) instead of a file-level `@file:OptIn(...)` makes the experimental dependency immediately visible to anyone editing the property. If the operator stabilises in a future coroutines release, the opt-in is the load-bearing line to delete; the file-level form would have hidden the dependency under unrelated edits. Generalised rule: scope `@OptIn` as tightly as the experimental API lets you — property > function > class > file.
- **The `Map<String, Message>` vs `Map<String, Message?>` choice is a contract choice, not a stylistic one.** `Map<String, Message?>` would mean "every recent-discussion id is in the map, value `null` if no last message". The chosen `Map<String, Message>` with `mapNotNull` filtering means "only ids with a non-null last message are in the map; consumers read `map[id]` as `Message?`". Both shapes encode the same information, but the second avoids the well-known Kotlin foot-gun of `Map.get(k)` returning `null` either because the key is absent or because the value is null — call sites that read `state.recentDiscussionLastMessages[conversation.id]` get unambiguous `Message?` semantics by construction. The trade-off is that "is this id known to the VM as recent?" is not directly answerable from the map — but `state.recentDiscussions.any { it.id == id }` already answers that question, and the parallel-map shape never claimed to be authoritative on slice membership.
