# #209 — feat(data): `recentWorkspaces()` on `ConversationRepository`

Lands the read-side data surface for the Workspace Picker bottom sheet's **Recent** section (split from #205). One new method on the [`ConversationRepository`](../features/conversation-repository.md) interface returning `Flow<List<String>>` of workspace folders previously bound to any conversation — deduped, ordered most-recent-first by the latest cwd-affecting write, with both the empty string `""` and [`DEFAULT_SCRATCH_CWD`](../features/data-model.md) excluded. Fake-backed via a sibling `MutableStateFlow<List<String>>` next to the existing `state` holder, bumped from every cwd-touching mutator. The write side (`createWorkspaceFolder`) is a separate follow-up ticket and will plug into the same `recents` storage without further refactor.

## Summary

- **Interface gained one method with a default implementation:** `fun recentWorkspaces(): Flow<List<String>> = flowOf(emptyList())`. Default returns an empty cold flow so the 11 anonymous `object : ConversationRepository` test fakes across `SettingsViewModelTest` / `ArchivedDiscussionsViewModelTest` / `DiscussionListViewModelTest` / `ChannelListViewModelTest` / `ThreadViewModelTest` inherit it untouched — zero mechanical-stub edits across the 5 test files. This is the **first deviation** from the existing "all methods abstract, explicit `TODO` stubs everywhere" interface convention; the deviation is deliberate and confined to one method. `flowOf(emptyList())` is semantically meaningful (a repo that doesn't track workspace history legitimately has no recents — the picker renders an empty Recent section, which is the correct behaviour for a fake that doesn't care).
- **`FakeConversationRepository` adds a sibling state-holder:** `private val recents = MutableStateFlow<List<String>>(initialRecents())` alongside the existing `state: MutableStateFlow<Map<String, ConversationRecord>>`. `recentWorkspaces()` returns `recents` directly — Kotlin upcasts `MutableStateFlow<List<String>>` to `Flow<List<String>>` at the call boundary, no `.asStateFlow()` needed because the consumer only collects.
- **Initial seed projection** lives in a private companion-object helper `initialRecents()`: `SEED_RECORDS.values.map { it.conversation }.sortedByDescending { it.lastUsedAt }.map { it.cwd }.filterNot { it.isEmpty() || it == DEFAULT_SCRATCH_CWD }.distinct()`. Current seeded list at class load is `["~/Workspace/pyrycode-mobile", "~/Workspace/joi-pilates", "~/Workspace/personal"]`. `distinct()` preserves first occurrence, which after the descending sort is the most-recent — correct dedup posture.
- **One bump helper, four call sites:** `private fun bumpWorkspace(cwd: String) { if (cwd.isEmpty() || cwd == DEFAULT_SCRATCH_CWD) return; recents.update { listOf(cwd) + it.filterNot { it == cwd } } }`. The "no bound workspace" filter is inside the helper at insert time — the sentinel pair never enters the list, so the read path returns the list as-is with no filter step. Called after each cwd-touching mutator's `state.update { }` block: `createDiscussion` (pass `conversation.cwd` — same as `workspace ?: ""`), `promote` (pass `updated.cwd`), `changeWorkspace` and `startNewSession` share the bump inside the `mintNewSession(...)` helper via a `lateinit var resultCwd: String` capture out of the `state.update` lambda. `rename`, `sendMessage`, `archive`, and `unarchive` don't touch cwd → don't bump.
- **Bump fires after `state.update { }`, not inside.** Two separate atomic writes are acceptable here — there's no invariant tying a specific `state` snapshot to a specific `recents` snapshot, and the sole consumer (Workspace Picker) reads `recents` only.
- **Three unit tests in `FakeConversationRepositoryTest`** (JUnit 4 + `runBlocking` + `Flow.first()`, no Turbine — matches the existing file's conventions). `recentWorkspaces_dedupes_repeatedCwds` (promote + `changeWorkspace` onto the same cwd asserts single occurrence at index 0, list grew by exactly one), `recentWorkspaces_ordersByMostRecentWrite` (asserts the exact seeded order `[pyrycode-mobile, joi-pilates, personal]` then re-bumps via `changeWorkspace` then `createDiscussion`, asserting the leading cwd after each), `recentWorkspaces_excludesEmptyStringAndDefaultScratch` (initial recents have neither sentinel; `createDiscussion(workspace = null)` and `createDiscussion(workspace = DEFAULT_SCRATCH_CWD)` don't add them).

## Patterns established

- **Default-implementation interface evolution to escape an unsplittable test-fake cascade.** This is the first method on `ConversationRepository` to ship with a body in the interface. The rule from prior tickets was "all members abstract, every fake stubs everything explicitly with `TODO('not used')`" — that rule held at 7 abstract methods × ~10 fakes because the per-fake cost stayed proportional. At 11 anonymous fakes across 5 files, adding another abstract method would have crossed the architect-side red line (> 10 simultaneous-update consumers normally trips a split) — but the cascade is structurally unsplittable in Kotlin (interface evolution is all-or-nothing). The escape valve is a default body. Pattern for future interface additions: **prefer abstract** as long as the cascade is bounded; **escalate to a default body** only when (a) the cascade exceeds the architect's red line *and* (b) the default body is semantically meaningful for the inheriting fakes (here: "a repo that doesn't track workspace history → empty recents"), not a placeholder. A default body whose meaning is `TODO("not used")` is still the wrong shape — the right shape is a default that *would* be correct for a hypothetical minimal implementation.
- **Sibling `MutableStateFlow` for a projection that's expensive to derive on the fly.** The architect's "fake builds recents by tracking a `MutableStateFlow` *or* by mapping over `observeConversations(All)`" choice landed on the explicit `MutableStateFlow`. Rationale: the derived alternative (`state.map { records -> records.values.sortedByDescending { … }.map { it.cwd }.filterNot { … }.distinct() }`) walks every conversation on every emission *and* doesn't preserve the most-recent-write semantics on its own (the seeded sort gives initial-load order, but the rule "the latest cwd-affecting write moves to index 0" doesn't fall out of resorting by `lastUsedAt` — `lastUsedAt` bumps on `sendMessage` too, which is not a cwd-affecting write). The explicit `recents` flow encodes the "cwd-affecting write" semantics at the call site of each mutator and skips the per-emission walk. Pattern applies to any future "derived collection whose update semantics differ from the underlying entity's update semantics" — when the projection rule doesn't match the entity's own update rule, store the projection.
- **Filter at insert time, not at read time.** The "no bound workspace" exclusion (`""` and `DEFAULT_SCRATCH_CWD`) lives inside `bumpWorkspace`'s early return, not in a read-path filter on `recentWorkspaces()`. Invariant: the sentinel pair never enters the list. Pattern keeps the read flow trivially correct (no transformation), keeps `MutableStateFlow.value` directly inspectable in tests / debugging without "apply the read filter in your head", and means a future consumer that reaches the underlying `recents` (e.g. the write-side `createWorkspaceFolder` ticket pinning a new folder explicitly) sees the same exclusion contract by construction. The alternative (insert anything, filter on read) would need every reader to remember the filter and would put two truths on disk.
- **Bump fires after the `state.update { }` block, with `lateinit var` to escape the relevant value out of the lambda.** Same idiom the file already uses for `mintNewSession`'s `lateinit var newSession: Session` and the post-#187 `lateinit var` patterns. Added `lateinit var resultCwd: String` inside `mintNewSession` so the bump can read `updatedConversation.cwd` after the atomic write commits. Two atomic writes (`state.update`, then `recents.update`) are acceptable when there's no cross-flow consistency invariant.

## Lessons learned

- **`MutableStateFlow.update { current -> listOf(cwd) + current.filterNot { it == cwd } }` is the right shape for a most-recent-first dedup-prepend, not `LinkedHashSet` + `.also { it.remove(cwd) }`.** The set-based alternative compiles, runs, and produces an identical result — but `MutableStateFlow.update` requires an *immutable*-style returning lambda that does CAS-equality on the produced value; mutating a captured `LinkedHashSet` and returning it would CAS against the same reference and risk a missed re-emission on the StateFlow side (the equals check is structural, but the back-loop sees the same `Set` instance). The `listOf(cwd) + filterNot { … }` shape allocates a fresh `List<String>` per update, which is what `MutableStateFlow.update`'s contract wants. Cost is one allocation per bump, which against the bounded "list of distinct workspace cwds" is negligible. **Pattern for any future `MutableStateFlow<List<T>>.update`:** return a freshly-allocated list literal, don't mutate-and-return. The flow's CAS loop assumes value equality on a stable snapshot.
- **`.distinct()` after `.sortedByDescending { … }` correctly preserves the most-recent occurrence.** Kotlin's `Iterable<T>.distinct()` preserves first occurrence per `kotlin.collections.distinct`. After `sortedByDescending { it.lastUsedAt }`, "first occurrence" is the most-recently-used cwd — which is what the AC's "ordered most-recent-first" wants. If the projection had instead sorted ascending and called `.distinct()`, the kept occurrence would be the *oldest* cwd at that path, which is wrong for the recents semantics. **Lesson:** when chaining `sortedBy` + `distinct`, the sort direction is load-bearing for which duplicate survives; it's not just a presentation choice. Add a comment if a future reader might flip the sort thinking it's cosmetic. (Comment skipped here because the surrounding code makes the intent obvious — `recentWorkspaces` literally returns this projection.)
- **`null`-workspace `promote` of a `DEFAULT_SCRATCH_CWD` discussion correctly does not enter the scratch sentinel into recents.** The bump is unconditional from the mutator side (`bumpWorkspace(updated.cwd)` always fires), but the filter inside `bumpWorkspace` catches the case where `updated.cwd == DEFAULT_SCRATCH_CWD` (because `promote(workspace = null)` preserves the existing cwd, and a scratch discussion's existing cwd *is* the sentinel). Same applies to `null`-workspace `startNewSession` on a scratch discussion. The unconditional-bump-with-internal-filter shape composes correctly across the "promote preserves cwd when workspace is null" and "scratch is the no-bound-workspace sentinel" rules without a special-case branch at each call site. **Pattern:** when a helper has an invariant filter and the call sites have a "no-op when the value is filtered" semantics, call unconditionally and let the helper do the early return — don't push the filter to each call site, that's where regressions land.
- **The architect's choice to skip a `Conversation.hasBoundWorkspace()` helper held up.** Only one site needs the predicate today (`bumpWorkspace`'s early-return guard) and the inline `cwd.isEmpty() || cwd == DEFAULT_SCRATCH_CWD` reads at the point of use — no jumping out to a helper file. Extract when a second site appears (the picker's row-rendering filter, or the future `RemoteConversationRepository`'s wire-parse). Out-of-scope for this ticket.
- **No `.distinctUntilChanged()` on the returned `Flow<List<String>>`.** Consistent with `observeConversations` / `observeMessages` / `observeLastMessage` (see [`conversation-repository.md`](../features/conversation-repository.md) → "What `observeMessages` does not do"). Re-emit on every state change is the contract; the consumer can dedupe at the collector if it cares.
