# #184 — feat(ui/thread): token-reveal animation + blinking caret on streaming assistant messages

Adds the streaming visualization that [#128](./128.md) / [#129](./129.md) / [#130](./130.md) explicitly deferred. When an assistant `Message.isStreaming = true`, the `Role.Assistant` arm of `MessageBubble` routes to a new private `StreamingAssistantBody` that reveals `content` progressively at ~50 chars/sec via `produceState`, with a `▎` (U+258E) caret glyph appended to the revealed prefix blinking on/off every 500 ms. The caret rides through [`MarkdownText`](../features/markdown-text.md) as inline text so it inherits `onSurface` from the surrounding `CompositionLocalProvider` and lands pixel-adjacent to the last revealed character regardless of which markdown block the prefix ends in. Completed messages take the unchanged static `MarkdownText` path — zero animation cost. `FakeConversationRepository`'s seed pipeline learns to carry `isStreaming` through `SeedMessage` and both message-construction paths, and one substantive streaming assistant message is seeded as the last entry of the Pyrycode Mobile channel's current session so the animation is visible end-to-end. The original streaming ticket was filed as #132 and renumbered / split during planning — every prior spec reference to `#132 streaming` resolves here.

## Summary

- `MessageBubble.kt` only — single file change on the UI side (plus the seed-pipeline edit in `FakeConversationRepository.kt`). No new files, no new module wiring, no new dependency. `produceState` and `kotlinx.coroutines.delay` are already on the classpath via `androidx.compose.runtime` + the project's existing coroutine dependency.
- `MessageBubble`'s top-level dispatch (`MessageBubble.kt:54`) now passes the full `Message` to the assistant arm instead of `message.content` — `AssistantMessage(message, modifier)`. The other two arms (`Role.User`, `Role.Tool -> Unit`) are unchanged. `AssistantMessage`'s outer `Box(...).padding(bottom = MessageRowVerticalSpacing)` + `CompositionLocalProvider(LocalContentColor provides onSurface) { … }` wrapper is preserved verbatim so spacing and ambient-colour contracts hold.
- Inside the `CompositionLocalProvider`, an `if (message.isStreaming)` branch picks between two paths:
  - `false` → existing `MarkdownText(markdown = message.content, modifier = Modifier.fillMaxWidth())` (identical to [#129](./129.md) — historical messages pay zero animation cost; no `produceState`, no coroutine, no extra recomposition).
  - `true` → new `StreamingAssistantBody(content = message.content, modifier = Modifier.fillMaxWidth())`.
- `StreamingAssistantBody` (private, same file) holds two pieces of derived state, both via `produceState`:
  - `revealedLength: State<Int>` keyed on `content`. Producer: `while (value < content.length) { delay(STREAMING_REVEAL_STEP_MS); value = (value + STREAMING_REVEAL_STEP_CHARS).coerceAtMost(content.length) }`. When Phase 4 grows `content` over time, the key flip cancels the running producer and starts a new one — the reveal restarts from 0, replaying the new content. Phase 0: content is static, the producer runs once until `value == content.length` then `produceState` retains the value.
  - `caretVisible: State<Boolean>` keyed on `Unit`. Producer: `while (true) { delay(STREAMING_CARET_BLINK_PERIOD_MS); value = !value }`. Independent of the reveal, so the caret keeps blinking after the prefix is fully revealed (visual "still live" signal) until `isStreaming` flips `false` and `AssistantMessage` un-mounts this composable.
  - The two producers are kept separate (not folded into one combined `produceState`) so the blink doesn't restart when `content` changes — keying separately is what makes the reveal restart-on-content-change without restarting the blink.
- Snapshot rendering split into a second private composable `StreamingAssistantBodyView(revealedText: String, caretVisible: Boolean, modifier: Modifier)`: computes `displayText = revealedText + (if (caretVisible) STREAMING_CARET_GLYPH else "")` and calls `MarkdownText(markdown = displayText, modifier = modifier)`. The split exists so previews can invoke the view directly with a pinned `(revealedText, caretVisible)` snapshot — Android Studio's `@Preview` runtime does run `LaunchedEffect` / `produceState`, but the captured snapshot for fast animations is non-deterministic, so the preview pins a mid-state instead of trying to capture the coroutine-driven version.
- Five file-private constants near the existing `MessageBubble.kt` spacing block:
  - `STREAMING_CARET_GLYPH = "▎"` (U+258E LEFT ONE QUARTER BLOCK) — matches the Figma frame `16:56` end-state visual.
  - `STREAMING_REVEAL_CHARS_PER_SECOND = 50` (AC1 — "approximately 50 chars/sec").
  - `STREAMING_REVEAL_STEP_CHARS = 1` — one character per tick. Simplest implementation; if a profiler ever flags 50 parses/sec as expensive, the tunable is `STREAMING_REVEAL_STEP_CHARS = 4` / `STREAMING_REVEAL_STEP_MS = 80L` (same effective rate, 4× fewer recompositions). Not pre-built.
  - `STREAMING_REVEAL_STEP_MS: Long = 1000L / STREAMING_REVEAL_CHARS_PER_SECOND` — 20 ms at 50 cps.
  - `STREAMING_CARET_BLINK_PERIOD_MS: Long = 500L` — 1 Hz toggle / 0.5 Hz full blink cycle, conventional terminal caret cadence.
- **Caret as text, not a sibling composable.** The AC requires the caret "immediately after the currently-revealed text". With [`MarkdownText`](../features/markdown-text.md) flowing content into a `Column` of mixed block elements (paragraphs, headings, lists, blockquotes, code fences), Compose has no straightforward way to position a side-rendered caret pixel-adjacent to the last glyph of the last revealed character. Appending the caret as a Unicode block character to the revealed source lets it ride through the same parser/renderer the rest of the text uses — the caret lands inline at the last position whether the prefix ends inside a paragraph, a heading, a list item, a blockquote, or a code fence. The Figma frame already designs the streaming end-state with this exact glyph in body-text flow.
- **Caret colour = `onSurface` by construction.** The caret is ordinary text inside `MarkdownText`, which reads `LocalContentColor.current`. The surrounding `AssistantMessage`'s `CompositionLocalProvider(LocalContentColor provides onSurface)` already pins the ambient — AC1's caret colour requirement holds without any explicit colour wiring inside `StreamingAssistantBody`.
- **`FakeConversationRepository` seed pipeline edits** (3 small, 1 substantive):
  - `SeedMessage` gains `val isStreaming: Boolean = false`. Defaulted so existing call sites are untouched.
  - `seedMsg(role, content, timestamp, isStreaming = false)` gains the flag with the same default.
  - Three `Message` construction paths (the `historicalMessages` and `currentMessageEntries` mappers inside `seedChannel`, plus the `messageEntries` mapper inside `seedDiscussion`) now read `msg.isStreaming` instead of the previously hard-coded `false`. Identical edit at three sites — keeps the data path uniform so a future "archived streaming snapshot" ticket doesn't have to re-touch the seed wiring.
  - The last assistant message of `seed-channel-pyrycode-mobile`'s current session (timestamp `2026-05-10T09:00:00Z`) becomes a substantive multi-paragraph reply (~480 chars: intro + three bulleted points using `**bold**`, inline `` `WorkspacePicker` `` / `Esc`, an ordered list, plus a closing question) with `isStreaming = true`. At 50 cps that's a ~10 s reveal — long enough to observe the animation visibly. AC4 ("≥ 3 sentences, including at least one markdown element") is met multiple times over.
- **Preview surface extended.** `MessageBubbleMarkdownPreviewBody` (the shared body invoked by both `MessageBubbleMarkdownLightPreview` and `MessageBubbleMarkdownDarkPreview`) now renders, top-to-bottom inside `Column(verticalArrangement = Arrangement.spacedBy(MessageRowVerticalSpacing))`:
  - A half-revealed streaming snapshot: `Box(Modifier.fillMaxWidth())` wrapping `CompositionLocalProvider(LocalContentColor provides onSurface) { StreamingAssistantBodyView(revealedText = MARKDOWN_PREVIEW_FIXTURE.take(MARKDOWN_PREVIEW_FIXTURE.length / 2), caretVisible = true) }` — exercises the caret glyph at the cut point in both themes; the surrounding `CompositionLocalProvider` is required here because the view is rendered outside an `AssistantMessage` wrapper in the preview.
  - The pre-existing completed message: `MessageBubble(previewMessage(Role.Assistant, MARKDOWN_PREVIEW_FIXTURE))` — exercises the `isStreaming = false` static path.
  - No new `@Preview` functions; no `previewStreamingMessage` helper. The two existing `@Preview`s (light + dark) automatically render the extended body. AC5 satisfied.
- No new tests. Same precedent as [#126](./126.md) / [#128](./128.md) / [#129](./129.md) / [#130](./130.md) — codebase has no Compose UI test infrastructure and the streaming wrapper is pure coroutine-driven rendering with no logic worth unit-testing. Visual verification is by `@Preview` rendering, plus running the app on a device/emulator with the new streaming seed (the last assistant message in the Pyrycode Mobile channel animates progressively at ~50 cps with the blinking caret), plus `./gradlew assembleDebug` + `./gradlew lint` passing.
- No effect on existing `FakeConversationRepositoryTest.kt` — the seed change replaces a one-liner with a multi-line streaming message at the same timestamp; nothing in the test suite asserts the exact `content` of that message.

## Patterns established

- **`produceState` is the right primitive for coroutine-driven Compose UI state with automatic-on-leave-composition cleanup.** No `LaunchedEffect`, no `DisposableEffect`, no `rememberCoroutineScope`. `produceState`'s built-in coroutine scope is tied to the composition's lifecycle — when the composable leaves composition (subtree swap, `LazyColumn` item scrolled out and disposed, parent dismount), the producer coroutine is cancelled automatically. Pattern applies to any future "leaf-level time-driven UI state" composable: shimmer placeholders, timed reveal animations, retry-backoff progress, idle-detection countdowns. Keep the state derivation inside `produceState`, leave higher-scope state to the `ViewModel`.
- **Two-`produceState` separation, not one combined, when keys have different restart semantics.** The reveal is keyed on `content` (restart when content changes); the blink is keyed on `Unit` (run for the lifetime of the composable). Folding both into a single `produceState(initialValue = ..., key1 = content)` would restart the blink every time content changes — visually jarring on a long-running stream. Pattern applies to any future "compose two time-driven values with different restart triggers" case: split the `produceState` blocks per-key, recombine at the render site.
- **State-derivation shell + pure-rendering view split for previewability.** `StreamingAssistantBody` (with `produceState`) calls `StreamingAssistantBodyView` (no state, takes a snapshot). Previews invoke the view directly with a pinned `(revealedText, caretVisible)` snapshot, bypassing the coroutine. Android Studio's `@Preview` runtime does run `LaunchedEffect` / `produceState`, but the captured snapshot for fast animations is non-deterministic — pre-pinning the snapshot gives a stable, reviewable preview. Pattern applies to any future time-driven composable where the preview needs to show a specific mid-state (animated progress indicators, typewriter effects, blink-rate teases). Cost is one extra private composable; benefit is deterministic preview rendering and trivial unit-testability if a Compose-host harness ever lands.
- **Animated visual effects route through the existing rendering pipeline by manipulating its input, not by side-rendering and positioning.** The caret is a Unicode block glyph appended to the revealed prefix; `MarkdownText` doesn't know streaming exists. Considered alternative was a separately-positioned `Box(width = 2.dp).background(onSurface)` placed by `Modifier.offset` after measuring the last text-baseline — would require either text-measurement APIs (`TextMeasurer`, `onTextLayout`) or a custom `Measurable` layout, both expensive, both block-element-specific (the offset calculation for a caret after the last line of a paragraph is different from the caret after the last item of a list, different from the caret inside a fenced code block). Routing the caret through the input string sidesteps all of that. Pattern applies to any future "visual marker that follows content position" need: prefer input-side rendering over output-side positioning when the underlying renderer is flexible enough to carry the marker as content.
- **`produceState(key1 = content)` is the contract for "restart the reveal animation when streaming source grows".** Phase 4's real streaming source will grow `Message.content` token-by-token. `produceState` semantics: changing a `key` cancels the running producer and starts a new one — the reveal restarts from `revealedLength = 0` with the new (longer) content. Pre-paying for "smarter" continuation (e.g. resume reveal at the previous `revealedLength`) is out of scope; the restart is the right Phase-0 behaviour and is also a reasonable Phase-4 fallback if the real WS feed ever sends a non-monotonic content snapshot. Pattern: when an animation's input can change underneath it, key the `produceState` on the input and accept the restart as the safe default; opt-in to continuation only if a profiler / UX traces ever flag the restart as the problem.
- **Defaulted-parameter passthrough is the right shape for adding a new optional field to a seed fixture.** `SeedMessage(val isStreaming: Boolean = false)` + `seedMsg(..., isStreaming: Boolean = false)`: dozens of existing call sites are unchanged because the default reproduces the prior behaviour exactly. The three mapper-site edits (`isStreaming = false` → `isStreaming = msg.isStreaming`) are mechanical. Pattern applies to any future seed-fixture extension (e.g. `Message.attachments`, `Message.editedAt`): defaulted field on the fixture + defaulted parameter on the builder + mapper edit that reads `msg.<field>` — three small symmetric edits, zero call-site churn.

## Lessons learned

- **`produceState` does NOT need `getValue`/`setValue` imports if you destructure the result — but it does need `androidx.compose.runtime.getValue` for the `by` delegate.** `val revealedLength by produceState(...)` requires `import androidx.compose.runtime.getValue`. Without it, the build error is `Property delegate must have a 'getValue(Nothing?, KProperty<*>)' method`. Same trap as the `remember { mutableStateOf(...) }` delegate; same fix. Filed as a lesson because the failure mode is unusual — Kotlin imports work file-wide, but Compose's `getValue` is in a function-shaped form (an extension function on `State<T>`) that IDEs don't always auto-import on first use of the `by` delegate against `produceState` specifically. If a future Compose-state composable's build fails with the property-delegate error, the first thing to check is the `androidx.compose.runtime.getValue` import.
- **`STREAMING_REVEAL_STEP_MS = 1000L / STREAMING_REVEAL_CHARS_PER_SECOND` integer division resolves to `20L` at 50 cps.** At other reveal rates `(60, 75, 100, 250)` the integer division yields `(16, 13, 10, 4)` ms which are all still > Compose's per-frame budget (16 ms at 60 Hz), so the reveal stays smooth. At extreme rates `(>500 cps → <2 ms per tick)` the `delay(...)` clamps to the next frame anyway; the visible cadence is frame-rate-bound. No need to switch to `Double` math or a fractional `Duration` until / unless we want sub-character reveal granularity, which we don't.
- **Android Studio `@Preview` runtime renders `produceState`-driven composables but captures a non-deterministic snapshot of fast animations.** The half-revealed snapshot in the preview is rendered via `StreamingAssistantBodyView` directly (pinned `revealedText.take(length / 2)` + `caretVisible = true`) precisely so a code reviewer opening the preview always sees the same visual: the caret glyph at the half-content mark. If the preview were `MessageBubble(previewMessage(Role.Assistant, ...).copy(isStreaming = true))`, the rendered snapshot would be "wherever the producer happened to be at the moment Studio captured the frame" — sometimes nearly empty, sometimes nearly full, never reliably at the half mark. The view-vs-body split is what makes the preview deterministic.
- **The reveal exits when `value == content.length` but `produceState` keeps the value pinned — the caret keeps blinking after the prefix is fully revealed.** This is intentional: streaming finishing the reveal doesn't mean `isStreaming` flips `false`; it means the producer has caught up. The blink continues as the "still live" signal until `AssistantMessage` flips branches. Verified visually in the running app — the streaming seed reveals over ~10 s, then the caret continues blinking until the user backs out (no real stream-end signal in Phase 0). When Phase 4's WS feed signals stream-end by flipping `isStreaming = false`, the branch swap removes the caret cleanly with no extra plumbing.
- **`StreamingAssistantBody`'s composable lifetime is bounded by `LazyColumn` item disposal.** When a streaming message scrolls off-screen in `ThreadScreen`'s `LazyColumn`, the item composable is disposed, `produceState`'s coroutines cancel, and the local `revealedLength` is lost. Scrolling back into view re-enters composition, `revealedLength` restarts from 0, and the message animates again from the top. Phase-0-acceptable — the streaming seed lives in the most-recently-used channel and the typical demo path keeps it on screen. Phase 4 (real backend streaming) may want to hoist `revealedLength` into the `ViewModel` so it persists across scroll-induced disposals; not built now. Document as a known limitation in the [`MessageBubble`](../features/message-bubble.md) feature doc so the next ticket touching this code doesn't re-derive the trade-off.
- **`Boolean` toggle via `value = !value` inside `produceState` is the right shape for periodic state.** Considered using `tickerFlow` / `flow { while (true) { delay(...); emit(...) } }`, a `Flow<Boolean>` collected with `collectAsStateWithLifecycle`. The `produceState` form is one fewer indirection (no `Flow`, no `.collectAsState`) and cancels on composition exit by the same `produceState` lifecycle as the reveal. Pattern: for "periodic boolean toggle scoped to a single composable's lifetime", `produceState(true, Unit) { while (true) { delay(...); value = !value } }` is the cleanest shape.
- **Caret placement inside an open markdown construct degrades to plain-text fallback.** If the reveal cut lands inside an open `**`, an open backtick, an open code fence (no closing ` ``` `), or an open `[link text`, the parser falls through to its `else` raw-text fallback (per [#129](./129.md)'s stance) and the caret renders inside the unstyled fallback. Acceptable per AC2's wording ("Markdown formatting continues to apply to the revealed prefix" — the formatting that *can* apply does; partial constructs go through the existing fallback). Two follow-up shapes if this ever looks visually rough in practice: (a) clamp the reveal cut to the end of the last closed construct, (b) parse a "stable prefix" up to the last token boundary. Not pre-built; pin this lesson against an observed failure mode rather than a theoretical one.
