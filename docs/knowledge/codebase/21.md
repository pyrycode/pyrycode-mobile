# #21 — Channel list TopAppBar (app name + settings gear)

First chrome on the channel list. `ChannelListScreen` (#46) is now wrapped in a Material 3 `Scaffold`; the `topBar` slot hosts a `TopAppBar` showing the `app_name` string resource as title and a single trailing settings-gear `IconButton`. Tapping the gear emits the new `ChannelListEvent.SettingsTapped` variant, which the NavHost destination translates into `navController.navigate(Routes.Settings)` — closing the loop on the previously-orphaned `Routes.Settings` placeholder from #16.

## Summary

- New sealed-interface variant `ChannelListEvent.SettingsTapped` (data object) joins the existing `RowTapped(conversationId)`. Adding a variant to a sealed interface is non-breaking for the VM unit tests (they don't pattern-match on the event type).
- `ChannelListScreen`'s body wraps the existing `when (state)` in a `Scaffold` whose `topBar` is a Material 3 `TopAppBar` with `title = { Text(stringResource(R.string.app_name)) }` and a single `actions = { IconButton { Icon(Icons.Default.Settings, …) } }`. The `Scaffold`'s `inner` padding is applied to each body branch via a local `bodyModifier = Modifier.padding(inner)` — inline form, no extraction into a private `ChannelListBody`.
- Chrome renders in **every** state (Loading / Empty / Loaded / Error) — falls out of wrapping the `when` inside the Scaffold body.
- New string resource `cd_open_settings = "Open settings"` for the gear's TalkBack `contentDescription`. Title text reuses the pre-existing `R.string.app_name` ("Pyrycode Mobile") rather than introducing a duplicate. `WelcomeScreen.kt:60`'s sibling string literal was deliberately not migrated (out of scope for this ticket — see "Lessons learned").
- Library catalog grew one line: `androidx-compose-material-icons-core` (group `androidx.compose.material`, name `material-icons-core`), version-managed by the existing Compose BOM. App-module `dependencies` block consumes it as `libs.androidx.compose.material.icons.core`. This is the first classpath entry for the icons package; previous placeholders used `TextButton("Back")` because no icon set was on the classpath (see Navigation feature doc's prior limitation note).
- `MainActivity.kt`'s `composable(Routes.ChannelList)` block extends its inline `when (event)` with a second branch: `ChannelListEvent.SettingsTapped -> navController.navigate(Routes.Settings)`. The compiler now enforces exhaustiveness on the sealed interface — no `else ->`.
- `ChannelListScreenLoadedPreview` unchanged in body; the existing call automatically renders the new chrome above the list (AC #5 satisfied without a new preview).
- `./gradlew assembleDebug` passes; existing `ChannelListViewModelTest` still passes.

## Patterns established

- **Screen-level chrome (TopAppBar/FAB/etc.) is the screen Composable's responsibility, not the NavHost's.** `ChannelListScreen` owns its `Scaffold`+`TopAppBar`; `MainActivity`'s outer `Scaffold` (which owns system-bar insets) does not push a shared `TopAppBar` slot down to destinations. Each screen renders its own chrome inside its own inner `Scaffold` so the chrome can vary per destination without a `TopAppBar`-content slot being threaded through the NavHost. Note: stacking two `Scaffold`s does mean double `innerPadding` plumbing (outer for system bars, inner for the screen's own bars); the inner Scaffold's `inner` padding is the right one to apply to body content.
- **Sealed-interface event variants append non-disruptively when no consumer pattern-matches on them yet.** Adding `SettingsTapped` did not require touching `ChannelListViewModelTest` — the VM neither produces nor consumes the event today. The `when` at the navigation destination is the only consumer; appending a branch makes the `when` exhaustive against the new variant at compile time, which is the desired safety net for the next variant addition.
- **Material 3 chrome components still carry `@OptIn(ExperimentalMaterial3Api::class)` at their composable boundary.** Annotated locally on `ChannelListScreen` rather than file-wide. Mirror this opt-in placement on the next screen that adopts `TopAppBar`/`BottomAppBar`/`ModalBottomSheet`/etc. — don't propagate a file-wide `@file:OptIn` until a fourth Material 3 experimental call site appears in the same file (rule of three).
- **Content-description strings use the `cd_` prefix in `strings.xml`.** First a-11y string in the project; the prefix groups them so a future translator can recognise the role at-a-glance, and grep-able for "what icon-only affordances do we have?". Future content descriptions for icon-only buttons should follow the same convention (`cd_search`, `cd_dismiss`, etc.).
- **Inline `bodyModifier = Modifier.padding(inner)` over extracting a `ChannelListBody(state, onEvent, modifier)` composable.** Picked the inline form because the body is short (four branches, each one line). Extract only when the body grows past a screenful — a private body composable is a refactor with no behaviour delta, easy to do later. The same call: at #22's FAB ticket, decide based on the body's then-size, not on the existence of the Scaffold.
- **App-name literals belong to `R.string.app_name`, not duplicated in code.** `WelcomeScreen.kt:60`'s literal "Pyrycode Mobile" is the lone surviving offender (deliberately not migrated this ticket — see "Lessons learned"). New screens or chrome adding the app name should use `stringResource(R.string.app_name)` from the start. The Welcome literal is the next sweep candidate when a ticket happens to touch that file.

## Lessons learned

- **`androidx.compose.material:material-icons-core` is the always-on icon set; `material-icons-extended` is the multi-MB megapack.** This ticket needed exactly one glyph (`Icons.Default.Settings`), and `-core` contains it. Default icons available in `-core` include Settings, ArrowBack, Menu, Search, Close, Add, Check, Clear, MoreVert, Done, Edit, Refresh, Share, Info, Warning, Email, Star, Favorite, Home, Person, Lock — the "obvious" UI verbs. Anything more specific (Coffee, Yoga, etc.) is `-extended`. The earlier #20 sleeping-dot lesson predicted this split; this is the first build-time confirmation. Future icon needs: try `-core` first, grep `androidx.compose.material.icons.Icons.Filled.*` / `Outlined.*` for the glyph, and only reach for `-extended` when a single ticket needs multiple glyphs absent from `-core` (the APK-size cost amortises across glyphs in the same ticket).
- **`-core` is BOM-managed; the catalog entry needs no `version.ref`.** The Compose BOM (`libs.compose.bom`) covers the `androidx.compose.material:material-icons-core` artifact. Adding `version.ref = "..."` to the catalog entry is wrong — the BOM resolution would be overridden, and the catalog would carry a version constant that has no other use. The shape is: `androidx-compose-material-icons-core = { group = "...", name = "..." }`, no version. Same shape as the other Compose entries (`androidx-compose-material3`, `androidx-compose-ui`, `androidx-compose-ui-graphics`, etc.).
- **Don't migrate `WelcomeScreen.kt:60`'s literal "Pyrycode Mobile" to `stringResource(R.string.app_name)` "while you're there".** Tempting because the ticket's AC explicitly named the inconsistency. Resisted, because the channel-list ticket's behavioural scope is the channel list — the Welcome literal already renders the correct text and migrating it touches an unrelated screen's pixels. CLAUDE.md's "Don't refactor adjacent code 'while you're there'" rule applied. The architect's spec called this out as out-of-scope; surfacing the lesson so the next ticket touching Welcome can sweep it intentionally.
- **The `Scaffold` body lambda receives `PaddingValues` as its sole argument — naming it `inner` over `innerPadding` saved one ident-width per call site.** Both names are fine; the project's outer Scaffold in `MainActivity` uses `innerPadding`. Consistency would prefer matching; brevity tied. Going forward, prefer `inner` for screen-level inner Scaffolds to disambiguate from `MainActivity`'s outer one when a future screen has both in scope. Minor — flag only because the next screen-level Scaffold has the same naming-fork choice.
- **`material-icons-core`'s `Icons.Default.X` is an alias for `Icons.Filled.X`.** Both expose the same `ImageVector`s; `Default` is the documented entry point that doesn't pre-commit to fill vs. outline. Used `Icons.Default.Settings` for forward-compat with a possible future M3 default-icon-style switch (currently filled). Mirrors most Compose sample code; no reason to bypass.
- **Compose's `TopAppBar` from `material3` is still flagged `@ExperimentalMaterial3Api` as of the pinned BOM.** Not a flicker — Google has had this annotated for ~2 years across Material 3 versions, with no apparent intent to graduate it. Stop treating "experimental" annotations as "imminent breaking change"; in Compose Material 3 they're a perma-flag on chrome surfaces (TopAppBar, NavigationBar, NavigationRail, FlexibleTopAppBar, et al.) and on bottom-sheet/sheet types. Add `@OptIn` and move on.
