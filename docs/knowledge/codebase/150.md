# #150 — Welcome logo radial snowflake arm geometry (Figma 9:2)

Geometry-only repair to `ic_pyry_logo.xml`. Six identical arm `pathData` strings rewritten so each arm shaft starts at the canvas-centre pivot and radiates outward, instead of straddling the pivot and overlapping with its opposite-rotation sibling. Inner/outer branch forks move from the pivot into the outer half of the shaft so the spike-near-tip silhouette in Figma node `9:2` is preserved. The drawable XML was the only file touched; `WelcomeScreen.kt` and the `Icon(tint = colorScheme.primary)` rendering pathway are unchanged.

## Summary

- Old arm `pathData` (shipped in #57): `M1.664,8.944 H43.264 ...` — a length-41.6 shaft *centred* on the group's pivot `(22.464, 8.944)`. After `translate(29.536, 43.056)` the pivot lands at canvas centre `(52, 52)`, so the shaft straddled centre with half on each side. The rotation-90 arm and the rotation+90 arm therefore overlapped entirely on top of each other instead of forming distinct radial spikes. Inner branches forked *at* the pivot, tangling visually with the centre-cursor rectangle.
- New arm `pathData`: `M22.464,8.944 H64.064 M47.424,8.944 L54.704,1.664 M47.424,8.944 L54.704,16.224 M58.864,8.944 L64.064,3.744 M58.864,8.944 L64.064,14.144`. Same length (41.6) but shifted entirely to one side of the pivot, with inner branches at ~60 % along the shaft and outer branches at ~87.5 %. After translate + rotation the shaft tip for the rot=−90 arm now lands at canvas `(52, 10.4)` — the `top-[10%]` Figma container edge for arm `9:4`. Six distinct arms radiate at 60° intervals; both branch pairs sit in the outer half so the Y-spike-at-tip pattern reads cleanly.
- Same string applied to all six `<group>` arms — the 60°-interval rotations on the groups (−90 / −30 / 30 / 90 / 150 / 210) carry the differentiation. The fix is a single `replace_all` on the buggy pathData.
- `Icon(painter = painterResource(R.drawable.ic_pyry_logo), tint = MaterialTheme.colorScheme.primary)` is untouched. The XML keeps `#FFFFFF` placeholder strokes / fills as the "tinting bed" that Compose `Icon`'s `BlendMode.SrcIn` filter replaces with `primary` at composition time — same idiom as `ic_qr_scan_frame.xml` and consistent with #57.

## Files touched

- `app/src/main/res/drawable/ic_pyry_logo.xml` — six `android:pathData` strings replaced (one per `<group>`). Centre-cursor `<path>`, all `<group>` pivot/translate/rotation attributes, viewport (`104×104`), stroke width (`3.328`), and stroke caps / joins are byte-identical. +6/−6 LOC.
- `docs/specs/architecture/150-welcome-logo-visual-fidelity.md` — architect spec.

## Patterns established

- **When a per-group `rotation` is combined with a `pivotX/Y`, place the path origin *at* the pivot rather than centred on it.** Android `<vector>` rotates a group's children around `(pivotX, pivotY)` *before* applying `translateX/Y`. If two opposite-angle groups share the same pivot, any path that straddles the pivot will draw on top of itself when one group rotates by 180° relative to the other. The fix is geometric: encode the path so its anchor coincides with the pivot, and let the path extend in one direction (`+x` here). Future radial / kaleidoscopic vector drawables in this project should follow the same shape.
- **`replace_all` over identical sibling strings is fine.** The six arm `<path>`s carry the same `pathData` value by construction (differentiation lives in the parent `<group>`'s rotation). A single bulk replace is the correct edit shape — don't atomize into six near-identical Edits.

## Lessons learned

- **Ticket-body framing can mis-name the root cause; fetch Figma before believing it.** The issue body asserted the logo "renders as a single-tint silhouette" because `Icon(tint = …)` "collapses every path to a single `primary` color … rather than differentiated stroke/fill treatment." But `mcp__plugin_figma_figma__get_variable_defs` on node `9:2` returned **only** `Schemes/Primary` — one bound colour across every arm and the cursor. The differentiation Figma actually encodes is *shape-primitive*: strokes for the six arms, a fill for the centre cursor rectangle. Both render in `primary`. The `Icon(tint = primary)` pathway was structurally correct all along; the visible defect was purely geometric (arms didn't radiate). A user-perceived "flat tinted glyph" was the downstream consequence of the knot-at-centre rendering, not evidence of a missing multi-colour treatment. Resolved by the architect — pattern for future Figma-anchored tickets: when the ticket body proposes a rendering-pathway change (Icon → Image, Canvas, etc.), validate against `get_variable_defs` / `get_design_context` before accepting the framing.
- **`#FFFFFF` is the intentional tinting-bed convention for tintable drawables in this project.** AC3 reads as "no hardcoded `Color(0xFF…)` literals in Kotlin"; drawable-XML literals are spec-permitted. Switching to `android:strokeColor="?attr/colorPrimary"` would compose with `Icon(tint = …)`'s `BlendMode.SrcIn` to produce a double-application that "mostly works" (both colours are `colorPrimary`) but conceptually muddles the contract and breaks any caller that uses the drawable without a tint. Keeping `#FFFFFF` as the placeholder + relying on `Icon(tint = …)` is the established idiom (`ic_qr_scan_frame.xml` follows the same shape; #57 documented it). Do not migrate these drawables to `?attr/colorPrimary` "while you're there."
- **Geometry contracts in specs are testable by tabulating canonical canvas positions.** The architect's spec included a 5-row table mapping each subpath through `(local coords) → translate → rotation` to canvas coords (e.g. shaft tip at canvas `(52, 10.4)` for rot=−90). That format makes the contract checkable without rendering — the developer can confirm the pathData numerically before running the build. Reuse the same table shape on future Figma-anchored vector-geometry tickets.
