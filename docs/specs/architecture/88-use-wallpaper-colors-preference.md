# Spec — #88 feat(theme): persist useWallpaperColors preference + dynamic ColorScheme branching

## Context

#86 landed the persisted `ThemeMode` preference plus reactive composition-root wiring (`MainActivity.setContent` now collects `themeMode` and passes `darkTheme: Boolean` into `PyrycodeMobileTheme`). This ticket adds the parallel plumbing for the second Appearance row — the Material You opt-in — using the same pattern. The visible switch UI lands in the sibling ticket #89; this slice is invisible-by-default.

A useful piece of luck: `PyrycodeMobileTheme` (PR #1, imported from Material Theme Builder) already exposes a `dynamicColor: Boolean = false` parameter that does the `Build.VERSION.SDK_INT >= Build.VERSION_CODES.S` check and picks between `dynamicLightColorScheme(context)` / `dynamicDarkColorScheme(context)` and the brand `lightScheme` / `darkScheme`. The work for this ticket is therefore strictly: persist a boolean, collect it at the composition root, pass it as `dynamicColor`. No `Theme.kt` edit.

## Design

### Package layout

```
de/pyryco/mobile/data/preferences/
└── AppPreferences.kt          (edited — one new key, one new flow, one new setter)
```

No new files. No new types — `Boolean` is the value, and the existing `pairedServerExists` shape is the template.

### `AppPreferences` additions

Contract (signatures only — bodies follow the existing `pairedServerExists` lines verbatim):

- `val useWallpaperColors: Flow<Boolean>` — backed by `booleanPreferencesKey("use_wallpaper_colors")`. When the key is absent, the flow emits `false`. Implementation: `dataStore.data.map { prefs -> prefs[USE_WALLPAPER_COLORS] ?: false }`.
- `suspend fun setUseWallpaperColors(enabled: Boolean)` — writes the boolean via `dataStore.edit { it[USE_WALLPAPER_COLORS] = enabled }`.
- New private companion constant: `val USE_WALLPAPER_COLORS = booleanPreferencesKey("use_wallpaper_colors")`.

No new imports — `booleanPreferencesKey` is already imported in this file.

### Composition-root wiring (`MainActivity.kt`)

The existing `setContent { … }` block already injects `AppPreferences` and collects `themeMode`. Add a second collection immediately after the `themeMode` collect, and add one named argument to the `PyrycodeMobileTheme` call:

1. Inject — already done. Reuse the existing `appPreferences` local.
2. Collect: `val useWallpaperColors by appPreferences.useWallpaperColors.collectAsStateWithLifecycle(initialValue = false)`. Initial value `false` matches the persisted default — no flash of dynamic-color-on-first-frame.
3. Theme call site becomes: `PyrycodeMobileTheme(darkTheme = darkTheme, dynamicColor = useWallpaperColors) { … }`.

The internal SDK gate in `PyrycodeMobileTheme` handles the "Android < 12 OR preference false → brand palette" branch correctly today; passing `dynamicColor = true` on a pre-S device still falls through to the brand scheme. No additional `Build.VERSION.SDK_INT` check is needed at the composition root.

### No changes to `Theme.kt`

The ticket's third AC says `PyrycodeMobileTheme` "accepts the resolved `ColorScheme` (either via an explicit parameter or via the composition root computing it before the call — match the convention #86 established)." The convention #86 established is "caller passes a `Boolean`; theme function does the resolution internally." Reusing the existing `dynamicColor: Boolean` parameter is the literal extension of that convention. Passing a pre-resolved `ColorScheme` would require changing `PyrycodeMobileTheme`'s signature, breaking all `@Preview` call sites, and inverting the encapsulation — strictly worse.

The fourth AC ("re-collecting the flow re-renders the theme without process restart") is satisfied by `collectAsStateWithLifecycle`'s standard semantics — `dynamicColor` becomes a state read inside the `setContent` composition, and any change triggers recomposition of `PyrycodeMobileTheme`, which re-runs the `when` block and selects the new `colorScheme`. Mark this in the PR description as a manual-smoke-test note ("toggled `setUseWallpaperColors` from a debug code path; observed live re-theme") per the ticket body's ask.

### Data flow

```
DataStore (disk)
   │
   ▼  Flow<Preferences>
AppPreferences.useWallpaperColors: Flow<Boolean>
   │
   ▼  collectAsStateWithLifecycle(initialValue = false)
MainActivity setContent
   │
   ▼  dynamicColor = …
PyrycodeMobileTheme — internal: if (dynamicColor && SDK_INT >= S) dynamic* else brand
   │
   ▼
MaterialTheme(colorScheme = …)
```

## State + concurrency model

- **Flow shape.** Cold `Flow<Boolean>` from `dataStore.data.map`. No `stateIn` at the data layer; the single consumer hot-converts with `collectAsStateWithLifecycle`. Identical shape to `pairedServerExists` and `themeMode`.
- **Dispatchers.** DataStore handles its own IO dispatcher; the projection is a primitive lookup. Nothing to schedule.
- **Setter scope.** `setUseWallpaperColors` is `suspend` and goes through `dataStore.edit`. This ticket has no caller for the setter — the sibling #89 will wire it up from the Settings row's `onCheckedChange` via `rememberCoroutineScope().launch { … }`. Leaving the setter unused on the main path is intentional and correct.
- **Lifecycle.** `collectAsStateWithLifecycle(initialValue = false)` is the same pattern used for `themeMode`. The initial value matches the persisted default, so there is no one-frame flash.
- **Cancellation.** Collector is tied to composition; no manual cleanup. The activity-level collector lives for the lifetime of the activity.

## Error handling

The only realistic failure modes touch DataStore IO, and existing precedent (`pairedServerExists`, `themeMode`) doesn't handle them — adding error handling here would be premature ([[Evidence-Based Fix Selection]] — no observed failure). The key-absent case is the explicit default-`false` branch and is covered by the test below.

No UI surface for errors. The setter is `suspend` and may throw `IOException` from `dataStore.edit`; the sibling #89 ticket decides whether to surface a snackbar or swallow on toggle.

## Testing strategy

All new coverage is unit (`./gradlew test`) in `AppPreferencesTest.kt`, alongside the existing tests. Same `TemporaryFolder` + real `PreferenceDataStoreFactory` setup. Three scenarios — written as bullet points; the developer translates into JUnit functions in the file's existing style (`runBlocking { … }`, `assertEquals(expected, actual)`).

- **`useWallpaperColors_defaultsToFalse`** — fresh DataStore, no writes. Assert `prefs.useWallpaperColors.first() == false`. Mirrors `pairedServerExists_defaultsToFalse`.
- **`setUseWallpaperColors_roundTripsBothValues`** — for each of `false`, `true` (in that order so the final write differs from the default): `prefs.setUseWallpaperColors(value)`, then assert `prefs.useWallpaperColors.first() == value`. A single test function with a small loop or two sequential asserts is preferred over two near-identical functions; AC says "round-trip of both boolean values."
- **`useWallpaperColors_flowReemitsAfterWrite`** — write `true`, assert `first() == true`; write `false`, assert `first() == false`. Strictly speaking this is subsumed by the round-trip test above, but the AC calls out flow re-emission explicitly. If the developer judges the round-trip already exercises it (which it does — each `first()` call resubscribes and would not see the new value unless DataStore re-emits), drop this test rather than duplicate. Single round-trip test satisfies both ACs.

No Compose UI test is added — there is no UI surface in this slice. No instrumented (`./gradlew connectedAndroidTest`) coverage needed.

The "verify with a manual smoke test note in the PR" requirement from the AC list goes into the PR description, not into the codebase. Suggested wording: *"Smoke test: temporarily called `prefs.setUseWallpaperColors(true)` from a debug code path on a Pixel 8 emulator (API 34); confirmed the theme re-rendered live without restart, and confirmed the brand palette returned after writing `false`."*

## Open questions

- **Should the setter be exercised from a debug code path in this PR?** No. The setter exists for #89 to consume; manually toggling it is the smoke test, not production code. Don't add a debug menu or a test-only entry point.
- **`booleanPreferencesKey` ordering inside the companion.** Cosmetic. The developer can place `USE_WALLPAPER_COLORS` after `THEME_MODE` (grouping by Appearance section) or between `PAIRED_SERVER_EXISTS` and `THEME_MODE` (grouping booleans). Either is fine.
