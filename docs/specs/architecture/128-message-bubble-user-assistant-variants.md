## Design source

**Figma:** https://www.figma.com/design/g2HIq2UyPhslEoHRokQmHG?node-id=16-8

The Figma frame `16:8` is the **full future-state thread screen**, containing several message rows. The two variants this ticket implements are:

- **User row** (Figma `16:23`, `16:39`, `16:51`): right-aligned bubble, background `Schemes/primary-container` (`#134a74` dark), text `Schemes/on-primary-container` (`#cfe4ff` dark), inner padding `14×12` (horizontal × vertical), corner radii `20/20/20/6` (the bottom-right corner is tighter — visual "tail" pointing right toward the user side), bubble width `330dp` on the 412dp canvas (~80% of column width). Body text uses M3 `bodyMedium`.
- **Assistant row** (Figma `16:25`, `16:32`, `16:42`, `16:54`): **The Figma shows a bubbled assistant** (background `Schemes/surface-container-high` `#272a2f`, mirrored asymmetric rounding `20/20/6/20`, width `330dp`). **The ticket body explicitly overrides this** — see "Figma-vs-AC conflict" below.

### Figma-vs-AC conflict — ticket body is authoritative

The ticket body says (twice, in both Context and AC3): assistant is **flat — no bubble shape, no background fill, left-aligned, full-width text wrap, `onSurface` text on the screen surface**. The Context paragraph names this "Claude-style (flat assistant, bubbled user)". This is a deliberate design choice that **supersedes the bubbled assistant tile in the Figma frame** — the Figma will be reconciled later (no follow-up ticket exists for the reconciliation yet; the architect for the eventual reconciliation should know that #128's flat-assistant decision is the current intent).

**Implication for the developer:** when verifying visual fidelity, the user-variant pixels should match Figma `16:23` exactly (color tokens, padding, corner radii, width). The assistant variant should ignore the Figma bubbled tiles and follow this spec's flat-text contract. Do **not** fall back to the Figma for the assistant variant's colors / corner radii / padding.

If a code reviewer pushes back on the flat assistant choice, route the question to PO via `needs-rework:po` rather than re-debating it in the PR — the ticket AC is the contract.

## Context

`#126` landed `ThreadScreen` with an empty `LazyColumn(reverseLayout = true) { items(emptyList<Unit>()) { } }` body. This ticket introduces the `MessageBubble` composable so downstream tickets can wire `items(state.messages) { MessageBubble(it) }` once the fake message-history repository (`#127`) lands. Markdown rendering (`#129`), code blocks (`#130`), tool-role rendering (`#131`), and the streaming-typing animation (`#132`) all build on top of this component.

Scope contract:

- **Component only, no wiring.** `ThreadScreen.kt` is **not** touched in #128. The LazyColumn keeps its empty placeholder body; #127 (or a follow-up) replaces `emptyList<Unit>()` with `state.messages` and adds the `MessageBubble` call. This isolates the component delivery from the data-flow change.
- **Plain text only.** `Message.content` is rendered via a single `Text(...)` call with no markdown parsing, no `buildAnnotatedString`, no code-block detection. Visual treatment of inline backticks / fenced code / links is #129/#130's scope.
- **Two roles only.** `Role.User` and `Role.Assistant` render; `Role.Tool` renders nothing (the `when` is exhaustive — see § Design / Role dispatch). #131 introduces the tool-card composable that ThreadScreen's dispatch will route `Role.Tool` to.
- **No streaming caret.** `Message.isStreaming` is ignored. #132 adds the caret + animation.
- **No timestamps in-bubble.** `Message.timestamp` is unused in this ticket. Timestamps are a per-bubble or per-session-delimiter concern handled elsewhere.

## Design

### File

**New:** `app/src/main/java/de/pyryco/mobile/ui/conversations/components/MessageBubble.kt`

One file, one public composable, two private helpers, two previews. **Production-file count: 1** (well under the size-S thresholds).

### Public API

```kotlin
@Composable
fun MessageBubble(
    message: Message,
    modifier: Modifier = Modifier,
)
```

- **Single parameter is `Message`** — the data class already on hand at the eventual call site (`state.messages.forEach { MessageBubble(it) }`). The dispatcher reads `message.role` and routes; consumers don't pre-split the list by role.
- **`modifier` defaulted to `Modifier`** — Compose convention. The caller (eventually a `LazyColumn` item slot) typically does not pass a modifier; the public API keeps the slot available.
- **No `onClick`** — message-level interaction (long-press for "Copy" / "Reply" / etc.) is not in this ticket's AC and not in Figma `16-8`. Adding it now is design-for-hypothetical-future per the project rules.
- **No `isStreaming` passthrough as a separate flag** — it's already on `Message`, and this ticket ignores it; #132 changes the rendering path internally without changing the public API.

### Role dispatch

```kotlin
when (message.role) {
    Role.User -> UserMessageBubble(message.content, modifier)
    Role.Assistant -> AssistantMessage(message.content, modifier)
    Role.Tool -> Unit  // #131 routes Role.Tool to a separate composable before reaching here
}
```

- **The `Role.Tool -> Unit` arm is deliberate.** The `when` must be exhaustive (Kotlin compiler enforces this against the `Role` enum), and rendering nothing is the right default while #131 is pending. **No `error("…")` throw, no debug placeholder Text** — the framework contract (ThreadScreen will filter or route Tool messages once #131 lands) is the source of truth; a defensive throw would crash exactly the path that has no observed failure mode.
- **No fallback default arm.** If a new `Role` value is added in the future (e.g. `Role.System`), the compiler flags the `when` as non-exhaustive and forces the architect/dev to make an explicit decision. That's the desired behaviour.

### `UserMessageBubble` (private)

Visual contract:

- **Layout:** outer `Row(Modifier.fillMaxWidth(), horizontalArrangement = Arrangement.End)` so the bubble pins to the right edge.
- **Bubble surface:** `Surface(modifier = Modifier.widthIn(max = UserBubbleMaxWidth), shape = UserBubbleShape, color = MaterialTheme.colorScheme.primaryContainer)`.
- **Inner padding:** `Modifier.padding(horizontal = BubbleHorizontalPadding, vertical = BubbleVerticalPadding)` applied to the `Text` inside the `Surface` (not on the `Surface` itself — keeps the bubble background painted under the padding).
- **Text:** `Text(text = content, style = MaterialTheme.typography.bodyMedium, color = MaterialTheme.colorScheme.onPrimaryContainer)`.

Token values (defined as file-private `val`s at the top of `MessageBubble.kt` — see § Spacing tokens):

- `UserBubbleMaxWidth = 320.dp` — matches Figma `w-[330px]` to within 3% on a 412dp canvas (≈ 80% of column width with a 16dp horizontal column padding). 320 is preferred over 330 because it falls on a Material 4dp grid step; the developer may use 330.dp if they prefer pixel-exact Figma match — both are acceptable.
- `UserBubbleShape = RoundedCornerShape(topStart = 20.dp, topEnd = 20.dp, bottomEnd = 6.dp, bottomStart = 20.dp)` — the tight bottom-right corner that makes the bubble look like it has a tail pointing toward the user side.
- `BubbleHorizontalPadding = 14.dp` — Figma `px-[14px]`.
- `BubbleVerticalPadding = 12.dp` — Figma `py-[12px]`.

### `AssistantMessage` (private)

Visual contract (**flat, per ticket AC — diverges from Figma**):

- **Layout:** `Box(modifier = Modifier.fillMaxWidth())` or simply `Row(Modifier.fillMaxWidth())` — the content is left-aligned by default (no `Arrangement.End`). No `Surface` wrapper, no `Modifier.background(...)`, no shape.
- **Text:** `Text(text = content, modifier = Modifier.fillMaxWidth(), style = MaterialTheme.typography.bodyMedium, color = MaterialTheme.colorScheme.onSurface)`.
- **No inner padding on the text itself** — the per-row vertical rhythm comes from the dispatcher-level padding (§ Spacing tokens below), and the horizontal alignment to the column edge is the caller's `LazyColumn` content padding, not this composable's responsibility.

The asymmetry between variants is intentional: the user bubble is a self-contained painted surface; the assistant message is text that flows in the column, with the screen's surface color behind it.

### Spacing tokens (AC4)

Material 3 does not expose a spacing scale. The codebase has no `LocalSpacing` provider, and the prior `#126` spec explicitly avoided introducing infrastructure for placeholder work. This ticket follows the same precedent.

**To satisfy AC4 ("Vertical spacing between consecutive messages uses M3 spacing tokens (not raw `.dp` literals)") without introducing a provider**, declare file-private named `val`s at the top of `MessageBubble.kt`:

- `private val MessageRowVerticalSpacing = 12.dp` — gap between consecutive message rows (Figma `gap-[12px]` on node `16:21`).
- `private val UserBubbleMaxWidth = 320.dp`
- `private val UserBubbleShape = RoundedCornerShape(topStart = 20.dp, topEnd = 20.dp, bottomEnd = 6.dp, bottomStart = 20.dp)`
- `private val BubbleHorizontalPadding = 14.dp`
- `private val BubbleVerticalPadding = 12.dp`

Why this satisfies AC4: the AC's spirit is "don't sprinkle arbitrary `.dp` literals at multiple sites; name them so the design system intent is legible". Named file-private `val`s achieve that without introducing a `LocalSpacing` provider that doesn't yet exist. Other components in the codebase use bare `.dp` literals freely (`ArchiveRow.kt:43`, `ConversationRow.kt:53`); this ticket's introduction of named constants is the first beachhead — do not back-fill them into other files in #128.

**Where does `MessageRowVerticalSpacing` get applied?** Two acceptable techniques:

- **A (recommended):** `MessageBubble` adds a single `Modifier.padding(bottom = MessageRowVerticalSpacing)` to the outer `Row`/`Box` of each variant. Self-contained — the eventual `LazyColumn` caller doesn't need to know about spacing.
- **B (alternative):** No padding inside `MessageBubble`; the eventual `LazyColumn` consumer uses `verticalArrangement = Arrangement.spacedBy(MessageRowVerticalSpacing)`. Cleaner separation of concerns but requires exposing the constant or duplicating it at the consumer site.

Pick **(A)** for this ticket. It keeps the spacing AC satisfied at the component level even though the component is the only thing #128 ships (no LazyColumn-wiring caller is part of this ticket). The bottom-padding-only approach (not vertical-symmetric `padding(vertical = 6.dp)`) avoids accumulating an extra leading gap at the top of the list.

The last message in the list will carry a trailing 12dp below it. That's acceptable visual padding before the (eventual) composer / status row.

### Previews (AC5)

Two `@Preview` composables at the bottom of `MessageBubble.kt`, mirroring `DiscussionPreviewRow.kt`'s shape (Light + Dark with `uiMode = Configuration.UI_MODE_NIGHT_YES` on the dark variant):

- `MessageBubbleLightPreview` — `PyrycodeMobileTheme(darkTheme = false) { Surface { Column(Modifier.padding(horizontal = 16.dp)) { /* four bubbles */ } } }`.
- `MessageBubbleDarkPreview` — same shape, `darkTheme = true`, `uiMode = Configuration.UI_MODE_NIGHT_YES`.

Each preview renders a four-message sequence to exercise both variants and alternation:

1. `MessageBubble(previewMessage(role = Role.User, content = "Can you help me think through the schema migration plan?"))`
2. `MessageBubble(previewMessage(role = Role.Assistant, content = "Sure — let me read the existing schema first."))`
3. `MessageBubble(previewMessage(role = Role.User, content = "Quick follow-up: what about edge cases?"))`
4. `MessageBubble(previewMessage(role = Role.Assistant, content = "Good catch. There are three classes of edge case here, the most important being null user_ids in the legacy table — let me walk through each."))` — the long content verifies multi-line wrap for the assistant variant.

`previewMessage` is a private helper in the same file:

```kotlin
private fun previewMessage(role: Role, content: String) =
    Message(
        id = "preview-${role.name}",
        sessionId = "preview-session",
        role = role,
        content = content,
        timestamp = kotlinx.datetime.Clock.System.now(),
        isStreaming = false,
    )
```

The preview wraps the sequence in `Surface { Column(...) }` so the assistant variant's transparent background renders against the theme's surface color (matching what `LazyColumn` will provide at runtime). The horizontal `16.dp` padding on the `Column` mimics the eventual column inset; without it, the user bubble would pin to the absolute right edge of the preview viewport.

**No preview for the `Role.Tool` arm.** The `when` renders nothing for it; a preview would be a blank screen with no visual signal.

## State + concurrency model

N/A. `MessageBubble` is a pure stateless composable: it consumes a `Message` and emits Compose nodes. No `remember`, no `LaunchedEffect`, no `DisposableEffect`, no coroutines, no flows, no `MutableState`.

**Stability and recomposition.** `Message` is a Kotlin `data class` with all stable fields (`String`, `Role` enum, `Instant`, `Boolean`), so it is a stable type from Compose's perspective. Passing the same `Message` instance into `MessageBubble` across recompositions will skip; passing a new instance with equal contents will skip via `equals`. No `@Stable` / `@Immutable` annotation needed — the compiler's stability inference handles it.

## Error handling

N/A. No I/O, no parsing, no network, no nullable inputs that could be missing. `Message.content: String` is non-null by the data class definition; an empty string renders an empty `Text`, which is the correct degenerate behaviour.

**No defensive throws.** Specifically: no `error("Tool role not yet handled")` in the `Role.Tool` arm — see § Role dispatch above for why. The `when`'s `Unit` arm is the contract.

## Testing strategy

### Unit tests — none

There is no behavior to unit-test: the composable is pure rendering with no state and no logic beyond a `when` over `Role`. The Role-dispatch branch coverage is visible directly in the previews. Adding a JVM unit test would require either Robolectric + Compose host (not on `testImplementation` today) or `ComposeTestRule` (no `androidTest` infrastructure today — see `#126` spec line 217). Building that infrastructure is over-scope for a 100-LOC component.

### Compose UI tests — none

Same precedent as `#126`: the codebase has no Compose UI test setup. Introducing it for this ticket would burn turns on tooling, not on the feature. Visual verification is by:

1. **Preview rendering in Android Studio** — both `MessageBubbleLightPreview` and `MessageBubbleDarkPreview` render without errors and visually match this spec's contract (user bubble right-aligned + primaryContainer; assistant text left-aligned + transparent + onSurface; ~12dp gap between rows; multi-line wrap on the long assistant message).
2. **`./gradlew assembleDebug`** must pass.
3. **`./gradlew lint`** must pass with no new warnings. Lint will not flag the unused-but-future-use `Role.Tool -> Unit` arm; if it does (e.g. "redundant statement"), suppress at the statement level with a one-line comment naming `#131`.

The developer reports the preview rendering check in the PR description; code-review treats it as the AC5 acceptance signal.

### `./gradlew test`

Runs existing tests; no new tests are added. All green.

## Open questions

- **Bubble max-width: `320.dp` vs `330.dp`.** Spec recommends `320.dp` (Material 4dp grid). Developer may use `330.dp` for pixel-exact Figma match. Either is acceptable; pick one and name the constant so the choice is documented in the file.
- **Long-press / selectable text.** Out of scope for #128. The composable uses `Text(...)`, not `SelectionContainer { Text(...) }`. If users want to copy assistant messages later, a follow-up wraps the screen-level body in a `SelectionContainer`. Do **not** add it here.
- **RTL languages.** `Arrangement.End` and `Alignment.End` in the user variant respect layout direction automatically — in RTL locales the user bubble pins to the left. No explicit `LayoutDirection` handling needed; the M3 layout primitives carry it. The corner-radius asymmetry (`bottomEnd = 6.dp`) also flips correctly under `RoundedCornerShape` semantics because `bottomEnd` is direction-aware. No further work for RTL.
- **Code-review pushback on flat assistant variant.** If a reviewer prefers the bubbled assistant (matching Figma), do not change the spec inline — the ticket AC governs. Route the question to PO with `needs-rework:po` and a one-line comment requesting design clarification.
