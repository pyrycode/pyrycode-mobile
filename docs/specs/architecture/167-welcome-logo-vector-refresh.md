# Spec — Welcome logo vector refresh per current Figma 6:32 (#167)

## Design source

**Figma:** https://www.figma.com/design/g2HIq2UyPhslEoHRokQmHG?node-id=6-32

The relevant subtree is **node `80:2` — "pyrycode-mark 1"**, a **92 × 104** mark (its `Group` child `80:3` measures 91.002 × 103.812 in viewBox units). Unlike the previous version, the new mark is a **single continuous filled path** (one `<path>` element in the source SVG), not a stroke-based composition of six rotated arm groups plus a centre rectangle. The visible shape is the same six-pointed snowflake topology with a centre cursor, but the encoding has been re-authored as one filled glyph; pixel-level details of the arm tips, branch positions, and centre cursor differ from the #150-shipped XML.

## Context

PR #153 (`fix(ui/welcome): radial snowflake arm geometry per Figma 9:2 (#150)`) merged 2026-05-16 10:01Z and shipped `ic_pyry_logo.xml` matching the Figma node that existed at the time (node `9:2`, a 104 × 104 mark built from six stroked arm groups plus a centre rectangle). After the merge, the Figma source-of-truth for the logo was re-edited: the mark is now under node `80:2` ("pyrycode-mark 1"), measures **92 × 104**, and is authored as a **single filled path** rather than a stroke-and-rectangle composition.

The encoding difference matters for the spec strategy:

- **#150/#153's approach** of "patch the arm `pathData` only, keep six groups + rotations + centre rectangle intact" cannot be reused here. The new Figma node has no rotation-symmetric subgroups; the entire glyph is one filled outline. Either the new drawable also goes single-path, or the architect re-derives six rotated stroke arms by hand from the filled outline — the latter is geometrically lossy and not what the design source specifies.
- **Decision: rewrite the drawable end-to-end as a single `<path>`** carrying the new Figma path data verbatim, plus the new 91.002 × 103.812 viewport at 92 dp × 104 dp intrinsic size. This is the direct translation of the design source and is the smaller spec.

The architect resolved the new vector by fetching the design context for the logo subtree (`mcp__plugin_figma_figma__get_design_context` on node `80:3`). The MCP design-context endpoint returned the logo as a flattened `<img>` asset URL rather than structured node JSON (because the mark is one Vector node, not a composition). The underlying asset URL was confirmed to be `Content-Type: image/svg+xml` (not PNG), so the architect downloaded the SVG directly via `curl` and extracted the single `<path d="…">` content. **The full path-data string is inlined verbatim into `## Design` → "New drawable contract — path data" below**, so the developer does not need to re-fetch from Figma. This pattern is the same one mobile lessons advise for design-token tickets (inline values, not "fetch them yourself" instructions) — see `docs/lessons.md` if present, or PO + architect notes around mobile #119 (2026-05-16).

### New drawable contract — root attributes

The rewritten `ic_pyry_logo.xml` uses these exact `<vector>` root attributes:

- `android:width="92dp"` (was `104dp`)
- `android:height="104dp"` (unchanged)
- `android:viewportWidth="91.002"` (was `104`)
- `android:viewportHeight="103.812"` (was `104`)

The viewport values come directly from the Figma SVG's `viewBox="0 0 91.002 103.812"`. The intrinsic dp size matches the Figma frame dimensions (92 × 104).

### New drawable contract — path element

The drawable contains exactly **one `<path>` element** (no `<group>` wrappers, no centre-cursor sub-rectangle — the new vector includes the centre cursor as part of its single filled outline):

```xml
<path
    android:fillColor="#FFFFFF"
    android:pathData="…verbatim copy of the d= attribute from the Figma SVG…" />
```

The `android:fillColor="#FFFFFF"` value is the tinting-bed placeholder (same idiom as #150 and #153 — see "Colour binding" below). The SVG source has `fill="var(--fill-0, #7AB8E8)"`; that fallback `#7AB8E8` is **discarded** in the Android port — `Icon`'s `BlendMode.SrcIn` tint overrides any per-pixel colour at composition time, so the XML colour is invisible to the user.

### WelcomeScreen size modifier

Change exactly one line in `WelcomeScreen.kt`:

- **Before** (line 75): `modifier = Modifier.size(104.dp),`
- **After:** `modifier = Modifier.size(width = 92.dp, height = 104.dp),`

Rationale: the drawable's intrinsic aspect ratio is now 92:104 (~0.885), not 1:1. With `Modifier.size(104.dp)` (square box) the Compose `Icon` composable defaults to `ContentScale.Fit` and would render the drawable at 91 × 104 dp visually inside a 104 × 104 dp bounding box — leaving ~6.5 dp horizontal padding inside the box on both sides. Since the Welcome screen's outer Column has 32 dp horizontal padding, that extra inner padding pushes the visible logo's left edge to 38.5 dp from the screen edge instead of 32 dp — visibly off-design against Figma where the logo's left edge aligns with the 32 dp column edge.

Sizing the Icon to the drawable's intrinsic dimensions removes the inner padding and aligns the logo's left edge with the column edge, matching Figma node `6:34` ("Pyry Logo (placeholder)") frame position `x=0` inside its Hero container.

### Colour binding — same #FFFFFF tinting bed as #150

AC3 reads: *"No hardcoded `Color(0xFF…)` literals introduced in Kotlin (drawable XML fills may use literal colors; if non-tokenized, architect documents rationale)."* The new drawable retains the `#FFFFFF` tinting-bed pattern. Rationale (carried forward from #150 spec lines 91–98):

- The visible colour comes from `Icon(tint = MaterialTheme.colorScheme.primary)`, which uses `BlendMode.SrcIn` to replace the drawable's per-pixel colour at composition time. The XML fill is a *placeholder* never visible to the user.
- `#FFFFFF` is the same value used by `ic_qr_scan_frame.xml` and the previous `ic_pyry_logo.xml` — the codebase's documented idiom for tintable drawables.
- The Figma SVG export's `fill="var(--fill-0, #7AB8E8)"` fallback value is **discarded**. The architect does not propagate the SVG fallback into the XML — see "Open questions" below for the colour-source caveat.

No Kotlin `Color(0xFF…)` literal is introduced. AC3 is satisfied as-is.

The developer should **NOT** preemptively change the tint; flag the visual mismatch (if any) in the PR description and let code-review + PO decide whether a follow-up ticket is needed. The current `tint = primary` matches the codebase's established idiom and the prior `WelcomeScreenDarkPreview` rendering.
